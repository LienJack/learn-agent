1. 図解タイプ
階層アーキテクチャ図。この図は CLI、Runtime、Provider Contract、Provider Adapter、実際の Model API の責任境界を説明するため。

2. 画面要素リスト
5 つの主要ノード：CLI、Runtime、Provider Contract、Provider Adapter、Model API。Provider Contract を淡い黄色で強調し、provider 詳細を隔離する耐荷重層であることを示す。Provider Adapter の横に小さな翻訳器アイコン、Model API の横にクラウドインターフェースアイコンを置く。背景にはごく淡いエンジニアリング回路線を入れてよい。

3. 正向画像提示詞
「最初のモデル呼び出しにおける Provider Contract 境界」というテーマの、記事内向け技術解説図を描く。

画風：アイボリーの紙背景、黒い手描きサインペンの線画、線幅は少し不均一、少量の淡い黄色ハイライト、editorial technical illustration、技術ブログ向けの手描きフローチャート、明快で抑制され、エンジニアリングスケッチ感がある。

構図：階層アーキテクチャ図。左から右、または上から下に 5 つのノードを示す。① CLI：端末ウィンドウとカーソル。ユーザー入力を示す。② Runtime：小さな制御台と歯車。ChatRequest の組み立てを示す。③ Provider Contract：プロトコルカードと橋、淡い黄色で強調し、短いラベルは「統一契約」。④ Provider Adapter：翻訳器、小さなプラグ、変換矢印。統一リクエストを特定 API へ翻訳することを示す。⑤ Model API：クラウドインターフェースとモデルチップ。実際の大規模モデルサービスを示す。

ハイライト：淡い黄色で Provider Contract、および Runtime から Provider Contract への矢印を強調する。

背景：ごく淡い回路線、ノード接続線、エンジニアリングスケッチの補助線。主題を邪魔しない。

文字：ノードラベルは短くする。下部に手書き名言スペースとして「Provider は能力を翻訳し、Core はタスクを進める」と入れられる一行を残す。日本語文字が不安定なら、後から追加できるよう空白領域だけ残す。

全体は技術ブログ内の手描きメカニズム図のようにし、製品 UI や正式なアーキテクチャ図にはしない。要素は明快に、階層は明確に、矢印関係も明確にする。
