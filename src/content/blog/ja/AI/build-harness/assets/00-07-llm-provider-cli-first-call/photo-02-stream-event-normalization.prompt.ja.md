1. 図解タイプ
横方向パイプライン図。この図は provider の生 stream が adapter によって Runtime が消費できる ModelEvent に正規化される流れを示すため。

2. 画面要素リスト
6 つの主要ノード：ユーザー入力、ChatRequest、Raw Stream、Adapter Normalize、ModelEvent、CLI Output / Session Record。Raw Stream は断片 token の点で表し、ModelEvent は淡い黄色で強調する。CLI Output と Session Record は小さな分岐にする。

3. 正向画像提示詞
「Streaming は raw chunk を直接表示することではなく、ModelEvent に正規化することである」というテーマの、記事内向け技術解説図を描く。

画風：アイボリーの紙背景、黒い手描きサインペンの線画、線幅は少し不均一、少量の淡い黄色ハイライト、editorial technical illustration、技術ブログ向けの手描きフローチャート、明快で抑制され、エンジニアリングスケッチ感がある。

構図：左から右への横方向パイプライン。① ユーザー入力：端末ウィンドウ内に短いカーソル。② ChatRequest：プロトコルフォームカード。③ Raw Stream：クラウドから一連の断片 token 点と小さなイベントブロックが流れ出る。provider 固有イベントを示す。④ Adapter Normalize：漏斗または変換器。断片イベントを統一形状に整理する。⑤ ModelEvent：イベントカードの束。淡い黄色で強調し、短いラベルとして text_delta、message_stop、error を含める。⑥ 出力分岐：上は CLI Output の端末文字表示、下は Session Record のログ帳。

ハイライト：淡い黄色で ModelEvent ノード、および Adapter Normalize から ModelEvent への矢印を強調する。

背景：ごく淡い回路線、ノード接続線、エンジニアリングスケッチの補助線。主題を邪魔しない。

文字：ノードラベルは短くする。下部に手書き名言スペースとして「CLI はテキストを表示し、Runtime はイベントを消費する」と入れられる一行を残す。日本語文字が不安定なら、後から追加できるよう空白領域だけ残す。

全体は技術ブログ内の手描きメカニズム図のようにし、製品 UI や正式なアーキテクチャ図にはしない。要素は明快に、階層は明確に、矢印関係も明確にする。

4. 負向提示詞
写真写実にしない、3D にしない、複雑な UI スクリーンショットにしない、大量のコードを入れない、密集した小さな文字を入れない、複雑な表を入れない、サイバーパンクにしない、ネオン色にしない、暗い背景にしない、カラフルな漫画調にしない、複雑な影を入れない、装飾を増やしすぎない、主要ノードを 7 個より多くしない、背景の回路線を主体より目立たせない、長い日本語文を生成しない。
