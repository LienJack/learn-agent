1. 図解タイプ
決定経路図。この図は provider の生エラーが runtime で判断可能なエラー分類へ変わり、停止、リトライ、圧縮、設定案内などの分岐を導くことを説明するため。

2. 画面要素リスト
7 つの主要ノード：Raw Error、Adapter Mapping、ProviderError、Runtime Decision、停止、バックオフリトライ、コンテキスト圧縮。ProviderError と Runtime Decision を淡い黄色で強調する。Raw Error は破損したクラウドインターフェースアイコン、Runtime Decision は分岐標識アイコンで表す。

3. 正向画像提示詞
「エラーマッピングにより、モデル呼び出し失敗が Runtime の意思決定入力になる」というテーマの、記事内向け技術解説図を描く。

画風：アイボリーの紙背景、黒い手描きサインペンの線画、線幅は少し不均一、少量の淡い黄色ハイライト、editorial technical illustration、技術ブログ向けの手描きフローチャート、明快で抑制され、エンジニアリングスケッチ感がある。

構図：決定経路図。左側は Raw Error で、破損したクラウドインターフェースとエラー稲妻として描く。矢印は Adapter Mapping へ入り、小さな漏斗または分類器として描く。中央は ProviderError の統一エラーカードで、淡い黄色で強調する。カード上の短いラベルは auth、rate_limit、context を含める。右側は Runtime Decision の分岐標識に入り、これも淡い黄色で強調する。分岐は 3 つの結果へ向かう：① 停止して設定を促す。停止標識と鍵アイコン。② バックオフリトライ。時計と回転矢印。③ コンテキスト圧縮。収縮ボックスと文書アイコン。

ハイライト：淡い黄色で ProviderError と Runtime Decision の 2 ノードを強調する。

背景：ごく淡い回路線、ノード接続線、エンジニアリングスケッチの補助線。主題を邪魔しない。

文字：ノードラベルは短くする。下部に手書き名言スペースとして「エラーは文字列ではなく、意思決定入力である」と入れられる一行を残す。日本語文字が不安定なら、後から追加できるよう空白領域だけ残す。

全体は技術ブログ内の手描きメカニズム図のようにし、製品 UI や正式なアーキテクチャ図にはしない。要素は明快に、階層は明確に、矢印関係も明確にする。

4. 負向提示詞
写真写実にしない、3D にしない、複雑な UI スクリーンショットにしない、大量のコードを入れない、密集した小さな文字を入れない、複雑な表を入れない、サイバーパンクにしない、ネオン色にしない、暗い背景にしない、カラフルな漫画調にしない、複雑な影を入れない、装飾を増やしすぎない、主要ノードを 8 個より多くしない、背景の回路線を主体より目立たせない、長い日本語文を生成しない。
