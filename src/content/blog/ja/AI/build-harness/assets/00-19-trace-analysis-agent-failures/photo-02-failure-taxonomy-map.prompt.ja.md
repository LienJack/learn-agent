1. 図解タイプ
decision path。要点は、Agent の失敗を責任境界に沿って分類し、それぞれ別の修復ルートへ落とすこと。

2. 画面要素リスト
中央の決定ノード「失敗原因」から、7 個の短いラベルへ分岐：モデル、context、ツール、権限、観測、検証、Join。各分岐の末端に小さな修復アイコンを置く。「観測」と「検証」を強調。背景には薄い回路線。下部に短い caption を置く余白。

3. ポジティブ画像プロンプト
記事内の技術解説図を描く。テーマは Agent の失敗分類が修復ルートをどう指し示すか。

画風：アイボリーの紙背景、黒い手描きサインペンの線画、線の太さは少し不均一、少量の淡い黄色ハイライト、editorial technical illustration、技術ブログ向け手描きフロー図、明快で抑制され、エンジニアリングスケッチ感がある。

構図：decision path。画面中央に菱形の決定ノード「失敗原因」を置き、周囲の 7 方向へ短い分岐を伸ばす。各分岐は責任境界ノードへつながり、さらに小さな修復アイコンへつながる。全体は手描きの事故原因ロゼットのように見せる。

ノード：
① モデル：小さな脳アイコン、判断ミスを表す。
② context：フォルダ + 目、事実がモデルに見えていなかったことを表す。
③ ツール：レンチ、実行または schema 問題を表す。
④ 権限：盾、allow / deny の誤判定を表す。
⑤ 観測：虫眼鏡 + 付箋、observation 投影ミスを表し、淡い黄色で強調。
⑥ 検証：チェック/バツ付きテストカード、verification の欠落または矛盾を表し、淡い黄色で強調。
⑦ Join：合流矢印、sub-agent 結果の統合ミスを表す。

ハイライト：失敗を成功に見せかけやすい「観測」と「検証」を淡い黄色で強調。

背景：ごく薄い回路線、ノード接続線、エンジニアリングスケッチ補助線を入れ、主題より目立たせない。

文字：ノードラベルは短くする。下部に手書きの日本語フレーズ用スペースを残してよい：「分類はラベル貼りではなく、どの層を直すかを決めること」。日本語文字が不安定なら、後入れ用の空白領域だけを残す。

全体は技術ブログ内の手描きメカニズム図の雰囲気で、製品 UI でも正式なアーキテクチャ図でもない。要素は明快で、階層と矢印関係がはっきりしている。
