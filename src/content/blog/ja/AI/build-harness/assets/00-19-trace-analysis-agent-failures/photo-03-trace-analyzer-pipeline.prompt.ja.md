1. 図解タイプ
horizontal pipeline。要点は、Trace Analyzer が event log から診断レポートと eval 候補へ進む線形処理フロー。

2. 画面要素リスト
7 個の主要ノード：Event Log、Trace 投影、事実抽出、ルール検査、失敗分類、診断レポート、Eval 候補。アイコンはログ、階層パネル、ピンセット、定規、タグツリー、レポートページ、テストカード。淡い黄色で「ルール検査」と「失敗分類」を強調。背景は薄い回路線。

3. 正向き画像プロンプト
記事内の技術解説図を描く。テーマは Trace Analyzer の処理パイプライン。

画風：アイボリーの紙背景、黒い手描きサインペンの線画、線の太さは少し不均一、少量の淡い黄色ハイライト、editorial technical illustration、技術ブログ向け手描きフロー図、明快で抑制され、エンジニアリングスケッチ感がある。

構図：horizontal pipeline。画面は左から右へ 7 個の処理ノードを並べ、手描き矢印で接続する。事実ログが分析器に入り、最後に診断レポートと eval 候補を出力することを表す。フロー中盤を前半の構造化寄り、後半の意味判断寄りの 2 段に見せる。

ノード：
① Event Log：ログ紙片の束、事実イベントを表す。
② Trace 投影：階層パネル、診断ビューへ整理することを表す。
③ 事実抽出：ピンセットで付箋を取り出す、重要事実の抽出を表す。
④ ルール検査：定規 + チェック/バツ、淡い黄色で強調し、決定的アサーションを表す。
⑤ 失敗分類：タグツリー、淡い黄色で強調し、原因 taxonomy を表す。
⑥ 診断レポート：レポートページ + 虫眼鏡、人間が読める振り返りを表す。
⑦ Eval 候補：テストカード、失敗サンプルを回帰に使えることを表す。

ハイライト：淡い黄色で「ルール検査」と「失敗分類」を強調。

背景：ごく薄い回路線、ノード接続線、エンジニアリングスケッチ補助線を入れ、主題より目立たせない。

文字：ノードラベルは短くする。下部に手書きの日本語フレーズ用スペースを残してよい：「先に構造化し、次に判断し、最後に回帰へ残す」。日本語文字が不安定なら、後入れ用の空白領域だけを残す。

全体は技術ブログ内の手描きメカニズム図の雰囲気で、製品 UI でも正式なアーキテクチャ図でもない。要素は明快で、階層と矢印関係がはっきりしている。
