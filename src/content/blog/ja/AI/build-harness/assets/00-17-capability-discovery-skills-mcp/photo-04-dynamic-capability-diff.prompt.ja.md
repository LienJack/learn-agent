1. 図解タイプ
state sequence。この図は、Agent loop の各ターンで能力集合がどのように変化し、capability diff として記録され、原因分析と replay に使われるかを示すため。

2. 画面要素リスト
6 個の主要ノード：前ターンの可視集合、タスクの新証拠、ポリシー再計算、能力 Diff、イベントログ、Replay 復元。状態カード、虫眼鏡、計算ギア、比較マーク、ログ巻物、再生ボタンのアイコンを使う。淡い黄色で「能力 Diff」と「イベントログ」を強調。

3. 正向き画像プロンプト
記事内の技術解説図を描く。テーマは動的な能力集合が capability diff として記録され、後続の原因分析と replay を支える仕組み。

画風：アイボリーの紙背景、黒い手描きサインペンの線画、線の太さは少し不均一、少量の淡い黄色ハイライト、editorial technical illustration、技術ブログ向け手描き状態シーケンス図、明快で抑制され、エンジニアリングスケッチ感がある。

構図：左から右への状態シーケンス。画面には 6 枚の状態カードを置き、手描き矢印で接続する。① 前回集合：数枚の小さなツールカードで、前ターンの visible set を表す。② 新証拠：虫眼鏡とエラーログ断片で、テスト出力が PR や CI に言及したことを表す。③ ポリシー再計算：ギアとふるいで、Discovery Policy が再計算されることを表す。④ 能力 Diff：比較マークとプラス/マイナス、淡い黄色で強調し、added、removed、changed を表す。⑤ イベントログ：ログ巻物とタイムスタンプ、淡い黄色で強調し、session event log への書き込みを表す。⑥ Replay：再生ボタンと復元矢印で、後から当時モデルが見ていた能力を復元できることを表す。

背景にはごく薄い回路線、ノード接続線、エンジニアリングスケッチ補助線を入れる。各ノードには短い日本語ラベルのみを置く。下部に手書きの日本語フレーズ用スペースを残す：「能力がなぜ現れたかも、事実にする」。全体は技術ブログ内の手描きメカニズム図の雰囲気で、製品 UI でも正式なアーキテクチャ図でもない。

4. ネガティブプロンプト
写真写実にしない、3D にしない、複雑な UI スクリーンショットにしない、大量のコードを入れない、密集した小文字を入れない、複雑な表にしない、サイバーパンクにしない、ネオン色にしない、暗い背景にしない、カラフルな漫画調にしない、複雑な影を入れない、装飾を増やしすぎない、主要ノードは 6 個を超えない、背景の回路線を主体より目立たせない、長い日本語文章を生成しない。
