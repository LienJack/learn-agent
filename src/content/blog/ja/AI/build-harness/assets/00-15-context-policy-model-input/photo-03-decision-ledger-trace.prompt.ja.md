1. 図解タイプ
横方向パイプライン。一回の context 投影が ledger を残し、失敗分析時に責任層を逆引きできることを示す。

2. 画面要素リスト
6 ノード：Context Sources、Policy Decision、Model Input、Model Action、Failure Trace、Fix Policy。下部に Decision Ledger 時間線。Decision Ledger と Failure Trace を強調する。

3. 正向き画像プロンプト
記事内の技術解説図を描く。テーマは「Context Decision Ledger によってモデル入力投影を trace して修正できる」。

画風：アイボリーの紙背景、黒い手描きサインペンの線画、線幅はわずかに不均一、少量の淡い黄色ハイライト、editorial technical illustration、技術ブログ向けの手描きフロー図、明瞭で抑制され、エンジニアリングのラフスケッチ感がある。

構図：横方向パイプライン。上部に左から右へ 6 ノード：Context Sources は複数のソースカード、Policy Decision は小チェックリスト、Model Input はモデル窓、Model Action は矢印とツールカード、Failure Trace は虫眼鏡で失敗箇所を調べるノードとして淡い黄色で強調、Fix Policy はレンチでポリシー修正を表す。下部に Decision Ledger の時間線を描き、すべての policy decision が下へ落ちてこの時間線に記録される。Ledger は淡い黄色で強調する。失敗後に「何が含まれ、除外され、要約されたか」を確認できることを表す。

背景にはごく薄い回路線、ノード接続線、エンジニアリングスケッチの補助線を入れるが、主題より目立たせない。

文字：ノードラベルは短い日本語または英語のみにする。下部には後から金句を追加できる空白の一行を残す。全体は技術ブログ中の手描きメカニズム図であり、製品 UI や正式なアーキテクチャ図ではない。要素は明瞭で、階層と矢印関係をはっきりさせる。
