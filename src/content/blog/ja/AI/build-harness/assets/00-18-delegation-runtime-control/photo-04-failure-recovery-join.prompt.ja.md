1. 図解タイプ

Decision path。要点は、子タスクの失敗を分類して回収し、直接メインタスクを失敗させるのではなく、統一的に親 Agent Join へ戻すこと。

2. 画面要素リスト

左側に「子タスク異常」の入口。中央で 5 種類の失敗に分岐：Validation、Capability、Runtime、Contract、Conflict。各分類は処理アクションに接続：タスクパッケージ修正、権限申請、リトライ/降格、出力書き直し、証拠仲裁。右側ですべてのアクションが「親 Agent Join」へ合流する。親 Agent Join と Capability/Permission 経路を強調。アイコンは壊れたギア、チェックリスト、ロック、タイマー、フォーマットカード、衝突矢印、合流ファネル。

3. ポジティブ画像プロンプト

記事内の技術解説図を描く。テーマは Delegation Runtime における子 Agent 失敗の分類回収メカニズム。

画風：アイボリーの紙背景、黒い手描きサインペンの線画、線の太さは少し不均一、少量の淡い黄色ハイライト、editorial technical illustration、技術ブログ向け手描き決定図、明快で抑制され、エンジニアリングスケッチ感がある。

構図：decision path。左側は「子タスク異常」の入口で、アイコンは壊れたギア。中央で 5 個の短いラベルの分岐に分ける：① Validation：チェックリストエラー、処理は「タスク修正」。② Capability：ロックとツール、処理は「権限申請/再委譲」、この分岐を淡い黄色で軽く強調。③ Runtime：タイマー、処理は「リトライ/降格」。④ Contract：フォーマットカード、処理は「出力書き直し」。⑤ Conflict：反対向き矢印、処理は「証拠仲裁」。右側ですべての処理アクションを 1 つの「親 Agent Join」ノードに合流させ、淡い黄色で強調し、合流ファネルと目標旗のアイコンを使う。

背景：ごく薄い回路線、ノード接続線、エンジニアリングスケッチ補助線を入れ、主題より目立たせない。ノードラベルは短く、長文を避ける。下部に手書きの日本語フレーズ用スペースを残してよい：「子タスクの失敗は、メインタスクの失敗ではない」。全体は技術ブログ内の手描きメカニズム図の雰囲気で、製品 UI でも正式なアーキテクチャ図でもない。
