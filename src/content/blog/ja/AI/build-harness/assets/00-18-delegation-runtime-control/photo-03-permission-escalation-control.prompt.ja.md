1. 図解タイプ

Decision path。要点は、子 Agent が高リスク動作に遭遇したら自分で承認せず、承認を上位へエスカレーションしなければならないこと。

2. 画面要素リスト

左側は子 Agent がタスクを実行する場面。中央で高リスク動作に遭遇して分岐：安全サブセットは続行し、高リスク動作は Permission Escalation へ入る。上部に親 Agent/ユーザー承認ゲート。右側に承認後に続行、拒否後に範囲を狭める、再計画の 3 つの出口。Permission Escalation と承認ゲートを強調。アイコンは小さな作業台、控えめな警告三角、盾ゲート、親 Agent 指揮台、分岐標識。

3. 正向き画像プロンプト

記事内の技術解説図を描く。テーマは Delegation Runtime で子 Agent の高リスク動作が必ず親 Agent またはユーザー承認へエスカレーションされる仕組み。

画風：アイボリーの紙背景、黒い手描きサインペンの線画、線の太さは少し不均一、少量の淡い黄色ハイライト、editorial technical illustration、技術ブログ向け手描き決定図、明快で抑制され、エンジニアリングスケッチ感がある。

構図：decision path。左側に「子 Agent」の小さな作業台を置き、右へ「局所タスク」を実行していく。中央で分岐ノード「高リスク動作？」に到達し、横に小さな警告三角を描く。下側の安全経路には「安全サブセット続行」と書く。上側の高リスク経路は「権限エスカレーション」ノードに入り、淡い黄色で強調し、盾と上向き矢印のアイコンを使う。さらに「親 Agent / ユーザー承認」ゲートにつなげ、これも淡い黄色で強調する。右側には 3 つの出口：① 承認して続行、② 拒否して範囲縮小、③ 再計画。矢印関係を明確にし、子 Agent が自己承認できないことを表す。

背景：ごく薄い回路線、ノード接続線、エンジニアリングスケッチ補助線を入れ、主題より目立たせない。ノードラベルは短く、長文を避ける。下部に手書きの日本語フレーズ用スペースを残してよい：「子 Agent は要求できるが、自分では承認できない」。全体は技術ブログ内の手描きメカニズム図の雰囲気で、製品 UI でも正式なアーキテクチャ図でもない。
