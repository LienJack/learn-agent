1. 図解タイプ
Decision path 図。この図は、いつフレームワークをそのまま使い、いつ拡張し、いつ局所抽象を迂回するかを読者が判断できるようにするため。

2. 画面要素リスト
入口ノードは「新しい Agent 要件」。判断ノードは順に、経路が確定しているか、現実の副作用があるか、デフォルト戦略で十分か、フレームワークに重要 hook があるか。結果ノードは、フレームワークをそのまま使う、薄いラッパー、フレームワーク拡張、自前 Harness 境界。「フレームワーク拡張」と「自前 Harness 境界」を強調する。

3. ポジティブ画像プロンプト
「Agent プロジェクトでフレームワークを使うか、拡張するか、抽象を迂回するかをどう判断するか」というテーマの、記事内向け技術解説図を描く。

画風：アイボリーの紙背景、黒い手描きサインペンの線画、線幅は少し不均一、少量の淡い黄色ハイライト、editorial technical illustration、技術ブログ向けの手描きフローチャート、明快で抑制され、エンジニアリングスケッチ感がある。

構図：上から下への決定経路図。最上部に「新 Agent 要件」の入口を置き、タスクカードアイコンで表す。下へ向かって 4 つのひし形判断ノードを通る：① 経路は確定？② 副作用あり？③ デフォルト戦略で十分？④ 重要 hook あり？各判断ノードは短い矢印で分岐する。左側の結果は「そのまま使う」と「薄いラッパー」で、アイコンは積み木と会話吹き出し。右側の結果は「フレームワーク拡張」と「自前 Harness 境界」で、それぞれスロット拡張モジュール、制御台ガードレールで表し、淡い黄色で強調する。

ハイライト：淡い黄色で「フレームワーク拡張」と「自前 Harness 境界」、およびそこへ向かうリスク経路を強調する。

背景：ごく淡い回路線、ノード接続線、エンジニアリングスケッチの補助線。主題を邪魔しない。

文字：ノードラベルは短くする。下部に手書き名言スペースとして「フレームワークは省力化するが、境界は自分で判断する」と入れられる一行を残す。日本語文字が不安定なら、後から追加できるよう空白領域だけ残す。

全体は技術ブログ内の手描きメカニズム図のようにし、製品 UI や正式なアーキテクチャ図にはしない。要素は明快に、階層は明確に、矢印関係も明確にする。

4. ネガティブプロンプト
写真写実にしない、3D にしない、複雑な UI スクリーンショットにしない、大量のコードを入れない、密集した小さな文字を入れない、複雑な表を入れない、サイバーパンクにしない、ネオン色にしない、暗い背景にしない、カラフルな漫画調にしない、複雑な影を入れない、装飾を増やしすぎない、主要ノードを 8 個より多くしない、背景の回路線を主体より目立たせない、長い日本語文を生成しない。
