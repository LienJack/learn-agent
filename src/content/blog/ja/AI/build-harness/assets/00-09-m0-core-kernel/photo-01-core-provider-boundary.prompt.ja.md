1. 図解タイプ
レイヤードアーキテクチャ図。この図は、実際の大規模モデル provider が core に接続されても、実行、状態、イベントログは runtime が制御し続けることを説明する。

2. 画面要素リスト
6 つの主ノード：CLI、Runtime Facade、Core Kernel、Provider Adapter、実際の大規模モデル、Tool Runtime。Core Kernel 内部には contracts、event bus、state の 3 つの短いラベルを置く。Core Kernel と ToolIntent 境界を強調する。アイコンは端末、コンソール、チップ、クラウドインターフェース、ツールボックス、ログ帳。

3. 正向き画像プロンプト
記事内の技術解説図を描く。テーマは「M0 Core Kernel は実際の大規模モデルをシステムへ接続するが、モデルにシステムを乗っ取らせない」。

画風：アイボリーの紙背景、黒い手描きサインペンの線画、線幅はわずかに不均一、少量の淡い黄色ハイライト、editorial technical illustration、技術ブログ向けの手描きフロー図、明瞭で抑制され、エンジニアリングのラフスケッチ感がある。

構図：レイヤードアーキテクチャ図。左側に CLI 端末アイコン、矢印で中央の Runtime Facade コンソールへ入る。中心には大きめの Core Kernel パネルを描き、内部に contracts、event bus、state の短いラベルを置く。右上に Provider Adapter と実際の大規模モデルのクラウドインターフェースを置き、矢印は ModelEvent と ToolIntent だけを返す。右下に Tool Runtime のツールボックスを置くが、後続章の実行層として破線で表す。provider が core を越えて直接ツールを実行できないことを示す明確な境界線を描く。

ハイライト：Core Kernel パネルと、Provider Adapter から Core Kernel へ ToolIntent を返す矢印を淡い黄色で強調する。

背景にはごく薄い回路線、ノード接続線、エンジニアリングスケッチの補助線を入れるが、主題より目立たせない。 各ノードには短い日本語または英語ラベルのみを置く。下部には後から文字を追加できる空白の一行を残す。全体は技術ブログ内の手描きメカニズム図であり、製品 UI や正式なアーキテクチャ図ではない。
