1. 図解タイプ

horizontal pipeline。この図は、ツールの生結果がどのように整理、予算処理、要約され、モデルコンテキストとイベントログへそれぞれ入るかを示すため。

2. 画面要素リスト

左から右へ 6 つのノード：Raw Result、Normalize、Budget、Observation、Prompt Context、Event Log。Observation と Prompt Context を強調する。Raw Result は端末ログ紙、Event Log は台帳、Prompt Context は投影光線で表す。

3. ポジティブ画像プロンプト

ツール結果が次ターンのモデルで使える observation へ変わる仕組みをテーマにした、記事内向け技術解説図を描く。

画風：アイボリーの紙背景、黒い手描きサインペンの線画、線幅は少し不均一、少量の淡い黄色ハイライト、editorial technical illustration、技術ブログ向けの手描きフローチャート、明快で抑制され、エンジニアリングスケッチ感がある。

構図：左から右への横方向パイプライン図。画面には 6 つの重要ノードを含め、手描き矢印で接続する：① Raw Result：端末ログ紙と失敗テストの X。生 stdout、ファイル内容、エラーを示す。② Normalize：漏斗 + フィールドカード。構造化整理を示す。③ Budget：はさみ + メーター。大きな結果の切り詰めを示す。④ Observation：要約カード + 目。淡い黄色で強調し、要約と重要証拠を示す。⑤ Prompt Context：投影光線 + モデルチップ。淡い黄色で強調し、次ターンのモデルから見えることを示す。⑥ Event Log：台帳 + タイムスタンプ。システムの事実記録を示す。

ハイライト：淡い黄色で Observation と Prompt Context、および Observation から Event Log へ分流する細い矢印を強調する。

背景：ごく淡い回路線、ノード接続線、エンジニアリングスケッチの補助線。主題を邪魔しない。

文字：ノードラベルは短くする。下部に手書き名言スペースとして「ツール結果はログではなく、次ターンの事実である」と入れられる一行を残す。日本語文字が不安定なら、後から追加できるよう空白領域だけ残す。

全体は技術ブログ内の手描きメカニズム図のようにし、製品 UI や正式なアーキテクチャ図にはしない。要素は明快に、階層は明確に、矢印関係も明確にする。

4. ネガティブプロンプト

写真写実にしない、3D にしない、複雑な UI スクリーンショットにしない、大量のコードを入れない、密集した小さな文字を入れない、複雑な表を入れない、サイバーパンクにしない、ネオン色にしない、暗い背景にしない、カラフルな漫画調にしない、複雑な影を入れない、装飾を増やしすぎない、主要ノードを 8 個より多くしない、背景の回路線を主体より目立たせない、長い日本語文を生成しない。
