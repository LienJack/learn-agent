1. 図解タイプ
Input-process-output。この図は、audit snapshot が scope、query plan、候補処理過程を復盤可能な証拠パッケージに変え、model input、replay、trace に供給することを説明するため。

2. 画面要素リスト
左側入力：Scope、Query Plan、Candidates。中央処理：Selected Items、Rejected Reasons、Visible Projection、Citation Map。右側出力：Model Input、Replay、Trace Audit。淡い黄色で Audit Snapshot 証拠箱と Citation Map を強調。アイコンは境界枠、計画チェックリスト、紙片の束、証拠箱、引用ラベル、再生ボタン、虫眼鏡。

3. 正向き画像プロンプト
記事内の技術解説図を描く。テーマは audit snapshot がキャッシュではなく検索証拠パッケージであること。

画風：アイボリーの紙背景、黒い手描きサインペンの線画、線の太さは少し不均一、少量の淡い黄色ハイライト、editorial technical illustration、技術ブログ向け手描きフロー図、明快で抑制され、エンジニアリングスケッチ感がある。

構図：input-process-output の 3 部構成。左側入力区には 3 つの小カードを描く：① Scope：境界枠アイコン、プロジェクト、権限、時間境界を表す。② Query Plan：チェックリストと分岐矢印、複数のサブクエリを表す。③ Candidates：紙片の束と虫眼鏡、リコール候補を表す。中央には大きな「Audit Snapshot」証拠箱を置き、淡い黄色で強調する。箱の中に 4 つの短いラベルを入れる：Selected、Rejected、Projection、Citation Map。Citation Map は引用ラベルアイコンで軽く強調する。右側出力区には 3 つの行き先：Model Input 作業台、Replay 再生ボタン、Trace Audit 虫眼鏡付き台帳を描く。手描き矢印で、同じ snapshot がモデル入力、再生、失敗原因分析を同時に支えることを表す。

ハイライト：「Audit Snapshot」証拠箱と「Citation Map」を淡い黄色で強調。

背景：ごく薄い回路線、ノード接続線、エンジニアリングスケッチ補助線を入れ、主題より目立たせない。

文字：ノードラベルは短くする。下部に手書きの日本語フレーズ用スペースを残す：「キャッシュは速度のため、スナップショットは事実のため」。日本語文字が不安定なら、後入れ用の空白領域だけを残す。

全体は技術ブログ内の手描きメカニズム図の雰囲気で、製品 UI でも正式なアーキテクチャ図でもない。要素は明快で、階層と矢印関係がはっきりしている。
