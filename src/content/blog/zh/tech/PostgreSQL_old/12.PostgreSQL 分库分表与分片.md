# PostgreSQL 分库分表与分片：先分区，还是直接拆库

当 PostgreSQL 单表越来越大时，很多人会直接想到：

```text
分库分表
```

但这里有一个很容易混淆的问题：

**PostgreSQL 分区表，不等于分库分表。**

分区表是在同一个 PostgreSQL 集群里，把一张逻辑表拆成多个物理分区。分库分表则是把数据拆到多个数据库实例、多个库或多个表里，应用或中间层需要负责路由。

这篇文章只回答一个问题：

**当 PostgreSQL 单表、单库、单主节点遇到边界时，应该如何区分分区、读写扩展、垂直拆分和真正的水平分片。**

我们继续用订单表：

```sql
CREATE TABLE orders (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  user_id BIGINT NOT NULL,
  seller_id BIGINT NOT NULL,
  status TEXT NOT NULL,
  amount NUMERIC(10, 2) NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

订单表会遇到几类增长：

1. 数据量越来越大，历史订单拖慢索引和维护。
2. 用户查询、商家查询、客服查询路径不同。
3. 写入都集中在主库。
4. 备份、恢复、DDL 时间越来越长。
5. 大促时某些用户、商家、商品形成热点。

这些问题不一定都要靠分库分表解决。

## 一、先判断瓶颈到底是什么

拆数据前，先问：

| 问题 | 更像什么方案 |
|------|--------------|
| 历史数据太多，按时间清理困难 | 分区表 |
| 单库读压力高 | 只读副本、缓存、读写分离 |
| 单主写入达到瓶颈 | 分片、业务削峰、写模型调整 |
| 单表 DDL 和 VACUUM 太慢 | 分区、在线变更、归档 |
| 不同业务域耦合太深 | 垂直拆分 |
| 单租户/单商家特别大 | 租户隔离、目录式分片 |
| 跨地域访问延迟高 | 多区域架构、异地副本 |

分库分表是重武器。它解决容量、写入和故障半径问题，同时会带来路由、事务、查询、迁移和运维复杂度。

不要因为单表大就立刻拆库。很多时候，先做分区、归档、索引和查询治理，就能撑很久。

## 二、分区表：还是一张逻辑表，只是底下拆开

PostgreSQL 声明式分区可以按范围、列表或哈希拆表。

按时间范围分区：

```sql
CREATE TABLE orders_partitioned (
  id BIGINT NOT NULL,
  user_id BIGINT NOT NULL,
  seller_id BIGINT NOT NULL,
  status TEXT NOT NULL,
  amount NUMERIC(10, 2) NOT NULL,
  created_at TIMESTAMPTZ NOT NULL
) PARTITION BY RANGE (created_at);

CREATE TABLE orders_2026_06
PARTITION OF orders_partitioned
FOR VALUES FROM ('2026-06-01') TO ('2026-07-01');

CREATE TABLE orders_2026_07
PARTITION OF orders_partitioned
FOR VALUES FROM ('2026-07-01') TO ('2026-08-01');
```

分区表适合：

- 时间序列数据。
- 日志、订单、流水这类按时间归档的数据。
- 大表需要按分区维护索引、清理历史数据。
- 查询经常带分区键。

分区的好处是：

1. 查询带分区键时可以 partition pruning，只扫相关分区。
2. 删除历史数据可以 `DROP` 或 `DETACH` 分区，比大 DELETE 更干净。
3. 每个分区的索引更小，维护成本更低。
4. 大表维护任务可以按分区分批做。

但分区不是无限扩展：

1. 所有分区仍在同一个 PostgreSQL 集群里。
2. 单主写入瓶颈仍然存在。
3. 查询不带分区键时可能扫很多分区。
4. 全局唯一约束会受到分区键限制。
5. 分区过多会增加规划和管理成本。

一句话：

**分区解决的是大表管理和局部扫描问题，不是把写能力横向扩到多台机器。**

## 三、垂直拆分：先按业务边界拆清楚

分库分表不一定一上来就水平切订单。

很多系统应该先做垂直拆分：

```text
user_db
order_db
payment_db
inventory_db
```

垂直拆分解决的是业务边界：

- 用户资料和订单生命周期不同。
- 支付数据安全等级更高。
- 库存写热点和订单查询路径不同。
- 后台报表不应该直接扫交易主库。

垂直拆分之后，每个库的表更少、职责更清晰、故障半径更小。

它也会带来新问题：

- 跨库事务变复杂。
- 跨域查询需要 API、事件或读模型。
- 数据一致性要通过消息、Outbox、对账来保证。

如果业务边界还没拆清楚，直接水平分片只会把混乱复制到更多分片里。

## 四、水平分片：分片键是长期架构决策

水平分片要回答：

```text
一条数据应该落在哪个分片？
一次查询应该去哪些分片？
未来扩容怎么迁移？
```

常见分片键：

| 分片键 | 优点 | 风险 |
|--------|------|------|
| `user_id` | 用户订单列表快 | 商家后台可能跨分片 |
| `seller_id` | 商家维度查询快 | 用户订单可能跨分片 |
| `order_id` | 点查简单，分布均匀 | 用户/商家列表需要二级索引表 |
| `tenant_id` | SaaS 租户隔离清晰 | 大租户热点明显 |
| 时间 | 归档方便 | 最新分片写热点 |
| hash 组合键 | 分布更均匀 | 查询路由更复杂 |

一个好的分片键通常满足：

1. 高基数。
2. 分布相对均匀。
3. 查询经常携带。
4. 不容易形成单点热点。
5. 长期稳定。
6. 支持迁移和扩容。

最麻烦的是：当前最常用的查询维度，不一定是未来最好的分片键。

例如订单：

```text
用户看订单 -> user_id 好
商家看订单 -> seller_id 好
客服查订单 -> order_id 好
履约扫状态 -> status + created_at 好
```

没有一个键同时完美服务所有查询。所以分片设计往往需要冗余读模型。

## 五、路由方式：Hash、Range 和目录表

### 1. Hash Sharding

```text
shard = hash(user_id) % 64
```

优点：

- 分布均匀。
- 实现简单。
- 适合点查和单用户范围查询。

缺点：

- 扩容时取模变化，迁移复杂。
- 范围查询不友好。
- 大用户仍然可能形成热点。

### 2. Range Sharding

```text
user_id 1-1000000 -> shard_01
user_id 1000001-2000000 -> shard_02
```

优点：

- 范围查询方便。
- 迁移某段数据直观。

缺点：

- 新数据可能集中写最新范围。
- 容易数据倾斜。
- 热点范围会拖垮单个分片。

### 3. Directory-based Sharding

用路由表记录每个 key 在哪个分片：

```sql
CREATE TABLE tenant_shard_map (
  tenant_id BIGINT PRIMARY KEY,
  shard_id TEXT NOT NULL,
  status TEXT NOT NULL,
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

优点：

- 可以单独迁移大租户。
- 支持热点租户拆分。
- 路由策略灵活。

缺点：

- 路由表本身要高可用。
- 每次访问前多一次路由或依赖缓存。
- 路由变更要处理缓存失效和双写窗口。

目录式分片更适合 SaaS 多租户和大客户隔离。

## 六、全局 ID：不要让自增主键困住分片

单库里用自增主键很方便：

```sql
id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY
```

但分片后，每个分片都有自己的自增序列，容易冲突。

常见方案：

| 方案 | 优点 | 缺点 |
|------|------|------|
| 中心化 ID 服务 | 全局有序、可控 | 多一个关键服务 |
| Snowflake 类 ID | 去中心化、吞吐高 | 时钟和机器号管理复杂 |
| UUID/ULID | 简单、全局唯一 | 索引局部性和存储成本要评估 |
| 分片号嵌入 ID | 路由方便 | ID 规则长期绑定分片设计 |

对于订单这类核心实体，ID 不是简单字段，而是路由、排序、追踪和排障的一部分。

一个常见做法是：

```text
order_id = 时间戳 + 机器号 + 序列号
```

或者在 ID 中保留 shard 信息：

```text
order_id -> decode shard_id -> 路由到对应分片
```

但一旦把 shard 信息编码进 ID，未来再分片和迁移就要考虑兼容性。

## 七、跨分片查询：用读模型，不要到处 scatter-gather

假设订单按 `order_id` 分片。用户查自己的订单列表怎么办？

如果每次查询都扫所有分片：

```text
查询 user_id = 10086
-> shard_00
-> shard_01
-> ...
-> shard_63
-> 合并排序
```

这就是 scatter-gather。分片少、低频后台查询还能接受；核心接口长期这样做，会把所有分片绑在一起。

更常见的做法是建立冗余索引表：

```sql
CREATE TABLE user_order_index (
  user_id BIGINT NOT NULL,
  order_id BIGINT NOT NULL,
  shard_id TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL,
  PRIMARY KEY (user_id, created_at, order_id)
);
```

写入订单时同时写：

```text
orders shard
user_order_index shard
seller_order_index shard
outbox_events
```

这样用户列表先查 `user_order_index`，拿到 `order_id` 和 `shard_id`，再批量点查订单详情。

代价是：索引表变成读模型，需要处理一致性。

可以通过：

- 本地事务写同分片索引。
- Outbox 事件异步构建读模型。
- 定期对账修复。
- 查询时容忍短暂不一致。

分片后，很多查询性能不再来自单条 SQL，而来自读模型设计。

## 八、跨分片事务：尽量通过业务设计避开

跨分片事务很难。

比如一个订单同时写：

```text
user shard
seller shard
payment shard
inventory shard
```

如果要求这些写入像单库事务一样同时成功或失败，就会进入分布式事务问题。

常见处理方式不是强行把所有东西塞进 2PC，而是：

1. 明确事实源：订单事实以订单库为准。
2. 用本地事务写事实和 outbox。
3. 用事件驱动更新库存、支付、搜索、报表等读模型。
4. 每个消费者幂等。
5. 用对账修复长尾不一致。

简化表：

```sql
CREATE TABLE outbox_events (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  aggregate_id BIGINT NOT NULL,
  event_type TEXT NOT NULL,
  payload JSONB NOT NULL,
  status TEXT NOT NULL DEFAULT 'READY',
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

本地事务：

```text
写订单
写 outbox_events
COMMIT
```

事务提交后再异步发布事件。

这不是说分布式事务永远不能用，而是核心 OLTP 路径里要非常谨慎。很多业务更适合最终一致、幂等和对账。

## 九、在线迁移和再分片：最难的是增量和切流

分片不是一次性动作。业务继续增长后，还会再分片。

典型迁移流程：

```text
确定迁移范围
-> 准备目标分片
-> 全量复制历史数据
-> 增量同步新写入
-> 双读校验
-> 切换路由
-> 观察
-> 清理旧数据
```

迁移最难的是三个点：

### 1. 增量同步

全量复制期间，源分片还在写。必须捕获这段时间的新变更。

方式包括：

- 应用双写。
- Outbox 事件。
- 逻辑复制。
- CDC。
- 触发器记录变更。

### 2. 数据校验

不能只比 `count(*)`。

还要按范围、按 hash、按关键字段抽样：

```sql
SELECT
  mod(id, 1024) AS bucket,
  count(*) AS rows,
  sum(amount) AS amount_sum
FROM orders
GROUP BY mod(id, 1024)
ORDER BY bucket;
```

### 3. 切流和回滚

切换路由要能灰度：

```text
1% 租户 -> 新分片
10% 租户 -> 新分片
100% -> 新分片
```

如果发现问题，要能回滚路由，或者至少停止扩大影响。

分片迁移本质上也是一次在线变更：全量、增量、校验、切换、回滚，一个都不能少。

## 十、PostgreSQL 生态里的分片选择

PostgreSQL 本身提供分区、外部数据包装器、逻辑复制等能力，但它不是内置透明分布式数据库。

常见路线：

| 路线 | 说明 |
|------|------|
| 单库 + 分区 | 优先解决大表管理和历史归档 |
| 单主 + 只读副本 | 优先解决读扩展 |
| 应用层分片 | 应用掌控路由，灵活但复杂 |
| 中间件/代理层分片 | 屏蔽部分路由复杂度，但要接受中间层限制 |
| Citus 等扩展 | 提供分布式 PostgreSQL 能力，适合特定工作负载 |
| 云数据库分布式产品 | 托管分布式能力，减少自运维但绑定平台 |

选择时要问：

- 查询是否大多带分片键？
- 是否有大量跨分片 JOIN？
- 是否需要强事务？
- 运维团队能否处理扩容、迁移、故障？
- 是否能接受某些 SQL 能力受限？
- 业务是否真的已经碰到单主写入瓶颈？

## 十一、一个决策顺序

遇到 PostgreSQL 单库压力，不要一步跳到分库分表。可以按这个顺序：

1. **优化 SQL 和索引**：先把明显浪费消掉。
2. **治理连接和慢事务**：避免并发压力放大。
3. **做归档和分区**：解决历史数据和大表维护。
4. **加只读副本**：解决读压力。
5. **按业务垂直拆分**：降低耦合和故障半径。
6. **构建读模型**：减少跨维度查询压力。
7. **再考虑水平分片**：解决单主写入、容量和故障半径。
8. **准备迁移体系**：没有迁移能力，不要轻易分片。

分片不是终点。分片之后，你还会面对：

- 分片扩容。
- 热点迁移。
- 跨分片查询。
- 全局唯一约束。
- 全局排序和分页。
- 备份恢复策略。
- 多分片 DDL。
- 多分片观测和告警。

所以它是长期架构承诺。

## 十二、一分钟复习

1. PostgreSQL 分区表不等于分库分表；分区仍在同一个集群里，主要解决大表管理和局部扫描。
2. 分库分表前先判断瓶颈：读压力、写压力、历史归档、DDL、业务边界，分别对应不同方案。
3. 垂直拆分通常应该先于水平分片，因为业务边界不清时，水平切分会复制混乱。
4. 分片键决定数据分布、查询路径、热点风险和未来迁移成本，是长期架构决策。
5. 跨分片查询要尽量通过读模型解决，不要让核心接口长期 scatter-gather。
6. 跨分片事务要优先通过事实源、Outbox、幂等和对账设计来降低强事务需求。
7. 再分片最难的是增量同步、数据校验、切流和回滚。
8. 没有迁移和观测能力时，不要急着把单库问题升级成多分片问题。

**分库分表不是数据库调优技巧，而是一次把数据路由、一致性、迁移和运维复杂度交给系统长期承担的架构选择。**
