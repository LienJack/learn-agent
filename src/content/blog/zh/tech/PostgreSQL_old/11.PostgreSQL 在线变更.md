# PostgreSQL 在线变更：DDL 锁、并发建索引和影子表迁移

线上数据库最怕的操作之一，是看起来很小的 DDL。

比如：

```sql
ALTER TABLE orders ADD COLUMN remark TEXT;
```

这条语句在小表上可能一瞬间完成。但在高并发大表上，真正危险的不是语句本身有多长，而是：

**它会拿什么锁？会不会挡住读写？会不会在等待锁时把后续请求排成长龙？**

很多线上事故不是 DDL 执行很久，而是 DDL 在等待锁时，后面的业务 SQL 被它挡住。

这篇文章只回答一个问题：

**PostgreSQL 在线变更的核心不是追求完全无锁，而是把锁窗口、回填压力、切换动作和回滚路径都控制在可预期范围内。**

## 一、先把 DDL 风险分成三类

线上变更前，不要只问"这条 SQL 能不能执行"。要先问三件事：

| 风险 | 典型问题 | 关注点 |
|------|----------|--------|
| 锁风险 | 是否阻塞 SELECT/INSERT/UPDATE/DELETE | 锁模式、锁等待、超时 |
| 重写风险 | 是否重写整张表 | 表大小、WAL、IO、复制延迟 |
| 回填风险 | 是否需要批量更新历史数据 | 批大小、节奏、索引、VACUUM |

观察锁等待：

```sql
SELECT
  pid,
  state,
  wait_event_type,
  wait_event,
  now() - query_start AS query_age,
  left(query, 160) AS query
FROM pg_stat_activity
WHERE state <> 'idle'
ORDER BY query_start;
```

变更前最好设置锁等待超时，避免 DDL 一直等锁并堵住队列：

```sql
SET lock_timeout = '3s';
SET statement_timeout = '10min';
```

这两个超时不是万能药，但能避免"悄悄等锁等到业务雪崩"。

## 二、不是所有 `ALTER TABLE` 都一样危险

不同 DDL 的风险差别很大。

| 操作 | 常见风险 | 说明 |
|------|----------|------|
| 添加可空列 | 通常较轻 | 元数据变更，仍需短锁 |
| 添加带非易变默认值列 | 新版本里很多场景更轻 | 仍要确认版本和具体表达式 |
| 修改列类型 | 可能重写表 | 大表非常危险 |
| 添加 NOT NULL | 需要验证历史数据 | 可先加约束再验证 |
| 建普通索引 | 会阻塞写入 | 大表慎用 |
| `CREATE INDEX CONCURRENTLY` | 不阻塞普通写入，但更慢 | 不能放在普通事务块里 |
| `VACUUM FULL` | 需要强锁 | 线上大表通常避免 |
| `CLUSTER` | 重写表并持锁 | 线上谨慎 |

这里不要死背，因为不同 PostgreSQL 版本对某些 DDL 做过优化。真正可靠的做法是：

1. 查当前版本文档。
2. 在同版本、同量级影子表上演练。
3. 变更时设置超时和监控。
4. 准备回滚或中止方案。

## 三、并发建索引：`CREATE INDEX CONCURRENTLY`

普通建索引可能阻塞写入：

```sql
CREATE INDEX idx_orders_user_id ON orders (user_id);
```

线上大表更常用：

```sql
CREATE INDEX CONCURRENTLY idx_orders_user_id
ON orders (user_id);
```

它的优势是：建索引期间允许普通读写继续发生。

但它不是免费午餐：

1. 需要更长时间。
2. 通常需要多次扫描表。
3. 会等待旧事务结束。
4. 失败后可能留下 invalid index，需要清理。
5. 不能在普通事务块里执行。

观察索引构建进度：

```sql
SELECT
  pid,
  datname,
  relid::regclass AS table_name,
  index_relid::regclass AS index_name,
  phase,
  blocks_done,
  blocks_total
FROM pg_stat_progress_create_index;
```

查看 invalid index：

```sql
SELECT
  indexrelid::regclass AS index_name,
  indrelid::regclass AS table_name,
  indisvalid,
  indisready
FROM pg_index
WHERE NOT indisvalid
   OR NOT indisready;
```

如果并发建索引失败，不要假装没发生。要确认 invalid index 是否需要 drop 后重建。

## 四、添加字段：先兼容，再回填，再收紧

大表添加字段和回填，推荐拆成多步。

假设要给订单增加风控评分：

```sql
ALTER TABLE orders ADD COLUMN risk_score INT;
```

第一步只加可空列，不立刻回填，不立刻加 NOT NULL。

第二步应用先兼容：

- 读到 `NULL` 时走默认逻辑。
- 新写入开始填 `risk_score`。
- 旧数据异步回填。

第三步分批回填：

```sql
UPDATE orders
SET risk_score = 0
WHERE id >= 1
  AND id < 10001
  AND risk_score IS NULL;
```

实际执行时由脚本按主键范围或时间窗口循环，每批控制大小和间隔。

第四步验证：

```sql
SELECT count(*)
FROM orders
WHERE risk_score IS NULL;
```

第五步再收紧约束。

可以先加检查约束并延后验证：

```sql
ALTER TABLE orders
ADD CONSTRAINT orders_risk_score_not_null
CHECK (risk_score IS NOT NULL) NOT VALID;

ALTER TABLE orders
VALIDATE CONSTRAINT orders_risk_score_not_null;
```

最后再决定是否改成真正 `NOT NULL`，并在低峰执行。

这套节奏的核心是：

```text
结构先兼容
数据慢慢补
读写双逻辑过渡
验证后再收紧
```

## 五、分批回填：不要用一个巨型 UPDATE 硬怼

最危险的回填方式是：

```sql
UPDATE orders
SET risk_score = 0
WHERE risk_score IS NULL;
```

如果表很大，这会带来：

1. 大事务。
2. 大量 WAL。
3. 长时间持锁。
4. 表膨胀。
5. 复制延迟。
6. 回滚成本巨大。

更稳的是按主键或时间分批：

```sql
WITH batch AS (
  SELECT id
  FROM orders
  WHERE risk_score IS NULL
  ORDER BY id
  LIMIT 1000
)
UPDATE orders
SET risk_score = 0
WHERE id IN (SELECT id FROM batch);
```

每批提交一次，批之间 sleep，观察指标：

- 主库 CPU 和 IO。
- WAL 生成量。
- 复制延迟。
- dead tuple 增长。
- autovacuum 是否跟得上。
- 应用错误率和延迟。

回填脚本要能暂停、恢复和幂等执行。不要写成只能从头跑到尾的一次性脚本。

## 六、影子表迁移：当原表结构变化太大时

如果变更会重写整表、改主键、拆列、改变分区策略，直接 `ALTER TABLE` 可能不合适。

这时可以考虑影子表迁移：

```text
创建新表 orders_new
-> 按新结构写入
-> 全量复制旧数据
-> 增量同步新写入
-> 校验数据
-> 短暂停写或双写收敛
-> 切换表名或路由
-> 保留旧表回滚窗口
```

简化示例：

```sql
CREATE TABLE orders_new (
  id BIGINT PRIMARY KEY,
  user_id BIGINT NOT NULL,
  status TEXT NOT NULL,
  amount NUMERIC(10, 2) NOT NULL,
  created_at TIMESTAMPTZ NOT NULL,
  risk_score INT NOT NULL DEFAULT 0
);
```

全量复制可以分批：

```sql
INSERT INTO orders_new (id, user_id, status, amount, created_at, risk_score)
SELECT id, user_id, status, amount, created_at, 0
FROM orders
WHERE id >= 1
  AND id < 10001
ON CONFLICT (id) DO NOTHING;
```

切换前必须校验：

```sql
SELECT count(*) FROM orders;
SELECT count(*) FROM orders_new;
```

还要抽样对账关键字段。

影子表迁移的难点不在建表，而在增量同步和切换：

- 是否双写？
- 是否用触发器记录增量？
- 是否用逻辑复制或 CDC？
- 切换时如何冻结写入？
- 切换失败如何回滚？
- 外键、索引、权限、触发器是否完整迁移？

没有这些答案，影子表只是另一个风险源。

## 七、`pg_repack`：在线整理表和索引膨胀

PostgreSQL 表膨胀严重时，很多人会想到：

```sql
VACUUM FULL orders;
```

但 `VACUUM FULL` 会重写表，并需要强锁，线上大表通常不适合直接跑。

`pg_repack` 是 PostgreSQL 常用的在线重整工具。它的目标是降低表和索引膨胀，同时尽量减少持有强锁的时间。

可以粗略理解成：

```text
创建重整后的副本
-> 捕获变更
-> 同步增量
-> 最后短锁切换
```

但它也不是无风险：

1. 需要安装扩展和对应版本工具。
2. 需要额外磁盘空间。
3. 会制造 IO 和 WAL。
4. 最后切换仍然需要锁。
5. 失败后要确认中间对象是否清理。

所以 `pg_repack` 适合有演练、有监控、有维护窗口的大表治理，不适合第一次在线上临时救火。

## 八、`gh-ost` 是 MySQL 工具，不是 PostgreSQL 的答案

用户经常把在线变更和 `gh-ost` 放在一起说。

这很正常，因为 `gh-ost` 是 MySQL 生态里非常著名的在线 schema migration 工具。它通过读取 binlog、维护影子表、最后切换表来降低 MySQL 大表变更风险。

但要注意：

**`gh-ost` 是 MySQL 在线变更工具，不是 PostgreSQL 工具。**

把它放进 PostgreSQL 学习里，更适合作为一种工程思想对比：

| 维度 | MySQL `gh-ost` 思路 | PostgreSQL 常见思路 |
|------|---------------------|---------------------|
| 增量来源 | binlog | 触发器、逻辑复制、应用双写、CDC |
| 影子表 | 是 | 可手工设计或由工具完成 |
| 切换 | rename / cut-over | rename、路由切换、短暂停写 |
| 在线建索引 | 依赖工具流程 | `CREATE INDEX CONCURRENTLY` |
| 膨胀整理 | 不是核心目标 | `pg_repack` 常用于 bloat |

所以 PostgreSQL 里不要问"怎么用 gh-ost"，而要问：

```text
这次变更需要的是并发建索引？
还是分批回填？
还是影子表迁移？
还是 pg_repack 清膨胀？
还是应该通过应用双写完成结构迁移？
```

工具名字不同，核心问题一样：全量、增量、校验、切换、回滚。

## 九、上线前的 DDL Checklist

线上 DDL 前，至少过一遍：

- [ ] 当前 PostgreSQL 版本下，这条 DDL 是否会重写表？
- [ ] 会拿什么锁？是否会阻塞读写？
- [ ] 是否设置了 `lock_timeout` 和 `statement_timeout`？
- [ ] 是否在同版本、同量级表上演练过？
- [ ] 是否需要 `CREATE INDEX CONCURRENTLY`？
- [ ] 是否有长事务会阻碍 DDL？
- [ ] 回填是否分批、可暂停、可恢复、幂等？
- [ ] 是否会造成复制延迟或 WAL 暴涨？
- [ ] 是否需要应用双写或读写兼容？
- [ ] 是否有数据校验 SQL？
- [ ] 失败后如何回滚？
- [ ] 是否准备了监控和维护窗口？

## 十、一分钟复习

1. PostgreSQL 在线变更的关键是控制锁窗口、重写风险、回填压力和回滚路径。
2. `CREATE INDEX CONCURRENTLY` 能让普通读写继续进行，但更慢、会等待旧事务，失败后可能留下 invalid index。
3. 大表添加字段通常按"可空列 -> 应用兼容 -> 分批回填 -> 验证 -> 收紧约束"推进。
4. 巨型 UPDATE 会制造大事务、大量 WAL、表膨胀和复制延迟，回填要分批、可暂停、可恢复。
5. 结构变化太大时考虑影子表迁移，但难点在增量同步、校验、切换和回滚。
6. `pg_repack` 可以在线治理表和索引膨胀，但仍需要额外空间、IO、WAL 和最后短锁。
7. `gh-ost` 是 MySQL 工具，在 PostgreSQL 系列里更适合作为在线变更思想对比。

**在线变更不是把 DDL 写得更短，而是把全量、增量、校验、切换和回滚变成可控流程。**
