## 二、功能介绍

### 2.1 并发正确性的五个维度

只说“线程安全”通常不够精确。一个并发设计至少要同时检查以下五个维度。

| 维度 | 核心问题 | 常见错误 |
|---|---|---|
| 原子性 | 一组状态变化能否作为不可分割整体完成？ | `count++` 被拆为读、加、写；余额检查与扣减分离 |
| 可见性 | 一个 goroutine 的写入，另一个何时保证可见？ | 通过日志、Sleep、轮询普通变量猜测“应该看见了” |
| 有序性 | 写入与读取之间是否建立了规定的先后关系？ | 先写数据再写 `done=true`，但两者都无同步 |
| 生命周期 | 谁启动、停止、等待、取消 goroutine？ | goroutine 泄漏、主流程提前返回、关闭时仍有发送者 |
| 所有权 | 谁可以读写、关闭、释放某个资源？ | 多方 close channel、多个 goroutine 同时修改同一 slice/map |

**高质量并发设计不是“到处加锁”，而是先明确状态、所有权和生命周期，再选择最小而充分的同步边。**

### 2.2 Data Race、Race Condition、Deadlock、Livelock、Starvation

#### 2.2.1 Data Race 的严格定义

Go Memory Model 对 Data Race 的核心定义是：

> 对同一内存位置，存在并发的读写或写写访问；至少一个访问是普通的非同步访问；两个访问在 happens-before 上无序。

更形式化地说：

- **读写 Race**：普通读 `r` 与写 `w` 访问同一位置，且既不是 `r happens-before w`，也不是 `w happens-before r`。
- **写写 Race**：两个写 `w`、`w'` 访问同一位置，且在 happens-before 上无序。
- 如果相关访问全部通过 `sync/atomic` 原子操作完成，它们不构成上述普通 Data Race；但业务逻辑仍可能有 Race Condition。

```go
var count int

func inc() {
    count++ // 读 count、计算、写 count；不是原子操作
}
```

多个 goroutine 调用 `inc`，至少会出现写写或读写冲突。

#### 2.2.2 Race Condition 不等于 Data Race

Race Condition 是更宽泛的逻辑概念：**程序结果依赖不受约束的执行时序，并可能违反业务不变量。**它可以完全没有 Data Race。

```go
var stock atomic.Int64

func Buy() bool {
    if stock.Load() <= 0 {
        return false
    }
    stock.Add(-1)
    return true
}
```

每次内存访问都是 atomic，因此 Race Detector 不会报告普通 Data Race；但两个购买者可能都看到库存为 1，然后都扣减，最终库存变为 -1。这是典型的 **check-then-act 业务竞态**。

正确方案可以是 CAS 循环：

```go
func Buy() bool {
    for {
        old := stock.Load()
        if old <= 0 {
            return false
        }
        if stock.CompareAndSwap(old, old-1) {
            return true
        }
    }
}
```

如果库存与订单、支付、优惠等多个实体共同构成不变量，仅靠单进程 CAS 仍不够，应进入数据库事务、条件更新、唯一约束或幂等协议层面。

#### 2.2.3 Deadlock、Livelock、Starvation

| 概念 | 定义 | 典型现象 |
|---|---|---|
| Deadlock | 参与者互相等待，系统无法继续取得进展 | A 持有 L1 等 L2，B 持有 L2 等 L1 |
| Livelock | 参与者仍在运行、重试或让步，但没有完成有效工作 | 两方持续 CAS 失败后同步退让，又同时重试 |
| Starvation | 某个参与者长期得不到锁、CPU、队列机会或资源 | 热门 goroutine 反复抢到锁，旧 waiter 长期等待 |

Go runtime 的 `fatal error: all goroutines are asleep - deadlock!` 只是在某些**全进程无可运行 goroutine**的场景下检测到全局停滞。它不是通用死锁证明器：

- 某个后台 goroutine 仍在运行时，局部锁环可能不会触发该错误；
- 正在等待外部 I/O、cgo、定时器或网络事件时，runtime 不能简单认定死锁；
- 分布式死锁、业务状态机卡死也不属于 runtime 的全局检测能力。

### 2.3 Memory Model：三种顺序与 DRF-SC

#### 2.3.1 sequenced-before

`sequenced-before` 描述**同一个 goroutine 内**由控制流、表达式求值顺序和语言规范确定的偏序。

```go
data = 42       // A
ready = true    // B
```

在同一 goroutine 中，A 通常 sequenced-before B。但这并不自动让其他 goroutine 看到 A 和 B；跨 goroutine 还需要同步关系。

#### 2.3.2 synchronized-before

`synchronized-before` 来自同步操作之间的“观察关系”。例如：

- channel send 与对应 receive；
- `Mutex.Unlock` 与后续成功返回的 `Mutex.Lock`；
- atomic 写被另一个 atomic 操作观察；
- `Once` 中 `f` 完成与所有 `Do` 返回。

它只描述同步操作之间的边。

#### 2.3.3 happens-before

`happens-before` 是下面两种关系并集的传递闭包：

```text
happens-before
    = transitive-closure(
          sequenced-before
        ∪ synchronized-before
      )
```

示例：

```go
var data int
ch := make(chan struct{})

go func() {
    data = 42          // A
    close(ch)          // B
}()

<-ch                   // C
fmt.Println(data)      // D
```

关系链为：

```text
A sequenced-before B
B synchronized-before C
C sequenced-before D
────────────────────────
A happens-before D
```

所以 D 保证观察到 A 的结果。

#### 2.3.4 DRF-SC

DRF-SC 是 **Data-Race-Free ⇒ Sequential Consistency**：

> 如果程序没有 Data Race，它的可观察结果可以用“各 goroutine 操作按某种顺序一致交错执行”来解释。

这并不意味着真实硬件只用一个 CPU，也不意味着 goroutine 按源码顺序串行运行；它给的是**可推理的语义保证**。

**面试要点：**不要在有 Data Race 的程序上继续讨论“究竟会读到 0 还是 1”。先判定程序错误，再谈实现可能出现的有限现象。特别是 slice、string、interface、map 等多机器字结构发生 Race 时，可能观察到不一致的指针/长度或指针/类型组合，后果不只是“丢一次更新”。

### 2.4 Go 中主要同步操作的保证

#### 2.4.1 Goroutine 创建与退出

- 启动新 goroutine 的 `go` 语句 synchronized-before 新 goroutine 开始执行。
- 因而在 `go f()` 之前完成的普通写，`f` 开始后可以通过这条边观察到。
- **goroutine 退出本身不保证 synchronized-before 其他任何事件。**要观察其结果，必须使用 channel、锁、`WaitGroup` 等同步机制。

```go
var x int

func main() {
    x = 1
    go func() {
        fmt.Println(x) // 保证看到 1；启动边建立了顺序
    }()
}
```

反向则不成立：

```go
var x int

go func() { x = 1 }()
fmt.Println(x) // Data Race；goroutine 退出没有自动 join 语义
```

#### 2.4.2 Channel

Go Memory Model 给出几条精确规则：

1. **send synchronized-before 对应 receive 完成。**
2. **close synchronized-before 因关闭而返回零值的 receive。**
3. 对无缓冲 channel，**receive synchronized-before 对应 send 完成。**
4. 对容量为 `C` 的缓冲 channel，第 `k` 次 receive synchronized-before 第 `k+C` 次 send 完成。

第三条使无缓冲 channel 成为双向 rendezvous；缓冲 channel 则不能简单理解成“send 返回就代表 receiver 已处理”。

```go
ch := make(chan int, 1)
ch <- 1 // 只说明值已进入缓冲区，不说明某个接收者已完成业务处理
```

#### 2.4.3 Mutex 与 RWMutex

对同一个 `Mutex` 或 `RWMutex`：

- 第 `n` 次 `Unlock` synchronized-before 第 `m` 次 `Lock` 返回，前提是 `n < m`。
- 成功的 `TryLock` 等价于 `Lock`；失败的 `TryLock` **没有任何同步效果**。
- `RWMutex` 还规定：某次 `RLock` 前存在一个对应的 `Unlock` 同步边；匹配的 `RUnlock` 又同步到后续写锁的 `Lock`。

关键不是“调用了 Lock”，而是**同一把锁上匹配的解锁与后续成功加锁形成顺序**。

#### 2.4.4 Once

`once.Do(f)` 中那一次 `f` 的完成 synchronized-before 任意一次 `Do` 返回。因此 `f` 发布的初始化结果，在所有返回的调用者中可见。

#### 2.4.5 WaitGroup

标准库契约规定：

- `Done` synchronized-before 它所解除阻塞的 `Wait` 返回；
- `WaitGroup.Go(f)` 中 `f` 的返回 synchronized-before 它所解除阻塞的 `Wait` 返回。

`WaitGroup` 只负责**等待任务结束**，不负责传递错误、结果或取消信号。

#### 2.4.6 Pool

- `Put(x)` synchronized-before 某次 `Get` 返回同一个 `x`；
- `New` 返回 `x` synchronized-before 某次 `Get` 返回该 `x`。

但 `sync.Pool` 可在任意时刻丢弃元素，不能用作业务对象仓库、连接所有权表或可靠队列。

#### 2.4.7 Cond

`Signal` 或 `Broadcast` synchronized-before 被其解除阻塞的 `Wait`。不过条件本身仍需在锁保护下检查与修改。

#### 2.4.8 sync.Map

`sync.Map` 文档按不同方法区分读操作和写操作。一个写操作 synchronized-before 任何观察到该写效果的读操作。`Range` **不是一致性快照**。

#### 2.4.9 Atomic

Go 的 `sync/atomic` 契约是：

- 如果 atomic 操作 A 的效果被 atomic 操作 B 观察到，则 A synchronized-before B；
- 程序中的所有 atomic 操作表现得像按某个**顺序一致（sequentially consistent）**的总序执行。

Go 标准库没有直接暴露 C++ 风格的 relaxed/acquire/release 选择。不要把别的语言的内存序 API 生搬过来。

### 2.5 为什么 Sleep 不是同步

```go
var ready bool
var data int

go func() {
    data = 42
    ready = true
}()

time.Sleep(time.Second)
if ready {
    fmt.Println(data)
}
```

无论 Sleep 多久，都没有创建 synchronized-before 边。它只能改变“较可能发生”的调度时机，不能提供：

- 可见性保证；
- 有序性保证；
- 完成保证；
- 对未来机器、负载、编译器优化的正确性保证。

正确做法是使用 channel、锁、atomic 或 WaitGroup。

---

### 2.6 为什么“我观察到先执行”不代表 happens-before

下面这些现象都不能构成规范上的同步证明：

- 日志时间戳先后；
- debugger 单步时的执行顺序；
- 单核机器上多次测试结果一致；
- `GOMAXPROCS=1`；
- goroutine A 似乎总比 B 先启动；
- 某次 profile 或 trace 中看到的实际调度顺序；
- CPU 在当前架构上“通常不会重排成那样”。

**happens-before 是程序语义中的偏序，不是一次运行的时间线截图。**一次执行恰好按某顺序发生，不等于程序对所有合法执行都建立了该顺序。

### 2.7 Channel、Mutex、Atomic 的第一层选型

| 需求 | 首选思路 | 原因 |
|---|---|---|
| 短临界区内维护多个字段不变量 | `Mutex` | 直接表达“这一组状态一起变” |
| 读多写少、读临界区较长且可证明收益 | `RWMutex` | 允许并发读，但需基准验证 |
| 单一计数、标志、版本号、指针发布 | typed atomic | 状态小、操作独立、语义明确 |
| 读极多写极少的整份配置 | immutable snapshot + `atomic.Pointer`/`Value` | 读路径无锁，写时整体替换 |
| 工作传递、所有权转移、背压、事件流 | channel | 将通信与生命周期显式化 |
| 等待一组任务结束 | `WaitGroup` | 计数栅栏，不承载结果 |
| 一次性初始化 | `Once` / `OnceValue` | 正确发布且避免 DCL |
| 可重复条件等待 | `Cond` | 同一共享谓词可多次变真/变假 |
| 特殊并发键值访问模式 | `sync.Map` | 专用结构；不是普通 map 的通用替代 |

**“Do not communicate by sharing memory; share memory by communicating” 是设计提醒，不是“所有共享状态都必须改成 channel”。**短小状态机和不变量通常用锁更清晰；流水线、所有权转移和背压通常用 channel 更自然。

## 三、底层实现

### 3.1 `sync.Mutex` 的公开语义

Go 1.26.4 的公开 `sync.Mutex`：

```go
type Mutex struct {
    _  noCopy
    mu internal/sync.Mutex
}
```

当前内部核心结构是：

```go
type Mutex struct {
    state int32
    sema  uint32
}
```

公开契约与使用规则：

1. **零值即未加锁状态，可直接使用。**
2. **第一次使用后不得复制。**`noCopy` 配合 `go vet -copylocks` 做静态提示，但不是运行时万能防线。
3. **不可重入。**同一 goroutine 再次 `Lock` 不会识别“这是自己”，通常会阻塞。
4. **锁不绑定 goroutine 所有权。**一个 goroutine 可以 Lock，另一个 goroutine Unlock；这是 API 允许的，但通常降低可读性，应谨慎设计。
5. 解锁未锁定的 Mutex 是运行时错误。
6. `TryLock` 存在，但官方文档明确提示：正确使用场景少见，频繁使用往往说明设计有更深问题。

#### 3.1.1 为什么复制 Mutex 危险

复制会复制 `state` 和 `sema` 的某一时刻快照。随后两个副本不再代表同一同步对象：

- 复制已加锁 Mutex，副本可能永远保持“锁定”；
- waiter 可能睡在原对象的 `sema` 地址，解锁副本不会唤醒它；
- 结构体 value receiver、按值传参、从容器取值再复制都可能无意中复制锁。

因此含锁结构通常使用指针接收者、指针传递，并避免复制。

---

### 3.2 Mutex 状态位

Go 1.26.4 `src/internal/sync/mutex.go` 中 `state` 的低位布局为：

```text
bit 0: mutexLocked      当前逻辑上已锁定
bit 1: mutexWoken       已有 waiter 被唤醒，避免重复唤醒
bit 2: mutexStarving    处于饥饿模式
bit 3...: waiter count  等待者数量
```

概念图：

```text
31                                      3  2  1  0
+----------------------------------------+--+--+--+
|              waiter count              | S| W| L|
+----------------------------------------+--+--+--+
                                           │  │  └─ locked
                                           │  └──── woken
                                           └─────── starving
```

`sema` 是交给 runtime semaphore 休眠/唤醒路径使用的地址标识与计数槽。

### 3.3 Mutex Lock：快路径、短暂自旋与慢路径

#### 3.3.1 快路径 CAS

无竞争时：

```go
if atomic.CompareAndSwapInt32(&m.state, 0, mutexLocked) {
    return
}
```

快路径只有一次 CAS，且可内联。它是 Mutex 在低竞争下很便宜的原因。

#### 3.3.2 为什么尝试自旋

如果锁预计很快释放，让当前 M/P 上运行的 goroutine 做少量 active spinning，可能比：

```text
入队 → gopark → 调度其他 G → goready → 再次调度
```

更便宜。

但 Go 不会无条件自旋。当前实现由 `runtime_canSpin(iter)` 判断是否值得，综合考虑迭代次数、可用并行度、当前 P 的运行队列等 runtime 状态。**具体启发式属于实现细节，不是语言保证。**

在普通模式下，如果锁处于 locked 且非 starving，慢路径可：

- 设置 `mutexWoken`，告诉 Unlock 已有人负责竞争，避免多唤醒；
- 调用 `runtime_doSpin()` 做有限自旋；
- 重新读取状态并尝试 CAS。

饥饿模式下不自旋，因为所有权应直接交给队首 waiter，新来者不能抢占。

#### 3.3.3 `lockSlow`

慢路径大致流程：

```text
读取 state
  ├─ 条件合适：有限自旋
  └─ 否则：
       计算新 state
       - 标记 locked 或 starving
       - waiter count +1
       - 必要时清 woken
       CAS 更新
          ├─ 原来可用：获得锁
          └─ 原来被占：
               runtime_SemacquireMutex(&sema, queueLifo, ...)
               goroutine 休眠
               被唤醒后重新竞争或接收直接 handoff
```

如果一个 waiter 已经睡过一次又重新入队，当前实现可使用 LIFO 插到前部，减少其再次长期落后。

---

### 3.4 正常模式与饥饿模式

#### 3.4.1 正常模式

正常模式中 waiter 大致按 FIFO 排队，但被唤醒不等于立即拥有锁。它还要和新到达且正在 CPU 上运行的 goroutine 竞争：

```text
Unlock 唤醒旧 waiter
           │
           ├─ 旧 waiter 尚未被调度
           └─ 新 goroutine 已在运行，CAS 抢走锁
```

这允许同一个或新来的 goroutine 连续快速获得锁，吞吐与局部性较好，但旧 waiter 可能多次失败。

#### 3.4.2 饥饿模式

Go 1.26.4 当前实现中，如果 waiter 等待超过：

```go
starvationThresholdNs = 1e6 // 1 ms
```

它可推动 Mutex 进入饥饿模式。此时：

- Unlock 将所有权直接 handoff 给队首 waiter；
- 新来的 goroutine 不尝试抢锁，也不自旋，而是排到队尾；
- waiter 获得锁后，如果它是最后一个 waiter，或其等待不足 1 ms，则退出饥饿模式。

**1 ms 是 Go 1.26.4 的实现阈值，不是 API 承诺。面试时必须说“当前实现”，不能说成 Go 语言永久规定。**

#### 3.4.3 为什么需要两种模式

- 只追求公平：频繁 handoff 与调度切换会降低吞吐；
- 只允许抢占：队首 waiter 在高竞争下可能长期饿死；
- 正常模式优化平均吞吐，饥饿模式限制尾延迟和病态不公平。

### 3.5 Mutex Unlock 如何唤醒 waiter

快路径先原子清除 locked 位：

```go
new := atomic.AddInt32(&m.state, -mutexLocked)
if new != 0 {
    m.unlockSlow(new)
}
```

#### 正常模式

`unlockSlow` 会检查：

- 是否根本没有 waiter；
- 是否已经有 waiter 被唤醒；
- 是否又有人获得锁；
- 是否已进入 starving。

若确实需要唤醒，它 CAS 减少 waiter 计数、设置 `mutexWoken`，再调用 `runtime_Semrelease(&m.sema, false, ...)` 唤醒一个等待者。

#### 饥饿模式

调用：

```go
runtime_Semrelease(&m.sema, true, ...)
```

`handoff=true` 允许 runtime 把 semaphore ticket 和当前 P 的执行机会直接交给 waiter，以尽快运行它。

### 3.6 runtime semaphore：不是业务信号量

`runtime/sema.go` 的注释明确提醒：不要把这里简单理解为传统计数信号量。它的目标是为同步原语提供**不会丢失唤醒的 sleep/wakeup 配对机制**，类似更简单语义的 futex。

核心思路：

1. 每个 `sema` 地址哈希到一个 `semaRoot`；
2. root 内按等待地址组织平衡树，同一地址上的 waiter 形成列表；
3. `semacquire1` 先走无锁/低成本消费路径；
4. 失败后增加 waiter 计数，再次检查，避免“检查后、入睡前”丢失唤醒；
5. 入队后 `goparkunlock` 休眠；
6. `semrelease1` 增加票据，若有 waiter 则出队并 `goready`；
7. 饥饿模式可直接 handoff 并让当前 goroutine yield。

```text
Mutex state CAS 失败
      │
      v
runtime_SemacquireMutex(sema)
      │
      ├─ 再检查票据，防 missed wakeup
      ├─ sudog 入队
      └─ gopark

Unlock
      │
      v
runtime_Semrelease(sema)
      ├─ 无 waiter：返回
      └─ 出队 waiter → goready / direct handoff
```

runtime semaphore 还为 block/mutex profile 记录等待时间与竞争归因。

### 3.7 `TryLock` 的边界

可能合理的少数场景：

- 调试或诊断路径，拿不到锁就跳过非关键统计；
- 某种有明确退路的 opportunistic fast path；
- 避免调用未知代码时的锁顺序反转，但更好的方案通常是重构锁层次。

危险用法：

```go
for !mu.TryLock() {
    // busy loop
}
```

问题包括：

- 空转占 CPU；
- 可能恶化锁持有者获得 CPU 的机会；
- 失败没有同步效果，不能借失败观察受保护状态；
- 易形成 livelock 或 starvation；
- 复杂的退避策略往往比直接阻塞锁更难正确。

**默认使用 `Lock`。只有“拿不到就做另一件有意义的事”时才考虑 `TryLock`。**

### 3.8 `sync.RWMutex` 数据结构

Go 1.26.4：

```go
type RWMutex struct {
    w           Mutex
    writerSem   uint32
    readerSem   uint32
    readerCount atomic.Int32
    readerWait  atomic.Int32
}
```

字段职责：

| 字段 | 作用 |
|---|---|
| `w` | 串行化写者；有 pending writer 时持有 |
| `readerCount` | 当前/待处理读者计数；负值表示有写者已宣布到来 |
| `readerWait` | 写者正在等待退出的旧读者数量 |
| `writerSem` | 最后一个旧读者退出时唤醒写者 |
| `readerSem` | 写者释放后唤醒被阻塞的新读者 |

内部使用：

```go
const rwmutexMaxReaders = 1 << 30
```

写者通过从 `readerCount` 减去这个大常数，把其变为负数，向后续读者发布“有写者等待”。

---

### 3.9 RWMutex 读写流程与 writer preference

#### 3.9.1 `RLock`

```text
readerCount.Add(1)
  ├─ 结果 >= 0：直接获得读锁
  └─ 结果 < 0：已有 pending writer，睡在 readerSem
```

#### 3.9.2 `Lock`

```text
w.Lock()                       // 先与其他写者竞争
readerCount -= maxReaders      // 宣告写者到来，阻止新读者
r = 原有活跃读者数量
readerWait += r
若 r != 0：睡在 writerSem，等待旧读者退出
```

#### 3.9.3 `RUnlock`

读者减计数。如果结果仍为负，说明写者在等；它再减少 `readerWait`。最后一个旧读者把 `readerWait` 降为 0，并唤醒写者。

#### 3.9.4 `Unlock`

写者：

1. 将 `readerCount` 加回 `rwmutexMaxReaders`；
2. 按被阻塞的新读者数量，逐个释放 `readerSem`；
3. 解开 `w`，允许下一个写者竞争。

#### 3.9.5 writer preference 的准确含义

一旦写者调用 `Lock` 并宣布 pending：

- 已持有读锁的读者可继续到 `RUnlock`；
- 后来的 `RLock` 会被阻塞；
- 写者等旧读者清空后取得锁。

这避免写者被持续到来的新读者无限推迟。

---

### 3.10 为什么不能升级锁、降级锁或递归读锁

#### 3.10.1 不能从 RLock 升级到 Lock

```go
rw.RLock()
// ...
rw.Lock() // 死锁风险
```

当前 goroutine 自己占一个读者名额；`Lock` 等待所有旧读者退出，其中包括它自己，但它只有在 `Lock` 返回后才会继续 `RUnlock`。

正确做法通常是：

1. `RUnlock`；
2. `Lock`；
3. **重新检查条件**，因为中间状态可能变化。

#### 3.10.2 不能原子降级

```go
rw.Lock()
rw.RLock() // 不支持原子“写锁降级为读锁”
```

需要先 `Unlock` 再 `RLock`，中间会有其他写者或读者介入，因此不能维持无缝所有权。若业务必须原子降级，需重构状态协议，而不是依赖 RWMutex。

#### 3.10.3 不要递归 RLock

看似多个读锁兼容，但若两次 `RLock` 之间有写者到来，第二次 `RLock` 会等待写者；写者又在等待当前 goroutine 释放第一次读锁，形成死锁。

标准库文档因此明确禁止把 `RLock` 当递归读锁使用。

---

### 3.11 RWMutex 为什么不一定比 Mutex 快

`RWMutex` 有额外成本：

- 每次读锁都要原子更新共享 `readerCount`，可能导致 cache line bouncing；
- 写者到来时要协调旧读者与新读者；
- 读临界区很短时，Mutex 的一次 CAS 可能更便宜；
- 写比例不低时，并发读优势被频繁的写者屏障抵消；
- 大量 CPU 核争用同一个原子计数也可能成为瓶颈；
- 数据访问本身可能受共享缓存、内存带宽限制，并非锁允许并行就能线性扩展。

经验规则：

- 读占绝大多数；
- 读临界区不是极短；
- 写相对稀少；
- 读之间确实有可并行工作；

才可能从 `RWMutex` 获益。**必须用代表真实读写比例与临界区长度的 benchmark 验证。**

### 3.12 `sync.WaitGroup` 当前实现

Go 1.26.4 核心结构：

```go
type WaitGroup struct {
    noCopy noCopy
    state  atomic.Uint64
    sema   uint32
}
```

`state` 当前实现的概念布局：

```text
高 32 位：task counter
中间 1 位：synctest bubble flag（内部测试机制）
低 31 位：waiter count
```

业务面试通常关注：counter、waiter 与 semaphore 如何协作，而不是内部测试标志。

#### 3.12.1 `Add` / `Done`

- `Add(delta)` 原子修改高位 counter；
- counter 变为负数会 panic；
- `Done()` 等价于 `Add(-1)`；
- counter 变为 0 且存在 waiter 时，清零 waiter 状态并多次 `Semrelease` 唤醒所有等待者。

#### 3.12.2 `Wait`

- 如果 counter 已为 0，立即返回；
- 否则 CAS 增加 waiter count；
- 睡在 `sema`；
- 被唤醒后校验 WaitGroup 是否被过早复用。

#### 3.12.3 为什么 Add 通常要在启动 goroutine 前

错误：

```go
for i := 0; i < n; i++ {
    go func() {
        wg.Add(1)
        defer wg.Done()
        work()
    }()
}
wg.Wait()
```

`Wait` 可能在任何子 goroutine 执行 `Add(1)` 前观察到 counter 为 0 并返回。之后任务仍在运行，主流程却认为已完成。

正确：

```go
for i := 0; i < n; i++ {
    wg.Add(1)
    go func() {
        defer wg.Done()
        work()
    }()
}
wg.Wait()
```

精确边界不是“任何 Add 都绝不能和 Wait 并发”：

- 当 counter 为 0 时，正增量 Add 必须发生在对应 Wait 之前；
- counter 已大于 0 时，进一步正增量可以在合规设计下发生；
- 重用 WaitGroup 时，新一轮 Add 必须在上一轮所有 Wait 返回后。

Go 1.25 起 `go vet` 增加了 WaitGroup 分析，可报告常见的“在新 goroutine 内 Add”错误。

#### 3.12.4 复用与复制

- 可复用，但必须等前一轮所有 `Wait` 返回后才能开始下一轮 Add/Go；
- 不得复制已使用的 WaitGroup；
- 把含 WaitGroup 的结构体按值传递同样有复制风险；
- 多个 goroutine 同时调用 `Wait` 是允许的，counter 归零时都会被唤醒。

---

### 3.13 `WaitGroup.Go`：Go 1.25+ 的当前语义

Go 1.26.4 可写：

```go
var wg sync.WaitGroup
wg.Go(task1)
wg.Go(task2)
wg.Wait()
```

它内部大意是：

```go
func (wg *WaitGroup) Go(f func()) {
    wg.Add(1)
    go func() {
        defer func() {
            if x := recover(); x != nil {
                // 当前实现重新抛出，并且不先 Done。
                panic(x)
            }
            // f 正常返回，或通过 runtime.Goexit 退出。
            wg.Done()
        }()
        f()
    }()
}
```

但要注意标准库文档的关键限制：

> **传给 `WaitGroup.Go` 的函数 `f` 不得 panic。**

Go 1.26.4 当前实现会在 wrapper 中 recover 到 panic 后重新 panic，并刻意不先调用 Done，避免 `Wait` 被解除后主 goroutine 与致命 panic 竞速甚至提前 `os.Exit(0)`。这属于当前实现为崩溃语义做的保护，不应写代码依赖其细节。

其他边界：

- 如果 WaitGroup 为空，`Go` 必须发生在 `Wait` 之前；
- 如果已非空，某个任务可继续调用 `wg.Go` 派生子任务；
- 它不传递返回值、错误或 context 取消；需要这些能力时使用自定义封装或合适的结构化并发库。

### 3.14 `sync.Once` 与一次性函数

当前结构：

```go
type Once struct {
    _    noCopy
    done atomic.Bool
    m    Mutex
}
```

`done` 放在首字段，方便热点路径生成更紧凑指令并被内联。

#### 3.14.1 快慢路径

```go
func (o *Once) Do(f func()) {
    if !o.done.Load() {
        o.doSlow(f)
    }
}
```

慢路径：

```go
o.m.Lock()
defer o.m.Unlock()
if !o.done.Load() {
    defer o.done.Store(true)
    f()
}
```

为什么不能只写 CAS：

```go
if done.CompareAndSwap(false, true) {
    f()
}
```

因为输掉 CAS 的调用者会立即返回，而 `f` 可能尚未完成；这破坏了“任何 Do 返回时，唯一一次 f 已完成”的保证。Mutex 让其他调用者等待完成，`done.Store(true)` 又必须推迟到 `f` 返回之后。

#### 3.14.2 panic 行为

- 如果 `f` panic，`Once.Do` 仍把它视为“已返回”；
- 后续 `Do` 不会重试 `f`；
- 因此 `Once` 不适合“失败后自动重试初始化”；
- `f` 内递归调用同一个 Once 的 `Do` 会死锁。

#### 3.14.3 `OnceFunc`、`OnceValue`、`OnceValues`

- `OnceFunc(f)` 返回一个可并发调用、只实际执行一次 `f` 的函数；
- `OnceValue(f)` 缓存一个返回值；
- `OnceValues(f)` 缓存两个返回值；
- 若 `f` panic，这些包装函数会让**每次调用都以相同 panic 值再次 panic**，与直接 `Once.Do` 后续静默返回的体验不同。

```go
loadConfig := sync.OnceValue(func() *Config {
    return mustLoadConfig()
})

cfg := loadConfig()
```

返回的对象若会被多 goroutine 读取，仍应在发布后保持不可变，或在对象内部继续同步。

### 3.15 `sync.Cond`

`Cond` 是围绕共享谓词的条件变量：

```go
type Cond struct {
    L      Locker
    notify notifyList
    // copy checker ...
}
```

典型模式：

```go
cond.L.Lock()
for !condition() {
    cond.Wait()
}
useCondition()
cond.L.Unlock()
```

`Wait` 做三件事：

1. 将当前 waiter 注册到通知列表；
2. **原子地解开 `L` 并休眠**；
3. 被唤醒后重新锁住 `L`，然后返回。

#### 为什么必须用 for

Go 的 `Cond.Wait` 不会无缘无故返回；但被唤醒时谓词仍可能为假：

- 多个 waiter 被 Broadcast 唤醒，先获得锁者消费了资源；
- Signal 只说明发生了通知，不把某个业务事实“锁定”给该 waiter；
- 从收到通知到重新获取锁之间，状态可能再次改变。

所以正确原则不是“防 POSIX 式伪唤醒”这么简单，而是：**等待的是谓词，不是通知本身。**

#### Signal 与 Broadcast

- `Signal` 唤醒一个 waiter；
- `Broadcast` 唤醒全部 waiter；
- 调用它们时 API 不强制持有 `L`；
- 但共享谓词必须在同一锁协议下修改。通常在持锁修改状态后通知，最容易避免丢失条件变化和推理错误。

#### Cond 与 Channel

| Cond | Channel |
|---|---|
| 围绕共享谓词反复等待 | 传递值、事件或所有权 |
| Signal/Broadcast 可重复使用 | close 是一次性广播 |
| 必须配合 Locker 与谓词循环 | send/receive 自带同步与可选背压 |
| 适合复杂条件或大量 waiter | 多数简单通知场景更清晰 |

### 3.16 `sync/atomic`：类型、语义与边界

#### 3.16.1 typed atomics

优先使用：

- `atomic.Bool`
- `atomic.Int32` / `atomic.Int64`
- `atomic.Uint32` / `atomic.Uint64`
- `atomic.Uintptr`
- `atomic.Pointer[T]`

这些类型提供：

- `Load`
- `Store`
- `Swap`
- `CompareAndSwap`
- 整数类型的 `Add`、`And`、`Or`

它们零值可用，但第一次使用后不得复制。`Int64`/`Uint64` 类型还通过编译器识别的对齐字段处理 32 位平台的 64 位对齐问题；直接用旧式函数操作裸 `int64` 时需要更谨慎。

#### 3.16.2 Atomic 适合什么状态

适合：

- 单调或独立计数；
- 标志位；
- 状态枚举；
- 版本号；
- 单指针发布；
- 能被编码进一个机器字的状态机；
- 读多写少的不可变快照指针。

不适合：

- 多字段共同不变量；
- 需要等待条件的复杂状态机；
- 需要公平性或阻塞队列；
- CAS 失败率高、竞争非常激烈的更新；
- 需要一次更新多个对象或跨进程资源。

#### 3.16.3 多个 atomic 字段不构成整体原子性

```go
type Stats struct {
    ok    atomic.Int64
    total atomic.Int64
}
```

写者依次更新两者，读者可能读到来自不同时间点的组合。即使每个字段本身无 Race，也不能保证 `ok <= total` 等跨字段不变量在观察瞬间成立。

解决方案：

- Mutex 保护多字段；
- 将状态打包进一个可 CAS 的整数；
- 构造完整不可变对象，通过一个 atomic pointer 一次发布；
- 使用版本校验重试，但要证明正确性。

---

### 3.17 `atomic.Pointer[T]`、`atomic.Value` 与不可变发布

#### 3.17.1 `atomic.Pointer[T]`

适合发布类型明确的 `*T`：

```go
type Config struct {
    Timeout time.Duration
    Hosts   []string
}

var current atomic.Pointer[Config]

func LoadConfig() *Config {
    return current.Load()
}

func PublishConfig(c *Config) {
    clone := &Config{
        Timeout: c.Timeout,
        Hosts:   slices.Clone(c.Hosts),
    }
    current.Store(clone)
}
```

**关键不是 Store 的那一刻，而是发布后不再修改 `clone` 及其可达可变对象。**只把指针原子化，不会让被指向对象的后续普通写自动安全。

#### 3.17.2 `atomic.Value`

特性：

- 零值 `Load` 返回 nil；
- 第一次 Store 后不得复制；
- 不能 Store nil；
- 同一个 Value 的所有 Store 必须使用相同具体类型，否则 panic；
- 可存放非指针的完整快照或接口值，但类型约束要一致。

```go
var v atomic.Value
v.Store(Config{Timeout: time.Second})
_ = v.Load().(Config)
```

`atomic.Pointer[T]` 类型更明确、避免断言，通常优先用于指针快照；`atomic.Value` 适合需要原子替换任意同一具体类型值的场景。

#### 3.17.3 copy-on-write / immutable snapshot

```text
writer:
  Load old snapshot
  Clone all需要修改的结构
  在私有副本上修改
  Atomic Store new snapshot

reader:
  Atomic Load snapshot
  只读，不加锁
```

优点：读路径非常便宜，不阻塞；缺点：写放大、分配和复制成本高，旧快照要等 GC 回收。适合配置、路由表、规则集等读极多写极少数据。

### 3.18 ABA 问题

CAS 只比较“当前值是否等于旧值”。如果状态经历：

```text
A → B → A
```

另一个 goroutine 只看到最终仍为 A，可能误认为从未变化。

典型风险：无锁栈节点弹出后被复用，地址再次变回同一值；CAS 成功却基于过期拓扑。

缓解方式：

- 值中携带递增版本号/tag；
- 不立即复用节点；
- epoch/hazard pointer 等安全回收机制；
- 不可变节点与受控生命周期；
- 在业务不值得承担无锁复杂度时直接用 Mutex。

Go 有 GC，不等于自动消除 ABA：对象地址复用、逻辑状态回环、索引或句柄复用仍可构成 ABA。

### 3.19 false sharing 与 cache line contention

两个逻辑上无关的原子变量若位于同一 cache line，不同 CPU 核频繁写各自字段时，整条缓存行仍需在核心间来回转移：

```text
Core 0 写 counterA ─┐
                     ├─ 同一 cache line 反复失效/转移
Core 1 写 counterB ─┘
```

这叫 false sharing。即使没有 Data Race，也可能造成吞吐下降和尾延迟。

而多个核心频繁写**同一个**原子计数，是真实 cache line contention。Atomic 避开了阻塞锁，却没有消除缓存一致性成本。

治理：

- 分片/每 P、每 worker 本地计数，低频聚合；
- 将高频独立写字段分离；
- 批量更新；
- 减少全局热点；
- 用 benchmark、CPU profile、硬件计数器验证。

不要随意硬编码“缓存行一定 64 字节”。平台与实现可能不同，填充会增加内存占用，必须基于目标平台与测量。

### 3.20 map + Mutex、分片锁与 `sync.Map`

#### 3.20.1 普通 map + Mutex

```go
type SafeMap[K comparable, V any] struct {
    mu sync.RWMutex
    m  map[K]V
}
```

优点：

- 类型安全；
- 可以在同一临界区维护 map 与其他字段不变量；
- 语义清晰；
- 适用于多数业务代码。

注意：锁只保护你在锁内访问的对象。若 value 是指针，取出后在锁外修改其字段，map 的锁并不会自动保护对象内部。

#### 3.20.2 分片锁

按 key 哈希到多个 shard：

```text
hash(key) % N → shard[i] → { Mutex, map }
```

适合大量独立 key 的并发访问，可降低单锁竞争。代价：

- 全局 Range/清空要协调所有 shard；
- 跨 key 不变量可能需要多锁；
- 必须规定多 shard 锁顺序；
- shard 数太多增加内存，太少仍热点；
- 热 key 仍集中在一个 shard。

#### 3.20.3 `sync.Map`：Go 1.26 当前实现

公开文档仍强调它是**专用结构**，主要优化：

1. 某 key 只写一次、之后读很多，如只增长缓存；
2. 多 goroutine 操作互不重叠的 key 集合。

Go 1.26.4 的公开 `sync.Map` 包装：

```go
type Map struct {
    _ noCopy
    m internal/sync.HashTrieMap[any, any]
}
```

当前 Hash-Trie 的要点：

- 根是 `atomic.Pointer`；
- 内部节点当前为 16 路 child 数组；
- child 用 atomic pointer 发布；
- 读路径沿哈希位无锁遍历；
- 写入锁定目标内部节点的局部 Mutex，二次校验后发布新 entry/子树；
- 冲突使用 overflow 链；
- `Clear` 可通过替换根节点完成；
- `Range` 不提供一致性快照。

**版本提醒：**Go 1.25 的默认实现仍是 `read`/`dirty` 双 map、`misses` 触发提升的经典方案，并有 `goexperiment.synchashtriemap` 实验路径。到了 Go 1.26，Hash-Trie 已成为当前默认实现。面试中可以先答 API 适用场景，再说明实现随版本变化；不要背旧字段冒充当前源码。

### 3.21 临界区、锁粒度与锁顺序

#### 3.21.1 临界区应保护不变量，而不是机械包住单行

太小：

```go
mu.Lock()
if balance >= n {
    mu.Unlock()

    mu.Lock()
    balance -= n
    mu.Unlock()
}
```

检查与修改分离，业务不变量被破坏。

太大：

```go
mu.Lock()
resp, err := httpClient.Do(req) // 锁内网络 I/O
mu.Unlock()
```

会放大等待时间、尾延迟和故障影响面。

正确粒度是：**足以覆盖一个不变量变换，但尽量不包含不可控阻塞。**

#### 3.21.2 锁内避免 I/O、回调和阻塞 Channel

- I/O 时延不可控；
- 回调可能重入当前对象，再次拿同一锁；
- 回调可能获取其他锁，形成隐藏锁顺序；
- channel send/receive 可能无限阻塞；
- 锁持有时间越长，竞争等待成本会乘以 waiter 数量。

常见做法：锁内只复制必要状态或登记任务，解锁后执行外部动作；执行后再以版本号/状态检查提交结果。

#### 3.21.3 统一锁顺序

例如规定：

```text
Account 锁按 accountID 升序获取
Shard 锁按 shard index 升序获取
全局锁必须先于对象锁
```

所有路径都遵守同一偏序，可消除一类循环等待。更好的方案常是减少嵌套锁、合并所有权或用消息传递串行化一个域。

#### 3.21.4 `defer Unlock`

推荐：

```go
mu.Lock()
defer mu.Unlock()
```

优点是异常路径、早返回和未来修改不易漏解锁。边界：

- `defer` 作用到当前函数返回，不是代码块结束；
- 在长循环中每轮 defer，会让锁一直持有到整个函数退出；
- 极端热点小函数可基准验证显式 Unlock 是否有收益，但不要用微小推测换正确性风险；
- 可提取小 helper，让 defer 的作用域与临界区一致。

### 3.22 双重检查锁为什么容易写错

错误：

```go
if instance == nil {
    mu.Lock()
    if instance == nil {
        instance = build()
    }
    mu.Unlock()
}
return instance
```

外层普通读与写 `instance` 发生 Data Race；即使某次读到非 nil，也不能由此证明对象全部字段已安全发布。

可用 `sync.Once`：

```go
once.Do(func() {
    instance = build()
})
```

或用 `atomic.Pointer` + Mutex 写出正确发布协议，但复杂度通常没有必要。**一次性初始化首选 Once；热更新首选不可变快照。**

### 3.23 业务层并发边界

进程内同步只能协调同一进程中的 goroutine：

| 问题范围 | 常用机制 |
|---|---|
| 同一变量的进程内访问 | Mutex、Atomic、Channel |
| 同一进程任务生命周期 | WaitGroup、Context、Channel |
| 同一数据库内多行/多表不变量 | 事务、锁、隔离级别、条件更新 |
| 重复请求或重试 | 幂等键、唯一约束、状态机 |
| 多实例争抢同一资源 | 数据库 CAS、租约、共识系统、经过证明的分布式锁 |
| 跨服务最终一致性 | outbox、幂等消费者、补偿、状态机 |

“加一个 Redis 分布式锁”不能自动解决：

- 锁过期后旧持有者继续执行；
- 网络分区；
- 进程暂停；
- fencing token 缺失；
- 数据库写入与锁状态不在同一原子域。

优先让最终资源本身执行条件更新或唯一约束；分布式锁需要租约、fencing token、容错模型和失败语义。

### 3.24 复杂度、编译器与实现边界

#### 3.24.1 复杂度不能只写一个 O(1)

| 原语 | 无竞争/常见路径 | 竞争或批量路径 | 主要隐藏成本 |
|---|---|---|---|
| Mutex | Lock/Unlock 当前快路径近似 O(1) | park、唤醒、调度；runtime semaphore 按地址查找等待队列 | cache line 转移、上下文调度、waiter 尾延迟 |
| RWMutex | RLock/RUnlock 原子 O(1)；写锁基本状态更新 O(1) | Unlock 可能按被阻塞读者数逐个唤醒，约 O(R) | 共享 readerCount 热点、写者屏障 |
| WaitGroup | Add/Done 常见 O(1) | counter 归零要唤醒 W 个 waiter，约 O(W) | 调度与误用导致的提前返回/永久等待 |
| Once | 完成后的 Load 快路径 O(1) | 首次调用与并发调用者在 Mutex 上等待 | 初始化函数本身耗时，panic 后不可重试 |
| Cond | Wait/Signal 常见队列操作近似 O(1) | Broadcast 至少要使全部 waiter 可运行，效果随 W 增长 | 惊群、重新竞争 Locker |
| Atomic | 单操作算法层面 O(1) | 高竞争 CAS 可能反复失败 | cache coherence、总线/缓存行争用 |
| `sync.Map` Hash-Trie | 哈希路径长度受哈希位数/分支因子约束 | 冲突链与并发写会增加成本；Range O(N) | 哈希、节点分配、局部锁、非快照语义 |

这些是算法与当前实现的近似分析，不是延迟上界。一次 O(1) 的原子操作在 64 核热点上可能比一个低竞争 Mutex 更慢。

#### 3.24.2 编译器做了什么

- `sync.Mutex.Lock` 的无竞争路径被设计为可内联，公开类型再转到 `internal/sync.Mutex`。
- `sync/atomic` 的底层操作会按目标架构降低为相应原子指令或运行时辅助；typed atomic 只是更安全的 API，不改变 Memory Model 的 SC 契约。
- `atomic.Int64`/`Uint64` 内的特殊对齐字段由编译器识别，以满足 32 位架构上的 64 位原子对齐要求。
- race 构建会对普通内存访问和同步操作加入检测器所需的插桩；这也是其显著时间与内存开销来源。
- `defer` 可能被编译器优化为 open-coded defer，但语义仍是函数返回/栈展开时执行；不能据此把循环内 defer 当作块级释放。
- `noCopy` 不受语言类型系统强制，它主要供 `go vet -copylocks` 等静态分析识别。

#### 3.24.3 哪些不能当永久保证

- Mutex 的状态位布局、自旋次数、1 ms 饥饿阈值；
- runtime semaphore 的树形结构与队列策略；
- RWMutex 的具体字段与唤醒循环；
- WaitGroup packed state 的位布局；
- `sync.Map` 的 Hash-Trie 节点宽度和内部锁粒度；
- 某架构上的具体原子指令与 cache line 大小。

可依赖的是公开 API 文档与 Memory Model 所写的同步关系。

## 五、常用场景

### 5.1 典型场景决策表

| 场景 | 推荐 | 不推荐 / 原因 |
|---|---|---|
| 请求总数、命中数 | `atomic.Uint64` | Mutex 也正确但可能更重；多个计数的一致快照另论 |
| 余额、库存等单值条件扣减 | CAS 循环或锁 | Load 后再 Add 存在业务竞态 |
| 账户余额 + 冻结额 + 版本 | Mutex/事务 | 多个 Atomic 不能保证组合不变量 |
| 启动后只读配置 | OnceValue 或 atomic immutable snapshot | 原子发布后继续改 slice/map 会 Race |
| 可热更新配置 | clone + `atomic.Pointer` | 锁外原地修改旧对象 |
| 任务等待 | `WaitGroup.Go` 或 Add-before-go | goroutine 内 Add |
| 需错误传播和取消的并发任务 | context + error aggregation | WaitGroup 不传错误，不自动取消 |
| 复杂条件队列 | `Cond` + 谓词循环 | 用 Sleep 轮询 |
| 简单一次性广播 | close(channel) | Cond 可能过重 |
| 普通业务 map | map + Mutex/RWMutex | 默认上 `sync.Map` 失去类型和跨字段不变量 |
| 独立 key 高并发 | 分片 map 或合适的 `sync.Map` | 单全局锁可能热点 |
| 对象临时复用 | `sync.Pool` | 存放必须保留的业务状态 |
| 事件流、所有权转移 | channel | 用锁模拟队列和唤醒容易复杂 |

### 5.2 一个正确的快照热更新模式

```go
package config

import (
    "slices"
    "sync/atomic"
    "time"
)

type Config struct {
    Timeout time.Duration
    Hosts   []string
}

var current atomic.Pointer[Config]

func init() {
    current.Store(&Config{Timeout: time.Second})
}

func Load() *Config {
    return current.Load() // 调用者只读
}

func Update(next Config) {
    snapshot := &Config{
        Timeout: next.Timeout,
        Hosts:   slices.Clone(next.Hosts),
    }
    current.Store(snapshot)
}
```

边界：如果 `Config` 内嵌 map、slice、pointer tree，必须深拷贝所有仍可能被修改的可达对象。浅拷贝结构体不等于不可变快照。

### 5.3 一个锁保护复合不变量的模式

```go
type Account struct {
    mu      sync.Mutex
    balance int64
    frozen  int64
}

func (a *Account) Freeze(n int64) bool {
    a.mu.Lock()
    defer a.mu.Unlock()

    if n <= 0 || a.balance < n {
        return false
    }
    a.balance -= n
    a.frozen += n
    return true
}
```

`balance + frozen` 的变换处于同一临界区，读者若要一致观察也必须使用同一锁。

### 5.4 Channel 与锁结合时的安全模式

不要持锁阻塞发送。可先在锁内生成不可变事件，再解锁发送：

```go
func (s *Service) Update(v int) {
    s.mu.Lock()
    s.value = v
    ev := Event{Value: v, Version: s.version + 1}
    s.version++
    s.mu.Unlock()

    select {
    case s.events <- ev:
    default:
        // 根据业务选择丢弃、落盘、计数或异步队列
    }
}
```

这仍需明确事件丢弃语义；不能为了避免阻塞就无条件 `default`，否则可能悄悄丢关键事件。

### 5.5 什么时候使用 RWMutex

先用 Mutex 写出正确版本，再做代表性基准：

```go
func BenchmarkReadMostly(b *testing.B) {
    // 控制：读写比例、临界区长度、key 分布、GOMAXPROCS
    b.RunParallel(func(pb *testing.PB) {
        for pb.Next() {
            // 执行真实读路径
        }
    })
}
```

同时看：

- ns/op 与吞吐；
- P95/P99 延迟；
- mutex/block profile；
- CPU profile 中 atomic/cache 相关热点；
- 不同 GOMAXPROCS 与读写比例。

只在“读锁可并行”的理论条件满足且实测有收益时采用。

## 六、代码陷阱题

> 以下题目中的死锁和 fatal 示例应在独立测试进程中验证，不要直接放进长期运行服务。对含 Data Race 的程序，标准答案首先是“程序错误”，而不是猜一个稳定输出。

### 题 1：`count++` 是否安全

```go
package main

import (
    "fmt"
    "sync"
)

func main() {
    var count int
    var wg sync.WaitGroup

    for g := 0; g < 2; g++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            for i := 0; i < 1000; i++ {
                count++
            }
        }()
    }
    wg.Wait()
    fmt.Println(count)
}
```

**先判断：**一定输出 2000 吗？是否有 Data Race？

**标准答案：**有 Data Race，结果不可靠；常见现象是丢更新，`go run -race` 会报告冲突。不能把某个观察到的数值范围当成语言保证。

**逐行分析：**

1. `count++` 至少包含读、加、写三个逻辑步骤。
2. 两个 goroutine 可同时读到同一个旧值，再各自写回同一个新值。
3. `WaitGroup` 只保证 main 等任务结束，不会自动保护 `count`。
4. 可改为 `atomic.Int64.Add(1)`，或用 Mutex 将整个更新置于临界区。

**追问方向：**Atomic 一定更快吗？多个计数需要一致快照怎么办？`WaitGroup.Wait` 建立了什么同步边？

### 题 2：锁内写、锁外读

```go
var (
    mu    sync.Mutex
    value int
)

func write() {
    mu.Lock()
    value = 42
    mu.Unlock()
}

func read() int {
    return value
}
```

**先判断：**写已经加锁，读不写，是否安全？

**标准答案：**不安全。若 `read` 与 `write` 并发，普通读和普通写无 happens-before，构成 Data Race。

**逐行分析：**

1. Mutex 只为遵守同一协议的访问建立同步。
2. `Unlock` 只同步到后续成功返回的 `Lock`，不会广播给所有普通读。
3. 读路径也应加同一把锁，或双方统一使用 atomic。

**追问方向：**一方 atomic Store、另一方普通读是否安全？只给写加 `RLock`/`Lock` 有何区别？

### 题 3：按值传参复制锁

```go
type Counter struct {
    mu sync.Mutex
    n  int
}

func snapshot(c Counter) int { // 按值复制
    c.mu.Lock()
    defer c.mu.Unlock()
    return c.n
}

func (c *Counter) Inc() {
    c.mu.Lock()
    c.n++
    c.mu.Unlock()
}
```

**先判断：**`snapshot` 加了锁，能否与 `Inc` 并发？

**标准答案：**不能。它锁的是复制出来的 Mutex，不是原对象的锁；复制本身若与原对象修改并发也可能发生 Race。

**逐行分析：**

1. `Counter` 按值传入时复制 `mu` 和 `n`。
2. `c.mu.Lock()` 只保护副本。
3. 已使用的锁不得复制，`go vet -copylocks` 通常能提示。
4. 应改为 `func snapshot(c *Counter) int` 或方法指针接收者。

**追问方向：**复制一个从未使用的零值锁是否语言层面禁止？为什么“第一次使用后不得复制”仍建议从设计上始终避免复制？

### 题 4：value receiver 隐式复制 Mutex

```go
type Registry struct {
    mu sync.Mutex
    m  map[string]int
}

func (r Registry) Put(k string, v int) { // value receiver
    r.mu.Lock()
    defer r.mu.Unlock()
    r.m[k] = v
}
```

**先判断：**方法内部有锁，多个 goroutine 调 `Put` 是否安全？

**标准答案：**不安全。每次调用复制一把 Mutex，但 map header 仍指向同一底层 map；不同副本锁无法互斥，可能 Data Race 或触发 `fatal error: concurrent map writes`。

**逐行分析：**

1. value receiver 复制整个 `Registry`。
2. map 是描述符，复制后仍共享底层 map。
3. 每次调用锁住不同 `r.mu` 副本。
4. 改成 `func (r *Registry) Put(...)`。

**追问方向：**如果字段是纯值 `int`，value receiver 会出现什么现象？为什么 copylocks 与逻辑 bug 可能同时存在？

### 题 5：嵌入 Mutex 后复制 struct

```go
type Cache struct {
    sync.Mutex
    data map[string]string

    return c
}
```

**先判断：**只是“复制一个缓存对象”，有什么风险？

**标准答案：**复制了已使用的嵌入 Mutex，同时两个 struct 共享同一 map。两个副本看起来都能 `Lock`，实际互不协调。

**逐行分析：**

1. 匿名嵌入不改变复制语义。
2. 提升出来的 `Lock` 方法容易让调用者误以为副本仍保护原数据。

### 题 6：双重检查初始化

```go
var (
    mu  sync.Mutex
    cfg *Config
)

func Get() *Config {
    if cfg == nil {
        mu.Lock()
        if cfg == nil {
            cfg = loadConfig()
        }
        mu.Unlock()
    }
    return cfg
}
```

**先判断：**这是经典 DCL，Go 中正确吗？

**标准答案：**错误。外层普通读取 `cfg` 与锁内写入并发，构成 Data Race；读到非 nil 也不能证明对象字段已安全发布。

**逐行分析：**

1. 外层 `cfg == nil` 没有同步。
2. 内层锁只保护初始化写者之间的互斥。
3. 返回处普通读取仍可能与写并发。
4. 一次初始化用 `sync.Once`/`OnceValue`；热更新用 immutable snapshot + atomic pointer。

**追问方向：**如何用 `atomic.Pointer` 写正确 DCL？为什么正确也通常不值得？

### 题 7：RWMutex 锁升级

```go
rw.RLock()
if needWrite() {
    rw.Lock()
    update()
    rw.Unlock()
}
rw.RUnlock()
```

**先判断：**会发生什么？

**标准答案：**可能永久阻塞。写锁等待所有读者退出，其中包括当前 goroutine 持有的读锁。

**逐行分析：**

1. `RLock` 增加读者计数。
2. `Lock` 宣告写者并等待旧读者清空。
3. 当前 goroutine 无法执行到 `RUnlock`，形成自我依赖。
4. 应先 `RUnlock`，再 `Lock`，并在写锁内重新检查条件。

**追问方向：**重新检查为什么必须？能否设计版本号避免重复计算？

### 题 8：递归读锁遇到写者

```go
rw.RLock()

go func() {
    rw.Lock()
    defer rw.Unlock()
}()

// 假设写者已经进入等待状态
rw.RLock()
rw.RUnlock()
rw.RUnlock()
```

**先判断：**读锁不是共享的吗，第二次 `RLock` 为什么可能卡住？

**标准答案：**`RWMutex` 有 writer preference。写者 pending 后，新读者会阻塞；当前 goroutine 的第二次 RLock 等写者，写者又等它释放第一次 RLock，形成死锁。

**逐行分析：**

1. 第一次 RLock 成为旧读者。
2. 写者减 `readerCount` 宣告 pending。
3. 第二次 RLock 看到负计数，等待 `readerSem`。
4. 写者等待第一次读锁释放，但当前 goroutine已阻塞。

**追问方向：**为什么文档禁止递归读锁？怎样用调用层次避免隐藏的重复 RLock？

### 题 9：在 goroutine 内 Add

```go
var wg sync.WaitGroup

for i := 0; i < 10; i++ {
    go func() {
        wg.Add(1)
        defer wg.Done()
        work()
    }()
}
wg.Wait()
fmt.Println("all done")
```

**先判断：**`all done` 是否一定最后打印？

**标准答案：**不一定。`Wait` 可在任何 Add 前看到 counter 为 0 并直接返回。Go 1.25+ 的 `go vet` 可报告这一常见模式。

**逐行分析：**

1. 创建 goroutine 不保证它立即运行。
2. main 可直接执行 `Wait`。
3. counter 尚为 0 时 `Wait` 返回。
4. 正确做法是在 `go` 语句前 Add，或使用 `wg.Go(work)`。

**追问方向：**正增量 Add 与 Wait 何时可以并发？`WaitGroup.Go` 是否处理错误和 panic？

### 题 10：Done 次数过多

### 题 11：Wait 与新一轮 Add 并发

```go
var wg sync.WaitGroup

wg.Add(1)
go func() { wg.Done() }()
wg.Wait()

// 两个 goroutine 无协议地开始“复用”
go wg.Wait()
wg.Add(1)
go func() { defer wg.Done(); work() }()
```

**先判断：**既然上一轮已经 Wait 完，后面为何仍有问题？

**标准答案：**新一轮 `Add(1)` 与一个在 counter 为 0 时开始的 `Wait` 并发，违反复用规则；Wait 可能提前返回，实现也可能检测到误用并 panic。

**逐行分析：**

1. WaitGroup 可复用，但新一轮必须有明确阶段边界。
2. 所有上一轮 Wait 返回后，才能开始新一轮 Add/Go。
3. 示例又启动了一个无归属的 Wait，与 Add 竞争。
4. 应由单一协调者推进 generation，或每轮使用独立 WaitGroup。

**追问方向：**为什么不能依赖 WaitGroup 的误用 panic？如何设计可循环 barrier？

### 题 12：`Once.Do` 中 panic

```go
var once sync.Once

func call(label string) {
    defer func() {
        fmt.Println(label, "recover:", recover())
    }()
    once.Do(func() {
        fmt.Println("init")
        panic("boom")
    })
    fmt.Println(label, "returned")
}

func main() {
    call("first")
    call("second")
}
```

**先判断：**第二次会再次打印 `init` 并 panic 吗？

**标准答案：**不会。第一次打印 `init` 后 panic；Once 把这次调用视为已完成。第二次 `Do` 不执行函数，继续打印 `second returned`，其 defer 中 `recover()` 得到 nil。

**逐行分析：**

1. `doSlow` 使用 defer 在 `f` 返回或 panic 展开时设置 done。
2. `Once` 不提供失败重试。
3. `OnceFunc`/`OnceValue` 的包装语义不同：首次 panic 后，后续调用会持续以同一值 panic。

**追问方向：**如何设计可重试初始化？初始化失败结果是否应缓存？

### 题 13：Cond 用 `if` 而不是 `for`

```go
type Queue struct {
    mu   sync.Mutex
    cond *sync.Cond
    n    int
}

func (q *Queue) Take() {
    q.mu.Lock()
    if q.n == 0 {
        q.cond.Wait()
    }
    q.n--
    q.mu.Unlock()
}
```

**先判断：**只要生产者 Signal，`n` 就一定大于 0 吗？

**标准答案：**不一定。另一个 waiter 或竞争者可在当前 goroutine 重新获得锁前消费资源，导致 `n` 已回到 0；此时继续 `n--` 破坏状态。

**逐行分析：**

1. Wait 被唤醒后还要重新竞争 `q.mu`。
2. 通知不把资源保留给某个 waiter。
3. 应使用 `for q.n == 0 { q.cond.Wait() }`。
4. 谓词读取与修改都要在同一锁下。

**追问方向：**Go Cond 是否存在“无通知伪唤醒”？为什么即便没有也必须循环？

### 题 14：未持锁调用 Cond.Wait

```go
mu := sync.Mutex{}
cond := sync.NewCond(&mu)
cond.Wait()
```

**先判断：**Wait 会自己先 Lock 吗？

**标准答案：**不会。调用者必须已持有 `cond.L`。`Wait` 首先尝试解锁，配合普通 Mutex 时会触发未锁定解锁的 runtime fatal。

**逐行分析：**

1. Wait 的原子动作是“登记 waiter + Unlock + sleep”。
2. 它不是“先 Lock 再等待”的便利函数。
3. 正确模式是 Lock → for predicate false → Wait → 使用状态 → Unlock。

### 题 15：多个 atomic 字段的一致性

```go
type Pair struct {
    x atomic.Int64
    y atomic.Int64
}

func (p *Pair) SetBoth(v int64) {
    p.x.Store(v)
    p.y.Store(v)
}

func (p *Pair) LoadBoth() (int64, int64) {
    return p.x.Load(), p.y.Load()
}
```

**先判断：**`LoadBoth` 是否只可能返回 `(0,0)` 或 `(v,v)`？

**标准答案：**不是。可返回 `(v,0)`，也可能在后续多次更新时读到不同 generation 的组合。每个字段原子，不代表二元组原子。

**逐行分析：**

1. 两次 Store 是两个独立同步操作。
2. 两次 Load 也是两个独立观察点。
3. SC 只给所有 atomic 操作一个总序，不会把相邻两次操作合并成事务。
4. 用 Mutex、单字打包或 immutable `PairSnapshot` 指针。

**追问方向：**seqlock 风格版本校验如何工作？它在 Go 中有哪些读重试和饥饿边界？

### 题 16：Atomic 发布后继续修改对象

```go
type Config struct {
    Rules map[string]bool
}

var current atomic.Pointer[Config]

func publish() {
    c := &Config{Rules: map[string]bool{"a": true}}
    current.Store(c)
    c.Rules["b"] = true
}

func read() bool {
    return current.Load().Rules["b"]
}
```

**先判断：**指针是 atomic 的，map 访问是否安全？

**逐行分析：**

1. `current.Store(c)` 后，其他 goroutine 可立即取得 `c`。

3. 后续普通写没有同步。
4. 发布前完成深拷贝，发布后保持整个可达对象图不可变。

### 题 17：map 加锁，但 value 指向对象未加锁

```go
type User struct {
    Name string
}

type Store struct {
    mu sync.RWMutex
    m  map[int]*User

func (s *Store) Get(id int) *User {
```

### 题 18：`defer Unlock` 放在循环里

```go
func updateAll(items []int) {
    for _, v := range items {
        mu.Lock()
        defer mu.Unlock()
        update(v)
    }
}
```

**先判断：**每轮结束时会解锁吗？

```go
func updateOne(v int) {
```

### 题 19：持锁发送 Channel

### 题 20：锁顺序反转

```go
func transferAB() {
```

### 题 21：TryLock 忙循环

### 题 22：用 Sleep 修 Race

```go
var result int

go func() { result = compute() }()

```

**先判断：**线上计算通常 1 ms，Sleep 100 ms 是否足够安全？

**标准答案：**不安全，仍是 Data Race。Sleep 不建立任何 happens-before；负载、停顿、调度和机器差异都可改变时序。

**逐行分析：**

1. goroutine 写 `result`。
2. main 普通读 `result`。
3. 中间只有计时器等待，没有同步边。
4. 使用 channel 接收结果、WaitGroup + 同步读，或受锁状态。

**追问方向：**为什么 `GOMAXPROCS=1` 也不能让它合法？测试里的 `Eventually` 与同步有什么区别？

### 题 23：共享底层数组上的 append

```go
base := make([]int, 0, 2)
a := base
b := base

var wg sync.WaitGroup
wg.Add(2)
go func() {
    defer wg.Done()
    a = append(a, 1)
}()
go func() {
    defer wg.Done()
    b = append(b, 2)
}()
wg.Wait()
```

**先判断：**`a`、`b` 是不同 slice 变量，是否安全？

**标准答案：**不安全。二者初始 len/cap 相同并共享底层数组，第一次 append 都可能写下标 0，发生写写 Race。

**逐行分析：**

1. slice descriptor 按值复制，不会复制 backing array。
2. cap 足够时 append 不分配新数组。
3. 两个 goroutine 修改各自 header，但写同一底层元素。
4. 使用独立 clone、预先划分不重叠索引，或通过锁/单一 owner 追加。

**追问方向：**两个 goroutine 写同一 slice 的不同固定索引何时安全？append 扩容后为何仍不能靠“可能分配”保证安全？

### 题 24：并发 close Channel

```go
ch := make(chan struct{})

var wg sync.WaitGroup
wg.Add(2)
for i := 0; i < 2; i++ {
    go func() {
        defer wg.Done()
        close(ch)
    }()
}
wg.Wait()
```

**先判断：**两次 close 会被合并吗？Race Detector 一定报告 Data Race 吗？

**标准答案：**不会合并；恰好一个 close 成功，另一个触发 `panic: close of closed channel`。这首先是所有权/生命周期 Race Condition，不应依赖 Race Detector 报告普通内存 Race。

**逐行分析：**

1. channel 内部同步保证操作安全执行，但 API 语义禁止重复 close。
2. close 应由能证明“再无发送者”的单一 owner 执行。
3. 真正共享的一次性关闭可用 `sync.Once`，但更优先梳理关闭所有权。

**追问方向：**发送者还是接收者负责关闭？send 与 close 并发为何危险？

### 题 25：Race Detector 测不到未执行路径

```go
func maybeRacy() {
    if os.Getenv("ENABLE_RACY_PATH") != "1" {
        return
    }
    go func() { shared++ }()
    shared++
}

func TestDefault(t *testing.T) {
    maybeRacy()
}
```

**先判断：**`go test -race` 没报告，能否证明 `maybeRacy` 安全？

**标准答案：**不能。Race Detector 是动态检测器，只能分析实际执行到的已插桩访问。默认环境未进入分支，冲突根本未发生。

**逐行分析：**

1. `-race` 不做所有路径的静态证明。
2. 要提高单测、集成、压力、故障注入与真实负载覆盖。
3. 可对 race 构建的二进制运行代表性工作负载。
4. 无报告只表示“本次执行未检测到”，不表示不存在。

**追问方向：**Race build 有多大开销？如何读取冲突栈与 goroutine 创建栈？

### 题 26：`atomic.Value` 混用具体类型

```go
var v atomic.Value
v.Store(int64(1))
v.Store(int(2))
```

**先判断：**二者都是整数，第二次 Store 是否允许？

**标准答案：**不允许，会 panic。`int64` 与 `int` 是不同具体类型；一个 `atomic.Value` 的所有 Store 必须保持同一具体类型。

**逐行分析：**

1. 第一次 Store 固定 Value 的 concrete type。
2. 后续不同 concrete type 被实现检测并 panic。
3. Store nil 也会 panic。
4. 可定义统一 snapshot struct 或使用 typed atomic。

**追问方向：**零值 Load 返回什么？Value 与 Pointer[T] 如何选？

### 题 27：把 `sync.Map.Range` 当快照

```go
var m sync.Map
m.Store("a", 1)

go func() {
    m.Store("a", 2)
    m.Store("b", 3)
}()

m.Range(func(k, v any) bool {
    fmt.Println(k, v)
    return true
})
```

**先判断：**Range 是否对应调用开始或结束时的完整快照？

**标准答案：**不保证。每个 key 最多访问一次，但并发 Store/Delete 时，不同 key 可能反映 Range 期间不同时间点的状态。

**逐行分析：**

1. `sync.Map` 设计为并发访问，不为 Range 停止所有写入。
2. 当前 Go 1.26 Hash-Trie 读路径同样不提供事务快照。
3. 若需要一致快照，应在更高层加锁、复制，或使用版本化 immutable snapshot。

**追问方向：**Go 1.26 `sync.Map` 与旧 `read/dirty` 实现有何变化？API 保证为何未随实现改变？

### 题 28：send 与 close 并发

```go
c := make(chan struct{}, 1)

go func() {
    c <- struct{}{}
}()

close(c)
```

**先判断：**close 和 send 都是 channel 操作，runtime 会自动排出安全顺序吗？

**标准答案：**不能依赖。send 与 close 之间没有建立顺序，Race Detector 可报告竞争，且 send 可能因向已关闭 channel 发送而 panic。

**逐行分析：**

1. 启动 goroutine 只建立 main 中 `go` 语句到发送 goroutine 开始的边，未建立 send 完成到 close 的边。
2. main 可在 send 前、send 中或 send 后执行 close。
3. 应由明确 owner 在确认所有发送者完成后 close，例如先 WaitGroup.Wait，再由协调者 close。
4. 若只是等待一次 send 完成，可先 receive 对应值，再 close；具体协议必须与业务一致。

**追问方向：**为什么“只有接收者 close”也不是绝对规则？多发送者场景如何证明再无 send？

## 七、面试高频问题

### 7.1 什么是 Data Race？

- **普通回答：**多个 goroutine 并发访问同一变量，至少一个写，且没有正确同步。
- **中高级回答：**严格说，是同一内存位置上的读写或写写操作在 happens-before 上无序，且至少一个是非同步访问。全 atomic 访问不构成普通 Data Race。
- **继续深挖：**说明 read-write/write-write race、happens-before、DRF-SC，以及多机器字结构 Race 可能导致不一致内部表示。
- **常见错误：**“同时写才叫 Race”“只读不加锁永远安全”“int 写是原子的所以没 Race”。
- **版本/边界：**Memory Model 页面当前仍标注 2022-06-06；定义是语言模型，不依赖某个 runtime 版本。

### 7.2 Race Condition 与 Data Race 有何区别？

- **普通回答：**Data Race 是内存访问未同步；Race Condition 是结果依赖时序的逻辑错误。
- **中高级回答：**Data Race 是可被内存模型精确定义的一类错误；Race Condition 更广，atomic check-then-act、重复 close、业务幂等失败都可能无 Data Race 但有竞态。
- **继续深挖：**用库存 `Load` 后 `Add(-1)` 举例，再扩展到数据库条件更新和唯一约束。
- **常见错误：**“`-race` 没报就没有并发问题”。
- **版本/边界：**Race Detector 只检测动态执行中的 Data Race，不验证业务不变量。

### 7.3 sequenced-before、synchronized-before、happens-before 是什么？

- **普通回答：**前者是 goroutine 内顺序，第二个是同步操作关系，HB 是两者并集的传递闭包。
- **中高级回答：**用“普通写 → channel send → 对应 receive → 普通读”的传递链说明可见性。
- **继续深挖：**解释同步读观察同步写产生 synchronized-before，以及普通读的可见写要求。
- **常见错误：**把 wall-clock 顺序、日志顺序或调度顺序当 HB。
- **版本/边界：**属于 Memory Model 保证，不是当前 CPU 的偶然行为。

### 7.4 什么是 DRF-SC？

- **普通回答：**无 Data Race 的 Go 程序可以按顺序一致的 goroutine 交错执行来理解。
- **中高级回答：**它是可推理性保证，不等于物理单核或没有编译器优化。
- **继续深挖：**说明有 Race 后不应继续依赖 SC 推理；Go 对小字长 Race 有有限实现约束，但程序仍错误。
- **常见错误：**“Go 所有并发程序都是顺序一致的”。
- **版本/边界：**只对 data-race-free 程序成立。

### 7.5 Goroutine 和 Channel 建立哪些同步边？

- **普通回答：**`go` 语句同步到新 goroutine 开始；send 同步到对应 receive；close 同步到因关闭返回的 receive。
- **中高级回答：**补充 goroutine 退出无自动 join；无缓冲 receive 同步到对应 send 完成；缓冲 channel 有第 k receive 到第 k+C send 的规则。
- **继续深挖：**解释为什么 buffered send 返回不代表消费者已处理。
- **常见错误：**“goroutine 结束后 main 自然能看到结果”“close 与所有此前/此后任意操作都全局同步”。
- **版本/边界：**Memory Model 契约稳定；具体调度顺序不保证。

### 7.6 为什么 Sleep 不能同步？

- **普通回答：**Sleep 只延迟当前 goroutine，不创建 HB。
- **中高级回答：**它不提供可见性、有序性或完成保证，负载与调度改变就会失效。
- **继续深挖：**说明 `GOMAXPROCS=1`、日志先后也不建立同步；应使用 channel、锁、atomic、WaitGroup。
- **常见错误：**“睡得足够久就行”。
- **版本/边界：**与版本无关，是语义问题。

### 7.7 Mutex 有哪些公开语义？

- **普通回答：**零值可用、互斥、使用后不可复制、不可重入。
- **中高级回答：**不绑定 goroutine owner；Unlock 可由另一 goroutine 调用；Unlock 未锁锁是运行时错误；失败 TryLock 无同步效果。
- **继续深挖：**解释 noCopy/vet 与实际复制风险，Unlock→后续 Lock 的 Memory Model 边。
- **常见错误：**“Go Mutex 会记录 goroutine ID”“锁可以递归”。
- **版本/边界：**公开文档是契约；内部字段不是。

### 7.8 Mutex 如何实现？

- **普通回答：**无竞争 CAS 快路径，竞争时进入慢路径并可能休眠。
- **中高级回答：**`state` 编码 locked/woken/starving/waiter count；有限自旋；runtime semaphore park/wakeup；正常与饥饿模式。
- **继续深挖：**讲 `lockSlow`、`unlockSlow`、防重复唤醒、队首 handoff 与调度成本。
- **常见错误：**“Mutex 就是一个 channel”“竞争后一直自旋”。
- **版本/边界：**Go 1.26.4 路径为 `src/internal/sync/mutex.go`；字段和启发式可变。

### 7.9 Mutex 为什么有饥饿模式？

- **普通回答：**避免旧 waiter 被新 goroutine 持续抢锁而长期饿死。
- **中高级回答：**正常模式允许竞争，吞吐高；等待超过当前 1 ms 阈值可转 starvation，Unlock 直接 handoff。
- **继续深挖：**讲退出饥饿条件和公平性/吞吐/尾延迟折中。
- **常见错误：**“Mutex 永远严格 FIFO”“1 ms 是语言规范”。
- **版本/边界：**1 ms 是 Go 1.26.4 当前实现常量。

### 7.10 TryLock 什么时候用？

- **普通回答：**只有拿不到锁时能安全走替代路径，才考虑使用。
- **中高级回答：**失败无 HB，不可据此读取受保护状态；忙循环会浪费 CPU并恶化公平性。
- **继续深挖：**讨论诊断路径、opportunistic work、带超时锁为何通常需要重新设计而不是轮询 TryLock。
- **常见错误：**“TryLock 循环比 Lock 快”。
- **版本/边界：**TryLock 已是标准库 API，但官方仍提示正确场景少见。

### 7.11 RWMutex 的五个字段如何协作？

- **普通回答：**`w` 串行写者，`readerCount` 统计/标记读者，两个 semaphore 分别唤醒读写方。
- **中高级回答：**写者先 `w.Lock`，再减 maxReaders 阻止新读者；`readerWait` 追踪旧读者，最后一个 RUnlock 释放 writerSem；写者 Unlock 释放 readerSem。
- **继续深挖：**解释负 readerCount、writer preference 和 race detector 精确同步注解。
- **常见错误：**“RWMutex 只是两个 Mutex”“读锁完全无写共享”。
- **版本/边界：**字段是 Go 1.26.4 实现，API 只承诺读写互斥与同步关系。

### 7.12 为什么 RWMutex 不一定比 Mutex 快？

- **普通回答：**RWMutex 本身更复杂，读写比例和临界区长度不合适时开销更大。
- **中高级回答：**读锁需共享原子计数，产生 cache line contention；写者协调与唤醒也有成本。
- **继续深挖：**要求用读写比例、GOMAXPROCS、尾延迟、profile 做基准，而非只比平均 ns/op。
- **常见错误：**“读多就无条件用 RWMutex”。
- **版本/边界：**性能结论与硬件、runtime 和工作负载相关。

### 7.13 WaitGroup 的正确规则是什么？

- **普通回答：**Add 后启动任务，任务 Done，协调者 Wait；不得复制。
- **中高级回答：**counter 从 0 做正 Add 必须在对应 Wait 前；复用时新一轮 Add 在上一轮所有 Wait 返回后；多个 Wait 可同时存在。
- **继续深挖：**讲 packed state、waiter count、semaphore、负 counter 和误用检测。
- **常见错误：**“所有 Add 都绝不能与 Wait 并发”“WaitGroup 会收集错误”。
- **版本/边界：**Go 1.25+ 有 `WaitGroup.Go` 和 vet 的 waitgroup 检查。

### 7.14 `WaitGroup.Go` 有什么限制？

- **普通回答：**自动 Add、启动 goroutine，并在任务完成时减少计数。
- **中高级回答：**`f` 不得 panic；空组时 Go 必须先于 Wait，非空组可由任务继续派生；不提供结果、错误、取消。
- **继续深挖：**Go 1.26.4 当前 wrapper 对 panic 重新抛出且不先 Done，避免 Wait 与致命 panic 竞速。
- **常见错误：**“等价于无条件 `defer wg.Done()`，panic 也完全一样”。
- **版本/边界：**API 自 Go 1.25 加入；旧版本需手写 Add/go/Done。

### 7.15 Once 为什么不只用一个 CAS？

- **普通回答：**CAS 输家必须等赢家执行完初始化，不能直接返回。
- **中高级回答：**atomic done 是快路径，Mutex 保证唯一执行和等待完成，done 在 `f` 返回后才发布。
- **继续深挖：**解释 Once 的 HB 保证、panic 仍标记 done、递归 Do 死锁。
- **常见错误：**“CAS 成功者执行，其他人返回就够了”。
- **版本/边界：**当前字段布局是实现优化；Do 的完成保证是 API 契约。

### 7.16 Cond 为什么必须循环检查条件？

- **普通回答：**被唤醒后条件可能已被别的 goroutine 改回去或消费。
- **中高级回答：**Wait 解锁休眠、醒后重新加锁；通知不是资源所有权，等待对象是谓词。
- **继续深挖：**比较 Signal/Broadcast、channel close/send，以及 missed condition 的锁协议。
- **常见错误：**“Signal 后条件一定成立”“Wait 会自动先加锁”。
- **版本/边界：**Go Cond 文档说 Wait 不会无故返回，但仍要求循环。

### 7.17 Go Atomic 的内存序是什么？

- **普通回答：**所有 atomic 操作表现为顺序一致。
- **中高级回答：**若 A 的效果被 B 观察，A synchronized-before B；全局存在与值一致的 atomic 总序。
- **继续深挖：**说明没有公开 relaxed/acquire/release 参数；SC 仍不把多个 atomic 合成事务。
- **常见错误：**“Load 是 relaxed，Store 是 release”“atomic 只保证不撕裂，不保证顺序”。
- **版本/边界：**以 Go Memory Model 和 `sync/atomic` 文档为准。

### 7.18 Atomic 适合哪些状态？

- **普通回答：**独立计数、标志、状态值、指针发布。
- **中高级回答：**状态必须能在一个原子对象上表达；CAS 循环要有可证明不变量与可接受的竞争。
- **继续深挖：**多字段使用锁、打包状态或 immutable snapshot；讨论 ABA 和高竞争 CAS 重试。
- **常见错误：**“Atomic 是无锁，所以一定更快且可替代 Mutex”。
- **版本/边界：**typed atomics 优先；老函数仍存在但更易错。

### 7.19 `atomic.Pointer` 与 `atomic.Value` 怎么选？

- **普通回答：**已知 `*T` 用 Pointer；同一具体类型的任意值可用 Value。
- **中高级回答：**Pointer 类型安全且无断言；Value 禁止 nil、禁止混用 concrete type，适合整份值快照。
- **继续深挖：**两者都只安全发布引用/值；发布后可达对象必须不可变或另行同步。
- **常见错误：**“原子指针使对象所有字段自动线程安全”。
- **版本/边界：**typed `Pointer[T]` 依赖泛型版本；当前实现细节不应外泄为契约。

### 7.20 什么是 ABA 和 false sharing？

- **普通回答：**ABA 是值变过又变回，CAS 误判未变化；false sharing 是独立变量共享 cache line 导致相互失效。
- **中高级回答：**ABA 需版本 tag/安全回收；false sharing 需分片、本地聚合或布局调整。
- **继续深挖：**Go GC 不自动消除逻辑 ABA；atomic 热点仍有缓存一致性瓶颈。
- **常见错误：**“无锁就没有竞争”“不同字段绝不互相影响”。
- **版本/边界：**cache line 大小与架构相关，不要把某个填充常量当语言保证。

### 7.21 map + lock、分片锁和 sync.Map 怎么选？

- **普通回答：**多数代码用普通 map + lock；独立 key 高并发可分片；特定模式可用 sync.Map。
- **中高级回答：**考虑类型安全、跨字段不变量、Range 快照、热 key、分片跨锁操作。
- **继续深挖：**Go 1.26 `sync.Map` 当前是 Hash-Trie：atomic child 读、局部节点锁写；Range 仍非快照。
- **常见错误：**“sync.Map 永远更快”“当前仍固定是 read/dirty/misses”。
- **版本/边界：**旧双 map 实现属于 Go 1.25 及以前默认路径；API 适用场景比实现更稳定。

### 7.22 Channel 与 Mutex 怎么选？

- **普通回答：**共享状态不变量用锁，消息/所有权转移/背压用 channel。
- **中高级回答：**看问题模型，不做宗教式选择；channel 本身也可能死锁、泄漏和争用。
- **继续深挖：**讨论 actor 单 owner、锁内发送、缓冲容量、关闭所有权和 cancellation。
- **常见错误：**“Go 中用 Mutex 就不地道”“channel 一定比锁慢/快”。
- **版本/边界：**属于设计取舍，不是规范强制。

### 7.23 如何系统避免死锁？

- **普通回答：**统一锁顺序、缩短临界区、避免持锁阻塞、减少嵌套锁。
- **中高级回答：**从 Coffman 条件入手，破坏循环等待；明确 owner、超时和取消；对动态锁集排序。
- **继续深挖：**用 goroutine dump、block profile、trace 找等待链；注意局部死锁不一定触发 runtime fatal。
- **常见错误：**“所有 Lock 都 defer Unlock 就不会死锁”。
- **版本/边界：**runtime 只能检测部分全局停滞。

### 7.24 Race Detector 能做什么、不能做什么？

- **普通回答：**`go test -race ./...` 动态发现实际执行中的 Data Race。
- **中高级回答：**报告冲突访问栈和 goroutine 创建栈；未执行路径、业务竞态、死锁不会自动发现。
- **继续深挖：**讲 GORACE、真实负载、平台限制、5–10 倍内存和 2–20 倍执行时间的典型开销范围。
- **常见错误：**“无报告即证明线程安全”“race build 只慢一点点”。
- **版本/边界：**具体支持平台与开销以当前官方文档为准。

### 7.25 Race Detector 与 concurrent map fatal 有何区别？

- **普通回答：**前者是插桩动态竞态检测；后者是 runtime 对普通 map 某些非法并发操作的内部检查。
- **中高级回答：**map fatal 不是完整 Data Race 检测器，不能覆盖普通变量或所有 map 竞态，也不应依赖其必然触发。
- **继续深挖：**fatal 通常直接终止进程；Race 报告给出访问与创建栈，适用于更广内存位置。
- **常见错误：**“没出现 concurrent map writes 就说明 map 安全”。
- **版本/边界：**runtime 检测属于实现防护，不是正确性协议。

### 7.26 mutex profile、block profile、trace 各看什么？

- **普通回答：**mutex 看锁竞争，block 看阻塞点，trace 看调度与延迟时间线。
- **中高级回答：**block 栈对应发生阻塞的位置；mutex 栈对应造成竞争的临界区结束/Unlock 位置；trace 适合看并行度、park/unpark、syscall、GC。
- **继续深挖：**说明采样率、生产开销、短窗口、结合 `/sync/mutex/wait/total:seconds` 判断全局趋势。
- **常见错误：**“mutex profile 顶部一定是 Lock 调用处”“trace 是 CPU hotspot 的首选工具”。
- **版本/边界：**Go 1.26 pprof 文档明确区分两类栈归因；profile 默认需启用采样。

### 7.27 进程内锁、数据库事务和分布式锁的边界是什么？

- **普通回答：**Mutex 只管单进程；跨进程不变量要由共享存储或分布式协议保证。
- **中高级回答：**优先数据库条件更新、事务、唯一约束、幂等键；分布式锁要处理租约过期和 fencing token。
- **继续深挖：**讨论旧 owner 在租约过期后继续写、网络分区、事务与锁非原子的问题。
- **常见错误：**“加 Redis 锁就全局 exactly-once”。
- **版本/边界：**这是系统模型边界，不由 Go runtime 提供。

## 八、深挖追问链

### 追问链 1：从 Data Race 深挖到 Memory Model

1. **什么是 Data Race？**  
   要点：同一内存位置的并发读写/写写，至少一个非同步访问，HB 无序。
2. **“并发”如何定义，必须物理同时吗？**  
   要点：不要求纳秒级同时；只要两操作在 happens-before 上无序，就可能并发冲突。
3. **happens-before 从哪里来？**  
   要点：sequenced-before 与 synchronized-before 的并集传递闭包。
4. **锁为什么能让普通字段可见？**  
   要点：写 sequenced-before Unlock，Unlock synchronized-before 后续 Lock 返回，Lock sequenced-before 读。
5. **Atomic 和 Mutex 的同步语义一样吗？**  
   要点：都可建立同步，但 atomic 是单对象低层操作且全局 SC；Mutex 表达临界区与多字段不变量。
6. **没有 Data Race 是否就没有 Race Condition？**  
   要点：不是；atomic check-then-act、重复请求、状态机跳转仍可能逻辑竞态。
7. **为什么 DRF-SC 很重要？**  
   要点：使无 Race 程序可用顺序一致交错推理，避免依赖硬件重排细节。
8. **`-race` 没报能否证明 DRF？**  
   要点：不能；动态覆盖有限，只能说明该执行未检测到。

### 追问链 2：从 Mutex API 深挖到 runtime semaphore

1. **Mutex 的零值能用吗？**  
   要点：能，零值未锁定。
2. **为什么使用后不能复制？**  
   要点：状态与 semaphore 地址被复制，waiter 与解锁不再对应。
3. **无竞争 Lock 的成本是什么？**  
   要点：当前实现主要是一次内联 CAS 快路径。
4. **CAS 失败后为何不立即休眠？**  
   要点：短临界区下有限自旋可避免 park/unpark 和调度成本。
5. **为什么不能无限自旋？**  
   要点：浪费 CPU、争抢锁持有者的执行机会；runtime 用启发式限制。
6. **正常模式为什么可能不公平？**  
   要点：被唤醒 waiter 还需与已在运行的新 goroutine 竞争。
7. **饥饿模式如何改善？**  
   要点：当前等待超过约 1 ms 可切换，Unlock 直接 handoff 给队首 waiter。
8. **runtime semaphore 如何防止丢失唤醒？**  
   要点：增加 waiter 计数后再检查票据，随后才入队休眠；release 先增加票据再看 waiter。
9. **生产如何找到锁持有者？**  
   要点：mutex profile 的栈归因到造成竞争的临界区结束/Unlock 位置，再结合源码和 trace。

### 追问链 3：从 RWMutex 深挖到性能反直觉

1. **RWMutex 比 Mutex 多了什么能力？**  
   要点：多个读者可并发，写者独占。
2. **如何阻止新读者让写者饿死？**  
   要点：写者把 readerCount 减去大常数，使后续 RLock 阻塞。
3. **旧读者如何唤醒写者？**  
   要点：readerWait 记录剩余旧读者，最后一个 RUnlock 释放 writerSem。
4. **为什么不能升级读锁？**  
   要点：写者等待包含自己在内的读者退出，产生自我依赖。
5. **为什么递归 RLock 也危险？**  
   要点：pending writer 阻塞第二次 RLock；写者又等第一次 RUnlock。
6. **读占 95% 就一定更快吗？**  
   要点：未必；读临界区太短、共享原子计数、核心数与 cache line contention 都会影响。
7. **如何正确 benchmark？**  
   要点：真实读写比例、临界区工作、key 分布、GOMAXPROCS、P99，并结合 profiles。
8. **读多写少还有什么替代？**  
   要点：immutable snapshot、分片、per-key ownership、sync.Map 特定模式。

### 追问链 4：从 Atomic 深挖到无锁设计边界

1. **`atomic.Add` 为什么没有 Data Race？**  
   要点：访问由 atomic API 完成，属于同步操作。
2. **Go atomic 是什么内存序？**  
   要点：顺序一致；没有公开 relaxed 参数。
3. **两个 atomic 字段为何仍不一致？**  
   要点：每个操作独立线性化，没有跨字段事务。
4. **如何发布多字段状态？**  
   要点：锁，或构造不可变 snapshot 后用一个 atomic pointer 发布。
5. **只保证顶层 struct 不改够吗？**  
   要点：不够；其 map/slice/pointer 可达对象也必须不可变或同步。
6. **CAS 循环有什么风险？**  
   要点：高竞争重试、CPU 消耗、livelock/饥饿、ABA、复杂回收。
7. **Go 有 GC 为何仍有 ABA？**  
   要点：逻辑值可 A→B→A，地址/索引/句柄也可能复用；GC 只解决部分内存回收问题。
8. **Atomic 一定比 Mutex 快吗？**  
   要点：否；热点 cache line 和失败重试可能更贵，且维护成本更高。

### 追问链 5：从 WaitGroup 深挖到生命周期设计

1. **WaitGroup 解决什么问题？**  
   要点：等待一组任务计数归零。
2. **为什么 Add 在 go 前？**  
   要点：避免 Wait 在 counter 仍为 0 时提前返回。
3. **WaitGroup 是否能传错误？**  
   要点：不能；需额外结果 channel、错误聚合或结构化并发方案。
4. **WaitGroup 是否取消任务？**  
   要点：不能；需 context/cancel，任务必须主动响应。
5. **何时可以复用？**  
   要点：上一 generation 的所有 Wait 返回后。
6. **`WaitGroup.Go` 和手写有什么差异？**  
   要点：封装 Add+start；Go 1.26.4 明确 f 不得 panic，当前实现对 panic 有特殊重抛逻辑。
7. **任务派生子任务安全吗？**  
   要点：组非空时可调用 `Go` 派生；仍需清晰的关闭与取消边界。
8. **怎样避免 goroutine leak？**  
   要点：所有阻塞点有完成/取消路径；错误早退时取消发送者或使用有容量且有收尾协议的结果通道。

### 追问链 6：从线上延迟抖动深挖到工具闭环

1. **P99 上升但 CPU 不高，先怀疑什么？**  
   要点：锁/Channel/I/O 阻塞、队列、外部依赖，不只看 CPU profile。
2. **先看哪些全局信号？**  
   要点：goroutine 数、请求并发、队列长度、`/sync/mutex/wait/total:seconds` 增速、CPU/GC/网络。
3. **mutex profile 与 block profile 怎么选？**  
   要点：前者找造成锁竞争的持有临界区；后者找阻塞发生点及 channel/WaitGroup/Cond 等等待。
4. **什么时候用 trace？**  
   要点：需要时间线、调度、并行度、park/unpark、syscall 和 GC 关联时，抓短窗口。
5. **如何确认是锁粒度问题？**  
   要点：profile 栈、临界区源码、等待时间与 waiter 数、按锁/业务维度打点，做负载复现。
6. **如何修复而不是只换 RWMutex？**  
   要点：移出 I/O/回调、拆状态域、分片、批处理、immutable snapshot、减少共享写。
7. **修复如何验证？**  
   要点：相同负载下比较吞吐、P99、profile、metrics；跑 race、压力、故障注入和回归。
8. **若技术 Race 已消失但重复订单仍在？**  
   要点：进入业务幂等、事务、唯一约束和跨实例并发控制层面。

### 9.2 Race Detector 实战

#### 9.2.1 基本命令

```bash
go test -race ./...
go test -race -count=1 -run TestName ./path/to/pkg
go run -race ./cmd/service
go build -race -o service-race ./cmd/service
```

对 race 构建的服务运行代表性集成或压测流量，比只跑几条 happy-path 单测更有效。

常用环境配置：

```bash
GORACE='halt_on_error=1 strip_path_prefix=/workspace/' go test -race ./...
```

#### 9.2.2 如何阅读报告

报告通常包含：

1. 当前冲突访问的读/写栈；
2. 之前冲突访问的读/写栈；
3. 创建相关 goroutine 的栈；
4. 文件、行号与访问类型。

处理顺序：

```text
定位同一内存位置
  → 找出两个访问路径各自认为的“同步协议”
  → 检查是否真用同一把锁/同一 atomic/匹配 channel
  → 修复所有访问，而不是只包住报告中的一行
  → 添加能稳定覆盖该交错的回归测试
```

#### 9.2.3 能发现什么

- 普通变量读写 Race；
- slice/map/对象字段的冲突；
- 循环变量、共享 error、共享 buffer 等常见 Race；
- send 与 close 等缺乏可识别同步的冲突模式；
- 用户代码与库代码实际执行路径中的冲突。

#### 9.2.4 不能发现什么

- 未执行路径；
- 无 Data Race 的业务竞态；
- 死锁、livelock、公平性问题；
- 所有 goroutine leak；
- 跨进程、数据库或分布式一致性错误；
- “这段代码已被形式化证明线程安全”。

#### 9.2.5 开销与部署边界

官方文档给出的典型范围：

- 内存开销约 **5–10 倍**；
- 执行时间约 **2–20 倍**；
- 每个 `defer`/`recover` 还可能有额外分配，直到 goroutine 退出才回收，且不一定反映在普通内存统计中。

所以常见策略是：

- CI 全量或分层跑 `-race`；
- 集成环境、压测环境长期运行 race 二进制；
- 生产采用小比例隔离 canary，先评估容量与延迟；
- 不把 race build 的性能数据与普通构建直接比较。

### 9.3 Race Detector 与 concurrent map fatal 的区别

| 对比 | Race Detector | runtime concurrent map fatal |
|---|---|---|
| 机制 | 编译插桩 + ThreadSanitizer | map 实现中的运行时并发误用检查 |
| 覆盖 | 多种内存位置 | 普通 map 的部分并发读写/写写情形 |
| 需要 | `-race` 构建 | 普通构建也可能触发 |
| 输出 | 冲突访问栈、创建栈 | fatal 错误与 goroutine 栈 |
| 完整性 | 动态、依赖覆盖 | 不是完整 Race 检测，不能依赖必触发 |
| 目标 | 找 Data Race | 防止/暴露 map 内部并发破坏 |

**“线上从未出现 concurrent map writes”绝不等于 map 的并发使用正确。**

### 9.4 Mutex profile

#### 9.4.1 它回答什么

Mutex profile 追踪 `sync.Mutex`、`sync.RWMutex` 和 runtime 内部锁的竞争。当前文档规定：

- 栈对应**导致竞争的临界区结束位置**，例如 `Unlock`；
- 样本值近似表示其他 goroutine 因该持有者而阻塞的累计时间；
- 若 5 个 waiter 各等 1 秒，归因可近似为 5 秒，而不是 1 秒。

这使它更接近“谁让别人等”，而不是只告诉你“谁在 Lock 处等”。

#### 9.4.2 启用与抓取

程序内：

```go
runtime.SetMutexProfileFraction(10) // 平均采样约 1/10 竞争事件
```

测试：

```bash
go test -run TestHotPath \
  -mutexprofile=mutex.out \
  -mutexprofilefraction=1 ./path/to/pkg

go tool pprof -http=:0 mutex.out
```

通过 `net/http/pprof` 暴露后：

```bash
go tool pprof -http=:0 \
  http://127.0.0.1:6060/debug/pprof/mutex
```

生产不要盲目把采样率设为 1 并永久开启；先用低比例评估开销，再在问题窗口提高。

#### 9.4.3 解读步骤

1. 看 cumulative/delay，而不只看 sample count；
2. 定位 Unlock 栈对应的临界区；
3. 查临界区是否含 I/O、日志、回调、序列化、channel；
4. 看 waiter 数和请求量，区分“少量超长持锁”与“高频短锁热点”；
5. 修复后用相同负载复测。

### 9.5 Block profile

Block profile 记录 goroutine 在同步原语上阻塞的时间，包括：

- Mutex/RWMutex；
- WaitGroup；
- Cond；
- channel send/receive/select；
- 其他可采样阻塞事件。

与 mutex profile 的关键区别：

- **block profile 栈是阻塞发生的位置**，例如 `Mutex.Lock` 或 channel send；
- **mutex profile 栈是造成锁竞争的临界区结束位置**。

启用：

```go
runtime.SetBlockProfileRate(1_000_000) // 目标：平均每阻塞 1 ms 采一个事件
```

测试：

```bash
go test -run TestPipeline \
  -blockprofile=block.out \
  -blockprofilerate=1 ./path/to/pkg

go tool pprof -http=:0 block.out
```

HTTP：

```bash
go tool pprof -http=:0 \
  http://127.0.0.1:6060/debug/pprof/block
```

Block profile 能告诉你“大家堵在哪里”，但不总能直接告诉你“谁应该唤醒却没唤醒”；需结合 goroutine dump、源码和 trace。

### 9.6 `go tool trace`

#### 9.6.1 适用问题

Execution trace 捕获：

- goroutine 创建、阻塞、解除阻塞；
- P/M/G 调度；
- syscall；
- GC 与堆变化；
- processor 启停；
- 用户 Task、Region、Log。

它适合回答：

- 为什么某段时间 CPU 没吃满？
- 哪个 goroutine 在何时被谁唤醒？
- 请求为何串行化？
- 锁、channel、syscall、GC 如何共同造成尾延迟？

它不是定位 CPU/内存热点的首选；先用 profile，时间关系不清时再用 trace。

#### 9.6.2 获取 trace

测试：

```bash
go test -trace=trace.out ./path/to/pkg
go tool trace trace.out
```

程序：

```go
f, _ := os.Create("trace.out")
if err := trace.Start(f); err != nil {
    log.Fatal(err)
}
// 只覆盖需要分析的短窗口
trace.Stop()
_ = f.Close()
```

HTTP：

```bash
curl -o trace.out \
  'http://127.0.0.1:6060/debug/pprof/trace?seconds=5'
go tool trace trace.out
```

#### 9.6.3 用户标注

用 `trace.NewTask`、`trace.WithRegion`、`trace.Log` 把 runtime 事件映射到业务请求、阶段和关键状态，能显著提高可读性。

#### 9.6.4 Flight Recorder（Go 1.25+）

`runtime/trace.FlightRecorder` 维护最近一段 trace 的滚动窗口，适合“偶发尖峰发生后再保存前几秒”。核心 API：

```go
fr := trace.NewFlightRecorder(trace.FlightRecorderConfig{
    MinAge:   5 * time.Second,
    MaxBytes: 32 << 20,
})
if err := fr.Start(); err != nil {
    log.Fatal(err)
}
defer fr.Stop()

// 发生异常时：fr.WriteTo(file)
```

当前同一进程最多一个活跃 Flight Recorder；`MaxBytes` 是提示性上界，不是严格内存保证。

### 9.7 Runtime Metrics：先看全局趋势

Go 1.26.4 的：

```text
/sync/mutex/wait/total:seconds
```

表示 goroutine 阻塞在 `sync.Mutex`、`sync.RWMutex` 或 runtime 内部锁上的近似累计时间，适合识别全局锁竞争是否在上升。

示例：

```go
samples := []metrics.Sample{
    {Name: "/sync/mutex/wait/total:seconds"},
}
metrics.Read(samples)
waitSeconds := samples[0].Value.Float64()
```

监控应看**单位时间增量/速率**，并结合流量归一化，例如：

```text
mutex_wait_seconds_delta / request_count
```

它只能说明全局趋势，不能给出具体锁栈；细节仍需 mutex/block profile。

### 9.8 Goroutine dump 与泄漏分析

抓取：

```bash
curl 'http://127.0.0.1:6060/debug/pprof/goroutine?debug=2' \
  > goroutines.txt
```

分析方法：

1. 按相同栈聚合数量；
2. 区分 runnable、chan send/receive、semacquire、select、IO wait；
3. 找创建者和生命周期 owner；
4. 检查取消信号、channel 关闭、错误早退、WaitGroup counter；
5. 连续多时点对比，确认栈是否只增不减。

#### Go 1.26 实验性 goroutine leak profile

构建时设置：

```bash

```

会增加 `goroutineleak` profile 和 `/debug/pprof/goroutineleak` endpoint。它利用 GC 可达性找一类“阻塞原语已不可能被任何可运行 goroutine 解除”的泄漏。

边界：

- **Go 1.26 仍是实验功能，不是默认启用。**
- 全局变量或可运行 goroutine 仍可达的原语可能让泄漏无法被识别。
- 它不是所有永久阻塞的完备检测器。

### 9.9 典型事故 1：锁内 RPC 导致服务雪崩

**现象：**下游延迟从 20 ms 升到 2 s；本服务 CPU 不高，goroutine 数暴涨，P99 超时。

**代码形态：**

```go
mu.Lock()
resp, err := client.Call(ctx, req)
state.apply(resp)
mu.Unlock()
```

**链路：**

```text
下游变慢
 → 锁持有时间变长
 → waiter 累积
 → 请求超时但 goroutine 仍排队
 → 连接/内存压力
 → 更高延迟与级联失败
```

**排查：**

- mutex profile 顶部指向包含 RPC 的 Unlock 栈；
- block profile 显示大量 Lock 等待；
- trace 显示请求从并行变串行；
- `/sync/mutex/wait/total` 增速异常。

**修复：**锁内只读取输入快照和预留版本；解锁后 RPC；返回后再短锁校验版本并提交。必要时把慢操作建模为显式状态机。

---

### 9.10 典型事故 2：RWMutex 长读锁阻塞全部写与新读

**现象：**一个报表请求持 RLock 遍历百万条记录；写者到来后，新短读请求也开始堆积。

**原因：**写者 pending 后 RWMutex 阻塞新读者；长旧读者不退出，写者无法获得锁，后续读者也等待，形成 convoy。

**修复选项：**

- 锁内复制必要索引/指针，解锁后遍历不可变数据；
- 分页或分片；
- immutable snapshot；
- 将长报表迁到副本/数据库；
- 不要仅通过“换回 Mutex”掩盖长临界区。

---

### 9.11 典型事故 3：错误早退导致发送者 goroutine 泄漏

```go
results := make(chan Result)
for _, job := range jobs {
    go func(j Job) {
        results <- run(j)
    }(job)
}

for range jobs {
    r := <-results
    if r.Err != nil {
        return r.Err // 剩余发送者可能永久阻塞
    }
}
```

**修复：**

- 使用 context cancel，让 worker 的 send 也 select `ctx.Done()`；
- collector 即使遇错也负责 drain，或启动明确的收尾者；
- 根据任务数合理缓冲，但不能只靠大 buffer 掩盖无限生产；
- 采用结构化并发，保证函数返回前子任务可终止。

---

### 9.12 典型事故 4：Atomic 热点导致扩核负收益

**现象：**从 8 核升到 32 核，吞吐不升反降，CPU profile 看到大量 atomic update/CAS。

**原因：**所有核心写同一 cache line，缓存行所有权反复转移；CAS 失败重试又放大流量。

**修复：**

- 每 worker/shard 局部计数；
- 批量聚合；
- 降低更新频率；
- 把强一致实时数改为近似或周期汇总；
- 用硬件工具验证 cache miss/line transfer，而非只凭猜测加 padding。

---

### 9.13 典型事故 5：技术无 Race，但业务超卖

**现象：**`go test -race` 全绿，库存仍偶发负数。

**错误模型：**

```go
if stock.Load() > 0 {
    stock.Add(-1)
    createOrder()
}
```

**单进程修复：**CAS 循环或 Mutex 把检查与扣减合并。

**多实例修复：**数据库条件更新：

```sql
UPDATE sku
SET stock = stock - 1
WHERE id = ? AND stock > 0;
```

检查受影响行数，并配合订单唯一键、事务或幂等键。若扣库存与建订单必须共同原子，需把它们放入正确事务/状态机，而不是只在 Go 进程里加锁。

### 9.14 生产锁竞争优化的正确闭环

```text
1. 定义症状与 SLO
2. 保存基线：流量、P50/P99、CPU、goroutine、mutex wait metric
3. 抓 profile/trace，定位具体锁与临界区
4. 识别根因：持锁太久、频率过高、锁域太大、热点 key、锁顺序
5. 选择结构性修复：移出慢操作、拆域、分片、批量、快照、owner 模型
6. 跑 race + 功能回归
7. 用同负载比较吞吐、尾延迟、profile 和资源
8. 灰度并监控回退指标
```

常见无效优化：

- 未测量就把 Mutex 全换成 RWMutex；
- 把锁换成多个 atomic，却破坏多字段不变量；
- 把同步改成 TryLock 忙循环；
- 随意 padding 所有结构体；
- 只缩短平均临界区，不处理少量超长尾部；
- 用更大 channel buffer 隐藏消费者停止。

## 十、面试回答模板

### 10.1 30 秒回答

> Go 并发正确性要同时看原子性、可见性、有序性、生命周期和所有权。严格的 Data Race 是同一内存位置上至少一个普通写与另一个读/写在 happens-before 上无序；Race Condition 更宽，可能所有访问都 atomic 但业务 check-then-act 仍错误。Go 通过 goroutine 启动、channel、Mutex/RWMutex、Once、WaitGroup 和 atomic 建立同步边；Sleep 和一次运行的日志顺序不算同步。实现上 Mutex 是 CAS 快路径加有限自旋、runtime semaphore 和正常/饥饿模式。工程上先用清晰的锁或 channel 保证不变量，再用 `-race`、mutex/block profile 和 trace 验证与排障。

### 10.2 2 分钟回答

> Memory Model 中，同一 goroutine 内有 sequenced-before，同步操作之间有 synchronized-before，两者并集的传递闭包是 happens-before。无 Data Race 的程序享有 DRF-SC，可以按某种顺序一致交错来推理。比如普通写在 channel send 前，send 同步到对应 receive，receive 后的普通读就保证看到写。
>
> `sync.Mutex` 零值可用、使用后不可复制、不可重入，也不绑定 goroutine owner。Go 1.26.4 当前实现用 `state` 编码 locked、woken、starving 和 waiter 数；无竞争时一次 CAS，竞争时可能短暂自旋，之后通过 runtime semaphore park。正常模式重吞吐，等待超过当前约 1 ms 可进入饥饿模式并直接 handoff。
>
> `RWMutex` 通过 `readerCount`、`readerWait`、`writerSem`、`readerSem` 和内部写锁实现 writer preference，所以不能升级、降级，也不应递归 RLock。它有共享 atomic 计数和协调开销，不一定比 Mutex 快。
>
> Atomic 适合独立计数、标志、状态或不可变指针发布；多个 atomic 字段不是事务。生产中用 `go test -race` 找实际执行的 Data Race，用 mutex profile 找造成竞争的 Unlock 临界区，用 block profile 找阻塞点，用 trace 看调度时间线；业务超卖等要靠 CAS、事务、幂等和唯一约束，而不是只看 `-race`。

### 10.3 5 分钟深入回答

> 我会先从正确性模型回答。Go 中跨 goroutine 可见性不能靠时间推测，而要构造 happens-before。`go` 语句同步到新 goroutine 开始，但 goroutine 退出没有自动 join；channel send 同步到对应 receive，close 同步到因关闭返回的 receive；同一锁的 Unlock 同步到后续 Lock 返回；Once 的初始化完成同步到所有 Do 返回；WaitGroup 的 Done 或 Go 任务返回同步到被解除的 Wait；atomic 全部表现为顺序一致。Sleep、日志先后和单核测试没有这种保证。
>
> Data Race 与业务 Race 要分开。普通 `count++` 是读改写，多个 goroutine 会 Data Race；而两个 atomic 字段或 atomic 库存的 Load+Add 可能无 Data Race，却因跨操作不变量被破坏而超卖。前者由锁或 atomic 修复，后者必须把状态变换做成单次 CAS、锁内事务或数据库条件更新。
>
> Mutex 当前实现有快慢路径。`state` 低位是 locked/woken/starving，高位是 waiter 数，`sema` 交给 runtime。CAS 0→locked 是快路径。慢路径在适合时有限自旋，失败后增加 waiter 并 `gopark`。正常模式允许唤醒者和新 goroutine 竞争以提高吞吐；旧 waiter 等待超过当前 1 ms 后可进入 starvation，Unlock 直接把 ticket 和执行机会 handoff。这个阈值是 Go 1.26.4 实现细节，不是规范。
>
> RWMutex 用负 `readerCount` 宣告 pending writer，阻塞新读者；旧读者通过 `readerWait` 清零后唤醒写者。因而读锁升级会等自己，递归 RLock 在写者到来时也能死锁。性能上每个 RLock 都更新共享 atomic，短读或核数高时 cache line 竞争可能比 Mutex 更差。
>
> WaitGroup 是 counter+waiter+semaphore 的生命周期栅栏；正 Add 从 0 开始必须在 Wait 前，Go 1.25 起可用 `WaitGroup.Go`，但当前文档要求 f 不得 panic。Once 用 atomic done 快路径加 Mutex 等待初始化完成，panic 后仍视为 done；需要值可用 OnceValue。Cond 等待谓词，Wait 必须在持锁时调用且醒后用 for 重查。
>
> 排障时先看 goroutine 数和 `/sync/mutex/wait/total` 增速，再抓 mutex/block profile。mutex 栈指向造成竞争的 Unlock/临界区结束，block 栈指向实际等待位置。需要看到 park/unpark、CPU 空洞、syscall 和 GC 时间关系时抓短 trace。Race Detector 是动态覆盖，典型内存 5–10 倍、时间 2–20 倍，适合 CI、集成和 canary，而不是把无报告当证明。

### 10.4 源码级回答

> 以 Go 1.26.4 为例，公开 `sync.Mutex` 包装 `internal/sync.Mutex`，后者是 `state int32` 与 `sema uint32`。`Lock` 先 CAS 0 到 `mutexLocked`；`lockSlow` 根据 `mutexLocked`、`mutexWoken`、`mutexStarving` 和 waiter count 计算新状态，调用 `runtime_canSpin`/`runtime_doSpin` 做有限自旋，竞争失败后用 `runtime_SemacquireMutex` 进入 `runtime/sema.go`。semaphore 路径先登记 waiter 再复查 ticket，避免 missed wakeup，随后把 sudog 挂到按地址组织的 semaRoot 并 gopark。`unlockSlow` 在正常模式设置 woken 并唤醒一个 waiter，在饥饿模式用 handoff=true 直接传票据；当前 `starvationThresholdNs` 是 1e6。
>
> `RWMutex` 有 `w Mutex`、`writerSem`、`readerSem`、`readerCount atomic.Int32`、`readerWait atomic.Int32`。写者先拿 `w`，再对 readerCount 减 `1<<30`，后来的 RLock 因结果为负而睡在 readerSem；旧读者退出减少 readerWait，最后一个释放 writerSem。
>
> `WaitGroup` 用 atomic Uint64 合并 task counter 与 waiter count，counter 归零时按 waiter 数释放 sema；Go 1.26.4 的 `Go` wrapper 对 panic 重新抛出且不先 Done。`Once` 的 done atomic 放首字段，快路径 Load，慢路径 Mutex，done Store 延迟到 f 返回。`sync.Map` 在 Go 1.26 已包装 `internal/sync.HashTrieMap`，根和 child 是 atomic Pointer，读沿 16 路 Hash-Trie 无锁遍历，写锁局部 indirect 节点并发布新 entry；旧 `read/dirty/misses` 已不是当前默认实现。

## 十一、本章速记

1. **Data Race：同一内存位置，读写/写写，至少一个普通访问，HB 无序。**
2. Race Condition 比 Data Race 更广；`-race` 全绿仍可能超卖、重复执行或重复 close。
3. `happens-before = transitive closure(sequenced-before ∪ synchronized-before)`。
4. DRF-SC 只保证无 Data Race 程序可按顺序一致交错推理。
5. `go` 语句同步到新 goroutine 开始；goroutine 退出没有自动 join。
6. Channel send 同步到对应 receive；buffered send 返回不代表业务已消费。
7. Sleep、日志顺序、GOMAXPROCS=1 都不是同步。
8. Mutex 零值可用、使用后不可复制、不可重入、不绑定 goroutine owner。
9. Mutex 无竞争是 CAS 快路径；竞争后可能有限自旋，再走 runtime semaphore。
10. Go 1.26.4 Mutex 饥饿阈值当前约 1 ms，属于实现细节。
11. TryLock 失败没有同步效果；不要忙循环。
12. RWMutex 有 writer preference；pending writer 会阻塞新读者。
13. RWMutex 不能升级、不能原子降级，也不要递归 RLock。
14. RWMutex 读锁更新共享 atomic，未必比 Mutex 快。
15. WaitGroup 正 Add 从 0 开始必须先于 Wait；复用要分 generation。
16. `WaitGroup.Go` 自 Go 1.25 存在；当前文档要求任务不得 panic。
17. Once 中 f panic 也视为已执行；后续 Do 不重试。
18. Cond.Wait 必须持锁，并始终在谓词循环中调用。
19. Go atomic 是顺序一致；多个 atomic 字段仍不是事务。
20. atomic pointer 发布后，对象及其可达结构应不可变；Atomic 也不消除 ABA 与缓存行争用。
21. 多数业务 map 先用 map+锁；**Go 1.26 `sync.Map` 当前默认是 Hash-Trie，而非旧 read/dirty 双 map。**
22. 锁内避免 I/O、回调和可能阻塞的 channel；多锁必须统一顺序。
23. Race Detector 是动态检测：无报告只代表本次执行未发现。
24. Mutex profile 看造成竞争的临界区结束点，block profile 看阻塞点，mutex wait metric 看全局趋势。
25. 进程内锁不能替代事务、幂等键、唯一约束和分布式并发协议。

### 12.4 统一答案

#### 12.4.1 简答题答案

1. **Data Race** 是同一内存位置上的读写或写写访问，至少一个为普通非同步访问，且二者在 HB 上无序。HB 是逻辑偏序，不要求物理时钟上完全重叠；只要没有建立先后，执行就可能以冲突方式交错。
2. sequenced-before 是单 goroutine 内规范顺序；synchronized-before 来自同步操作观察；happens-before 是两者并集的传递闭包。
3. `go` 语句 synchronized-before 新 goroutine 开始，因此启动前写可通过该边发布给新 goroutine。goroutine 退出没有自动同步到其他事件，必须显式 join/等待。
4. send→对应 receive 完成；close→因关闭返回零值的 receive；无缓冲 receive→对应 send 完成；容量 C 的 channel 上第 k receive→第 k+C send 完成。缓冲 send 返回只表示入队成功，消费者可能尚未开始处理。
5. 当前 `state` 含 locked、woken、starving 和 waiter count。0→locked CAS 是快路径；慢路径可有限自旋，失败后登记 waiter 并经 runtime semaphore 休眠。正常模式允许竞争以提高吞吐；等待超过当前约 1 ms 可进入饥饿模式，由 Unlock 直接 handoff。
6. 升级时写锁等待所有读者，包括调用者自己持有的读锁；调用者又等写锁返回才释放，形成环。递归 RLock 在 pending writer 出现后会被阻塞，而写者等第一次 RUnlock，也形成环。
7. counter 从 0 做正 Add 必须在对应 Wait 前；Done 与计数一一对应；复用要等上一轮所有 Wait 返回；`WaitGroup.Go` 自 Go 1.25 可用，但任务不得 panic，不负责错误/结果/取消。
8. `Once.Do` 中 f panic 仍将 Once 标为完成，后续 Do 不再调用 f。OnceFunc/OnceValue/OnceValues 会在后续调用中以同一 panic 值再次 panic。
9. 两个 atomic 操作是两个独立线性化点，不形成事务。替代：一个 Mutex；把状态打包到一个 CAS 字；完整 immutable snapshot 通过一个 atomic pointer 发布。
10. Race Detector 找实际执行路径的 Data Race；mutex profile 找造成锁竞争的持有临界区；block profile 找 goroutine 阻塞位置和累计时间；trace 看调度、阻塞/唤醒、syscall、GC 和请求阶段的时间关系。

#### 12.4.2 代码题答案

#### 代码题 1 答案

原代码对 `ready` 和 `data` 都有 Data Race；忙循环也不保证观察到写。把 `ready` 改为 `atomic.Bool`，并让 writer 在写完 data 后 `ready.Store(true)`、reader 用 `ready.Load()`，在 Go 的 SC atomic 语义下，reader 观察到 true 后可通过同步边看到此前 data 写入：

```go
var data int
var ready atomic.Bool

func writer() {
    data = 7
    ready.Store(true)
}

func reader() int {
    for !ready.Load() {
        runtime.Gosched()
    }
    return data
}
```

不过忙等仍可能浪费 CPU；若等待可能较长，channel 或 Cond 更合适。若 `data` 在 ready 为 true 后还会继续修改，则仍需额外同步。

#### 代码题 2 答案

问题至少包括：

1. `Add(1)` 在新 goroutine 内，`Wait` 可提前返回；
2. 循环中的 `task` 捕获应确保每次闭包绑定正确任务。Go 1.22+ range 变量每轮新建，但显式传参仍更清楚，也兼顾旧版本阅读；
3. 全局 WaitGroup 让多个 `run` 调用互相干扰且可能错误复用。

Go 1.24 及更早常规写法：

```go
func run(tasks []func()) {
    var wg sync.WaitGroup
    for _, task := range tasks {
        task := task
        wg.Add(1)
        go func() {
            defer wg.Done()
            task()
        }()
    }
    wg.Wait()
}
```

Go 1.25+：

```go
func run(tasks []func()) {
    var wg sync.WaitGroup
    for _, task := range tasks {
        task := task
        wg.Go(task)
    }
    wg.Wait()
}
```

前提是 task 不 panic；若需错误与取消，应使用更合适的结构，而不是只靠 WaitGroup。

#### 代码题 3 答案

`p.Store(s)` 只发布指针；随后修改 `s.Names` 会与 reader 共享 slice header/backing array，可能 Data Race。正确做法是在发布前构造独立副本，发布后不再修改：

```go
func update(s *Snapshot) {
    next := &Snapshot{
        Names: slices.Clone(s.Names),
    }
    p.Store(next)
}

func load() []string {
    snap := p.Load()
    if snap == nil {
        return nil
    }
    return snap.Names // 调用者也必须只读
}
```

若不能信任调用者只读，可返回 clone，或提供只读查询方法而不暴露 slice。嵌套 map、pointer tree 也要深拷贝可变部分。

#### 代码题 4 答案

在同一进程、两把锁均成功获得后，余额变换处于同一临界区，具有原子性。但两个并发转账若参数方向相反，会按 `from→to` 的动态顺序加锁，可能锁顺序反转。应按稳定 ID 排序加锁：

```go
func transfer(from, to *Account, n int64) bool {
    if n <= 0 {
        return false
    }
    if from == to {
        return true // 或按业务规则直接拒绝同账户转账
    }

    first, second := from, to
    if first.id > second.id {
        first, second = second, first
    }

    first.mu.Lock()
    second.mu.Lock()
    defer second.mu.Unlock()
    defer first.mu.Unlock()

    if from.balance < n {
        return false
    }
    from.balance -= n
    to.balance += n
    return true
}
```

特殊输入是 `from == to`：若不提前处理，会对同一非重入 Mutex 加锁两次。跨实例时该进程锁仍不足以保证全局正确。

#### 代码题 5 答案

`if` 只检查一次，醒来重新拿锁时 `open` 可能已变回 false；正确写法使用循环：

```go
func (g *Gate) Wait() {
    g.mu.Lock()
    defer g.mu.Unlock()
    for !g.open {
        g.cond.Wait()
    }
}

func (g *Gate) Open() {
    g.mu.Lock()
    g.open = true
    g.cond.Broadcast()
    g.mu.Unlock()
}
```

这里在持锁时修改谓词并 Broadcast，使 waiter 醒来后在同一锁协议下重新检查。若 Gate 后续可 Close，Close 也应在同一锁下修改 `open=false`。

#### 12.4.3 系统设计题答案

#### 系统题 1：高读低写配置中心

推荐 **immutable snapshot + `atomic.Pointer[Config]`**：

1. watcher 收到新版本，在私有内存中解析、校验和深拷贝；
2. Config 包含版本号、校验值，map/slice/嵌套对象在发布后全部不可变；
3. 用一次 `Store` 原子替换完整根指针，读者一次 `Load` 得到同一版本；
4. 回滚等价于发布一个先前保留的不可变版本指针；
5. 读 API 不暴露可修改引用，或文档/类型封装强制只读；
6. 限制历史版本数，否则旧 snapshot 被长请求持有会增加内存；
7. 更新失败不替换当前指针，指标记录版本、更新时间和校验失败；
8. 若写频率或快照很大导致复制过重，再考虑结构共享，但必须保持发布后不可变。

不建议每次读加 RWMutex，除非配置很小且实测足够；也不应原地修改当前 map。

#### 系统题 2：多实例账户转账

正确性主边界应放在数据库事务：

1. 请求带全局唯一幂等键，数据库对幂等键建唯一约束；
2. 事务内锁定或条件更新两个账户，采用固定账户 ID 顺序降低数据库死锁；
3. 检查余额，扣款、入账、流水和幂等结果共同提交；
4. 事务冲突/死锁按可识别错误重试，重试仍使用同一幂等键；
5. 进程内 Mutex 只能减少本实例同账户热点，不能作为全局正确性依据；
6. 分布式锁不是首选原子性基础。若为削峰使用，必须有租约、fencing token，最终写仍由数据库版本/条件拒绝旧 owner；
7. 崩溃恢复依赖已提交事务和幂等记录，不依赖内存状态；
8. 下游事件通过 transactional outbox 等方式可靠发布，消费者幂等。

#### 系统题 3：P99、goroutine 与 mutex wait 同升

完整步骤：

1. **止损：**限流、熔断慢路径、回滚最近变更、扩容只作为暂时措施；保护 pprof endpoint 权限。
2. **保存现场：**连续抓三份 goroutine dump；记录流量、P50/P99、CPU、内存、GC、队列、连接池、mutex wait metric。
3. **抓 profile：**短时提高 Mutex/Block 采样；获取 mutex、block、CPU profile。
4. **抓时间线：**问题窗口抓 5–10 秒 trace，或从 Flight Recorder 落盘。
5. **定位：**mutex profile 找 Unlock 临界区；block 找 Lock/channel 等待点；dump 看大量相同 `semacquire` 栈；trace 看何时开始串行化。
6. **检查变更：**是否把 RPC/日志/序列化/回调移入锁，是否新增全局 map 锁、长 RLock、锁顺序或 channel 依赖。
7. **复现：**用生产分布的 key、读写比例与下游延迟做压力测试；保留基线。
8. **修复：**移出不可控阻塞、缩小但完整覆盖不变量的临界区；按状态域分片；采用快照或批处理；修正生命周期。
9. **正确性验证：**单测、压力、`-race`、故障注入、死锁/取消路径回归。
10. **性能验证：**相同负载比较 P99、吞吐、mutex wait 增速、profile 顶部和 goroutine 稳态。
11. **灰度：**小流量发布，设置自动回退阈值；问题消失后再逐步恢复采样率和容量。
