## 二、功能介绍

### 2.1 Channel 是什么

**[规范]** Channel 是一种带元素类型、可选方向和可选容量的并发通信类型。它用于在 Goroutine 之间发送和接收值，并在通信点建立同步关系。

```go
var bi chan int       // 双向：可发送、可接收
var out chan<- int    // 只发送
var in <-chan int     // 只接收

unbuffered := make(chan int)    // 容量为 0
buffered := make(chan int, 16)  // 容量为 16
```

需要区分三个层面：

- **语言层面**：规定何时阻塞、何时 panic、关闭后如何接收、`select` 如何选分支。
- **运行时层面**：当前用 `hchan`、环形缓冲区、等待队列和调度器实现。
- **工程层面**：开发者额外约定“谁拥有发送权、谁负责关闭、如何取消”。

**[规范]** 同一个 Channel 可以被任意数量的 Goroutine 并发用于发送、接收以及查询 `len`/`cap`，无需调用者再为 Channel 内部一致性加锁；但这不等于关闭所有权、send/close 顺序或通过指针共享的业务对象自动安全。

Channel 值可以被复制、比较并作为 Map Key。复制 Channel 值只会复制一个指向同一通信对象的句柄，不会复制缓冲区。

```go
ch1 := make(chan int, 1)
ch2 := ch1

fmt.Println(ch1 == ch2) // true
ch2 <- 7
fmt.Println(<-ch1)      // 7
```

### 2.2 零值与创建

**[规范]** Channel 的零值是 `nil`。nil Channel 没有对应的可通信对象：

- 向 nil Channel 发送：永久阻塞。
- 从 nil Channel 接收：永久阻塞。
- 关闭 nil Channel：panic。
- 在 `select` 中，涉及 nil Channel 的通信 case 永远不会就绪。
- 对 nil Channel 执行 `len`、`cap`：均为 0。

“永久阻塞”不等于一定立刻出现 `fatal error: all goroutines are asleep - deadlock!`。只有 runtime 判断整个程序已无可运行 Goroutine、无可唤醒事件时，才会报告全局死锁；在线上服务中，一个 Goroutine 可能只是静默泄漏。

### 2.3 无缓冲与有缓冲 Channel

#### 无缓冲 Channel

```go
ch := make(chan int)
```

发送必须与接收配对。发送方和接收方在交接点同步：

```text
Sender G ── value ──► Receiver G
         同一次 rendezvous
```

- 没有接收者时，发送阻塞。
- 没有发送者时，接收阻塞。
- 适合表达交接、确认、所有权转移和严格背压。

#### 有缓冲 Channel

```go
ch := make(chan int, 3)
```

发送和接收可在容量范围内解耦：

```text
sendx                          recvx
  │                              │
  ▼                              ▼
+------+------+------+
|  A   |  B   | empty|
+------+------+------+
        环形队列
```

- 缓冲区未满时，发送通常不等待接收者。
- 缓冲区非空时，接收通常不等待发送者。
- 满时发送阻塞；空时接收阻塞。
- 在单发送者与单接收者的通信序列中，值按发送顺序被接收；存在多个并发发送者或接收者时，谁先完成、谁得到哪个值取决于实际交错，不能自行假设业务顺序。
- 容量表达的是可承受的在途工作量，不是“越大越快”。

**关键区别**：无缓冲 Channel 把发送方与接收方的进度直接耦合；有缓冲 Channel 允许短时速率差，但缓冲区最终仍会满，因此它提供的是**有界解耦和背压**。

### 2.4 双向与单向 Channel

方向是静态类型约束：

```go
func produce(out chan<- int) {
    out <- 1
    close(out) // send-only Channel 可以 close
}

func consume(in <-chan int) {
    for v := range in {
        fmt.Println(v)
    }
}
```

**[规范]** 双向 Channel 可以赋值或转换为同元素类型的只发送/只接收 Channel；反向不能恢复：

```go
bi := make(chan int)
var send chan<- int = bi
var recv <-chan int = bi

// var bi2 chan int = send // 编译错误
// close(recv)             // 编译错误：不能关闭只接收 Channel
```

方向不会创建新的运行时 Channel，`bi`、`send`、`recv` 仍指向同一个对象。方向主要用于缩小 API 权限，减少误用。

### 2.5 nil、正常、关闭 Channel 的完整行为表

| 操作 | nil Channel | 正常无缓冲 Channel | 正常有缓冲 Channel | 已关闭但仍有缓冲数据 | 已关闭且已排空 |
|---|---|---|---|---|---|
| `ch <- v` | 永久阻塞 | 等待接收者 | 有空位则入队，否则阻塞 | panic | panic |
| `<-ch` | 永久阻塞 | 等待发送者 | 有数据则出队，否则阻塞 | 立即返回队首值 | 立即返回元素零值 |
| `v, ok := <-ch` | 永久阻塞 | 收到发送值，`ok=true` | 收到发送值，`ok=true` | 返回缓冲值，`ok=true` | 返回零值，`ok=false` |
| `close(ch)` | panic | 首次关闭成功 | 首次关闭成功 | 再次关闭 panic | 再次关闭 panic |
| `for v := range ch` | 永久阻塞 | 持续接收，直到关闭并排空 | 同左 | 继续排空 | 退出循环 |
| `select` 中通信 case | 永不就绪 | 仅可配对时就绪 | 满足发送/接收条件时就绪 | 接收 case 就绪 | 接收 case 永远就绪 |
| `len(ch)` | 0 | 0 | 当前排队元素数的瞬时快照 | 剩余缓冲元素数 | 0 |
| `cap(ch)` | 0 | 0 | 创建时容量 | 容量不因关闭改变 | 容量不因关闭改变 |

> **面试易错点**：已关闭 Channel 的接收不是 panic；只有向已关闭 Channel 发送、关闭 nil Channel、重复关闭才会 panic。

### 2.6 send、receive、close 的语言语义

#### 发送

```go
ch <- expr
```

**[规范]** 进入发送语句时，Channel 操作数和待发送表达式会先求值；之后通信才开始。即使最终因为 Channel 已关闭而 panic，表达式副作用也已经发生。

- 向 nil Channel 发送永久阻塞。
- 向已关闭 Channel 发送 panic。
- 无缓冲 Channel 要等接收者就绪。
- 有缓冲 Channel 只要缓冲区有空位即可完成。

#### 接收

```go
v := <-ch
v, ok := <-ch
```

- `ok=true`：值来自某次成功发送，即使 Channel 随后已关闭。
- `ok=false`：Channel 已关闭且缓冲区已经排空，此时 `v` 为元素类型零值。
- 不能仅根据 `v` 是否为零值判断 Channel 是否关闭，因为零值也可能是发送方合法发送的数据。

#### 关闭

```go
close(ch)
```

关闭表达的是：**今后不会再有值发送到该 Channel**。

- close 不发送一个普通元素。
- close 不清空缓冲区。
- close 不等待接收者消费完成。
- close 会让等待接收的 Goroutine 被唤醒；缓冲区排空后，它们得到零值和 `ok=false`。
- close 会让等待发送的 Goroutine 被唤醒并 panic。
- close 不是析构函数，也不负责释放 Channel 内存。

### 2.7 `for range` 的退出条件

```go
for v := range ch {
    use(v)
}
```

等价语义可理解为：

```go
for {
    v, ok := <-ch
    if !ok {
        break
    }
    use(v)
}
```

因此，`range` 只有在 **Channel 已关闭且缓冲区排空** 后退出。发送方停止发送但不关闭，接收方会一直等待下一项。

### 2.8 谁应该关闭 Channel

**[工程约定]** 最稳妥的规则是：

> **创建并不自动等于拥有关闭权；知道“不会再发生发送”的一方才有资格关闭。通常这个角色是发送方或发送方协调者。接收方一般不关闭数据 Channel。**

原因不是语法限制，而是接收方通常无法证明其他发送者已经结束。接收方贸然关闭会使尚未完成的发送 panic。

#### 单发送者

发送者完成最后一次发送后关闭：

```go
func producer(out chan<- int) {
    defer close(out)
    for i := 0; i < 10; i++ {
        out <- i
    }
}
```

#### 多发送者

多个生产者各自只发送，不关闭；独立协调者等待所有生产者结束后关闭：

```go
func merge(inputs ...<-chan int) <-chan int {
    out := make(chan int)
    var wg sync.WaitGroup
    wg.Add(len(inputs))

    for _, in := range inputs {
        in := in
        go func() {
            defer wg.Done()
            for v := range in {
                out <- v
            }
        }()
    }

    go func() {
        wg.Wait()
        close(out)
    }()

    return out
}
```

该示例只突出“多发送者统一关闭”模式；若下游可能提前停止消费，转发时还应同时选择 `ctx.Done()`，否则转发 Goroutine 仍可能阻塞在 `out <- v`。

`sync.Once` 只能避免“重复 close”，不能自动解决“发送与 close 并发”。若某个发送者还可能发送，即使 close 被 `Once` 包住，仍可能发生 send-on-closed-channel panic。

### 2.9 close 是广播，不是单播信号

关闭一个 Channel 后，所有等待接收者最终都能观察到关闭：

```go
stop := make(chan struct{})

for i := 0; i < 100; i++ {
    go func() {
        <-stop // close(stop) 后全部被唤醒
    }()
}

close(stop)
```

相比之下：

```go
stop <- struct{}{}
```

通常只会被一个接收者消费。若需要取消树、超时、错误传播和跨 API 生命周期管理，优先使用 `context.Context`，其 `Done()` 本质上提供了一个可关闭广播信号，但还携带截止时间和原因语义。

### 2.10 `select` 的基本语义

```go
select {
case v := <-in:
    use(v)
case out <- result:
    // 发送完成
case <-ctx.Done():
    return ctx.Err()
default:
    // 当前没有通信可立即完成
}
```

**[规范]** `select` 的关键规则：

1. 进入 `select` 时，所有 case 的 Channel 操作数，以及发送 case 的右值表达式，按源码顺序求值一次；接收 case 的赋值左侧表达式要等该 case 真正被选中后才求值。
2. 判断哪些通信可以立即进行。
3. 若一个或多个 case 就绪，从中做一次**均匀伪随机选择**。
4. 若没有通信就绪且有 `default`，执行 `default`。
5. 若没有通信就绪且没有 `default`，当前 Goroutine 阻塞。
6. 空 `select {}` 永久阻塞。

**伪随机选择不等于严格公平**。规范没有承诺轮询、优先级、最大等待时间或长期无饥饿。当前 runtime 会随机化轮询顺序，但调度时机和就绪状态不断变化，仍不能据此证明业务级公平。

#### nil Channel 动态禁用分支

```go
var out chan<- int
var next int

if hasPending {
    out = destination
    next = pendingValue
}

select {
case out <- next: // out == nil 时该 case 永不就绪
    hasPending = false
case v := <-input:
    pendingValue = v
    hasPending = true
}
```

把 Channel 变量设为 nil 可以在不重建 `select` 的情况下动态关闭某个 case。对于已关闭的输入 Channel，常见做法是在检测到 `ok=false` 后将对应局部变量设为 nil，避免该 case 永远就绪导致忙循环。

#### `default` 是非阻塞，不是“低优先级等待”

```go
select {
case ch <- v:
    // 立即发送成功
 default:
    // 此刻不能立即发送，直接走这里
}
```

这是“try send”。若放在无休止循环中而没有任何阻塞、退避或限速，会形成忙轮询并占满 CPU。

### 2.11 `time.After`、Timer、Ticker 与版本边界

`time.After(d)` 适合一次性、清晰的超时分支：

```go
select {
case v := <-ch:
    return v, nil
case <-time.After(500 * time.Millisecond):
    return 0, context.DeadlineExceeded
}
```

但在高频循环中，每轮调用都会创建新 Timer，可能带来分配和定时器管理开销。热路径通常应评估复用 `time.Timer`。

#### Go 1.23 前后的重要变化

下表的“Go 1.23+”指程序启用了新的 Timer Channel 语义；通常要求主模块 `go.mod` 的 `go` 版本至少为 1.23，且没有通过 `GODEBUG=asynctimerchan=1` 强制恢复旧行为。

| 维度 | 旧语义 | 启用 Go 1.23+ 新语义后的行为 |
|---|---|---|
| 未停止且不再可达的 Timer/Ticker | 通常要等触发或 Stop 后才可回收，旧面经常称 `time.After` 循环“泄漏到到期” | 未引用的未停止 Timer/Ticker 可被 GC 回收 |
| Timer Channel 表现 | 异步缓冲语义，可能观察到旧值 | 对外表现为同步 Channel，`len`/`cap` 为 0 |
| `Stop`/`Reset` 后旧值 | 需要谨慎 Stop、检查返回值并排空 | 对 Channel Timer 提供更强的“不再收到旧值”保证 |

因此，在 Go 1.26.4 中说“`time.After` 一定造成不可回收的永久泄漏”已经过时；正确说法是：**一次性使用通常没问题，高频循环仍可能产生不必要的 Timer 分配和 GC 压力，应基于版本和性能数据选择复用 Timer。**

Ticker 在语义结束时仍应调用 `Stop`，目的不仅是 GC，还包括停止未来 tick 和释放关联调度工作。停止 Ticker 不会关闭 `C`，不能靠 `for range ticker.C` 自动退出。

### 2.12 Channel 的 happens-before 规则

Go Memory Model 中，Channel 相关的关键同步关系包括：

1. **某次发送完成，先于对应接收完成。**
2. **关闭 Channel，先于因该关闭而返回零值的接收完成。**
3. **对无缓冲 Channel，接收完成，先于对应发送完成。**
4. **容量为 C 的 Channel，第 k 次接收完成，先于第 k+C 次发送完成。**

结合 Goroutine 内部的程序顺序，可以推导 happens-before。例如：

```go
var x int
ch := make(chan struct{})

go func() {
    x = 42
    ch <- struct{}{}
}()

<-ch
fmt.Println(x) // 能看到 42，不构成对 x 的 data race
```

但发送指针只复制指针值，并不会自动冻结或深拷贝对象：

```go
p := &Payload{}
ch <- p
p.Field++ // 若接收方同时访问 p.Field，仍可能 data race
```

“通过 Channel 传递指针”只有在业务协议明确完成所有权转移、发送后不再访问，或另有同步保护时才安全。

## 三、底层实现

### 3.1 从语法到 runtime 调用

**[当前实现]** 编译器会把 Channel 操作降级为 runtime 调用：

| Go 语法 | 编译器/runtime 入口（概念） |
|---|---|
| `make(chan T, n)` | `makechan` 或 `makechan64` |
| `ch <- v` | `chansend1` → `chansend` |
| `<-ch` | 接收辅助入口 → `chanrecv` |
| `close(ch)` | `closechan` |
| `len(ch)` / `cap(ch)` | `chanlen` / `chancap` |
| 一般 `select` | 构造 case 数组 → `selectgo` |
| 单个通信 case + `default` | 可能优化成 `selectnbsend` / `selectnbrecv` |
| `select {}` | 直接降级为永久阻塞入口 |

这说明源码阅读不能只看 `runtime/chan.go`：还要看编译器如何识别简单 `select` 并避免通用路径开销。

### 3.2 `hchan` 的核心结构

**[当前实现：Go 1.26.4]** 每个非 nil Channel 在 runtime 中由一个 `hchan` 对象表示。可用下面的概念图理解：

```text
hchan
├── qcount      当前缓冲区元素数
├── dataqsiz    缓冲区容量
├── buf         环形缓冲区起始地址
├── elemsize    单个元素大小
├── elemtype    元素类型元数据
├── closed      是否关闭
├── sendx       下一次入队索引
├── recvx       下一次出队索引
├── sendq       阻塞发送者队列
├── recvq       阻塞接收者队列
├── timer       Timer Channel 关联信息
└── lock        保护 Channel 状态与等待者字段
```

字段含义：

| 字段 | 作用 | 面试推导 |
|---|---|---|
| `qcount` | 当前已排队元素数 | 解释 `len(ch)` 的来源，但它只是瞬时快照 |
| `dataqsiz` | 环形缓冲区容量 | 0 表示无缓冲；不因 close 改变 |
| `buf` | 缓冲区内存地址 | 有缓冲 Channel 的值存储位置 |
| `sendx` | 下一发送槽位 | 到尾部后回绕，形成环形队列 |
| `recvx` | 下一接收槽位 | 保证单个发送序列的 FIFO |
| `sendq` | 等待发送的 `sudog` 队列 | 满缓冲或无接收者时发送者排队 |
| `recvq` | 等待接收的 `sudog` 队列 | 空缓冲或无发送者时接收者排队 |
| `closed` | 关闭标记 | close 后发送 panic，排空后接收零值 |
| `lock` | runtime mutex | 保护队列、索引、计数和关闭状态的一致性 |

runtime 源码维护的重要不变量是：通常至少有 `sendq`、`recvq` 中一个为空；对有缓冲 Channel，若缓冲区有数据，通常没有等待接收者；若缓冲区未满，通常没有等待发送者。极少数涉及同一 Goroutine 在 `select` 中同时等待同一 Channel 多个方向的情况会让不变量表述更复杂。上图只列面试相关核心字段，省略了当前版本用于内部测试等用途的非核心字段。

### 3.3 `makechan` 与内存分配

`makechan` 首先检查：

- 元素大小能否由 runtime 字段表示。
- 元素对齐是否满足要求。
- `capacity × element size` 是否溢出或超过可分配上限。

随后根据容量与元素是否包含指针选择分配方式：

1. **容量为 0，或元素大小为 0**：主要分配 `hchan` 本体；零大小元素不需要真实数据区。
2. **元素不含指针**：当前实现可将 `hchan` 与缓冲区放在一次连续分配中，减少分配和 GC 扫描负担。
3. **元素包含指针**：`hchan` 与按元素类型分配的缓冲区分开处理，使 GC 能正确扫描缓冲区中的指针。

这也是为什么“大容量、含大量指针的 Channel”会明显增加堆保留与 GC 扫描成本。

### 3.4 `waitq` 与 `sudog`

`sendq`、`recvq` 都是 `waitq`，内部维护队头和队尾，队列节点为 `sudog`。

`sudog` 不是 Goroutine 本身，而是“某个 Goroutine 在某个同步对象上的一次等待记录”。一个 Goroutine 可同时出现在多个等待关系中，例如一般 `select` 会为多个 case 分别准备等待节点，因此不能把等待链表直接嵌入 G。

概念字段包括：

```text
sudog
├── g          等待的 Goroutine
├── elem       发送值或接收目标的地址，可能指向 Goroutine 栈
├── next/prev  Channel 等待队列链表
├── isSelect   是否来自 select
├── success    唤醒原因是否为成功通信
└── c          关联 Channel
```

当前实现中 `elem` 使用可被 runtime 正确追踪的指针包装，而不是简单把它当普通 Go 堆指针。源码特别谨慎，是因为它可能指向另一个 Goroutine 的栈。

### 3.5 `chan.lock` 为什么存在

多个 Goroutine 可并发发送、接收和查询 Channel，runtime 必须原子地维护：

- 关闭状态。
- 缓冲区计数及索引。
- 发送/接收等待队列。
- `sudog` 中与 Channel 交互的关键字段。

因此当前实现用 `hchan.lock` 串行化关键状态变化。**“Channel 可并发使用”不代表“完全无锁”**。

runtime 还遵守一个重要实现约束：通常不会在持有 `hchan.lock` 时直接把另一个 G 置为 runnable，因为唤醒路径可能与栈收缩等机制交互。常见做法是先收集待唤醒 G，释放锁后再 `goready`。

### 3.6 无缓冲 Channel 的直接交接

无缓冲 Channel 没有环形队列。若发送方发现已有接收者等待，可直接把发送值复制到接收者目标地址：

```text
Sender stack/value
        │
        │ runtime direct copy
        ▼
Receiver stack/destination
```

大致流程：

1. 发送方锁住 `hchan`。
2. 从 `recvq` 取出等待接收者的 `sudog`。
3. 将发送值直接复制到接收者目标地址。
4. 标记等待成功。
5. 解锁。
6. 将接收者 Goroutine 置为 runnable。

反向也类似：接收方若发现 `sendq` 有等待发送者，可从发送者地址直接取值。

栈到栈复制不是普通 Go 代码能随意做的操作。runtime 必须配合写屏障和栈移动规则，确保 GC 与并发栈管理正确。

### 3.7 有缓冲 Channel 的环形队列

当没有可直接配对的等待者且缓冲区有空间，发送流程为：

```text
buf[sendx] = value
sendx = (sendx + 1) % dataqsiz
qcount++
```

接收流程为：

```text
value = buf[recvx]
clear(buf[recvx])
recvx = (recvx + 1) % dataqsiz
qcount--
```

清空槽位很重要：若元素含指针，不清理会让已消费对象继续被缓冲区引用，延长其生命周期。

一个容易被忽略的路径是：缓冲区已满且有发送者等待时，接收方取走队首元素后，可把等待发送者的值直接填回腾出的槽位，并唤醒发送者。这样在持锁期间完成“出队一个、补入一个”，维持满缓冲区状态并减少额外步骤。

### 3.8 `chansend` 主流程

可以把 `chansend` 概括为：

```text
chansend(ch, elem, block)
  ├─ ch == nil ?
  │    ├─ 非阻塞：失败返回
  │    └─ 阻塞：gopark，永不被 Channel 唤醒
  ├─ 非阻塞快速判断：当前无法发送则返回 false
  ├─ lock(ch)
  ├─ closed ? panic
  ├─ recvq 有等待者？
  │    └─ 直接交接，解锁并唤醒接收者
  ├─ qcount < dataqsiz？
  │    └─ 写入 buf[sendx]，推进索引，解锁返回
  ├─ 非阻塞？
  │    └─ 解锁，返回 false
  └─ 封装 sudog → 加入 sendq → gopark
       ├─ 被接收者唤醒：发送成功
       └─ 被 close 唤醒：panic
```

非阻塞快速路径读到的是瞬时状态，它只用于判断“此刻不可能立即完成”，真正修改状态仍需锁内复核。源码中的无锁检查不能被照搬成业务代码中的 `if len(ch) < cap(ch)`，后者无法与后续发送组成原子操作。

### 3.9 `chanrecv` 主流程

```text
chanrecv(ch, elem, block)
  ├─ ch == nil ?
  │    ├─ 非阻塞：失败返回
  │    └─ 阻塞：gopark
  ├─ 非阻塞快速判断
  ├─ lock(ch)
  ├─ closed && qcount == 0？
  │    └─ 目标清零，返回 selected=true, received=false
  ├─ sendq 有等待者？
  │    └─ 直接交接，或“取缓冲头 + 发送者补缓冲尾”
  ├─ qcount > 0？
  │    └─ 从 buf[recvx] 取值并清槽，推进索引
  ├─ 非阻塞？
  │    └─ 解锁，返回未选中
  └─ 封装 sudog → 加入 recvq → gopark
       ├─ 正常发送唤醒：received=true
       └─ close 唤醒：零值，received=false
```

这里的两个布尔概念要区分：

- 是否选中了该操作，供非阻塞 `select` 等内部路径使用。
- 是否收到真实发送值，对应用户代码中的 `ok`。

### 3.10 `closechan` 主流程

```text
closechan(ch)
  ├─ ch == nil → panic
  ├─ lock(ch)
  ├─ already closed → unlock + panic
  ├─ closed = 1
  ├─ 取出全部 recvq 等待者
  │    ├─ 接收目标清零
  │    └─ success = false
  ├─ 取出全部 sendq 等待者
  │    └─ success = false
  ├─ unlock(ch)
  └─ goready 所有等待者
       ├─ 接收者返回零值、ok=false
       └─ 发送者恢复后 panic
```

**[当前实现]** close 的成本与当时等待在 Channel 上的发送者和接收者数量相关，可近似看作 `O(S+R)`。这是实现特征，不是规范给出的复杂度承诺。

close 不遍历并销毁缓冲区。已有缓冲值继续按 FIFO 接收；只有排空后才返回零值和 `ok=false`。

### 3.11 `gopark`、`goready` 与调度器协作

当 Channel 操作必须阻塞时，runtime 不会让当前 G 在 CPU 上自旋等待，而是：

1. 准备 `sudog`，记录等待原因和通信地址。
2. 将 `sudog` 加入 Channel 等待队列。
3. 调用 `gopark`，把当前 G 从 running 转为 waiting。
4. M/P 可继续执行其他 runnable G。
5. 通信对端或 close 找到等待者后调用 `goready`。
6. G 进入 runnable 队列，之后由调度器安排再次运行。

因此，“Channel 阻塞”通常是 Goroutine 级阻塞，不等于操作系统线程一直睡在该 Channel 上。它仍有入队、调度、上下文切换和缓存失效成本，不能把阻塞当成免费操作。

### 3.12 `selectgo` 的大致流程

一般 `select` 的当前 runtime 流程可分为三遍：

```text
编译器准备 scase 数组
        │
        ▼
1. 生成 pollorder：随机化 case 扫描顺序
2. 生成 lockorder：按 Channel 地址排序
3. 按统一顺序锁住涉及的 Channel
4. 第一遍：扫描是否有立即就绪 case
   ├─ 有：完成一个通信，解锁，返回所选 case
   └─ 无且有 default：解锁，返回 default
5. 第二遍：为各通信 case 建立 sudog 并分别入队
6. gopark 当前 G
7. 某一个 case 唤醒 G
8. 第三遍：重新加锁，移除其余未胜出的 sudog
9. 解锁并返回胜出 case
```

#### 为什么要统一加锁顺序

一个 `select` 可能同时操作多个 Channel。若不同 Goroutine 按源码顺序随意加锁：

```text
G1: lock(A) → lock(B)
G2: lock(B) → lock(A)
```

会形成 runtime 内部死锁。当前实现按 Channel 的稳定排序键生成 `lockorder`，所有 `select` 以相同全局顺序加锁；同一 Channel 出现在多个 case 时只需按规则处理一次锁。

当前实现为构造 lock order 使用堆排序，因此一般 `select` 除线性扫描外还包含约 `O(n log n)` 的排序工作；小型和简单 `select` 可被编译器走专用路径。复杂度属于版本实现细节。

#### 为什么不能承诺严格公平

规范只规定在“当前同时就绪”的通信中做均匀伪随机选择。它没有保证：

- 每个 case 在 N 次内必定被选中。
- 先等待者一定先服务。
- 一个热 Channel 不会长期压制偶发就绪 Channel。
- Goroutine 调度本身绝对公平。

若业务要求租户级、任务级或权重级公平，应显式实现队列、配额、轮询或调度器，不能把公平性外包给 `select`。

### 3.13 时间与空间复杂度

| 操作 | 常见当前实现成本 | 说明 |
|---|---:|---|
| 无等待者且有缓冲空间的 send | `O(1)` | 锁、复制、索引更新 |
| 缓冲区有值的 receive | `O(1)` | 锁、复制、清槽、索引更新 |
| 已有对端等待的直接交接 | `O(1)` + 调度成本 | 元素复制及唤醒 |
| 需要阻塞的 send/receive | `O(1)` 入队 + 调度成本 | 后续还包含唤醒与重新调度 |
| close | `O(S+R)` | 当前等待发送者与接收者都要处理 |
| 一般 `select` | 约 `O(n log n)` + 排队成本 | 当前实现需构造锁顺序；简单形式可能优化 |
| Channel 空间 | `O(capacity × elementSize + waiters)` | 还包括 `hchan`、`sudog` 与被引用对象 |

元素复制成本也不可忽略。向 `chan [1<<20]byte` 发送会复制大值；传指针减少复制，却把所有权和 data race 风险交给业务协议。

### 3.14 Channel 是否一定比 Mutex 慢

不能脱离问题模型比较：

- 若只是保护一个计数器或 Map，Mutex/Atomic 通常更直接，额外通信与调度可能没有价值。
- 若要转移工作、串行化状态所有权、构建 Pipeline 或自然表达背压，Channel 可以减少共享可变状态，整体设计可能更快、更容易正确。
- 在高争用下，两者都可能发生等待和调度；Channel 还涉及值复制、队列和协议成本，Mutex 则可能发生锁竞争、缓存行抖动和临界区串行化。
- 性能结论必须用接近真实负载的 benchmark、block/mutex profile 和 execution trace 验证。

正确面试回答不是“Channel 慢”或“不要通过共享内存通信”，而是：

> **Channel 更适合通信、所有权转移和流程编排；Mutex 更适合保护同一进程内的共享状态。先选语义最清晰的工具，再测量关键路径。**

### 3.15 版本变化与边界总结

1. **规范语义稳定**：nil、close、发送/接收、`select` 和 `range` 的核心规则不依赖 `hchan` 具体布局。
2. **runtime 字段会演化**：例如当前 `hchan` 包含 Timer 和测试同步相关字段，旧文章的结构体截图可能已不完整。
3. **`sudog.elem` 的具体类型会演化**：应理解它“记录通信值地址并可能指向栈”，不要死记某个旧版本字段声明。
4. **Timer Channel 在 Go 1.23 有重大变化**：旧版关于 `time.After`、`len(timer.C)`、Stop/Reset 排空的经验不能无条件套用。
5. **启用 Go 1.22 新循环变量语义后，`for range` 中用 `:=` 声明的迭代变量按迭代创建**；这影响闭包捕获，但不改变 Channel 的关闭与排空语义。
6. **Timer Channel 的 API 视图可能与内部缓冲不同**：启用 Go 1.23 新 Timer 语义时，`len`/`cap` 对外报告 0；不要从当前 `hchan.dataqsiz` 反推语言层可观察行为。

| 15 | `proc.go` | `gopark`、`goready`、`ready` | G 状态迁移与 run queue | Channel 阻塞为何不等于线程阻塞 |
| 16 | `runtime/select.go` | `selectgo` | pollorder、lockorder、三遍扫描 | 公平性、锁序、复杂度、清理败选 case |
| 17 | Timer 文档与源码 | Timer Channel | Go 1.23 语义切换 | 如何纠正旧版 `time.After` 面经 |

### 4.3 阅读源码时的五个问题

每看一个函数，都用以下问题约束阅读：

1. **入口前提是什么？** 调用者是否已求值、是否持锁、是否允许阻塞？
2. **线性化点在哪里？** 发送、接收、关闭真正对其他 Goroutine 生效的时刻是什么？
3. **快路径与慢路径如何分开？** 哪些路径不 park，哪些需要 `sudog`？
4. **如何与 GC、栈和调度器协作？** 是否持有栈指针、何时唤醒、是否有写屏障？
5. **这是规范还是实现？** 字段名、队列算法、复杂度和锁序都可能变化；panic、阻塞和关闭语义才是稳定契约。

## 五、常用场景

### 5.1 Worker Pool：限制并发并集中回收结果

Worker Pool 适合任务数量较多、单任务相互独立、需要限制并发度的场景。

```go
package pool

import (
    "context"
    "sync"
)

type Job struct {
    ID int
}

type Result struct {
    JobID int
    Err   error
}

type Handler func(context.Context, Job) error

func Run(
    ctx context.Context,
    workers int,
    jobs <-chan Job,
    handle Handler,
) <-chan Result {
    results := make(chan Result)

    if workers <= 0 {
        close(results)
        return results
    }

    var wg sync.WaitGroup
    wg.Add(workers)

    for i := 0; i < workers; i++ {
        go func() {
            defer wg.Done()

            for {
                select {
                case <-ctx.Done():
                    return
                case job, ok := <-jobs:
                    if !ok {
                        return
                    }

                    result := Result{
                        JobID: job.ID,
                        Err:   handle(ctx, job),
                    }

                    // 消费者退出或取消后，worker 不能永久卡在发送结果。
                    select {
                    case results <- result:
                    case <-ctx.Done():
                        return
                    }
                }
            }
        }()
    }

    // 唯一关闭者：等所有 worker 确认不再发送后关闭。
    go func() {
        wg.Wait()
        close(results)
    }()

    return results
}
```

设计要点：

- `jobs` 的关闭责任属于其上游生产者或协调者，不属于 worker。
- 所有 worker 都可能发送 `results`，因此由独立协调者 `wg.Wait()` 后关闭结果 Channel。
- 发送结果也必须监听取消，否则调用方提前退出时 worker 会泄漏。
- worker 数不是越多越好：CPU 密集任务通常接近可用 CPU；I/O 密集任务应根据下游并发限制、连接池、延迟和负载测试设定。
- 一个共享工作 Channel 不提供严格任务公平或租户隔离；有此要求应增加分租户队列和调度层。

### 5.2 Pipeline：把处理阶段串成可取消的数据流

```go
func generate(ctx context.Context, values ...int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for _, v := range values {
            select {
            case out <- v:
            case <-ctx.Done():
                return
            }
        }
    }()
    return out
}

func square(ctx context.Context, in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for {
            select {
            case <-ctx.Done():
                return
            case v, ok := <-in:
                if !ok {
                    return
                }
                select {
                case out <- v * v:
                case <-ctx.Done():
                    return
                }
            }
        }
    }()
    return out
}
```

Pipeline 的关键不是“把几个 Channel 连起来”，而是建立完整生命周期：

1. 每个阶段知道输入何时结束。
2. 每个阶段只关闭自己的输出。
3. 下游提前退出时，取消能传播到所有上游。
4. 阶段内部的阻塞 I/O 也应接受 Context，否则仅在 Channel 发送处监听取消仍无法及时退出。

不带取消的 Pipeline 很容易出现经典泄漏：下游只取一个结果就返回，上游仍永久阻塞在向下游发送。

### 5.3 Fan-out / Fan-in

- **Fan-out**：多个 worker 从同一输入 Channel 竞争任务，提高并行度。
- **Fan-in**：把多个结果 Channel 汇合到一个输出 Channel，统一消费。

Fan-in 的关闭协议通常是：每个转发 Goroutine 不关闭共享输出；协调者等待全部转发结束后关闭输出。前文 `merge` 示例正是这一模式。

适用：

- 同质、可并行、顺序要求较弱的任务。
- 每个输入流最终可关闭或可通过 Context 取消。

不适用：

- 必须严格保持全局输入顺序。
- 不同租户需要强公平或配额。
- 任务有依赖图，简单竞争消费会破坏先后关系。

### 5.4 Semaphore：用容量表达并发上限

```go
func runLimited(
    ctx context.Context,
    limit int,
    tasks []func(context.Context) error,
) error {
    if limit <= 0 {
        return fmt.Errorf("limit must be positive")
    }

    sem := make(chan struct{}, limit)
    // 容量覆盖最多一次/任务的结果，避免等待 wg 时 worker 卡在汇报错误。
    errCh := make(chan error, len(tasks))
    var wg sync.WaitGroup
    var launchErr error

launchLoop:
    for _, task := range tasks {
        task := task

        select {
        case sem <- struct{}{}: // acquire
        case <-ctx.Done():
            launchErr = ctx.Err()
            break launchLoop
        }

        wg.Add(1)
        go func() {
            defer wg.Done()
            defer func() { <-sem }() // release，必须与成功 acquire 配对
            errCh <- task(ctx)
        }()
    }

    // 即使停止继续启动，也回收已经启动的 Goroutine。
    wg.Wait()
    close(errCh)

    var firstTaskErr error
    for err := range errCh {
        if err != nil && firstTaskErr == nil {
            firstTaskErr = err
        }
    }
    if launchErr != nil {
        return launchErr
    }
    return firstTaskErr
}
```

容量为 N 的 Channel 可表达最多 N 个并发许可。它利用了有缓冲 Channel 的阻塞与 memory model 规则。

边界：

- 许可没有权重，不适合每个任务消耗不同资源量的场景。
- 不保证等待者的业务级严格公平。
- 忘记 release 会永久耗尽许可。
- 任务数极大时，“先为每个任务创建 Goroutine，再在 Goroutine 内 acquire”会产生大量等待 G；更稳妥的是 acquire 成功后再创建，或使用固定 worker pool。

### 5.5 背压：让过载显式化

Channel 的容量定义了生产者可领先消费者多少项：

| 设计 | 过载时行为 | 适用场景 | 风险 |
|---|---|---|---|
| 无缓冲 | 生产者立即跟随消费者速度 | 严格交接、低积压 | 吞吐受握手和调度影响 |
| 小型有界缓冲 | 吸收短突发，满后阻塞 | 大多数进程内队列 | 需选择容量和超时策略 |
| 非阻塞发送 + 丢弃 | 满时降级或丢样 | 指标、日志采样、可丢事件 | 必须统计丢弃量 |
| 超时发送 | 等待有限时间后失败 | 延迟有 SLO 的调用链 | 需明确重试和幂等 |
| 无界内存队列 | 生产者长期不受限 | 极少数可控场景 | OOM、GC 压力、尾延迟扩大 |
| 持久化消息系统 | 落盘、跨进程、重放 | 可靠异步业务 | 运维与一致性成本更高 |

**缓冲区不能修复长期吞吐不平衡。** 若平均到达率持续大于处理率，再大的有限缓冲区也只会推迟满载时间；无界队列则把“阻塞”改成“内存耗尽”。

生产设计应明确满载策略：

- 阻塞并向上游传播背压。
- 超时失败。
- 丢弃最新、丢弃最旧或采样。
- 降级处理。
- 溢写磁盘或外部消息系统。
- 水位触发扩容或限流。

### 5.6 有界队列与无界队列

Go 原生 Channel 必须有固定容量，因此天然是有界队列。所谓“无界 Channel”通常是一个中间 Goroutine 维护可增长 Slice：

```text
producers → input channel → queue goroutine + growing slice → output channel → consumers
```

它并不真正无界，只是把上限推到进程内存。风险包括：

- Slice 扩容和对象引用长期保留。
- GC 扫描成本与暂停辅助工作增加。
- 消费延迟随队列长度增长。
- 中间 Goroutine 成为串行瓶颈。
- 进程重启后队列丢失。

除非业务已经定义最大内存、水位、丢弃、落盘和观测策略，否则应优先使用有界队列。

### 5.7 Context 取消：避免单向通知协议失控

Channel 适合传数据；Context 适合跨调用边界传取消、截止时间和请求级元数据。

发送端的基本模式：

```go
select {
case out <- value:
    return nil
case <-ctx.Done():
    return ctx.Err()
}
```

接收端的基本模式：

```go
select {
case value, ok := <-in:
    if !ok {
        return nil
    }
    return consume(value)
case <-ctx.Done():
    return ctx.Err()
}
```

注意：

- 不要关闭 `ctx.Done()`；它由 Context 实现管理。
- `Done()` 可能为 nil，例如 `context.Background()`；放在 `select` 中正好自动禁用取消分支。
- 取消只是一条信号。下游函数必须实际检查 Context，无法被取消的系统调用仍可能长期阻塞。
- `context.WithCancel`、`WithTimeout` 等返回的 cancel 函数应调用，以释放关联资源。

### 5.8 Channel 缓冲区大小如何选择

不能用“CPU 核数”“经验取 1000”作为通用答案。应从四个量出发：

1. 到达速率 `λ`。
2. 服务速率 `μ`。
3. 允许吸收的突发持续时间 `T`。
4. 单个排队元素实际保留的内存 `M`。

若突发期 `λ > μ`，吸收该突发所需容量可先估算为：

```text
capacity ≈ (λ - μ) × T
```

Little's Law `L = λW` 可用于估计稳定系统中的平均在途量，但它不是容量上界保证。最终容量还要同时满足：

- 排队等待时间不突破 SLO。
- `capacity × M` 在内存预算内。
- 满载策略可接受。
- 下游故障时不会无限等待。
- 实际压测中的吞吐、P99/P999 延迟和 GC 指标合理。

建议从小容量起步，通过队列占用率直方图、发送阻塞时间、消费延迟和丢弃量逐步调参。`len(ch)` 可用于观测趋势，但不能用于同步决策。

### 5.9 传值、传指针与所有权

| 方式 | 优点 | 代价/风险 | 建议 |
|---|---|---|---|
| 传小值 | 语义清晰，天然快照 | 发生值复制 | 默认优先 |
| 传大值 | 无共享对象 race | 复制成本高，Channel 缓冲占用大 | 衡量大小，必要时改指针或索引 |
| 传指针 | 复制成本小 | 共享可变对象、生命周期延长、data race | 明确所有权转移或加锁 |
| 传不可变句柄/ID | 队列轻量，状态集中管理 | 需额外查表 | 大对象、缓存对象常用 |

“发送后不再访问，接收方独占”是常见所有权转移约定，但 Go 类型系统不会强制它。若发送者继续修改对象，Channel 只同步了指针值的传递，并未同步未来任意修改。

### 5.10 Channel 与 Mutex 如何选择

| 问题形态 | 更自然的工具 | 原因 |
|---|---|---|
| 保护 Map、计数器、缓存元数据 | `sync.Mutex` / `RWMutex` / Atomic | 共享状态本身需要原地访问 |
| 把任务交给固定拥有者处理 | Channel | 转移工作与状态所有权 |
| Pipeline、Worker Pool、Fan-in | Channel | 流程和背压自然表达 |
| 等待一组任务完成 | `sync.WaitGroup` / `errgroup` | 不需要自行设计完成计数协议 |
| 一次性初始化 | `sync.Once` | 语义直接，避免 close 协议误用 |
| 多条件等待共享状态变化 | Mutex + `sync.Cond` | 条件与共享状态在同一锁下检查 |
| 高频单值计数/状态位 | `sync/atomic` | 避免不必要的队列与调度 |
| 跨进程可靠消息 | 消息队列/日志系统 | Channel 不持久、不跨进程、不可重放 |

不要为了遵循口号把所有共享状态都搬进“actor Goroutine + Channel”。单拥有者模型可以减少锁，但也可能制造单点瓶颈、复杂请求/响应协议和难以取消的队列。正确选择应基于不变量、所有权、性能和可维护性。

### 5.11 关闭广播与发送一个信号

| 需求 | 推荐方式 |
|---|---|
| 通知一个消费者执行一次工作 | 发送一个值 |
| 通知所有观察者永久停止 | 关闭只读广播 Channel，或取消 Context |
| 重复发布状态变化 | 发送事件，或使用受锁状态 + 通知机制 |
| 必须携带错误原因 | Context cause、错误 Channel，或结构化结果 |
| 需要可重置广播 | 不能“重新打开”已关闭 Channel；需创建新一代 Channel 或使用其他原语 |

已关闭 Channel 永远保持关闭，不能复用。把 close 当成可重置事件是协议设计错误。

### 5.12 不适合使用进程内 Channel 的场景

- 需要持久化、重放、事务、跨进程或跨机器可靠投递。
- 需要严格优先级、租户公平和复杂调度策略。
- 需要随机访问、更新和查询一个共享数据结构。
- 不能接受进程崩溃后在途消息丢失。
- 工作量可能无限积压而又没有背压或丢弃策略。
- 需要向动态大量订阅者广播并管理慢订阅者；单个 Channel 的竞争接收不是发布订阅。

## 六、代码陷阱题

> 以下题目默认按 Go 1.26.4 语义分析。判断时先区分：**编译错误、panic、当前 Goroutine 阻塞、全局死锁、data race、Goroutine 泄漏、忙循环**。

### 题 1：向 nil Channel 发送

```go
package main

func main() {
    var ch chan int
    ch <- 1
}
```

**问题**：程序输出什么？会 panic、阻塞还是退出？

**标准答案**：`ch` 为 nil，发送永久阻塞。此程序没有其他可运行 Goroutine 或未来唤醒事件，runtime 最终报告全局死锁，通常看到 `fatal error: all goroutines are asleep - deadlock!`。

**逐行分析**：

1. `var ch chan int` 得到 Channel 零值 nil，没有分配 `hchan`。
2. `ch <- 1` 的值表达式先求值，然后通信尝试开始。
3. nil Channel 不可能就绪；阻塞发送路径会 park 当前 G，且没有任何 Channel 对端能唤醒它。
4. main G 被 park 后，全程序无进展，触发 runtime 死锁检测。

**追问方向**：若同一进程还有 HTTP 服务 Goroutine，是否仍会立刻报全局死锁？答案是不一定；这个 G 可能只是永久泄漏。nil Channel 放进带 `default` 的 `select` 又会怎样？该 case 被禁用，走 `default`。

---

### 题 2：从 nil Channel 接收

```go
package main

func main() {
    var ch <-chan string
    _ = <-ch
}
```

**问题**：只接收 Channel 的 nil 值是否会返回空字符串？

**标准答案**：不会。nil Channel 接收永久阻塞；本程序最终报告全局死锁。只有**已关闭且排空**的非 nil Channel 才会立即返回元素零值。

**逐行分析**：

1. `<-chan string` 的方向不改变零值，`ch` 仍为 nil。
2. `_ = <-ch` 不是“读取零值变量”，而是一次 Channel 通信。
3. nil Channel 永远没有发送者，也不存在关闭状态可观察，因此接收不会完成。

**追问方向**：nil 与 closed 的核心区别是什么？nil 表示“没有通信对象，永不就绪”；closed 表示“对象存在且永久进入结束状态”，其接收永远就绪。

---

### 题 3：关闭 nil Channel

```go
package main

func main() {
    var ch chan int
    close(ch)
}
```

**问题**：是静默无操作、阻塞还是 panic？

**标准答案**：运行时 panic：关闭 nil Channel。

**逐行分析**：

1. `ch` 的动态值为 nil，但静态类型允许调用 `close`。
2. `close` 只允许接收一个可发送方向的 Channel；这里类型检查通过。
3. runtime 的 `closechan` 首先检查 nil，发现没有可关闭对象，直接 panic。

**追问方向**：为什么不把 close nil 设计成幂等？因为 close 表示一次协议状态转移；静默吞掉 nil/重复 close 会掩盖所有权和生命周期错误。用 `recover` 包装“安全关闭”是否推荐？通常不推荐，它不能消除并发 send/close。

---

### 题 4：反复接收已关闭的空 Channel

```go
package main

import "fmt"

func main() {
    ch := make(chan int)
    close(ch)

    a, ok1 := <-ch
    b, ok2 := <-ch
    fmt.Println(a, ok1, b, ok2)
}
```

**问题**：输出是什么？第二次接收会 panic 吗？

**标准答案**：输出 `0 false 0 false`。从已关闭且排空的 Channel 接收永不阻塞，也不会 panic，每次都返回元素零值和 `ok=false`。

**逐行分析**：

1. `make(chan int)` 创建非 nil、无缓冲 Channel。
2. `close(ch)` 把状态设为 closed，此时没有缓冲数据。
3. 第一次接收命中“closed + empty”，得到 `0, false`。
4. closed 状态不可逆，第二次仍得到 `0, false`。

**追问方向**：为什么只写 `v := <-ch` 容易误判？因为合法发送的 0 与关闭返回的 0 无法区分。关闭 Channel 放在 `select` 中为什么可能导致 busy loop？因为接收 case 永远就绪。

---

### 题 5：关闭有缓冲 Channel 后如何接收

```go
package main

import "fmt"

func main() {
    ch := make(chan int, 2)
    ch <- 10
    ch <- 20
    close(ch)

    for i := 0; i < 3; i++ {
        v, ok := <-ch
        fmt.Println(v, ok)
    }
}
```

**问题**：三行输出分别是什么？close 会清空缓冲区吗？

**标准答案**：

```text
10 true
20 true
0 false
```

close 不清空缓冲区。已有值继续按 FIFO 被接收，且 `ok=true`；排空后才返回零值和 `ok=false`。

**逐行分析**：

1. 两次发送填满容量为 2 的环形缓冲区。
2. close 只设置结束状态并唤醒等待者，不删除 `10`、`20`。
3. 前两次接收对应真实发送值。
4. 第三次接收观察到 closed 且 `qcount==0`。

**追问方向**：`for range ch` 会在 close 时立即退出吗？不会，先排空。若缓冲元素是指针，未消费的数据会怎样？只要 Channel 可达，缓冲槽中的指针就会继续保留对象。

### 题 6：向已关闭 Channel 发送时的求值顺序

```go
package main

import "fmt"

func value() int {
    fmt.Println("evaluate value")
    return 1
}

func main() {
    ch := make(chan int)
    close(ch)
    ch <- value()
}
```

**问题**：会先 panic，还是先打印？

**标准答案**：先打印 `evaluate value`，随后 panic：向已关闭 Channel 发送。

**逐行分析**：

1. 发送语句进入时，Channel 操作数和发送右值按规范先求值。
2. `value()` 的副作用因此已经发生。
3. runtime 在真正执行发送时检查到 `closed`，触发 panic。
4. 不会有任何值进入 Channel。

**追问方向**：`select` 中多个发送 case 的右值是否只求值被选中的一个？不是；进入 `select` 时所有发送右值都会先求值一次。如何避免昂贵或有副作用的求值？在 `select` 外预计算或重新设计控制流。

### 题 7：重复关闭

```go
package main

func main() {
    ch := make(chan struct{})
    close(ch)
    close(ch)
}
```

**问题**：第二次 close 是幂等的吗？

**标准答案**：不是。第二次 close panic。

**逐行分析**：

1. 第一次 close 完成 open → closed 的唯一状态转移。
2. `hchan.closed` 已被标记。
3. 第二次 `closechan` 在锁内发现已经关闭，解锁并 panic。

**追问方向**：多个 Goroutine 都可能 close 时能否先 `if !closed(ch)`？Go 没有可靠的通用 `closed` 查询；即使有，检查与 close 之间仍有 TOCTOU。正确办法是单一关闭所有者或通过 WaitGroup/协调者证明所有发送结束。

### 题 8：`range` 未关闭导致阻塞

```go
package main

import "fmt"

func main() {
    ch := make(chan int)

    go func() {
        ch <- 1
        ch <- 2
        // 忘记 close(ch)
    }()

    for v := range ch {
        fmt.Println(v)
    }
}
```

**问题**：是否打印 1、2 后正常退出？

**标准答案**：通常打印 1、2，随后 main 在下一次接收处永久阻塞；发送 Goroutine 已退出，程序最终报告全局死锁。

**逐行分析**：

1. 发送 Goroutine 完成两次交接后返回。
2. `range` 并不知道“发送者函数已经返回”，它只观察 Channel 状态。
3. Channel 仍为 open，因此下一轮接收等待未来发送。
4. 没有任何 Goroutine 再发送或关闭，无法进展。

**追问方向**：是否所有消费循环都必须 close？若接收方通过 Context 或固定次数退出，不一定；但使用 `range` 表达“直到数据流结束”时，上游必须提供关闭协议。接收方能否自行 close？通常不能，因为它无法证明没有并发发送。

### 题 9：`select` 中已关闭 Channel 的 busy loop

```go
package main

func main() {
    ch := make(chan int)
    close(ch)

    for {
        select {
        case _, ok := <-ch:
            if !ok {
                continue
            }
        }
    }
}
```

**问题**：程序会阻塞、死锁还是持续占用 CPU？

**标准答案**：它会形成忙循环。已关闭且排空的 Channel 接收永远就绪；`continue` 立即进入下一轮，再次选择同一 case。

**逐行分析**：

1. `close(ch)` 后接收无需等待。
2. `select` 每轮都有一个立即就绪 case，因此不会 `gopark`。
3. `ok=false` 后执行 `continue`，没有退出、休眠或禁用分支。
4. 循环高速执行，可能表现为单核接近 100% CPU。

**正确修复**：

```go
case _, ok := <-ch:
    if !ok {
        ch = nil // 多路输入时动态禁用；若无需继续，直接 return
        continue
    }
```

**追问方向**：把 case 改成 `<-ch` 而不检查 `ok` 会怎样？更隐蔽地永久就绪。goroutine dump 可能看到什么？该 G 多数时间是 runnable/running，而不是 `[chan receive]`，因此应结合 CPU profile 和源码定位。

### 题 10：用 nil 动态关闭 `select` 分支

```go
package main

import "fmt"

func main() {
    a := make(chan int, 1)
    b := make(chan int, 1)
    a <- 1
    b <- 2
    close(a)
    close(b)

    for a != nil || b != nil {
        select {
        case v, ok := <-a:
            if !ok {
                a = nil
                continue
            }
            fmt.Println("a", v)
        case v, ok := <-b:
            if !ok {
                b = nil
                continue
            }
            fmt.Println("b", v)
        }
    }
}
```

**问题**：会不会因为两个 Channel 都关闭而忙循环？输出顺序是否固定？

**标准答案**：不会忙循环，最终正常退出；会各打印一次 `a 1`、`b 2`，顺序不固定。

**逐行分析**：

1. 初始时两个接收 case 都就绪，`select` 伪随机选一个。
2. 对应缓冲值先被取出，`ok=true`。
3. 下一次选到该 Channel 时得到 `ok=false`，把局部变量设为 nil。
4. nil Channel case 不再就绪，相当于从后续选择中移除。
5. 两个变量都为 nil 后循环条件失败。

**追问方向**：若没有外层条件，两个 case 最终都变 nil 且无 `default` 会怎样？`select` 永久阻塞。动态 nil 技巧常用于 Fan-in，但必须另有退出条件或取消分支。

### 题 11：`default` 造成忙轮询

```go
package main

func main() {
    ch := make(chan int)

    for {
        select {
        case v := <-ch:
            _ = v
        default:
        }
    }
}
```

**问题**：没有发送者时，该 Goroutine 会在 Channel 上阻塞吗？

**标准答案**：不会。接收 case 未就绪时立即执行空 `default`，循环没有任何阻塞，形成忙轮询并持续消耗 CPU。

**逐行分析**：

1. `ch` 为正常无缓冲 Channel，但没有发送者。
2. 因存在 `default`，`select` 不会 park。
3. 空 `default` 立即返回，下一轮重复相同检查。
4. 这不是高效“监听”，而是用户态自旋。

**正确替代**：直接阻塞接收；或添加 Ticker/Timer、退避、`runtime.Gosched` 之外的真实等待机制。`runtime.Gosched` 只能让出执行权，不能把错误的轮询协议变成可靠同步。

**追问方向**：何时 `default` 是合理的？实现 try-send、try-receive、可丢指标或受控的非阻塞状态机时合理，但必须定义失败策略并监控丢弃/重试。

### 题 12：`len(ch)` 检查后再发送

```go
package main

import "sync"

func main() {
    ch := make(chan int, 1)
    checked := make(chan struct{}, 2)
    start := make(chan struct{})

    var wg sync.WaitGroup
    for i := 0; i < 2; i++ {
        wg.Add(1)
        go func(v int) {
            defer wg.Done()
            if len(ch) < cap(ch) {
                checked <- struct{}{}
                <-start
                ch <- v
            }
        }(i)
    }

    <-checked
    <-checked
    close(start)
    wg.Wait()
}
```

**问题**：两次 `len(ch) < cap(ch)` 都为真，是否保证两个发送都不阻塞？

**标准答案**：不保证。代码有意让两个 Goroutine 都在缓冲区为空时完成检查；放行后只有一个值能进入容量 1 的 Channel，另一个发送永久阻塞，`wg.Wait()` 也无法返回，最终全局死锁。

**逐行分析**：

1. `len` 和 `cap` 的调用本身允许并发，但结果只是调用瞬间的快照。
2. 两个 Goroutine 都在任何发送发生前看到 `0 < 1`。
3. `close(start)` 同时放行；第一个成功发送后，缓冲区满。
4. 第二个在 `ch <- v` 阻塞，说明“检查 + 操作”不是原子事务。
5. 这通常是逻辑竞态/TOCTOU；它不一定被 race detector 报为普通内存 data race。

**正确写法**：

```go
select {
case ch <- v:
    // 原子地完成 try-send
 default:
    // 满载策略：丢弃、计数、返回错误或重试
}
```

**追问方向**：`len(ch)==0` 后再接收是否同样不可靠？是。`len` 适合指标和调试，不适合建立同步正确性。即使 Channel 只有一个生产者，消费者仍可能改变长度。

### 题 13：多个发送者各自 close

```go
package main

import (
    "fmt"
    "sync"
)

func main() {
    ch := make(chan int)
    var wg sync.WaitGroup
    wg.Add(2)

    for i := 0; i < 2; i++ {
        go func(v int) {
            defer wg.Done()
            ch <- v
            close(ch) // 两个发送者都错误地认为自己有关闭权
        }(i)
    }

    for v := range ch {
        fmt.Println(v)
    }
    wg.Wait() // 确保错误发送者不能被 main 提前退出掩盖
}
```

**问题**：是否能稳定打印两个值后退出？

**标准答案**：不能。该版本用 `wg.Wait()` 保证两个发送者都必须走完，因此最终一定会因“重复 close”或“向已关闭 Channel 发送”而 panic；打印了一个还是两个值取决于调度。

**逐行分析**：

1. 两个 Goroutine 都认为自己拥有关闭权，但它们无法证明另一个已停止发送。
2. 若第一个发送完成并 close，第二个尚未发送，则第二个 send panic。
3. 若两个发送都恰好完成，两个 deferred close 中后执行的一个 panic。
4. `range` 的存在不构成发送者间协调。

**安全模式**：

```go
var wg sync.WaitGroup
wg.Add(2)
for i := 0; i < 2; i++ {
    go func(v int) {
        defer wg.Done()
        ch <- v
    }(i)
}
go func() {
    wg.Wait()
    close(ch)
}()
```

**追问方向**：用 `sync.Once` 包 close 是否足够？只能防止重复 close，不能阻止另一发送者与 close 并发。更根本的条件是 close 前已经证明所有 send 路径结束。

### 题 14：超时后消费者退出，发送者泄漏

```go
package service

import (
    "context"
    "time"
)

func waitResult(ctx context.Context) error {
    result := make(chan int)

    go func() {
        time.Sleep(100 * time.Millisecond)
        result <- 42
    }()

    select {
    case <-result:
        return nil
    case <-ctx.Done():
        return ctx.Err()
    }
}
```

**问题**：若 `ctx` 在 10ms 后取消，会发生什么？

**标准答案**：`waitResult` 返回，但后台 Goroutine 在约 100ms 后永久阻塞于 `result <- 42`。在长寿命服务中这是 Goroutine 泄漏，并可能连带保留栈上/堆上的请求对象。

**逐行分析**：

1. `result` 是无缓冲 Channel，发送必须有接收者配对。
2. Context 先取消，调用函数不再执行接收分支。
3. 工作 Goroutine 不知道取消，也没有备用接收者。
4. 它完成工作后卡在发送，Channel 和相关对象因该 G 的栈引用而继续存活。

**修复方式**：

```go
select {
case result <- 42:
case <-ctx.Done():
    return
}
```

对于严格的一次性结果，也可使用容量 1 的结果 Channel，使工作 Goroutine 能完成一次无人接收的发送；但仍应让昂贵工作本身响应取消。

**追问方向**：main 返回时这个 Goroutine 会怎样？进程直接结束，不会优雅清理；因此短小示例可能掩盖服务中的泄漏。如何检测？观察 goroutine 数趋势并抓取 dump，常见栈顶为 `[chan send]`。

### 题 15：无缓冲结果 Channel 的经典泄漏

```go
package query

import (
    "context"
    "time"
)

type Result struct {
    Value string
    Err   error
}

// 示例中的慢查询；真实代码应让底层调用接收并响应 ctx。
func slowQuery() (string, error) {
    time.Sleep(200 * time.Millisecond)
    return "ok", nil
}

func Query(ctx context.Context) Result {
    out := make(chan Result)

    go func() {
        value, err := slowQuery()
        out <- Result{Value: value, Err: err}
    }()

    select {
    case r := <-out:
        return r
    case <-time.After(50 * time.Millisecond):
        return Result{Err: context.DeadlineExceeded}
    }
}
```

**问题**：超时后 `slowQuery` 返回，工作 Goroutine 是否能退出？

**标准答案**：不能。若超时分支先返回，`out` 再无接收者，无缓冲发送永久阻塞。

**逐行分析**：

1. 启动 Goroutine 本身不会继承“父子生命周期”。
2. `time.After` 只让调用方停止等待，不会自动取消 `slowQuery`。
3. 返回后，局部变量看似离开作用域，但工作 G 仍持有 `out`。
4. 发送不完成，工作 G 及其引用对象都无法回收。

**适合一次性结果的改法**：

```go
out := make(chan Result, 1)
```

再配合把 `ctx` 传给 `slowQuery`。容量 1 能容纳这一条最终结果，使发送方不依赖调用方仍在接收；Context 则避免超时后继续浪费外部资源。

**追问方向**：为什么不能只在超时后 `close(out)`？调用方一旦关闭，工作 G 随后发送会 panic。数据 Channel 的关闭权仍应由确认不再发送的一方持有。

### 题 16：容量 1 是否一定解决泄漏

```go
package main

import "time"

func startWorkers() <-chan int {
    out := make(chan int, 1)

    for i := 0; i < 2; i++ {
        go func(v int) {
            time.Sleep(100 * time.Millisecond)
            out <- v
        }(i)
    }

    return out
}

func main() {
    _ = startWorkers()
    time.Sleep(time.Second)
}
```

**问题**：容量 1 是否保证两个 worker 都能退出？

**标准答案**：不保证。没有接收者时，最多一个结果进入缓冲区；另一个 worker 在进程继续运行期间无法自行退出，本示例中会一直阻塞到 main 结束进程。容量 1 只对“恰好一个可能被放弃的一次性发送”有效。

**逐行分析**：

1. 两个 worker 最终都执行一次发送。
2. Channel 容量只有 1，且 main 丢弃返回值，不接收。
3. 一个发送成功并退出；另一个看到缓冲区满而 park。
4. 缓冲中的值还会保留其引用对象，直到 Channel 不再可达；被阻塞 G 又会继续持有 Channel。

**追问方向**：容量应该改成 worker 数吗？这只能处理当前已知数量的单次发送，仍不能代替取消和消费协议。若每个 worker 可能发送多次、数量动态变化或结果很大，大缓冲只会把泄漏推迟并放大内存保留。

### 题 17：循环中的 `time.After`

```go
func run(jobs <-chan int) {
    for {
        select {
        case job, ok := <-jobs:
            if !ok {
                return
            }
            _ = job // 处理任务
        case <-time.After(30 * time.Second):
            // 上报空闲
        }
    }
}
```

**问题**：这段代码在 Go 1.26.4 中是否“一定发生 Timer 内存泄漏”？

**标准答案**：不能这样绝对表述。Go 1.23+ 中，不再可达的未停止 Timer/Ticker 可以被 GC 回收，旧版“每个未触发 `time.After` 都无法回收直到到期”的结论已过时。但每轮仍创建一个新 Timer；在 `jobs` 高频就绪时会产生显著分配、定时器维护和 GC 开销，热路径更适合复用 `time.Timer`。

**逐行分析**：

1. 每次进入循环都会调用 `time.After` 并创建新的 Timer Channel。
2. 若 `jobs` 先被选择，本轮 Timer Channel 随循环结束变得不可达。
3. Go 1.23+ runtime 可在 Timer 触发前回收这种不可达对象。
4. “可回收”不等于“零成本”；频繁创建仍会增加分配率和 runtime Timer 工作。

**追问方向**：旧模块是否一定使用新语义？Go 1.23 Timer 语义与主模块的 `go` 版本及 `GODEBUG=asynctimerchan` 兼容开关有关，诊断时要确认运行环境。Ticker 为什么仍应 Stop？为了停止未来事件和工作，而不只是为了 GC。

### 题 18：send 与 close 并发

```go
package main

import "sync"

func main() {
    ch := make(chan int, 1)

    var wg sync.WaitGroup
    wg.Add(2)

    go func() {
        defer wg.Done()
        defer func() { _ = recover() }()
        ch <- 1
    }()

    go func() {
        defer wg.Done()
        close(ch)
    }()

    wg.Wait()
}
```

**问题**：加了 `recover` 后是否就并发安全？发送一定成功吗？

**标准答案**：不安全，结果依赖调度。发送可能先完成，随后 close；也可能 close 先发生，发送 panic 后被 recover。`recover` 只吞掉了症状，没有建立 send 与 close 的顺序；使用 race detector 通常会报告该并发冲突。

**逐行分析**：

1. `hchan` 内部锁保证 runtime 数据结构不会被破坏，但不保证业务协议正确。
2. 两个 Goroutine 之间没有 happens-before。
3. close 的语义是“以后不再发送”，而此处该事实尚未成立。
4. recover 不能撤销已经出现的不确定性，也可能掩盖数据丢失。

**正确方案**：关闭者先通过 WaitGroup、完成 Channel、状态机或单拥有者事件循环确认全部发送路径结束，再执行 close。

**追问方向**：并发 close 与 receive 是否允许？允许且是正常广播模式；并发 send 与 receive 也是 Channel 的核心用途。危险的是没有协议地让 close 与潜在 send 重叠。

### 题 19：复制 Channel

```go
package main

import "fmt"

func main() {
    a := make(chan int, 1)
    b := a

    b <- 7
    close(a)

    v1, ok1 := <-b
    v2, ok2 := <-b
```

**问题**：`a` 与 `b` 是否有各自独立的缓冲区？

**逐行分析**：

1. `b := a` 复制 Channel 值，即复制同一对象的引用。
2. 通过 `b` 发送的值进入共同缓冲区。

### 题 20：Channel 作为 Map Key

```go
package main

import "fmt"

func main() {
    a := make(chan int)
    b := a
    c := make(chan int)

    owners := map[chan int]string{
        a: "first",
        c: "second",
    }

    fmt.Println(owners[b], a == b, a == c)
}
```

**问题**：能否编译？输出是什么？

**标准答案**：可以编译，输出 `first true false`。Channel 值可比较，因此可作为 Map Key；相等表示引用同一个 Channel 对象，或二者都为 nil。

**逐行分析**：

1. `a`、`b` 是同一 Channel 身份。
2. `c` 来自另一次 `make`，即使元素类型和容量相同也不相等。
3. Map 查询使用 Channel 身份的相等性，不比较缓冲内容或关闭状态。

**追问方向**：close 后 Key 是否变化？不会。Channel 身份不变。以 Channel 为 Key 维护订阅者注册表时应如何防泄漏？必须在生命周期结束时删除 Map 项，否则 Map 本身会让 Channel 持续可达。

---

### 题 21：双向转单向与反向恢复

```go
package main

import "fmt"

func main() {
    bi := make(chan int, 1)

    var send chan<- int = bi
    var recv <-chan int = bi

    send <- 9
    fmt.Println(<-recv)
    close(send)

    // var again chan int = send // 编译错误
    // close(recv)               // 编译错误
}
```

**问题**：哪些语句合法？方向转换是否复制 Channel？

**标准答案**：示例中的运行语句合法并打印 `9`。双向可收窄为单向，且不复制运行时对象；不能从 send-only 恢复为双向，不能关闭 receive-only Channel。send-only Channel 可以 close，因为它拥有发送方向能力。

**逐行分析**：

1. `send`、`recv`、`bi` 都指向同一对象。
2. 静态类型只限制允许的操作，不改变缓冲和状态。
3. 收窄权限可用于函数签名，表达调用者不应执行的操作。
4. 反向恢复会绕过 API 权限，因此类型系统拒绝。

**追问方向**：单向 Channel 是否可比较、可传入 `select`？可以，只要执行方向与类型一致。函数返回 `<-chan T` 能否阻止调用方 close？可以在编译期阻止普通代码关闭它。

### 题 22：`select` 多个 case 指向同一个 Channel

```go
package main

import "fmt"

func value(v int) int {
    fmt.Println("eval", v)
    return v
}

func main() {
    ch := make(chan int, 1)

    select {
    case ch <- value(1):
        fmt.Println("case 1")
    case ch <- value(2):
        fmt.Println("case 2")
    }

    fmt.Println("recv", <-ch)
}
```

**问题**：哪些输出固定，哪些不固定？第一个 case 是否优先？

**标准答案**：前两行求值输出固定为：

```text
eval 1
eval 2
```

随后可能是：

```text
case 1
recv 1
```

也可能是：

```text
case 2
recv 2
```

两个发送 case 在容量为 1 的空 Channel 上都能立即完成，`select` 从就绪 case 中伪随机选择；源码顺序不构成优先级。

**逐行分析**：

1. 进入 `select` 时，两条发送右值按源码顺序各求值一次。
2. 两个 case 指向同一 Channel，但仍是两个独立候选通信。
3. 当前缓冲区有一个空位，因此任一发送都可完成。
4. 只有被选中的值写入缓冲区，其 case body 执行。

**追问方向**：若 Channel 已满会怎样？没有 `default` 和接收者时当前 G 阻塞，并为两个 case 建立等待关系；其中一个胜出后 runtime 清理另一个。若两个接收 case 都从已关闭 Channel 接收，也会随机选一个。

## 七、面试高频问题

### 7.1 Channel 是什么？它只是线程安全队列吗？

**普通回答**：Channel 是 Go 中用于 Goroutine 间发送和接收类型化值的通信原语，可有缓冲或无缓冲，也可以关闭来表示数据流结束。

**中高级回答**：它不只是队列。无缓冲 Channel 表达同步交接；有缓冲 Channel 表达有界排队和背压；send、receive、close 还会建立 memory model 中的同步关系。Channel 值是可比较的引用型句柄，复制后仍指向同一通信对象。

**深挖回答**：当前 runtime 以 `hchan` 保存环形缓冲区、发送/接收等待队列、关闭状态和锁；阻塞者通过 `sudog` 入队，并由 `gopark`/`goready` 与调度器协作。语言规范只保证通信语义，不保证 `hchan` 布局。

**常见错误**：“Channel 是线程安全的，所以通过它发送指针后对象怎么改都安全。”Channel 只同步通信事件，指针指向对象的后续并发修改仍可能 race。

**版本/边界**：Channel 不跨进程、不持久化、不保证重放或严格公平；这些需求应使用消息系统或显式调度组件。

---

### 7.2 无缓冲和有缓冲 Channel 的本质区别是什么？

**普通回答**：无缓冲发送要等接收者；有缓冲在未满时可以先把值放入缓冲区。

**中高级回答**：无缓冲把生产者与消费者的进度耦合在一次 rendezvous 上；有缓冲允许最多 `cap` 个值在途，吸收短时速率差。满缓冲仍会把背压传播给发送者，因此容量不是吞吐不足的永久解决方案。

**深挖回答**：当前实现中，无缓冲可在等待者之间直接栈到栈复制；有缓冲用 `sendx`、`recvx` 管理环形队列。Memory Model 还规定无缓冲 Channel 的对应接收完成先于发送完成，以及容量 C 的第 k 次接收先于第 k+C 次发送完成。

**常见错误**：“有缓冲就完全异步”“容量越大越快”。容量只改变何时阻塞，过大会放大排队延迟、对象保留和 GC 成本。

**版本/边界**：具体直接复制与环形字段属于当前实现；阻塞条件和 FIFO 通信语义属于规范。

---

### 7.3 nil、正常、closed Channel 的发送和接收分别怎样？

**普通回答**：nil 的发送和接收永久阻塞，close nil panic；正常 Channel 按缓冲和对端状态决定是否阻塞；closed Channel 接收先排空再返回零值和 `ok=false`，发送和重复 close panic。

**中高级回答**：要额外指出 closed Channel 的接收永远就绪，可能让 `select` 忙循环；`range` 只在 closed 且排空后退出；close 不丢弃已缓冲值。

**深挖回答**：`chanrecv` 在锁内区分 closed+empty、缓冲有值和等待发送者；`closechan` 唤醒全部等待接收者和发送者，但接收者以失败接收返回，发送者恢复后 panic。

**常见错误**：“从 closed Channel 接收会 panic”“nil 接收会得到零值”“close 会清空队列”。

**版本/边界**：全局死锁报错不是每次 nil 阻塞都发生；长寿命服务器中可能只泄漏一个 G。

---

### 7.4 `value, ok` 与 `for range` 如何判断 Channel 结束？

**普通回答**：`v, ok := <-ch` 中 `ok=false` 表示 Channel 已关闭且排空；`for range ch` 会持续接收，直到这一条件成立。

**中高级回答**：`ok=true` 只说明值来自真实发送，即使 Channel 已经 close；不能用 `v == zero` 判断结束，因为零值可能是合法数据。`range` 不知道发送者是否退出，只识别 Channel 状态。

**深挖回答**：关闭后缓冲值仍由普通发送-接收同步关系覆盖；排空后的零值接收由 close-接收同步规则覆盖。`range` 是编译器实现的控制流形式，但语义由规范定义。

**常见错误**：“发送方函数 return 后 range 自动结束”“close 后 range 立即停止，不消费剩余值”。

**版本/边界**：Go 1.22 的循环变量变化会影响闭包捕获，但不改变 Channel range 的结束条件。

---

### 7.5 为什么通常由发送方关闭 Channel？

**普通回答**：因为发送方最清楚何时不会再发送；接收方关闭可能让仍在发送的 Goroutine panic。

**中高级回答**：关闭权应属于能证明“不再存在未来 send”的角色。单发送者可在最后一次发送后 close；多发送者则由协调者等待全部发送者结束后 close。创建者不必然就是关闭者，关键是协议所有权。

**深挖回答**：close 与潜在 send 重叠没有正确的检查后关闭模式。即使设计一个 `closed(ch)` 查询，也会在检查和 send/close 之间产生 TOCTOU。应通过 WaitGroup、单拥有者事件循环或状态机建立 happens-before。

**常见错误**：“谁发现没数据谁 close”“接收方 defer close 最保险”。

**版本/边界**：用于纯广播的 `done` Channel 通常由广播源关闭；Context 的 `Done` 则由 Context 实现管理，业务代码不能关闭。

---

### 7.6 多发送者如何安全关闭 Channel？

**普通回答**：每个发送者只负责发送；使用 `sync.WaitGroup` 等待所有发送者退出，再由单独 Goroutine 关闭输出 Channel。

**中高级回答**：关闭必须发生在所有 send 路径完成之后，包括错误分支、重试分支和异步回调。常见 Fan-in 模式是 `wg.Add(n)`，每个转发者 `Done`，协调者 `Wait` 后 close。

**深挖回答**：`sync.Once` 只保证 close 最多执行一次，不保证 close 与其他 send 不并发；`recover` 也只是掩盖 panic。真正的线性化条件是：关闭者观察到所有生产者都完成，且这些完成与后续 close 有同步顺序。

**常见错误**：“每个发送者都 defer close”“先看 `len(ch)==0` 再 close”“close 包在 Once 就万事大吉”。

**版本/边界**：若输出本就由 Context 取消而不是 `range` 消费，可不关闭数据 Channel，但仍必须确保生产者能退出。

---

### 7.7 close 是否用于释放资源？为什么说它是广播？

**普通回答**：close 用于表示不会再发送，不负责释放内存。关闭后所有接收者最终都能观察到 `ok=false`，所以它可作为广播信号。

**中高级回答**：Channel 在不可达时由 GC 回收，与是否 close 没有一一对应关系。反过来，已 close 但仍被全局变量或 Goroutine 引用的 Channel 也不会回收。一个普通发送值通常只被一个竞争接收者消费，close 则是永久状态，所有观察者都能看到。

**深挖回答**：当前 `closechan` 设置 `closed`、唤醒全部等待者；后续所有接收都立即完成。Memory Model 明确 close 先于因关闭而返回零值的接收，为广播前写入的状态建立可见性。

**常见错误**：“所有 Channel 都必须 defer close，类似文件句柄”“close 会回收缓冲区”。

**版本/边界**：无需向 Channel 发送且接收方不依赖关闭结束时，可以不 close；但 Timer、文件、连接等资源仍需调用它们自己的 Stop/Close。

---

### 7.8 `select` 的执行步骤是什么？

**普通回答**：找出可立即通信的 case；有多个就随机选一个；都不就绪时，有 `default` 就执行它，否则阻塞。

**中高级回答**：进入 `select` 时，所有 Channel 操作数和发送右值按源码顺序求值一次，未选中的发送右值也会求值。nil Channel case 永不就绪；已关闭 Channel 的接收 case 永远就绪。

**深挖回答**：编译器会优化空 select、单 case 和单通信+default；一般形式构造 case 数组进入 `selectgo`。runtime 随机化 poll order，按 Channel 地址生成 lock order，先扫描就绪；不能立即完成时把同一 G 的多个 `sudog` 入队并 park，唤醒后清理败选节点。

**常见错误**：“case 从上到下优先”“只有被选中的发送表达式才求值”“default 会等待一小会儿”。

**版本/边界**：通用 `selectgo` 的数据结构和复杂度是实现细节；求值与选择语义是规范保证。

---

### 7.9 `select` 是否公平？

**普通回答**：多个同时就绪 case 会伪随机选择，但不保证严格公平。

**中高级回答**：规范保证的是单次选择时对就绪通信做均匀伪随机选择，不是轮询、先来先服务或最大等待时间。case 的就绪集合会因调度和并发不断变化，长期仍可能出现偏斜或饥饿。

**深挖回答**：当前 runtime 的 `pollorder` 用随机排列减少固定源码顺序偏差；Channel 等待队列虽通常按队列处理，也不能上升为跨版本公平性契约。业务级公平需显式维护队列、权重、配额和轮转状态。

**常见错误**：“随机就等于绝不饥饿”“把高优先级 case 写前面即可”。

**版本/边界**：概率上长期趋于均衡不等于工程上有延迟上界。金融撮合、租户调度等不能依赖 `select` 随机性满足公平 SLO。

---

### 7.10 nil Channel 为什么能动态禁用 `select` 分支？

**普通回答**：nil Channel 的发送和接收永不就绪，所以把某个 Channel 变量设为 nil 后，对应 case 不会再被选中。

**中高级回答**：常用于 Fan-in：输入关闭后设置为 nil，避免 closed Channel 永远就绪；也可在有待发送值时才启用输出 case，实现状态机。

**深挖回答**：这利用的是规范语义，不需要 runtime 特殊“删除 case”。`selectgo` 在扫描时把 nil Channel case 排除；若所有通信 case 都为 nil 且无 default，G 永久 park。

**常见错误**：“close 和设 nil 等价”。close 让接收永远就绪；nil 恰好相反，让通信永远不就绪。

**版本/边界**：将共享 Channel 变量并发设 nil 本身需要同步；通常只在一个事件循环 Goroutine 的局部变量中使用该技巧。

---

### 7.11 `hchan` 中哪些字段最重要？

**普通回答**：`qcount`、`dataqsiz`、`buf` 表示缓冲；`sendx`、`recvx` 是环形索引；`sendq`、`recvq` 是等待队列；`closed` 表示关闭；`lock` 保护状态。

**中高级回答**：还应提到元素大小和类型元数据，用于复制、清零与 GC；当前 Timer Channel 还有 timer 关联字段。`qcount` 是 `len` 的基础，但只提供瞬时状态。

**深挖回答**：源码不变量解释了为什么缓冲有数据时通常不会有等待接收者、未满时通常不会有等待发送者。`lock` 也保护与等待 `sudog` 相关的关键字段；唤醒一般在解锁后执行。

**常见错误**：死记旧版本完整结构体，把字段布局说成 Go 规范；或说 Channel 完全 lock-free。

**版本/边界**：字段会随 runtime 演进。面试应先讲职责，再明确“以下是 Go 1.26.4 当前实现”。

---

### 7.12 `chansend` 与 `chanrecv` 的核心流程是什么？

**普通回答**：发送先找等待接收者，再看缓冲空位，否则阻塞；接收先找等待发送者或缓冲值，否则阻塞；closed 状态按规范处理。

**中高级回答**：发送流程包括 nil、非阻塞快速判断、锁内 closed 检查、直接交接、写环形缓冲、构造 `sudog` 入 `sendq` 并 park。接收对称地处理 closed+empty、等待发送者、读缓冲和 `recvq`。

**深挖回答**：满缓冲且有等待发送者时，接收者可取走 `recvx` 槽位，并把等待发送者的值补入同一轮环形位置；无缓冲路径则可能直接在两个 G 的栈之间复制，需配合写屏障与栈管理。

**常见错误**：“所有发送都先写缓冲”“无缓冲 Channel 内部也有一个容量 1 的槽”。

**版本/边界**：算法细节可变；发送/接收何时阻塞、FIFO 和关闭行为是规范契约。

---
### 7.13 `closechan` 为什么要分别处理接收者和发送者？

**普通回答**：等待接收者被唤醒后返回零值和 `ok=false`；等待发送者被唤醒后 panic，因为向 closed Channel 发送非法。

**中高级回答**：`closechan` 在锁内标记关闭并摘下两个等待队列，清理接收目标、设置等待结果；随后解锁并统一 `goready`。已有缓冲值不会被清除，之后接收仍先走缓冲路径。

**深挖回答**：不能在持有 `hchan.lock` 时随意直接 ready 其他 G，源码注释强调这会与栈收缩和调度器锁顺序交互。close 的当前复杂度与等待者数成正比，成千上万等待者的广播可能形成唤醒洪峰。

**常见错误**：“close 只是把一个布尔值设为 true，因此一定 O(1)”“close 后所有接收立刻只返回零值”。

**版本/边界**：`O(S+R)` 是当前实现分析，不是规范复杂度；生产中应评估惊群和下游同时恢复的冲击。

---

### 7.14 `sudog`、`waitq` 和 `chan.lock` 分别解决什么问题？

**普通回答**：`sudog` 表示一个 G 在 Channel 上的一次等待；`waitq` 排列等待发送者或接收者；`chan.lock` 保护 Channel 内部状态。

**中高级回答**：一个 G 在 `select` 中可同时等待多个 Channel，因此 G 与同步对象是多对多关系，需要独立 `sudog` 节点。节点还记录发送值/接收目标地址、是否属于 select、唤醒是否成功等。

**深挖回答**：`sudog.elem` 可能指向 Goroutine 栈，runtime 必须保证 park 期间栈和 GC 可正确处理。select 的败选节点需要在唤醒后从其他队列清理，当前实现用选择完成状态避免多个 case 同时胜出。

**常见错误**：“`sendq` 直接存 G 指针就够了”“等待队列与业务 FIFO 是规范保证”。

**版本/边界**：字段类型和池化方式会变化；核心理解应是“等待记录将 G、同步对象与通信地址关联”。

---

### 7.15 Channel 阻塞时，G、M、P 分别发生什么？

**普通回答**：当前 G 被挂起，调度器让 P/M 去运行其他可运行 G；对端通信或 close 后，该 G 被唤醒等待重新调度。

**中高级回答**：慢路径把 `sudog` 入队后调用 `gopark`，G 由 running 变 waiting；`goready`/`ready` 把它变 runnable 并放入运行队列。它不会保证立即继续执行，只是获得再次调度资格。

**深挖回答**：park 必须与释放 Channel 锁形成无丢失唤醒协议；否则对端可能在 G 真正休眠前发出唤醒。runtime 用专门的 park commit 路径保证状态转换与锁释放正确。

**常见错误**：“Channel 阻塞会独占一个 OS 线程”“`goready` 后立刻在当前 CPU 运行”。

**版本/边界**：若 G 正在不可中断的系统调用或 cgo 中，行为涉及更完整的调度器机制；普通 Channel park 是用户态 G 调度路径。

---

### 7.16 `selectgo` 为什么要排序锁？

**普通回答**：一个 select 可能涉及多个 Channel，统一按 Channel 地址顺序加锁可避免不同 Goroutine 反向加锁导致 runtime 内部死锁。

**中高级回答**：当前实现分别生成随机 `pollorder` 和稳定 `lockorder`：前者减少选择偏置，后者保证锁顺序。锁定后扫描就绪 case；若需阻塞，给各 case 入队，park 后再清理败选 case。

**深挖回答**：同一 Channel 可出现在多个 case，排序时要保持 case 随机性又避免重复锁；当前使用基于 Channel 排序键的堆排序，通用路径大致含 `O(n log n)` 排序和 `O(n)` 扫描/排队。

**常见错误**：“按源码 case 顺序加锁”“随机顺序同时用于锁顺序”。随机锁顺序反而会增加死锁风险。

**版本/边界**：编译器可把单 case、单通信+default 降成专用非阻塞函数，不一定进入通用 `selectgo`。

---

### 7.17 Channel 的 happens-before 规则有哪些？

**普通回答**：对应发送先于对应接收完成；close 先于接收方因关闭得到零值；无缓冲接收先于发送完成；容量 C 的第 k 次接收先于第 k+C 次发送完成。

**中高级回答**：要用“同步于”和程序顺序组合推导可见性，而不是笼统说“经过 Channel 就安全”。例如写共享变量后发送信号，接收完成后读可见；但两个发送者对同一对象的并发写仍无顺序。

**深挖回答**：容量规则解释了用 buffered Channel 实现计数 Semaphore 的正确性；无缓冲的反向规则使发送完成可视为接收者已经接手该值。close 只同步那些因关闭而完成的接收，缓冲值仍由各自发送同步。

**常见错误**：“发送语句开始就同步”“只要两个 G 都使用同一个 Channel，它们的所有操作都有全序”。

**版本/边界**：Memory Model 规则稳定，但不替代对象级不变量设计；仍应运行 `go test -race` 验证实现。

---

### 7.18 Channel 中传指针会自动转移所有权吗？

**普通回答**：不会自动。Channel 只复制指针值；所有权转移是业务约定。

**中高级回答**：若发送者在 send 完成后不再访问对象，接收者独占访问，可形成清晰所有权转移。若双方继续并发读写，就需要额外 Channel 往返确认、Mutex、Atomic 或不可变设计。

**深挖回答**：发送前对 pointee 的写可通过发送-接收关系对接收方可见；但发送完成后的新写不自动排序。缓冲 Channel 还允许发送方快速继续，因此更容易误以为对象已经无人使用。

**常见错误**：“指针经过 Channel 后 runtime 会复制对象”“Channel 自带对象锁”。

**版本/边界**：传大值能减少共享 race，但增加复制和缓冲内存；传指针降低复制但增加 GC 扫描、生命周期与协议复杂度，应基准测试并明确所有权。

---

### 7.19 为什么 `len(ch)`、`cap(ch)` 不能用于同步判断？

**普通回答**：它们只是调用瞬间的状态；返回后其他 Goroutine 可能立即发送或接收。

**中高级回答**：`if len(ch) < cap(ch) { ch <- v }` 不是原子 try-send，仍可能阻塞；`if len(ch)>0 { <-ch }` 同理。非阻塞通信应使用带 `default` 的 `select`。

**深挖回答**：runtime 内部也有无锁快速检查，但它只在受严格证明的条件下做保守判断，并在锁内复核；业务代码无法把一个公开快照与后续操作绑定为同一线性化点。

**常见错误**：“Channel 内部有锁，所以 len 后 send 也是安全事务”“race detector 没报就说明逻辑正确”。

**版本/边界**：`len` 可用于监控占用趋势、测试辅助和启发式负载均衡，但结果不应决定正确性。Timer Channel 的 `len/cap` 还受 Go 1.23 新语义影响。

---

### 7.20 Channel 和 Mutex 应如何选择？

**普通回答**：传递任务、事件和所有权用 Channel；保护共享 Map、计数器和结构状态用 Mutex。

**中高级回答**：先看不变量在哪里。若多个操作必须在共享状态上原子完成，Mutex 能把条件检查与修改放在同一临界区；若状态可由单一 Goroutine 拥有，其他方提交命令，Channel 可减少共享可变状态。

**深挖回答**：Channel 自身也有锁、复制和调度成本；Mutex 在无竞争路径通常很轻，但高争用可能导致阻塞和缓存抖动。Actor 化会换来串行瓶颈、请求响应 Channel 和取消复杂度。应通过 benchmark、block/mutex profile、trace 验证。

**常见错误**：“Do not communicate by sharing memory 意味着禁止 Mutex”“Channel 一定比锁慢/快”。

**版本/边界**：Atomic 适合简单状态；`sync.Cond` 适合在锁保护状态上等待条件；跨进程可靠通信需外部系统。

---

### 7.21 Channel 缓冲区应该设置多大？

**普通回答**：根据生产速率、消费速率、突发量、允许延迟和内存预算测量，而不是固定经验值。

**中高级回答**：可先用 `(到达率-服务率)×突发持续时间` 估算突发容量，并结合 Little's Law 估计平均在途量；随后用占用率、发送阻塞、P99 延迟、丢弃量和 GC 数据校准。

**深挖回答**：容量增大会降低短时发送阻塞，却增加排队等待、缓存对象保留和故障恢复时的陈旧工作量。若平均到达率长期大于服务率，有限缓冲必满；无界缓冲只是推迟 OOM。

**常见错误**：“容量等于 worker 数就是最优”“取 1024 因为是 2 的幂”“把缓冲调大就能提高下游吞吐”。

**版本/边界**：大元素 Channel 的内存约为容量乘元素大小，还要考虑指针指向对象；实际分配与 GC 扫描由元素类型影响。

---

### 7.22 常见 Goroutine 泄漏模式有哪些？

**普通回答**：无人接收的发送、无人发送的接收、永不结束的 `range`、消费者提前退出、缺少 Context、nil Channel、Timer/Ticker 生命周期错误。

**中高级回答**：重点看“谁还能让阻塞点变为就绪”。一次性无缓冲结果 Channel 在调用方超时后最典型；Pipeline 下游只取部分结果也会让所有上游逐级卡在 send。

**深挖回答**：泄漏不仅占一个 G，还会保留栈、Channel、`sudog`、缓冲值和对象图。排查时先看 `/sched/goroutines` 趋势，再抓 goroutine dump；`[chan send]`、`[chan receive]`、`[select]` 的重复栈聚类最有价值，execution trace 可还原阻塞时间线。

**常见错误**：“Goroutine 很轻，泄漏几个没关系”“加 buffer=1 总能解决”“main 退出会自动清理，所以服务函数也没问题”。

**版本/边界**：某些常驻后台 G 是有意设计，应有明确 owner、退出机制和基线数量；诊断看增长趋势而非孤立总数。

---

### 7.23 `time.After`、Timer 和 Ticker 在 `select` 中怎么选？

**普通回答**：一次性简单超时可用 `time.After`；需要复用、取消或重置时用 `time.Timer`；周期事件用 `time.Ticker`，结束时 Stop。

**中高级回答**：热循环中反复 `time.After` 会重复分配 Timer；复用 Timer 可降低分配。Ticker 的 Channel 不会因 Stop 自动关闭，消费循环还需要 Context 或其他退出条件。

**深挖回答**：Go 1.23 起 Timer Channel 对外采用同步语义，未引用的未停止 Timer/Ticker 可被 GC，Stop/Reset 的旧值保证也增强。诊断旧程序要检查主模块 `go` 版本和 `GODEBUG=asynctimerchan`。

**常见错误**：“当前 Go 中 `time.After` 必然永久泄漏”“`ticker.Stop()` 会 close `ticker.C`”“用 `len(timer.C)` 判断是否需要 drain”。

**版本/边界**：`AfterFunc` 没有可接收的普通 Timer Channel，Reset 语义与 Channel Timer 不同；面试时不要把两类 Timer 混为一谈。

---

### 7.24 如何设计一个不会泄漏的 Worker Pool？

**普通回答**：固定数量 worker 从 jobs 接收；生产者关闭 jobs；worker 结束后由协调者关闭 results；所有阻塞 send/receive 都监听 Context。

**中高级回答**：还要定义任务失败、取消传播、结果无人消费、panic 隔离、重试、顺序、队列满载和优雅停机。worker 不应各自关闭共享 results。

**深挖回答**：关闭顺序通常是停止接收新任务 → 取消或排空在途任务 → 等待 worker → 关闭结果；若 handler 不响应 Context，池仍无法及时停机。为每个任务先创建 G 再抢 semaphore 会产生大量等待 G，应固定 worker 或先 acquire 再启动。

**常见错误**：“close jobs 会自动取消正在执行的 handler”“收到第一个错误后直接 return，不管剩余 worker 发送结果”。

**版本/边界**：需要错误聚合和结构化取消时，可使用经过审查的并发组合原语；无论使用何种库，最终仍要验证每条 Goroutine 路径可退出。

---

### 7.25 Channel 的性能和 GC 成本来自哪里？

**普通回答**：值复制、内部锁、缓冲内存、阻塞/唤醒调度、`sudog` 和被缓冲对象的生命周期都会产生成本。

**中高级回答**：无竞争且缓冲可用的操作通常 O(1)，但大元素复制昂贵；高争用时 `hchan.lock` 可能成为热点；阻塞路径涉及 park/ready 和缓存迁移；一般 select 随 case 数增长。

**深挖回答**：含指针的大缓冲区增加 GC 扫描，未消费槽位持续保留对象；大量阻塞 G 又保留栈和通信值。close 大量等待者会形成 O(waiters) 唤醒和 runnable backlog。应同时看 CPU、block profile、trace、heap profile 与 scheduler metrics。

**常见错误**：“Goroutine 和 Channel 都很轻，所以可以无限创建”“只看 ns/op，不看 allocs、尾延迟和 GC”。

**版本/边界**：具体纳秒和分配量高度依赖 Go 版本、架构、元素类型和竞争度；必须在目标版本与负载上基准测试。

---

### 7.26 Channel 如何导致死锁、活锁和饥饿？

**普通回答**：死锁是通信依赖形成环或永远没有对端；活锁是 Goroutine 一直重试却没有业务进展；饥饿是某个任务长期得不到被选择或服务。

**中高级回答**：典型死锁包括 nil Channel、无接收者 send、未 close 的 range、锁内等待 Channel 而对端需要同一锁。`default` 重试、双方反复撤销动作可能活锁；热点 case、无公平队列或慢租户隔离缺失可能饥饿。

**深挖回答**：runtime 的“all goroutines asleep”只检测全局无进展；局部死锁、泄漏和业务活锁通常不会触发。需要结合 goroutine 状态、CPU profile、block profile、trace 和业务进度指标判断。

**常见错误**：“没有 fatal deadlock 就没有死锁”“select 随机所以绝不会饥饿”“加 default 可以避免阻塞，因此更安全”。

**版本/边界**：分布式等待、外部 I/O 和 Context 超时会让症状跨出 Channel 本身；排查应画完整等待图，而不是只看一行栈。

## 八、深挖追问链

### 追问链 1：从“无缓冲 Channel”追到调度器

1. **无缓冲 Channel 是什么？**  
   答题点：容量为 0；发送与接收必须配对；表达同步交接，不是“内部有一个隐藏槽”。

2. **发送方先到会怎样？**  
   答题点：若无等待接收者，发送方构造等待记录加入 `sendq` 并 park；若是非阻塞发送则失败返回。

3. **接收方后来到达，值放在哪里？**  
   答题点：当前实现可从发送方通信地址直接复制到接收方目标地址，不经过环形缓冲区。

4. **为什么能跨 Goroutine 栈复制？**  
   答题点：由 runtime 专用代码完成，必须处理写屏障、栈移动和 GC 可见性；普通 Go 代码不能任意持有另一个 G 的栈指针。

5. **发送方阻塞是否占住 OS 线程？**  
   答题点：通常不占；`gopark` 把 G 置为 waiting，P/M 可运行其他 G。

6. **接收方如何唤醒发送方？**  
   答题点：从等待队列取 `sudog`，完成复制和状态更新，解锁后 `goready`，发送方进入 runnable 而非保证立即执行。

7. **无缓冲发送返回时，接收方是否已经处理完业务？**  
   答题点：只保证对应接收操作已完成，不保证接收方后续 `handle(v)` 已完成；若要业务确认，需要反向 ack Channel、WaitGroup 或更高层协议。

8. **这条同步关系能做什么？**  
   答题点：发送前写入对接收完成后的读取可见；无缓冲接收完成还先于发送完成，可用于所有权交接和同步点。

---

### 追问链 2：从 close 追到多发送者优雅停机

1. **close 的语义是什么？**  
   答题点：宣布今后不会再发送；不是发送特殊值，也不是释放资源。

2. **为什么接收方通常不 close 数据 Channel？**  
   答题点：它通常无法证明所有发送者结束；关闭后任何后续 send 都 panic。

3. **多发送者如何关闭？**  
   答题点：生产者只发送；协调者用 WaitGroup 等待全部生产者完成，再执行唯一一次 close。

4. **用 `sync.Once` 包 close 是否充分？**  
   答题点：不充分；它只避免重复 close，不能保证没有发送与 close 并发。

5. **服务停机时先 cancel 还是先 close jobs？**  
   答题点：取决于“排空还是立即终止”策略。排空模式先停止接入并 close jobs，让 worker 处理完；立即终止模式先 cancel。协议必须明确，不能同时无序操作。

6. **worker 完成后谁关闭 results？**  
   答题点：等待所有可能发送结果的 worker 结束的协调者；任何单 worker 都不能证明其他 worker 已结束。

7. **消费者收到第一个错误就返回，会有什么问题？**  
   答题点：其他 worker 可能阻塞发送结果。应触发 Context 取消，并让发送结果也选择 `ctx.Done()`；或者确保结果缓冲覆盖所有确定的单次发送并等待回收。

8. **close 后为何可能有唤醒洪峰？**  
   答题点：当前 `closechan` 遍历并 ready 所有等待者；大量 G 同时 runnable 会增加调度延迟和下游冲击，应分批、分层或用限流控制恢复。

---

### 追问链 3：从 `select` 追到 `selectgo`

1. **多个 case 同时就绪时如何选？**  
   答题点：规范要求在可进行的通信中做均匀伪随机选择，无源码顺序优先级。

2. **发送 case 的表达式何时求值？**  
   答题点：进入 select 时，所有 Channel 操作数和发送右值按源码顺序求值一次；未选中也会发生副作用。

3. **nil Channel case 怎样处理？**  
   答题点：永不就绪；可动态禁用。全部 case 都 nil 且无 default 时永久阻塞。

4. **closed Channel case 呢？**  
   答题点：接收永远就绪；若不检查 `ok` 并退出/设 nil，会忙循环或压制其他分支。

5. **runtime 为什么同时需要 poll order 和 lock order？**  
   答题点：poll order 随机化选择扫描；lock order 全局稳定以防多 Channel 反向加锁死锁，职责不同。

6. **没有 case 就绪时怎样等待多个 Channel？**  
   答题点：为每个通信 case 创建/取得 `sudog` 并加入对应等待队列，当前 G park；任一 case 胜出后清理其余节点。

7. **select 是否保证每个 case 最终被选中？**  
   答题点：不保证。单次伪随机不提供长期延迟上界；调度和就绪集合都在变化。

8. **若业务需要优先级怎么办？**  
   答题点：显式两阶段 select、优先队列、配额/轮询调度器或独立事件循环；同时谨慎避免“先试高优先级”造成低优先级永久饥饿。

---

### 追问链 4：从 happens-before 追到指针 data race

1. **发送和接收建立什么关系？**  
   答题点：某次发送同步于对应接收完成，结合程序顺序形成 happens-before。

2. **发送前写变量，接收后读取是否安全？**  
   答题点：若这条 Channel 通信确实连接二者，写先于发送、发送先于接收、接收先于读，因此可见且无该变量上的 race。

3. **发送的是指针时，对象是否被冻结？**  
   答题点：不会，只复制指针；发送前的写可见，发送后的并发写未自动同步。

4. **发送方在 send 返回后修改对象，接收方同时读会怎样？**  
   答题点：可能 data race，尤其有缓冲 Channel 中 send 很早返回。需要所有权转移约定或额外同步。

5. **无缓冲 Channel 是否就自动安全？**  
   答题点：也不是。send 返回只表示接收操作完成，不表示接收方已完成后续对象使用。

6. **如何做真正的所有权归还？**  
   答题点：接收方处理完后通过 ack Channel 返回确认，或对象沿 Channel 反向传回；也可使用 Mutex 管理共享生命周期。

7. **close 能同步哪些写？**  
   答题点：close 前写入可通过“close 先于因关闭返回零值的接收”对观察者可见；接收缓冲值则由对应 send 建立同步。

8. **如何验证没有遗漏 race？**  
   答题点：写出 happens-before 图，运行 `go test -race`，并用压力测试覆盖不同交错；race detector 无报告也不证明协议无死锁或泄漏。

---

### 追问链 5：从缓冲区大小追到 GC 与容量规划

1. **为什么要加缓冲？**  
   答题点：吸收短突发、减少阶段间同步等待、允许有限在途任务；不是提升下游固有吞吐。

2. **容量如何估算？**  
   答题点：基于到达率、服务率、突发持续时间和延迟/内存预算，可用 `(λ-μ)×T` 初估并压测校准。

3. **容量很大有什么副作用？**  
   答题点：排队延迟、陈旧任务、故障恢复耗时、内存占用、含指针元素的对象保留和 GC 扫描增加。

4. **`len(ch)` 能否驱动自动扩容？**  
   答题点：可做观测信号，但不能作为同步正确性；应使用一段时间的占用率分布、发送阻塞和消费延迟，而非一次快照。

5. **平均生产率一直高于消费率会怎样？**  
   答题点：任何有限缓冲最终都会满；扩大容量只延迟故障。需要限流、扩容、降级、丢弃或持久化队列。

6. **为什么无界队列危险？**  
   答题点：把背压转成堆增长和尾延迟；Slice 中的引用让对象长期存活，最终可能 OOM。

7. **如何确认 Channel 是内存积压源？**  
   答题点：看队列水位、heap profile 的对象保留路径、goroutine dump、GC 扫描指标和消费延迟；必要时在入队对象上减少大对象引用。

8. **传指针能否降低内存？**  
   答题点：只降低槽位复制与直接存储大小，不一定降低总存活内存；指针仍让整个对象图可达，还增加扫描与 race 风险。

---

### 追问链 6：从“Goroutine 数上涨”追到生产修复

1. **先看什么指标？**  
   答题点：Goroutine 总数和 waiting/runnable 趋势、请求量、错误率、队列水位、内存与 GC；判断是否随流量回落。

2. **如何快速定位阻塞点？**  
   答题点：抓 `/debug/pprof/goroutine?debug=2`，按栈聚类；关注 `[chan send]`、`[chan receive]`、`[select]` 及具体源码行。

3. **发现大量 G 卡在结果发送，下一步问什么？**  
   答题点：谁应接收？调用方是否超时返回？发送是否监听 Context？结果 Channel 是否无缓冲？是否存在下游提前退出？

4. **block profile 能否替代 goroutine dump？**  
   答题点：不能。dump 更适合看当前仍卡住的 G；block profile 汇总阻塞事件和时间，需启用采样，二者互补。

5. **execution trace 提供什么？**  
   答题点：G 的创建、阻塞、唤醒、调度延迟和 runnable backlog 时间线，可判断是 Channel 对端慢、调度拥塞还是外部 I/O。

6. **怎样止血？**  
   答题点：限流、停止新建 G、取消上游、临时扩大消费者、缩短超时或丢弃可丢任务；避免直接通过 recover 隐藏发送 panic。

7. **怎样根治？**  
   答题点：为每个 G 定义 owner 和退出条件；所有可能阻塞的 send/receive 加取消分支；统一关闭所有权；补充泄漏测试和指标。

8. **如何验证修复？**  
   答题点：压力与超时故障注入后，G 数回归基线、无重复泄漏栈、race 通过、队列受界、P99 与堆稳定，并验证优雅停机。

## 九、生产故障与排查

### 9.1 先按症状分类

| 线上症状 | Channel 相关高概率原因 | 首要证据 | 第一处修复方向 |
|---|---|---|---|
| Goroutine 数持续上涨 | 超时后发送者泄漏、未关闭的 range、nil Channel、下游提前退出 | goroutine dump 中大量相同 `[chan send]` / `[chan receive]` / `[select]` | 补 Context、关闭协议和发送取消分支 |
| 单核或多核 CPU 突升 | closed Channel 永远就绪、空 `default` 忙轮询、无退避重试 | CPU profile 指向 select 循环；G 多为 runnable | 退出、设 nil、删除 default 或增加阻塞/退避 |
| Heap/RSS 持续上涨 | 大缓冲积压、Channel 中保存大对象指针、阻塞 G 保留请求对象 | heap profile、队列水位、GC scan/live 指标 | 限流、缩小载荷、排空/丢弃、修复泄漏 |
| P99/P999 延迟升高 | 缓冲排队、下游变慢、唤醒洪峰、单 Channel 高争用 | trace、block profile、队列等待时间 | 背压、分片、批处理、限制 fan-out |
| 偶发 `send on closed channel` | 多关闭者、send/close 并发、接收方误关 | panic 栈、race detector、关闭调用链 | 单一关闭者，建立完成顺序 |
| 程序挂住但无全局 deadlock | 局部等待环、服务中部分 G 泄漏、外部 I/O 不返回 | goroutine dump、等待图、trace | 打破依赖环、加超时和取消 |
| 数据错乱或 race | 传指针后双方继续修改、以 Channel 信号替代完整对象同步 | `go test -race`、对象访问栈 | 所有权转移、不可变值、Mutex/Atomic |
| 吞吐下降且 runtime 占比高 | 过细任务、频繁 park/unpark、巨大 select、高 fan-in 争用一个 `hchan.lock` | CPU/block profile、trace | 批量化、分片、减少 case、调整工作粒度 |
| 停机超时 | worker 卡在结果发送、handler 不响应 Context、Ticker 循环不退出 | 停机期间 goroutine dump | 取消优先、双侧 select、明确 drain 策略 |

### 9.2 标准排障闭环

#### 第一步：确认是全局问题还是局部泄漏

- 是否出现 runtime 的全局 deadlock 报错？
- Goroutine 数是否随请求量上涨后不回落？
- 业务进度指标是否停滞，但进程仍能处理健康检查？
- runnable、waiting、CPU、内存哪个先变化？

不要把“没有 fatal deadlock”误当成“没有 Channel 死锁”。线上服务通常还有 netpoll、Timer、GC 和 HTTP Goroutine，局部等待环不会触发全局检测。

#### 第二步：抓取 Goroutine dump 并做栈聚类

启用 `net/http/pprof` 后：

```bash
curl -s 'http://127.0.0.1:6060/debug/pprof/goroutine?debug=2' > goroutines.txt
```

重点状态：

- `[chan send]`：谁不再接收？是否调用方已超时？
- `[chan receive]`：谁应发送或关闭？是否 Channel 为 nil？
- `[select]`：所有分支都在等什么？是否遗漏 `ctx.Done()`？
- `[semacquire]`：可能是 WaitGroup/Mutex，继续画依赖关系。
- runnable/running 且栈在 select 循环：考虑 busy loop。

按“函数 + 行号”聚类比逐条阅读更有效。若几万个 G 都卡在同一行，先分析这条生命周期路径。

#### 第三步：画等待图

示例：

```text
G1 持有 mu，等待向 ch 发送
          │
          ▼
G2 想先获取 mu，再从 ch 接收
```

这是锁与 Channel 混合形成的等待环。仅看 `G1 [chan send]` 会误以为“加大缓冲”可修复；根因是持锁跨越阻塞通信。

画图时列出：

- 每个 G 当前持有什么锁/许可。
- 正在等待哪个 Channel 或外部资源。
- 哪个 G 能让它就绪。
- 那个 G 是否又依赖当前 G。
- 是否有超时、取消或关闭能打破环。

#### 第四步：使用 block profile 看累计阻塞热点

测试环境：

```bash
go test -run '^$' -bench . -benchmem \
  -blockprofile=block.out -blockprofilerate=1 ./...

go tool pprof -http=:0 block.out
```

在线服务中可按可接受采样率调用：

```go
runtime.SetBlockProfileRate(1000)
```

再采集：

```bash
go tool pprof -http=:0 \
  'http://127.0.0.1:6060/debug/pprof/block'
```

Block profile 适合回答“哪些 Channel send/receive/select 累计阻塞时间最多”。当前仍永久卡住的 G，goroutine dump 往往更直接；二者应结合。

#### 第五步：用 execution trace 还原调度时间线

测试：

```bash
go test -trace=trace.out ./...
go tool trace trace.out
```

或在可控程序区间使用 `runtime/trace`。关注：

- G 何时创建、阻塞和被唤醒。
- 唤醒后多久才真正运行，即 scheduler latency。
- 某个 Channel 是否形成串行瓶颈。
- close 是否瞬间制造大量 runnable G。
- 下游变慢时上游是否正确受到背压。

Trace 数据量和开销较高，应限定采集窗口，不要无界长期运行。

#### 第六步：运行 race detector

```bash
go test -race ./...
go test -race -count=100 -run TestPipelineCancel ./...
```

重点捕获：

- send 与 close 并发。
- 通过 Channel 传指针后双方并发访问。
- 共享 Channel 变量被并发设为 nil 或替换。
- 错误地把一次通知当成对象完整生命周期同步。

Race detector 不能发现所有死锁、泄漏或业务数据丢失；无报告只说明本次执行未观察到受支持类型的内存竞争。

#### 第七步：关联 Heap 与队列水位

采集：

```bash
go tool pprof -http=:0 \
  'http://127.0.0.1:6060/debug/pprof/heap'
```

同时必须有业务指标：

- Channel 当前长度与容量。
- 入队/出队速率。
- 满队列次数、发送超时、丢弃数。
- 项目从入队到处理的等待时间。
- 最老队列项年龄。
- 每个载荷估算大小。

Heap profile 未必直接写着“由某 Channel 槽位保留”。需要结合对象类型、分配栈、队列水位和 goroutine 引用链推断。

### 9.3 推荐的运行时指标

Go 1.26.4 的 `runtime/metrics` 可关注：

```text
/sched/goroutines:goroutines
/sched/goroutines-created:goroutines
/sched/goroutines/runnable:goroutines
/sched/goroutines/running:goroutines
/sched/goroutines/waiting:goroutines
/sched/latencies:seconds
/gc/heap/live:bytes
/gc/scan/heap:bytes
/memory/classes/heap/objects:bytes
/sync/mutex/wait/total:seconds
```

指标集合会随版本演进，生产采集器应以 `metrics.All()` 检查目标 Go 版本实际支持项，不要硬编码不存在的名称。

runtime 没有提供“每个业务 Channel 的通用名称和水位”指标。关键队列应在封装层自行埋点：

```go
type QueueMetrics interface {
    ObserveDepth(name string, depth, capacity int)
    IncSendBlocked(name string)
    IncDropped(name string)
    ObserveQueueLatency(name string, d time.Duration)
}
```

`len(ch)` 用于指标采样是合理的，只要不把它用于同步决策。

### 9.5 故障案例一：超时率上升后 Goroutine 数单调增长

**现象**：下游数据库变慢，接口超时率升高；流量恢复后 Goroutine 数仍不下降，Heap 也缓慢上涨。

**证据**：goroutine dump 中成千上万条相同栈：

```text
service.query.func1
    result <- response   [chan send]
```

**根因**：调用方超时返回，结果 Channel 无缓冲；后台查询完成后仍试图发送，永久没有接收者。

**止血**：限制并发查询、取消下游请求、临时让发送选择 Context；不要仅扩大进程内存。

**根治**：

```go
result := make(chan Response, 1)
go func() {
    r := query(ctx)
    select {
    case result <- r:
    case <-ctx.Done():
    }
}()
```

同时确保 `query(ctx)` 真正响应取消。补充故障注入测试：超时后 Goroutine 数应回归基线。

### 9.6 故障案例二：发布后 CPU 单核 100%

**现象**：没有明显流量上涨，某实例单核持续满载。

**证据**：CPU profile 指向：

```go
for {
    select {
    case _, ok := <-updates:
        if !ok {
            continue
        }
    }
}
```

**根因**：上游关闭 `updates` 后，该 case 永远就绪；循环没有阻塞。

**修复**：若数据流结束就应退出，直接 `return`；若还要等待其他输入，执行 `updates = nil`。增加“上游关闭后 CPU 不持续增长”的测试。

### 9.7 故障案例三：队列看似提高吞吐，实际引发 OOM

**现象**：把容量从 100 调到 100000 后，短期发送阻塞下降，但数小时后 RSS、GC CPU 和 P99 都上升。

**证据**：

- Channel 深度长期接近高水位。
- 出队速率始终低于入队速率。
- Heap profile 显示大批请求对象和字节切片存活。
- `/gc/scan/heap:bytes` 上升。

**根因**：容量掩盖了持续的服务率不足；每个槽位保存指向大请求对象的指针，队列成为对象保留根。

**修复**：恢复有界小队列并传播背压；限制入队；扩大真正瓶颈的处理能力；只传必要字段或 ID；定义超时、丢弃和持久化策略。

### 9.8 故障案例四：偶发 `send on closed channel`

**现象**：低概率 panic，只在滚动重启或超时高峰出现。

**证据**：发送栈和关闭栈来自不同 Goroutine；`go test -race` 在故障注入下报告 send/close 并发。

**根因**：关闭者只观察了一个“停止接入”标志，却没有等待已在途发送者完成；或多个 worker 都 defer close 共享输出。

**修复**：把关闭权移给协调者：停止新任务 → 等待在途发送结束 → close。不要用 recover 继续运行，也不要尝试先 `len`/“检查是否关闭”。

### 9.9 故障案例五：Channel 与 Mutex 组成死锁环

错误模式：

```go
mu.Lock()
ch <- update // 可能阻塞
mu.Unlock()
```

而消费者：

```go
mu.Lock()
state := snapshot()
mu.Unlock()

v := <-ch
_ = v
_ = state
```

若消费者必须先拿 `mu` 才能进入接收，而生产者持有 `mu` 等待发送，就形成环。

**修复原则**：

- 不要在持锁时执行可能阻塞的 Channel 通信。
- 若必须保持状态快照，锁内复制最小数据，解锁后发送。
- 或由单一事件循环拥有状态，取消锁与 Channel 的交叉依赖。
- 对“锁顺序 + Channel 等待”统一画等待图并写入设计文档。

### 9.10 故障案例六：高 fan-in 造成吞吐平台期

**现象**：增加生产者后吞吐不再提升，runtime Channel 函数和调度开销上升。

**可能原因**：

- 所有生产者争用一个 `hchan.lock`。
- 每个任务过小，通信成本高于有效工作。
- 无缓冲交接导致频繁 park/unpark。
- 一个巨型 `select` 每轮处理大量 case。
- 消费者本身是单点瓶颈。

**优化顺序**：

1. 先确认业务处理能力是否已饱和。
2. 批量发送多个任务，增加每次通信的有效工作。
3. 按 key/worker 分片 Channel，减少单点争用。
4. 适当小缓冲吸收调度抖动，但不掩盖长期过载。
5. 比较 Mutex+共享队列、每 worker 私有队列等替代设计。
6. 用目标 Go 版本和真实元素大小重新 benchmark。

## 十、面试回答模板

### 10.1 30 秒回答：Channel 是什么

> Channel 是 Go 的类型化并发通信原语。无缓冲 Channel 用发送与接收的同步交接来协调 Goroutine；有缓冲 Channel 提供容量受限的排队和背压。nil Channel 的收发永久阻塞，关闭后先排空缓冲，再返回零值和 `ok=false`；向关闭 Channel 发送、重复 close、close nil 都会 panic。工程上通常由发送方或发送方协调者关闭，多发送者先 Wait 再由唯一角色 close。Channel 通信还建立 happens-before，但发送指针不等于对象自动并发安全。

### 10.2 2 分钟回答：语义、实现与工程边界

> 我会从三个层面理解 Channel。
>
> 第一是规范语义。无缓冲必须发送接收配对，有缓冲未满可先入队；nil Channel 永不就绪；closed Channel 接收先排空，之后返回零值和 `ok=false`，发送和重复关闭 panic。`select` 会先求值所有 Channel 操作数和发送右值，从当前就绪 case 中伪随机选一个；`default` 是立即非阻塞，不提供等待。close 表示以后不会再发送，不是释放资源。
>
> 第二是当前 runtime。Go 1.26.4 用 `hchan` 保存 `qcount`、`dataqsiz`、`buf`、`sendx`、`recvx`、`sendq`、`recvq`、`closed` 和锁。发送先找等待接收者，再尝试环形缓冲，最后以 `sudog` 入队并 `gopark`；接收对称处理。close 会唤醒全部等待者，接收者返回关闭，发送者恢复后 panic。一般 select 通过 `selectgo` 随机化扫描，并按统一 Channel 顺序加锁。
>
> 第三是工程协议。关闭权属于能证明再无发送的一方；多生产者由协调者 Wait 后 close。所有可能因下游退出而阻塞的 send 都应监听 Context。缓冲用于吸收突发，不解决持续过载；`len(ch)` 只做观测，不能做检查后发送。共享状态保护通常选 Mutex，任务流、所有权转移、Pipeline 和背压更适合 Channel。

### 10.3 5 分钟回答：从状态机讲到生产排障

> Channel 可以先看成三态：nil、open、closed。nil 没有运行时对象，收发永久阻塞，close panic；open 状态按容量和对端决定是否就绪；closed 状态不再允许发送，缓冲值仍能接收，排空后每次接收都立即得到零值和 `ok=false`。所以 closed Channel 放在 select 中会永久就绪，必须 return 或把局部变量设 nil。`for range` 只有在 closed 且排空后才结束。
>
> 无缓冲 Channel 是 rendezvous，发送和接收在同一次通信上同步；有缓冲 Channel 用固定容量环形队列解耦短时速率差。Go Memory Model 规定对应发送先于对应接收完成，close 先于因关闭返回零值的接收，无缓冲接收还先于发送完成，容量 C 的第 k 次接收先于第 k+C 次发送完成。它们能证明通信前写入可见，但发送指针后双方继续并发改对象仍会 race。
>
> 当前 Go 1.26.4 runtime 里，`hchan` 保存缓冲计数、容量、缓冲地址、发送/接收索引、两类等待队列、关闭标记和锁。`chansend` 先处理 nil 与非阻塞快路径，锁内检查 closed；有等待接收者就直接交接，有缓冲空位就写 `sendx`，否则以 `sudog` 加入 `sendq` 并 park。`chanrecv` 对称地处理 closed+empty、等待发送者和 `recvx`。`gopark` 只挂起 G，P/M 可运行其他 G；`goready` 只是让等待者进入 runnable。
>
> `select` 的规范重点是求值顺序和伪随机选择。一般实现由 `selectgo` 生成随机 poll order 和稳定 lock order，按统一顺序锁多个 Channel，扫描就绪；若没有就绪 case，则把一个 G 以多个 `sudog` 挂到不同 Channel，唤醒后清理败选 case。随机不等于严格公平。
>
> 工程上我会明确数据 Channel 的 owner。单生产者在最后发送后 close；多生产者只发送，由协调者 `Wait` 后唯一 close。`sync.Once` 不解决 send 与 close 并发。Pipeline 和 Worker Pool 的每个阻塞 send/receive 都要能被 Context 打断，尤其是调用方超时后的一次性结果发送；容量 1 只对恰好一条被放弃的结果有帮助，不是通用防泄漏方案。
>
> 线上若 Goroutine 数上涨，我先看 goroutine dump 的 `[chan send]`、`[chan receive]` 和 `[select]` 聚类，再用 block profile 看累计阻塞、trace 看 park/ready 和调度延迟、race detector 查 send/close 与指针共享、heap profile 配合队列水位查对象积压。CPU 突增则优先排查 closed Channel 或空 default 的忙循环。缓冲区大小根据速率差、突发时长、延迟和内存预算决定，不能用任意大容量掩盖下游吞吐不足。

### 10.4 源码级回答：`hchan`、发送接收和 `selectgo`

> 以下是 Go 1.26.4 当前实现，不是规范承诺。编译器把 `make(chan)`、send、receive、close 和一般 select 分别降到 `makechan`、`chansend`、`chanrecv`、`closechan`、`selectgo` 等 runtime 路径；简单 select 会有专用优化。
>
> `hchan` 的核心字段包括当前元素数 `qcount`、容量 `dataqsiz`、环形缓冲 `buf`、下一发送和接收索引 `sendx/recvx`、等待队列 `sendq/recvq`、关闭标记和互斥锁。等待队列节点是 `sudog`，它把 G、Channel 和通信元素地址关联起来；一个 G 在 select 中可能同时拥有多个 `sudog`。
>
> `chansend` 在锁内先拒绝 closed Channel；若 `recvq` 有等待者，直接把值交给接收者并在解锁后唤醒；否则有缓冲空间就写环形槽位；再否则将发送者加入 `sendq` 并 `gopark`。`chanrecv` 对称处理：closed 且缓冲空时清零目标并返回 `ok=false`；有等待发送者就直接交接，缓冲满时还可一边取队首一边让等待发送者补入；有缓冲值则读 `recvx` 并清槽；否则入 `recvq` park。
>
> `closechan` 在锁内设置 closed，摘下所有接收和发送等待者，接收者标记为失败接收，发送者标记为失败发送；解锁后统一 ready。接收者最终得到零值和 false，发送者恢复后 panic，所以当前 close 成本与等待者数量相关。
>
> 一般 `selectgo` 生成随机 poll order 和按 Channel 排序的 lock order，按统一顺序锁住涉及 Channel。第一遍查立即就绪 case；无结果时第二遍为各 case 入队并 park；某一 case 唤醒后第三遍清除其他队列中的败选 `sudog`。统一锁序防止多 Channel 反向加锁，随机扫描减少固定偏置，但不提供严格公平。

## 十一、本章速记

1. **Channel 的零值是 nil；nil 收发永久阻塞，close nil panic。**
2. **无缓冲 Channel 是同步交接，不是容量为 1 的队列。**
3. **有缓冲 Channel 只吸收有限突发，不能修复长期生产率大于消费率。**
4. **向 closed Channel 发送 panic；从 closed Channel 接收不 panic。**
5. **关闭后先排空缓冲值，之后才返回零值和 `ok=false`。**
6. **`for range ch` 在 closed 且排空后退出；只停止发送但不 close 不会退出。**
7. **close 表示“不再发送”，不是释放资源或清空缓冲。**
8. **关闭权属于能证明再无 send 的角色，通常是发送方或协调者。**
9. **多发送者：发送者只 send，协调者 Wait 后唯一 close。**
10. **`sync.Once` 防重复 close，但不防 send 与 close 并发。**
11. **closed Channel 的接收在 select 中永远就绪，需 return 或设 nil。**
12. **nil Channel case 永不就绪，可用于动态禁用 select 分支。**
13. **`select default` 是立即非阻塞；空 default 循环会忙轮询。**
14. **多个就绪 case 伪随机选择，不保证轮询、优先级或无饥饿。**
15. **进入 select 时，所有 Channel 操作数和发送右值都会先求值一次。**
16. **`len(ch)`、`cap(ch)` 是快照，不能与后续 send/receive 组成原子判断。**
17. **Channel 值可比较、可作 Map Key；复制只会别名同一对象。**
18. **双向 Channel 可收窄成单向，不能从单向普通恢复为双向。**
19. **Channel 同步通信事件；发送指针不会自动冻结或深拷贝对象。**
20. **当前 runtime 用 `hchan`、环形缓冲、`sudog`、等待队列和锁实现。**
21. **阻塞路径通过 `gopark` 挂起 G，唤醒通过 `goready` 进入 runnable。**
22. **Channel 适合通信和所有权转移；Mutex 更适合保护共享状态。**
23. **容量 1 只解决特定的一次性弃收结果，不是通用泄漏修复。**
24. **Go 1.23+ 的 Timer Channel 语义已变化；旧版 `time.After` 结论要注明版本。**
25. **排查 Channel 问题要联合 goroutine dump、block profile、trace、race 和 heap/队列指标。**

### 12.4 参考答案

#### 简答题答案

1. **nil Channel 没有可通信对象，接收永远不就绪；closed Channel 是已有对象进入终止状态，排空后接收永远就绪并返回零值、`ok=false`。**
2. close 不清空缓冲区。缓冲中的值都来自成功发送，接收它们时 `ok=true`；只有排空后才 `ok=false`。
3. 各发送者只负责 send，并在退出时 `wg.Done()`；独立协调者 `wg.Wait()` 后执行唯一一次 `close(out)`。关闭前还要保证所有异步发送分支都结束。
4. 不优先。规范要求在当前可进行的通信中做均匀伪随机选择；源码顺序只影响进入 select 时表达式的求值顺序，不构成优先级。
5. 可答：对应 send 先于对应 receive 完成；close 先于因关闭返回零值的 receive；无缓冲 Channel 的 receive 完成先于对应 send 完成；容量 C 的第 k 次 receive 完成先于第 k+C 次 send 完成。
6. `len`/`cap` 是瞬时快照，检查后其他 G 可改变队列；检查与发送不是原子操作。非阻塞发送应使用 `select { case ch <- v: default: }`。
7. 不复制缓冲区，只复制指向同一 Channel 对象的值。Channel 可比较，因此可作为 Map Key；同一对象相等，不同 `make` 的对象不相等。
8. send 同步了指针值及 send 前的写，不会同步 send 后的任意写。接收方可能在发送者后续修改时同时访问同一对象，需要所有权转移、ack、Mutex 或不可变设计。
9. Go 1.23+ 未引用的未停止 Timer/Ticker 可以被 GC；Timer Channel 对外为同步语义，`len/cap` 为 0，Stop/Reset 对旧值有更强保证。因此“`time.After` 必然永久泄漏”已过时，但高频循环仍有重复分配和 Timer 管理成本。
10. 工作交接、事件流、Pipeline、背压和状态所有权转移优先 Channel；保护共享 Map、计数器或必须在同一临界区检查和修改的不变量优先 Mutex/Atomic。最终以语义清晰和测量结果为准。

#### 代码题答案

**代码题 1**

输出：

```text
0 true
0 false
```

第一个 0 是真实发送值，所以 `ok=true`；第二个 0 是 closed+empty 返回的元素零值，所以 `ok=false`。这正说明必须使用第二返回值区分关闭。

**代码题 2**

输出：

```text
closed
```

已关闭 Channel 的接收立即就绪，因此不会走 `default`。`default` 不是低优先级的候选通信；它只在没有任何通信 case 可立即完成时执行。

**代码题 3**

输出：

```text
default
```

两个 nil Channel 通信 case 都永不就绪，因此选择 default。删除 default 后，所有 case 都无法就绪，main 永久阻塞并最终触发全局死锁检测。

**代码题 4**

存在 data race，输出可能是 0 或 1。发送 `d` 只保证 send 前的初始化对接收方可见；`d.N = 1` 发生在 send 之后，与 main 中的 `fmt.Println(got.N)` 没有先后关系。可修改为：发送前完成所有写并在发送后不再访问；或让接收方处理完后通过 ack 归还所有权；或用 Mutex 保护 `N`。如果把 `<-done` 移到读取 `got.N` 之前，close/receive 会为写入建立顺序，也能消除这次竞争。

**代码题 5**

不会执行到打印。第一次发送填满容量 1 的缓冲区；第二次发送没有空位且没有接收者，main 被阻塞。由于接收语句位于第二次发送之后，永远无法执行，最终报告全局死锁。

#### 系统设计题答案

**系统题 1：可取消 Worker Pool**

- 创建 `jobs` 输入和 `results` 输出；输入由任务生产者在停止提交后关闭。
- 启动固定 32 个 worker，每个循环同时选择 `ctx.Done()` 和 `jobs`。
- worker 处理任务时把 Context 传给图片解码、存储和网络调用。
- 发送结果时再次选择 `results <- r` 与 `ctx.Done()`，避免结果消费者退出后阻塞。
- worker 不关闭共享 `results`；`WaitGroup` 等待 32 个 worker，全结束后由唯一协调者 close results。
- 若要求排空，停机顺序为停止接入 → close jobs → 等 worker 处理完 → close results；若要求立即取消，则 cancel 后 worker 尽快退出。两种策略要在 API 中明确。
- 设置有界 jobs 缓冲和满载策略，监控深度、等待时间、丢弃/拒绝量、Goroutine 数和处理延迟。

**系统题 2：Goroutine 与 Heap 同涨**

1. 先关联请求量、超时率、队列水位、Goroutine/Heap/GC 指标，确认增长不回落。
2. 抓多份 goroutine dump，按函数和行号聚类，确认 `[chan send]` 是否在超时调用的结果返回路径。
3. 开启合适采样的 block profile，确认累计阻塞热点；用 trace 观察调用方取消后工作 G 仍运行并阻塞发送。
4. 用 heap profile 看被这些 G 和 Channel 缓冲保留的请求/响应对象；运行 race detector 排除 send/close 和指针共享问题。
5. 止血：限制并发、取消上游、拒绝新请求或增加临时消费者；避免盲目扩大缓冲。
6. 根治：让昂贵操作接收 Context；结果发送选择 `ctx.Done()`；对严格单结果可用容量 1，但不把它当通用方案；统一关闭所有权。
7. 验证：超时故障注入和压力测试后，G 数回归基线、Heap 稳定、无重复阻塞栈、race 通过、P99 恢复，并检查优雅停机。

**系统题 3：严格多租户与持久化**

不应只依赖一个 `select` 的随机性和进程内 Channel。`select` 不保证权重、公平延迟上界；Channel 不持久化、不能跨重启恢复。

更合适的设计是：

- 使用持久化消息日志/队列保存任务、确认和重试状态。
- 在调度层维护每租户独立队列、权重、配额和 deficit round-robin/weighted fair queue 等明确算法。
- 进程内可用 Channel 把调度器选出的任务交给 worker，并用有界容量提供局部背压。
- 状态与 checkpoint 落在可恢复存储；定义幂等、至少一次/至多一次语义和死信策略。
- 指标按租户跟踪排队时延、吞吐、拒绝和饥饿告警。

Channel 在该系统中是进程内执行组件，而不是持久化公平调度系统本身。
