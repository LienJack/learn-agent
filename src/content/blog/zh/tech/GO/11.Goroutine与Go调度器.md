## 二、功能介绍

### 2.1 Goroutine 是什么

**[规范]** `go` 语句会在同一地址空间内启动一个独立、并发执行的控制流，称为 Goroutine。它不是语言层面定义的“轻量线程对象”，规范也没有规定必须使用 GMP、必须有多大栈、必须如何映射到操作系统线程。

**[实现]** 在官方 gc/runtime 中，Goroutine 由内部 `runtime.g` 表示，由用户态调度器把大量 G 多路复用到较少的操作系统线程 M 上；M 要执行普通 Go 代码，通常还必须持有一个 P。

**最重要的面试边界**：

- “Goroutine 由 Go runtime 调度”是当前官方实现事实。
- “Goroutine 一定比线程快多少倍”“每个 Goroutine 永远只占 2 KiB”都不是规范保证。
- Goroutine **轻量，但不是免费**：至少有 G 元数据、初始栈、调度记录、可能的闭包对象、GC 根扫描成本及队列操作成本。

### 2.2 Goroutine 与操作系统线程

| 维度 | Goroutine | OS Thread |
|---|---|---|
| 抽象层 | Go 语言并发执行单元 | 内核调度执行单元 |
| 创建者 | Go runtime | 操作系统内核/线程库 |
| 调度者 | Go runtime 先调度 G，内核再调度承载它的线程 | 内核直接调度 |
| 栈 | 当前实现为可增长、可收缩的连续 Go 栈 | 通常预留较大的固定或按页增长线程栈，策略由 OS 决定 |
| 切换 | G 间切换可在用户态完成 | 线程切换涉及内核调度 |
| 数量级 | 通常可远多于线程，但受内存、GC、调度和业务资源限制 | 数量通常更受线程栈、内核对象和切换成本限制 |
| 阻塞 I/O | 可借助 netpoll 只挂起 G | 阻塞系统调用会阻塞线程 |
| 亲和性 | 默认不绑定某个线程 | 线程本身就是内核实体 |

**不要把二者理解成简单的一一映射。** 一个 G 在生命周期内可能先后运行于不同 M；一个 M 也会顺序执行多个 G。只有 `runtime.LockOSThread`、cgo 回调、某些 GUI/图形库或依赖线程局部状态的代码，才需要显式线程亲和性。

### 2.3 并发与并行

- **并发（concurrency）**：多个任务的生命周期重叠，可以交替推进；一个 CPU 上也能并发。
- **并行（parallelism）**：同一时刻确实有多个任务在不同 CPU 上执行。

```text
GOMAXPROCS=1：可以有成千上万个并发 G，但普通 Go 代码同一时刻通常只在一个 P 上执行。
GOMAXPROCS=8：最多可有 8 个线程同时执行用户 Go 代码，但是否真正并行还取决于可运行任务、CPU 配额、亲和性和内核调度。
```

### 2.4 `go` 语句的精确语义

考虑：

```go
go f(a(), b())
```

**[规范]**：

1. 函数值 `f` 和参数 `a()`、`b()`在**发起 `go` 语句的 Goroutine 中**按通常规则求值。
2. 调用本身在一个新 Goroutine 中独立开始。
3. 当前 Goroutine不等待新 Goroutine 完成。
4. 被调用函数的返回值会被丢弃。
5. 新 Goroutine 在函数终止时结束。

因此，“异步的是函数体执行”，不是“连参数求值也异步”。参数求值产生的 panic 或阻塞也发生在发起方的求值阶段。

### 2.5 `go` 语句不保证立即执行

**[规范]** 只承诺启动独立并发执行，不承诺：

- 在下一条语句之前开始；
- 在调用方返回之前完成；
- 按创建顺序运行；
- 与其他 G 轮流执行；
- 在新 OS 线程上执行；
- 在某个固定时间内获得 CPU。

**[实现]** 当前 `newproc` 通常把新 G 以 `next=true` 放入当前 P 的 `runnext` 候选位，这是一种降低通信延迟的优化，但它仍不是语言层面的“立即运行”保证；`runnext` 也可能被替换、转入本地队列或被其他 P 窃取。

### 2.6 生命周期与可观察状态

面向业务代码，可以把 Goroutine 生命周期抽象为：

```text
创建 → Runnable → Running → Waiting/阻塞 → Runnable → Running → 退出
                     ↘ Syscall ↗
```

**[实现]** `runtime2.go` 中状态更细，包括 `_Gidle`、`_Grunnable`、`_Grunning`、`_Gsyscall`、`_Gwaiting`、`_Gdead`、`_Gcopystack`、`_Gpreempted` 等。Go 1.26 的实验性泄漏检测还引入了 `_Gleaked`。这些名字和编号是 runtime 内部协议，不是公开 API。

| 典型状态 | 含义 | 常见来源 |
|---|---|---|
| Runnable | 已可运行但尚未拿到执行资源 | 刚创建、被唤醒、被抢占 |
| Running | 正在某个 M 上执行 | 获得 P 后由 `execute` 切入 |
| Waiting | 在 runtime 内等待事件 | Channel、锁、Timer、netpoll、`select{}` |
| Syscall | 正在系统调用 | `syscall`、部分 cgo 路径 |
| Dead | 已退出或位于复用缓存 | `goexit0` 后等待复用 |

### 2.7 Goroutine 没有业务上的父子层级

创建者会在 runtime 调试字段中留下 `parentGoid`、创建 PC 等信息，但**这不等价于结构化的父子生命周期**：

- 父 G 退出，子 G 不会自动取消。
- 父 G 的 `defer recover()` 不能捕获子 G 的 panic。
- 子 G 的返回值不会自动传给父 G。
- runtime 不提供“join 某个 G”的公开句柄。

业务代码需要用 `Context`、Channel、`sync.WaitGroup`、`errgroup.Group` 或自定义任务组显式建立取消、等待和错误传播关系。

### 2.8 `main`、panic、recover 与 `Goexit`

#### `main` 返回

**[实现/API 约定]** 当 `main.main` 正常返回，程序退出；其他用户 Goroutine 不会被等待，也不会保证执行 defer 或清理逻辑。不要用 `Sleep` 猜测后台任务何时结束。

#### panic/recover

**[规范]** panic 的展开只沿**同一个 Goroutine**的调用栈执行 defer。只有同一 G 中、由发生 panic 的调用链触发的直接 deferred function 调用 `recover`，才能停止该 panic。一个未恢复的子 G panic 通常会终止整个进程。

#### `runtime.Goexit`

**[API]** `Goexit` 终止调用它的当前 G，并执行该 G 的 defer；它不是 panic，因此 defer 中的 `recover()` 返回 `nil`。在 main G 中调用 `Goexit` 不等同于 `main` 返回：其他 G 仍可运行；当其他 G 都结束后，程序会因无可运行用户 G 而崩溃。

### 2.9 常用公开 API 的边界

| API | 作用 | 不保证/不控制 |
|---|---|---|
| `runtime.GOMAXPROCS(n)` | 设置同时执行用户 Go 代码的最大并行度 | 不限制 G 数、总 M 数、阻塞 syscall/cgo 线程数 |
| `runtime.SetDefaultGOMAXPROCS()` | Go 1.25 起恢复运行时默认值及自动更新 | 不是“固定为 NumCPU” |
| `runtime.Gosched()` | 当前 G 主动让出执行机会，之后会自动恢复 | 不是同步原语，不建立 happens-before |
| `runtime.LockOSThread()` | 将当前 G 固定到当前 OS 线程 | 不等于加锁；调用必须与 `UnlockOSThread` 平衡 |
| `runtime.NumGoroutine()` | 返回当前存在的 G 数量 | 只是瞬时值，不直接等于泄漏数或 runnable 数 |
| `runtime.NumCPU()` | 启动时可用逻辑 CPU 数快照 | 不等同于当前 `GOMAXPROCS`，也不直接代表容器 CPU quota |

## 三、底层实现

### 3.1 为什么需要 GMP

如果每个 G 都绑定一个线程，则百万级 G 会变成百万级线程，线程栈、内核对象与上下文切换成本不可接受。若只使用一个全局队列和固定线程池，又会产生严重的全局锁竞争、缓存抖动和唤醒延迟。

当前 runtime 使用 GMP，把问题拆成三层：

```text
           可运行 G
        ┌──────────────┐
        │ local runq P0│──┐
        │ local runq P1│──┼── work stealing / global runq / netpoll / timers
        │ local runq Pn│──┘
        └──────────────┘
              │
          M 必须持有 P
              │
        OS Thread 由内核调度
```

- **G**：待执行的 Go 任务及其运行现场。
- **M**：machine，runtime 对 OS 线程的表示。
- **P**：processor，执行 Go 代码所需的调度与本地资源载体。

`GOMAXPROCS` 大体决定 P 的数量，从而决定同时执行普通 Go 代码的最大并行度。

### 3.2 G：Goroutine 的运行现场

`src/runtime/runtime2.go` 的 `type g struct` 主要保存：

- `stack [lo, hi)`：实际连续栈范围；
- `stackguard0/1`：函数序言栈检查与抢占触发相关；
- `_panic`、`_defer`：当前 G 的 panic/defer 链；
- `m`：当前承载它的 M；
- `sched gobuf`：切换时保存的 SP、PC、G 等上下文；
- `syscallsp/syscallpc/syscallbp`：系统调用期间供回溯和 GC 使用；
- `atomicstatus`：内部调度状态；
- `goid`：runtime 内部 ID；
- `schedlink`：调度链表链接；
- `waitreason`、`waiting`：阻塞原因与等待结构；
- `preempt`、`preemptStop`、`asyncSafePoint`：抢占相关；
- `parentGoid`、`gopc`、`startpc`：创建关系和诊断信息；
- pprof labels、trace、GC assist 等附加状态。

**面试要点**：G 不只是“函数 + 栈”，还必须保存可恢复执行的寄存器上下文、调度状态、等待原因、panic/defer、GC 和 profiling 元数据。

### 3.3 M：操作系统线程的 runtime 表示

`type m struct` 的关键字段包括：

- `g0`：该 M 的调度栈/系统栈 Goroutine；
- `curg`：当前正在执行的用户 G；
- `p`：当前持有的 P；
- `nextp`：即将接管的 P；
- `oldp`：进入 syscall 前关联的 P；
- `procid`：OS 线程 ID；
- `gsignal`：信号处理 G；
- `spinning`：是否正在主动寻找工作；
- `lockedg/lockedExt/lockedInt` 等：线程绑定状态；
- `incgo`、`ncgocall`：cgo 状态；
- `locks`、`mallocing`、`preemptoff`：是否允许抢占等内部约束。

**关系结论**：

1. 一个 M 基本对应一个 OS 线程。
2. M 可以没有 P，例如阻塞在 syscall/cgo 中或处于休眠状态。
3. M 要执行普通用户 Go 代码，通常必须持有 P。
4. M 数量可以大于 `GOMAXPROCS`，尤其在大量阻塞系统调用、cgo、`LockOSThread` 时。

### 3.4 P：并行执行权与本地资源

`type p struct` 不只是“CPU 核心映射”。它保存或关联：

- 本地可运行队列 `runq` 与 `runnext`；
- `mcache`、`pcache` 等每 P 内存分配缓存；
- per-P timer heap；
- defer/sudog 等本地缓存；
- GC work buffer 与 mark worker 状态；
- 调度 tick、syscall tick、sysmon 观察值；
- trace buffer、随机状态等。

**为什么 P 很重要**：把频繁使用的调度、分配和 GC 资源下沉到每 P，可减少全局锁和跨核缓存争用。P 不是线程；一个 P 可以先后被不同 M 持有。

### 3.5 本地队列、全局队列与 `runnext`

#### 本地 run queue

**[实现]** Go 1.26 的每 P 本地环形队列当前是 `runq [256]guintptr`，通过 `runqhead/runqtail` 管理。所有者 P 的常见入队出队路径可避免全局调度锁。

- 队列未满：新 G 放本地。
- 队列满：`runqputslow` 会把一批 G 转移到全局队列，而不是简单丢弃或无限扩容。

#### 全局 run queue

全局队列位于共享调度状态中，用于：

- 本地队列溢出；
- 批量注入可运行 G；
- 维持跨 P 的可见性和公平性；
- 某些唤醒、系统调用返回或 runtime 路径。

访问通常需要调度锁，因此不会让所有普通操作都依赖它。

#### `runnext`

每个 P 当前有一个 `runnext` 槽。被当前 G 唤醒的新 G可能被优先放到这里，并继承当前时间片剩余部分。这对“通信后立即等待”的 handoff 模式有利：

```text
G1 发送/解锁 → 唤醒 G2 → G2 很快运行 → 减少队尾等待
```

但 `runnext` 不是严格优先级机制：

- 只能容纳一个候选；
- 旧候选可能被挤入普通本地队列；
- 其他 P 可原子地窃取它；
- 连续 `runnext` 链共享同一调度 tick，`sysmon` 仍会防止长期霸占。

### 3.6 Goroutine 创建：`newproc` 与 `newproc1`

编译器把 `go` 语句降低为调用 runtime 创建路径。当前核心流程：

```text
调用方求值函数值和参数
        ↓
runtime.newproc(fn)
        ↓ systemstack
runtime.newproc1(fn, callergp, callerpc, ...)
        ↓
从 P 的 gFree/全局空闲池复用 G，或 malg(stackMin)
        ↓
初始化新 G 的栈、sched.pc/sp、goexit 返回帧、goid、parentGoid、gopc
        ↓
状态转为 _Grunnable
        ↓
runqput(currentP, newg, true)
        ↓
wakep()（必要时唤醒/创建执行者）
```

`newproc1` 当前在无可复用 G 时调用 `malg(stackMin)`，其中 `stackMin` 当前为 2048 字节。**这表示初始分配下限的实现值，不表示每个 G 永远只使用 2 KiB，也不等于总内存成本。**

### 3.7 调度主循环：`schedule → findRunnable → execute`

#### `schedule`

`schedule()` 运行在 M 的调度栈上，完成一轮调度且不返回到原调用点。它会处理锁定 G、GC worker、trace 等特殊情况，然后寻找可运行 G。

#### `findRunnable`

`findRunnable()` 是核心找工作逻辑，可能检查：

1. stop-the-world/GC 标记任务；
2. 当前 P 到期 timer；
3. 为公平性定期检查全局队列；
4. 当前 P 的 `runnext` 和本地队列；
5. 全局队列；
6. netpoll 已就绪 G；
7. 从其他 P work stealing；
8. 无工作时释放 P、让 M spinning/stop，并在休眠前再次防丢唤醒检查。

**[实现]** 当前源码中每个 P 的 `schedtick % 61 == 0` 时会优先看一次全局队列，避免两个持续互相唤醒的 G 长期占据本地调度而饿死全局任务。`61` 是实现常数，不是语言保证。

#### `execute`

`execute(gp, inheritTime)` 大致执行：

- 把 G 从 `_Grunnable` 切换为 `_Grunning`；
- 绑定 `m.curg` 与 `g.m`；
- 恢复正常 stack guard；
- 根据是否继承时间片更新调度 tick；
- 发出 trace 事件；
- 最终通过 `gogo(&gp.sched)` 恢复 G 的寄存器现场。

### 3.8 `gopark` 与 `goready`

它们是许多阻塞原语与调度器之间的底座。

#### `gopark`

当前 G 需要等待时，runtime 会记录：

- 解锁/提交等待的回调；
- 等待对象；
- `waitreason`；
- trace 阻塞原因；

随后通过 `mcall(park_m)` 切到 g0，令 G 从 Running 进入 Waiting，并让 M/P 去执行别的工作。

#### `goready`

事件满足时，`goready` 在 system stack 上调用 `ready`：

```text
_Gwaiting → _Grunnable → runqput(..., next=true) → wakep()
```

这就是 Channel 收发、锁释放、Timer 到期、netpoll 就绪等事件“唤醒 Goroutine”的共同调度骨架。

### 3.9 Work stealing 与调度公平性

当某 P 没有本地工作时，它不会立刻休眠，而会尝试从其他 P 窃取工作。当前 `runqsteal` 通常尝试窃取受害 P 可运行队列的大约一半，以平衡：

- 偷太少：频繁窃取，开销高；
- 偷太多：破坏受害者局部性，反向失衡。

调度公平性来自多套机制共同作用：

- 全局队列周期性检查；
- work stealing；
- netpoll 与 timer 注入；
- 时间片与 sysmon 抢占；
- 阻塞点自然让出；
- spinning M 控制，避免无工作时无限忙转；
- 抢占及 STW safe point。

**不要回答“Go 调度器严格公平”。** 它追求吞吐、延迟与局部性的工程平衡，不提供 FIFO、优先级或最大等待时间的语言保证。

### 3.10 `sysmon`：不依赖 P 的系统监控线程

`sysmon` 是 runtime 的监控循环，通常运行在不需要 P 的特殊 M 上。其职责随版本演进，当前重要工作包括：

- 检查长时间运行的 P/G并触发抢占；
- 从长时间系统调用中 retake P；
- 处理 netpoll、timer 相关唤醒；
- 触发周期性 GC；
- 输出 `schedtrace`；
- 处理 runtime 的若干后台维护任务。

当前 `retake` 中 `forcePreemptNS` 为约 10 ms，用于判断同一调度 tick 是否运行过久并请求抢占。它不是“每个 G 精确运行 10 ms”的保证：检测有周期、抢占需满足安全条件、run queue 与系统负载也会影响实际延迟。

### 3.11 Netpoller：为什么网络 I/O 通常不独占线程

以支持 runtime poller 的 `net.Conn.Read` 为例：

1. fd 被设为适合事件轮询的模式并注册到平台 poller（Linux 常见 epoll，BSD/macOS 常见 kqueue，Windows 使用相应机制）。
2. 读操作暂不可完成时，`internal/poll` 进入 runtime poll wait。
3. `netpollblock` 把当前 G登记到 `pollDesc.rg/wg` 并调用 `gopark`。
4. G 进入 Waiting；原 M/P 可运行其他 G。
5. 内核报告 fd 就绪，平台 `netpoll` 调用 `netpollready` 收集相应 G。
6. G 被改为 Runnable，重新进入调度队列。

因此，**等待网络事件的主要成本是挂起 G，而不是为每个连接长期占住一个 M**。边界包括：

- 只有可被 Go poller 管理的描述符和路径才有此效果；
- DNS、文件系统、设备 I/O、第三方库、直接 syscall/cgo 可能走不同路径；
- I/O 就绪只表示可重试，不保证一次读写满足全部业务数据；
- 网络处理函数中的 CPU 计算、锁等待仍会占用执行资源。

### 3.12 Timer 与调度器

**[实现]** 当前 timer 集合是 per-P 的 `timers`，核心包含受锁保护的最小堆，按到期时间排序。P 在找工作时会检查到期 timer；无普通工作时，调度器会把下一到期时间与 netpoll 等待时间结合，避免忙轮询。

Timer 到期可能：

- 使等待 `time.Timer.C` / deadline 的 G 变为 Runnable；
- 执行 `time.AfterFunc` 注册的函数；
- 周期性推进 Ticker。

Timer 并不意味着“精确在该时刻运行”：到期只是变为可执行条件，真正执行还受调度、CPU 配额、STW、系统时钟与负载影响。

### 3.13 阻塞系统调用与 P handoff

当 G 进入可能阻塞的系统调用：

```text
G Running on M+P
    ↓ entersyscall / entersyscallblock
G = _Gsyscall，M 进入内核
    ↓
P 可被 sysmon 或显式路径交给其他 M（handoff/retake）
    ↓
其他 G 继续使用该 P 执行
    ↓ syscall returns
exitsyscall 尝试拿回 oldp 或空闲 P
    ↓
成功：继续运行；失败：G 变 Runnable，M 进入调度/休眠
```

关键结论：

- 阻塞 syscall 会占住一个 M/OS thread，但不应长期占住 P。
- 因此大量阻塞 syscall 可使 M 数明显超过 `GOMAXPROCS`。
- `handoffp` 根据本地/全局工作、GC、trace、timer 等判断是否立即为 P 启动另一个 M。
- syscall 返回的 G 不能无视 `GOMAXPROCS` 直接执行 Go 代码，必须重新获得 P。

### 3.14 cgo 对调度和线程的影响

`runtime.cgocall` 通常调用 `entersyscall`，切到 M 的 g0/OS 栈执行 C，再通过 `exitsyscall` 回到 Go。结果是：

- 长时间 C 调用会占用其 M/OS thread；
- P 可供其他 M 执行 Go；
- 大量并发 cgo 阻塞可造成 OS 线程数上升；
- C 代码阻塞、线程局部状态、回调 Go、信号与指针规则使路径比普通 netpoll 更复杂；
- `GOMAXPROCS` 仍只限制同时执行 Go 代码的并行度，不直接限制处于 C 中的线程数。

**Go 1.26 版本说明**：发布说明称 cgo 调用的基线 runtime 开销降低约 30%，但这不消除跨语言边界、线程占用、调度和数据转换成本。

### 3.15 Goroutine 栈：连续、动态增长与复制

#### 初始栈

当前 `stackMin` 为 2048 字节。新 G 的栈通常从小尺寸开始，避免为大量短小 G 预留大块内存。

#### 连续栈

现代 Go 使用可移动的**连续栈**，不是早期版本的分段栈。连续内存有利于普通调用约定和局部性，但增长时需要搬迁。

#### `morestack → newstack`

编译器通常在函数序言生成 stack bound check。空间不足时：

```text
函数序言检查失败
   ↓ runtime.morestack（汇编入口）
切到 g0/system stack
   ↓ runtime.newstack
计算需要的新尺寸，通常按倍数增长并确保容纳目标帧
   ↓ copystack
复制已使用栈数据并调整指针
   ↓ 恢复 G，在新栈继续执行
```

`newstack` 还会识别 `stackPreempt`：同一套栈检查机制也可成为同步抢占入口。

#### 栈收缩

runtime 在安全时机可收缩过大的 G 栈。当前实现通常在已用空间明显小于容量时复制到更小栈，但不会低于最小固定栈。处于 syscall、异步 safe point 或某些 Channel 交互窗口时，不能随意移动栈。

### 3.16 栈上指针如何调整

复制栈不能只做 `memmove`。旧栈中的地址发生变化，runtime 必须：

1. 计算新旧栈地址差 `delta`；
2. 复制当前实际使用区域；
3. 根据编译器生成的 stack map/GC metadata 找出指针槽；
4. 调整指向旧栈区间的指针；
5. 更新 `g.sched`、frame pointer、defer 链、panic 链；
6. 处理 `sudog.elem` 等可能指向 G 栈的 Channel 等待结构，并与并发收发同步；
7. 更新栈边界、SP、stack guard；
8. 释放或缓存旧栈。

这也是 Go 对 `unsafe.Pointer`、把 Go 指针伪装成 `uintptr` 并跨调用长期保存等行为有严格边界的原因之一：runtime 只有在可识别的位置才能安全追踪和搬迁指针。

### 3.17 协作式抢占、异步抢占与 safe point

#### Go 1.14 以前的主要问题

早期抢占高度依赖函数调用序言的 stack check、阻塞调用或显式让出。一个没有调用、没有阻塞的长 CPU 循环可能较长时间不让出，影响调度公平性和 STW。

#### Go 1.14 起的异步抢占

Go 1.14 引入基于信号等 OS 机制的异步抢占能力。当前 `preempt.go` 将 safe point 分成：

1. **Blocked safe point**：G 已因同步、syscall 等不运行；
2. **Synchronous safe point**：G 主动执行函数序言等抢占检查；
3. **Asynchronous safe point**：用户代码中 runtime 可安全暂停、保守扫描寄存器/栈并注入 `asyncPreempt` 的指令位置。

异步抢占大致为：runtime 请求抢占 → 暂停目标 M → 检查 PC、SP、runtime 锁、写屏障原子区、栈空间等是否安全 → 修改信号上下文使其像调用了 `asyncPreempt` → 保存寄存器 → 进入调度器。

#### 长 CPU 循环还能否被抢占

**通常可以，但不是“任意指令无条件立即抢占”。** 下列情况会影响时机：

- 当前指令不是 async safe point；
- runtime 正处于不可抢占区、持有关键锁或执行写屏障原子序列；
- 平台不支持或通过 `GODEBUG=asyncpreemptoff=1` 关闭；
- cgo/内核代码不在普通 Go 异步抢占路径中；
- 调度请求和信号处理本身有延迟。

### 3.18 `runtime.Gosched` 与 `runtime.LockOSThread`

#### `Gosched`

`Gosched` 让当前 G 交还执行机会，之后自动重新变为可运行。它适合极少数主动协作或 runtime/算法实验场景，但**不能代替 Channel、Mutex、atomic、WaitGroup**；它不建立可见性或完成顺序。

#### `LockOSThread`

`LockOSThread` 把当前 G 固定到当前 M/OS thread；调用可嵌套，必须由等量 `UnlockOSThread` 解除。若 G 在仍锁定时退出，runtime 会终止该 OS 线程，而不是把受线程局部状态污染的线程交给其他 G。

适用场景：

- UI 主线程要求；
- OpenGL、某些驱动或库的 thread-local context；
- 调用依赖 per-thread namespace/credential/state 的非 Go 库；
- 与 C 线程亲和性协议集成。

不适用：为了“让代码更快”、保护共享变量、实现普通互斥。

### 3.19 `GOMAXPROCS` 的准确含义

**[API]** `GOMAXPROCS` 设置可同时执行的 CPU/用户 Go 代码并行度，当前通常对应活跃 P 数。

它控制：

- 同时执行普通 Go 代码的最大线程数；
- P 数量和 per-P 资源规模；
- CPU-bound Go 工作可获得的并行上限；
- GC 可用并行资源的若干部分。

它不直接控制：

- Goroutine 总数；
- runtime 拥有的 OS 线程总数；
- 阻塞在 syscall/cgo 的线程数；
- 外部 C 库自己创建的线程；
- 内核是否对容器进行 CPU throttling；
- 请求并发、数据库连接数、下游 QPS。

### 3.20 Go 1.25/1.26 的容器感知 `GOMAXPROCS`

从 Go 1.25 起，默认值在 Linux 上会综合：

- 机器逻辑 CPU 数；
- 进程 CPU affinity mask；
- cgroup CPU quota/period 得到的平均 CPU throughput limit。

当前 Go 1.26.4 文档给出的主要实现规则：

1. 若环境变量 `GOMAXPROCS` 为正整数，使用它并停止自动更新。
2. 否则通常取逻辑 CPU、affinity 数和 cgroup throughput limit 的较小值。
3. cgroup 结果为小数时向上取整。
4. 除非逻辑 CPU 或 affinity 本身低于 2，否则默认值不会因 cgroup 计算降到 2 以下。
5. 默认值可随 CPU 数、affinity 或 cgroup quota 变化而周期更新，最多约每秒一次，空闲时更少。
6. 显式调用 `runtime.GOMAXPROCS` 会关闭自动更新；`runtime.SetDefaultGOMAXPROCS()` 可恢复默认和更新。
7. `GODEBUG=containermaxprocs=0` 可退回旧式默认；`GODEBUG=updatemaxprocs=0` 可关闭自动更新。
8. 对 `go.mod` 语言版本 1.24 及以下，兼容性默认会关闭这两项新行为，除非显式覆盖。
9. 使用的是 **CPU limit/quota**，不是 Kubernetes CPU request。

**工程边界**：CPU quota 是一段时间内的吞吐预算，而 `GOMAXPROCS` 是瞬时并行度。把 1.5 CPU quota 向上取到 2，是在可用并行与 throttling 风险之间的默认折中，不表示任意时刻都有两个完整 CPU。

### 3.21 Goroutine 为什么轻量但不免费

设创建 `N` 个 G，成本大体包括：

- `N` 个 `g` 元数据和初始/增长栈；
- 闭包捕获及参数逃逸造成的堆对象；
- GC 对所有活跃 G 栈及根的扫描；
- 运行队列入队、出队、窃取和唤醒；
- 任务自己的缓冲、连接、Timer、Context、Channel；
- runnable 过多导致 scheduler latency；
- waiting 过多导致内存驻留和诊断成本；
- 栈增长时的复制成本；
- trace/profile 开启后的额外记录成本。

大量 G 的真正大头常常不是 2 KiB 初始栈本身，而是**每个 G 关联的业务对象、缓冲区、请求上下文和下游资源**。

### 3.22 CPU-bound 与 I/O-bound 并发度

#### CPU-bound

起始值通常接近有效 `GOMAXPROCS`，再用 benchmark 和生产指标调整。远超 P 数的 runnable CPU G 不会增加物理算力，反而可能增加：

- 排队和调度延迟；
- cache/TLB 失效；
- 共享数据竞争；
- GC root 与栈扫描；
- 尾延迟。

#### I/O-bound

并发度可高于 `GOMAXPROCS`，因为大量 G 在等待 I/O；但应按以下最小瓶颈限流：

```text
连接池容量、下游限额、内存预算、文件描述符、单请求缓冲、超时预算、目标尾延迟
```

**不是“越多越好”**。常用 Little's Law 估算：`并发量 ≈ 到达率 × 平均在途时间`，然后以压测验证。

### 3.23 False sharing 与 cache contention

即使调度器让 CPU 工作完全并行，共享缓存行仍可能让吞吐下降：

```go
type Counters struct {
    a atomic.Int64 // CPU 0 高频写
    b atomic.Int64 // CPU 1 高频写，可能与 a 同一 cache line
}
```

两个核反复写同一 cache line，会产生缓存一致性流量；这不是 data race，却可能比单线程更慢。常见优化：

- 每 P/每 worker 分片计数，最后聚合；
- 减少共享写；
- 批量更新；
- 谨慎做 cache-line padding，并以目标架构 benchmark 证明收益；
- 不要把 runtime 内部的 `P` 当作稳定公开 shard ID。

### 3.24 Scheduler latency 与 runnable backlog

- **scheduler latency**：G 已经 Runnable 到真正 Running 的等待时间。
- **runnable backlog**：已可运行、但等待 P/M 执行的 G 积压。

Go 1.26 的 `runtime/metrics` 提供：

- `/sched/latencies:seconds`：Runnable 状态等待时间直方图；
- `/sched/goroutines/runnable:goroutines`：近似 Runnable G 数；
- `/sched/goroutines/running:goroutines`；
- `/sched/goroutines/waiting:goroutines`；
- `/sched/goroutines/not-in-go:goroutines`；
- `/sched/goroutines-created:goroutines`；
- `/sched/goroutines:goroutines`；
- `/sched/gomaxprocs:threads`；
- `/sched/threads/total:threads`。

这些实现定义的指标集合可能演进，采集器应先检查 `metrics.All()`，而不是假定所有 Go 版本都存在同名指标。

**判断思路**：

| 现象 | 可能解释 |
|---|---|
| runnable 高、CPU 已满、latency 高 | CPU 饱和、GOMAXPROCS/配额不足、CPU 任务过量 |
| waiting 高、CPU 低 | I/O/锁/Channel/Timer 等等待；不一定是调度器问题 |
| thread 总数升高、not-in-go 高 | 阻塞 syscall、cgo、LockOSThread 或线程泄漏 |
| G 总数持续升高但完成率不变 | 可能存在 Goroutine 泄漏或无界排队 |
| runnable 不高但请求延迟高 | 下游、锁、GC、I/O、应用队列或 tracing 范围之外的问题 |

### 3.25 复杂度与设计取舍总结

| 操作 | 当前典型成本 | 关键边界 |
|---|---|---|
| 创建 G | 摊销接近 O(1)，但可能分配栈/G/闭包 | 大量创建仍有 GC 和内存成本 |
| 本地 runq 入/出队 | O(1) | 满时批量转全局 |
| 全局队列操作 | O(1) 或批量 O(k)，需要共享锁 | 竞争随 P/负载上升 |
| work stealing | 扫描 P 并批量窃取 | 无工作时可能有探测开销 |
| G 栈增长 | 与当前已使用栈大小 O(used stack) | 复制和指针调整，不是 O(1) |
| netpoll 等待 | G park；内核事件轮询集中处理 | 仅适用于可轮询 fd/path |
| 阻塞 syscall | 占一个 M，P 可 handoff | 大量调用导致线程数增长 |
| G 抢占 | 请求和切换近似常数，但时机依赖 safe point | 不保证严格最大等待时间 |

## 四、源码阅读路径

阅读调度器源码最容易犯的错误，是从 `schedule` 直接往下钻，最后陷入汇编、GC 与平台分支。更有效的方法是先建立**状态转换和调用链**，再按一条业务路径纵向阅读。

### 4.1 阅读前先固定三个边界

1. **固定版本**：本章源码均以 `go1.26.4` tag 为准。不要把 `master`、旧博客和当前二进制混在一起。
2. **固定平台**：优先看 `linux/amd64` 或你的生产平台；`*_windows.go`、`*_wasm.go`、汇编入口可能不同。
3. **固定问题**：一次只追一条链，例如“新 G 如何开始运行”或“网络读如何被唤醒”。

推荐先掌握这些源码记号：

| 记号 | 含义 |
|---|---|
| `gp` / `mp` / `pp` | `*g` / `*m` / `*p` 的惯用变量名 |
| `getg()` | 取得当前 G；编译器/runtime 特殊处理 |
| `mcall` | 从用户 G 切到当前 M 的 g0 栈执行 runtime 函数 |
| `systemstack` | 在系统栈执行闭包 |
| `casgstatus` | 原子转换 G 状态并维护相关约束 |
| `acquirem` / `releasem` | 临时禁止当前 M 被抢占，以保护运行时临界步骤 |
| `throw` | runtime 内部不可恢复错误，不等同于业务 `panic` |
| `//go:nosplit` | 函数序言不插入普通栈增长检查；使用受严格约束 |
| `//go:nowritebarrierrec` | 禁止当前函数及递归调用链写屏障，常用于调度/GC 关键路径 |

### 4.3 路径一：`go f(x)` 如何变成可运行 G

建议按下面顺序阅读：

```text
源代码 go f(x)
  ↓ 编译器先在当前 G 求值函数值和参数，并生成调用包装
runtime.newproc
  ↓ 切到 system stack
runtime.newproc1
  ↓ 从空闲 G 池取 G 或分配 G，准备栈和 sched 上下文
runqput(pp, newg, true)
  ↓ 优先尝试 runnext，必要时入本地队列
wakep()
  ↓ 若存在空闲 P 且没有自旋 M，尝试唤醒/创建 M
```

阅读时回答以下问题：

1. 参数求值为什么不在 `newproc1` 中发生？
2. `newproc1` 为什么必须在 system stack 上运行？
3. 新 G 的 `startpc`、`gopc`、`parentGoid` 在哪里设置？
4. 新 G 为什么先从 `_Gdead` 转为 `_Grunnable`？
5. `runqput(..., true)` 中的 `true` 为什么只是 `runnext` 候选，不是立刻执行承诺？
6. 什么情况下需要 `wakep`？为什么不能每创建一个 G 都新建线程？

### 4.4 路径二：调度循环如何找到下一个 G

```text
schedule()
  ├─ 检查当前 M/P 是否满足调度约束
  ├─ findRunnable()
  │    ├─ 偶尔检查 global runq
  │    ├─ 检查本地 runnext/local runq
  │    ├─ 检查 GC worker、finalizer、trace 等特殊工作
  │    ├─ 检查 netpoll 与 timer
  │    ├─ work stealing：从其他 P 窃取
  │    └─ 实在无工作：释放 P、让 M 休眠或阻塞 netpoll
  └─ execute(gp, inheritTime)
       ├─ _Grunnable → _Grunning
       ├─ 建立 gp.m / mp.curg
       └─ gogo(&gp.sched) 恢复用户代码
```

重点不要只背“先本地、再全局、再偷取”。当前实现为了公平性，会周期性检查全局队列；查找顺序还会受 GC、trace、timer、netpoll、自旋状态等影响。面试中应说“**大致优先本地，但存在公平性和系统任务插入点**”。

推荐在 `proc.go` 中依次搜索：

```text
func schedule
func findRunnable
func execute
func runqget
func globrunqget
func runqsteal
func stopm
func startm
func wakep
```

然后画出两条状态转换：

```text
_Grunnable --execute--> _Grunning
_Grunning  --park/preempt/Gosched--> _Gwaiting/_Grunnable
```

### 4.5 路径三：Channel/锁等待如何 park 和 ready

业务原语通常不会“让线程睡眠”，而是把当前 G 挂到原语的等待队列，再调用调度器：

```text
用户 G 发现条件不满足
  ↓ 把 sudog/等待节点挂到 Channel、sema、pollDesc 等结构
runtime.gopark(unlockf, lock, reason, traceReason, traceskip)
  ↓ mcall(park_m)，切到 g0
_Grunning → _Gwaiting
  ↓ schedule 运行其他 G

事件到达/对端操作
  ↓ 从等待队列取出目标 G
runtime.goready / ready
_Gwaiting → _Grunnable
  ↓ 放入某个 P 的可运行队列，必要时 wakep
```

阅读 `gopark` 时重点看 `unlockf`：它用于在 G 已安全进入等待状态后释放关联锁，避免经典的“先解锁后 park 导致丢失唤醒”竞态。

### 4.6 路径四：网络 I/O、Timer 与调度器的汇合

#### 网络 I/O

```text
net.(*netFD).Read
  ↓ internal/poll.FD.Read
  ↓ 非阻塞 fd 返回 EAGAIN
poll_runtime_pollWait
  ↓ netpollblock：把 G 挂到 pollDesc 的读/写等待位
  ↓ gopark

epoll/kqueue/IOCP 报告就绪
  ↓ runtime.netpoll
  ↓ netpollready
  ↓ 将等待 G 组成 runnable 列表
  ↓ injectglist / ready
```

要同时阅读 `internal/poll` 与 `runtime/netpoll.go`，否则只看 runtime 很难理解 fd 为什么已经设置为非阻塞。

#### Timer

当前实现中 Timer 主要归属于 P 的 timer 集合。阅读：

```text
src/runtime/time.go:
  type timer
  type timers
  (*timer).maybeAdd
  (*timers).addHeap
  (*timers).check
  (*timers).run
```

然后回到 `findRunnable`，观察下一 Timer 截止时间如何影响 netpoll 的阻塞时长，以及到期 Timer 如何形成可运行工作。

### 4.7 路径五：阻塞系统调用与 P handoff

```text
用户 G
  ↓ entersyscall / entersyscallblock
_Grunning → _Gsyscall
  ↓ M 进入内核；P 可被释放或由 sysmon retake
  ↓ 其他 M 接管 P，继续执行 Go 代码

系统调用返回
  ↓ exitsyscall
  ├─ 快路径：重新获得原 P/空闲 P，继续执行
  └─ 慢路径：G 变为 runnable，M 可能休眠
```

需要对比三类路径：

| 路径 | G | M | P |
|---|---|---|---|
| netpoll 等待 | park | 通常不被该等待长期占用 | 可运行其他 G |
| 阻塞 syscall | `_Gsyscall` | 被 syscall 占用 | 可 handoff 给其他 M |
| cgo 调用 | 类似跨 runtime 边界，M 可能长期在 C | 被 C 调用占用 | 通常释放给其他 M；还涉及回调与线程状态 |

### 4.8 路径六：栈增长与异步抢占

先从普通函数序言理解：

```text
检查 SP 是否接近 stackguard0
  ├─ 否：继续函数体
  └─ 是：调用 morestack
           ↓ 切到 g0
         newstack
           ├─ 若是抢占请求：在安全条件下处理抢占
           └─ 若空间不足：分配更大栈、copystack、调整栈内指针
```

再阅读 `preempt.go`：异步抢占会向目标 M 发送平台信号，在可异步抢占的 PC 上把执行流改写到异步抢占入口。面试时必须保留边界：

- 抢占不是任意机器指令处强行切走；runtime 要判断 PC、栈、锁和 unsafe-point 状态。
- `//go:nosplit`、runtime 临界区、持有内部锁等区域可能暂时不可抢占。
- 抢占改进了公平性和 GC STW 响应，但不构成实时调度保证。

### 4.9 推荐的断点与观测顺序

调试 runtime 通常不宜直接对生产二进制做单步；更实用的是“观测 → 定位 → 最小复现 → 源码映射”：

1. `go test -race` 排除明显数据竞争。
2. goroutine profile 查看大量 G 堆栈与 wait reason。
3. `GODEBUG=schedtrace=1000,scheddetail=1` 查看 P/M/G 快照。
4. `go tool trace` 看 runnable 到 running、syscall、netpoll、GC 和 processor 视图。
5. `runtime/metrics` 连续采样 `/sched/*` 指标。
6. 用最小 benchmark/reproducer 固定工作负载。
7. 最后对照 `proc.go`、`netpoll.go`、`stack.go` 的对应路径。

## 五、常用场景

### 5.1 先定义结构化并发的四个不变量

一个生产级并发任务组通常应同时满足：

1. **有所有者**：谁创建，谁负责等待或移交所有权。
2. **可停止**：超时、取消、错误或服务关闭时能退出。
3. **有上限**：并发数、队列长度和资源使用有明确边界。
4. **可观测**：任务数、队列深度、错误、延迟和取消原因可监控。

“启动一个 Goroutine”只是实现动作，不是并发设计。

### 5.2 请求内并行：`errgroup` + `Context` + 限流

`golang.org/x/sync/errgroup` 不是标准库，但它提供了常用的等待、首错传播和 Context 取消。下面的代码让多个后端请求并行，并把并发数限制在 `limit`：

```go
package fanout

import (
    "context"
    "errors"
    "fmt"
    "io"
    "net/http"

    "golang.org/x/sync/errgroup"
)

type Result struct {
    URL  string
    Body []byte
}

func FetchAll(
    ctx context.Context,
    client *http.Client,
    urls []string,
    limit int,
) ([]Result, error) {
    if client == nil {
        return nil, errors.New("nil http client")
    }
    if limit <= 0 {
        return nil, fmt.Errorf("invalid concurrency limit %d", limit)
    }

    g, ctx := errgroup.WithContext(ctx)
    g.SetLimit(limit)

    results := make([]Result, len(urls))
    for i, rawURL := range urls {
        // Go 1.22+ 对 := 声明的循环变量已按迭代重新创建；显式复制仍便于兼容旧语言版本。
        i, rawURL := i, rawURL
        g.Go(func() error {
            req, err := http.NewRequestWithContext(ctx, http.MethodGet, rawURL, nil)
            if err != nil {
                return err
            }
            resp, err := client.Do(req)
            if err != nil {
                return err
            }
            defer resp.Body.Close()

            if resp.StatusCode < 200 || resp.StatusCode >= 300 {
                return fmt.Errorf("GET %s: status %s", rawURL, resp.Status)
            }
            body, err := io.ReadAll(resp.Body)
            if err != nil {
                return err
            }
            results[i] = Result{URL: rawURL, Body: body}
            return nil
        })
    }

    if err := g.Wait(); err != nil {
        return nil, err
    }
    return results, nil
}
```

关键点：

- `WithContext` 的派生 Context 会在首个任务报错或 `Wait` 返回时取消。
- `SetLimit` 限制的是该 group 的活跃函数数，不是进程全局并发。
- `g.Go` 可能因达到上限而阻塞，因此不要在持有业务锁时调用。
- 每个任务写不同的 slice 元素，不产生同一内存位置并发写；`Wait` 后再读取结果。
- HTTP 客户端仍需配置连接池、超时和响应体上限；Goroutine 限制不能代替下游保护。

### 5.3 固定 Worker Pool：适合持续任务流

```go
package workerpool

import (
    "context"
    "errors"
    "sync"
)

type Job struct {
    ID int
}

type Result struct {
    JobID int
    Err   error
}

type Handler func(context.Context, Job) error

func Run(
    ctx context.Context,
    workers int,
    jobs <-chan Job,
    handle Handler,
) (<-chan Result, error) {
    if workers <= 0 {
        return nil, errors.New("workers must be positive")
    }
    if handle == nil {
        return nil, errors.New("nil handler")
    }

    out := make(chan Result)
    var wg sync.WaitGroup
    wg.Add(workers)

    for range workers { // Go 1.22+：可对整数 range
        go func() {
            defer wg.Done()
            for {
                select {
                case <-ctx.Done():
                    return
                case job, ok := <-jobs:
                    if !ok {
                        return
                    }
                    result := Result{JobID: job.ID, Err: handle(ctx, job)}
                    select {
                    case out <- result:
                    case <-ctx.Done():
                        return
                    }
                }
            }
        }()
    }

    go func() {
        wg.Wait()
        close(out)
    }()
    return out, nil
}
```

**所有权约定必须写进 API 文档**：

- 调用方拥有并关闭 `jobs`，或通过取消 `ctx` 终止。
- `Run` 拥有并关闭 `out`。
- Worker 不关闭输入 Channel。
- 消费方如果提前停止读取 `out`，必须取消 `ctx`，否则 Worker 可能卡在发送结果。

### 5.4 短任务限流：信号量模式

```go
sem := make(chan struct{}, maxConcurrent)
var wg sync.WaitGroup

for _, item := range items {
    item := item

    select {
    case sem <- struct{}{}:
    case <-ctx.Done():
        break
    }

    wg.Add(1)
    go func() {
        defer wg.Done()
        defer func() { <-sem }()
        process(ctx, item)
    }()
}
wg.Wait()
```

这段示例还不够严谨：`break` 只跳出 `select`，不会跳出外层 `for`。生产代码应使用带标签的退出，或封装成函数：

```go
func ProcessAll(ctx context.Context, items []Item, n int) error {
    if n <= 0 {
        return fmt.Errorf("invalid limit %d", n)
    }

    sem := make(chan struct{}, n)
    var wg sync.WaitGroup

loop:
    for _, item := range items {
        item := item
        select {
        case sem <- struct{}{}:
        case <-ctx.Done():
            break loop
        }

        wg.Add(1)
        go func() {
            defer wg.Done()
            defer func() { <-sem }()
            process(ctx, item)
        }()
    }

    wg.Wait()
    return ctx.Err()
}
```

更进一步应决定：发生首个错误时是否停止、是否收集全部错误、已开始任务是否可取消。没有错误传播需求时才适合只用 `WaitGroup`。

### 5.5 Pipeline：每一级都必须可取消

```go
func square(ctx context.Context, in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for {
            select {
            case <-ctx.Done():
                return
            case x, ok := <-in:
                if !ok {
                    return
                }
                select {
                case out <- x * x:
                case <-ctx.Done():
                    return
                }
            }
        }
    }()
    return out
}
```

只在接收端检查 `ctx.Done()` 不够；发送端也必须可取消，否则下游停止消费时上游会永久阻塞。

### 5.6 服务后台循环：显式 Start/Stop/Wait

```go
type Refresher struct {
    cancel context.CancelFunc
    done   chan struct{}
}

func StartRefresher(parent context.Context, interval time.Duration, refresh func(context.Context) error) *Refresher {
    ctx, cancel := context.WithCancel(parent)
    r := &Refresher{cancel: cancel, done: make(chan struct{})}

    go func() {
        defer close(r.done)
        ticker := time.NewTicker(interval)
        defer ticker.Stop()

        for {
            select {
            case <-ctx.Done():
                return
            case <-ticker.C:
                _ = refresh(ctx) // 生产代码应记录/分类错误并防止一次执行无限悬挂
            }
        }
    }()
    return r
}

func (r *Refresher) Stop() { r.cancel() }
func (r *Refresher) Wait() { <-r.done }
```

服务关闭顺序通常是：停止接收新流量 → 取消后台任务 → 等待有界时间 → 刷新/关闭资源 → 退出进程。不要依赖 `main` 返回替你清理。

### 5.7 Actor/单所有者模型

当一个状态对象由单个 G 独占，其他 G 通过消息交互，可以避免多处共享写：

```go
type command struct {
    delta int64
    reply chan int64
}

func counter(ctx context.Context) chan<- command {
    cmds := make(chan command)
    go func() {
        var value int64
        for {
            select {
            case <-ctx.Done():
                return
            case cmd, ok := <-cmds:
                if !ok {
                    return
                }
                value += cmd.delta
                select {
                case cmd.reply <- value:
                case <-ctx.Done():
                    return
                }
            }
        }
    }()
    return cmds
}
```

它不是天然更快：所有请求会串行化，消息分配和 Channel 调度也有成本。适合状态机、连接会话和需要明确所有权的对象，不适合高频独立计数。

### 5.8 CPU 并行：按任务粒度分块

```go
func ParallelSum(xs []int64, workers int) int64 {
    if len(xs) == 0 {
        return 0
    }
    if workers < 1 {
        workers = 1
    }
    if workers > len(xs) {
        workers = len(xs)
    }

    partial := make([]int64, workers)
    chunk := (len(xs) + workers - 1) / workers

    var wg sync.WaitGroup
    for w := 0; w < workers; w++ {
        lo := w * chunk
        hi := min(lo+chunk, len(xs))
        if lo >= hi {
            break
        }
        w := w
        wg.Go(func() { // Go 1.25+
            var sum int64
            for _, x := range xs[lo:hi] {
                sum += x
            }
            partial[w] = sum
        })
    }
    wg.Wait()

    var total int64
    for _, x := range partial {
        total += x
    }
    return total
}
```

实践中要 benchmark：任务太小会被 G 创建、同步和 cache miss 吞掉；任务太大又降低负载均衡。CPU worker 起点通常取有效 `GOMAXPROCS`，不是输入元素个数。

### 5.9 Fire-and-forget 的严格适用条件

只有同时满足以下条件，才考虑真正“不等待”的 G：

- 任务丢失可接受；
- 不持有必须释放的外部资源；
- 有独立超时/取消；
- panic 有本 G 内部的隔离策略；
- 有并发上限；
- 有错误与数量监控；
- 进程退出无需等待它完成。

日志、审计、扣款、库存、消息确认等通常不满足这些条件，应交给有持久化和重试语义的队列/任务系统，而不是裸 `go func()`。

### 5.10 `runtime.LockOSThread` 的合理场景

典型场景包括：

- 图形库要求主线程调用；
- C 库依赖 thread-local state；
- 操作系统 API 要求同一线程建立和使用上下文；
- namespace、credential 或其他线程级状态切换。

```go
func runThreadAffine(ctx context.Context, f func(context.Context) error) error {
    runtime.LockOSThread()
    defer runtime.UnlockOSThread()
    return f(ctx)
}
```

不要把它用于“让 Goroutine 更快”或“避免被调度”。它减少调度器灵活性，错误使用会增加线程数，甚至让线程无法复用。

## 六、代码陷阱题

> 以下题目默认以 Go 1.26、`go.mod` 语言版本 1.26 讨论。凡涉及调度顺序，除非代码建立了同步关系，否则不要根据一次运行结果下结论。

### 6.1 `main` 返回会等待子 Goroutine 吗

**题目：判断可能输出。**

```go
package main

import "fmt"

func main() {
    go fmt.Println("child")
    fmt.Println("main")
}
```

**标准答案**

可能只输出 `main`；也可能输出两行，顺序可能是 `child/main` 或 `main/child`。程序不保证等待子 G 完成。

**逐行分析**

1. `go fmt.Println("child")` 先在 main G 中求值函数值和参数，再创建可运行 G。
2. 创建完成不代表它已经获得 P；调度器也允许它在 main G 下一条语句前运行。
3. main G 执行第二个 `Println`。
4. 一旦 `main` 返回，进程结束；仍存在的普通 G 不会让进程自动等待。

**继续追问**

- 怎样保证输出？使用 `WaitGroup`、Channel 或任务组等待，而不是 `Sleep`。
- 子 G 的 defer 是否一定执行？若因进程退出而被截断，不保证执行。
- `go` 后面立即 `runtime.Gosched()` 是否足够？仍不是完成同步。

### 6.2 `go` 调用的参数何时求值

**题目：哪些输出顺序受保证？**

```go
package main

import "fmt"

func arg() int {
    fmt.Println("arg")
    return 42
}

func work(x int) {
    fmt.Println("body", x)
}

func main() {
    go work(arg())
    fmt.Println("next")
}
```

**标准答案**

`arg` 必须在 `next` 之前打印；`body 42` 可以在 `next` 前或后，也可能因 main 返回而看不到。

**逐行分析**

1. 规范要求 `work` 的函数值和 `arg()` 在发起 `go` 语句的 main G 中求值。
2. `arg()` 因此同步打印 `arg`，并返回 `42`。
3. runtime 得到已准备好的调用参数后创建新 G。
4. main G 与新 G 从此并发；二者没有完成顺序。

**继续追问**

- `arg()` panic 会怎样？panic 发生在调用方 G，调用目标 G 尚未正常启动。
- `arg()` 阻塞会怎样？`go` 语句本身不能完成，调用方仍会阻塞。
- `go obj.Method()` 中接收者何时求值？与函数值和参数一样，在调用方求值。

### 6.3 Go 1.22 之后的循环闭包

**题目：下面代码是否仍会打印三个 `3`？**

```go
package main

import (
    "fmt"
    "sync"
)

func main() {
    var wg sync.WaitGroup
    for i := 0; i < 3; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            fmt.Println(i)
        }()
    }
    wg.Wait()
}
```

**标准答案**

当模块语言版本为 Go 1.22 或更高时，`i` 是每次迭代新建的变量，通常会以任意顺序各打印一次 `0、1、2`，且这里没有对 `i` 的数据竞争。语言版本低于 1.22 时，经典语义是各闭包共享同一个 `i`，结果不确定，常见为三个 `3`，并伴随数据竞争。

**逐行分析**

1. `i := 0` 在 Go 1.22 新循环语义下产生 per-iteration 变量。
2. 每个闭包捕获本次迭代对应的 `i`。
3. `wg.Add(1)` 在启动 G 前执行，避免 WaitGroup 计数竞态。
4. `Wait` 只保证全部完成，不保证打印顺序。

**继续追问**

- 代码使用哪个语义由工具链版本还是 `go.mod` 决定？关键是模块/包的语言版本，不能只看本机安装的 Go 版本。
- 为什么仍常写 `i := i`？兼容旧语言版本、突出捕获意图，也适用于更复杂的复用变量。
- `for _, p := range []*T{...}` 中是复制指针还是复制对象？复制每次迭代的指针值，不会深拷贝对象。

### 6.4 Go 1.22 也救不了预先声明的循环变量

**题目：当前 Go 版本下是否安全？**

```go
package main

import (
    "fmt"
    "sync"
)

func main() {
    var wg sync.WaitGroup
    var i int
    for i = 0; i < 3; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            fmt.Println(i)
        }()
    }
    wg.Wait()
}
```

**标准答案**

不安全。`i` 在循环外声明，`for` 使用赋值 `=`，各次迭代和闭包共享同一个变量；存在读写数据竞争，输出不受保证。

**逐行分析**

1. Go 1.22 的 per-iteration 规则主要作用于循环子句或 range 子句中以 `:=` 声明的变量。
2. 此处只有一个 `i`。
3. main G 在 `i++` 写，子 G 在 `Println(i)` 读，没有同步。
4. `WaitGroup` 只同步任务结束，不会消除任务执行期间的共享读写竞态。

**继续追问**

- 最小修复是什么？循环体中 `i := i`，或把 `i` 作为参数传给闭包。
- `go func(v int) { ... }(i)` 的 `i` 何时复制？在调用方执行 `go` 语句时求值。

### 6.5 父 Goroutine 的 `recover` 能否捕获子 Goroutine 的 panic

**题目：程序会打印 `recovered` 吗？**

```go
package main

import "fmt"

func main() {
    defer func() {
        if v := recover(); v != nil {
            fmt.Println("recovered:", v)
        }
    }()

    done := make(chan struct{})
    go func() {
        defer close(done)
        panic("boom")
    }()
    <-done
}
```

**标准答案**

不会由 main G 的 defer 恢复。panic 的栈展开只发生在子 G；如果子 G 内没有有效 `recover`，进程会因未恢复 panic 终止。

**逐行分析**

1. main G 注册自己的 defer。
2. 子 G 有独立调用栈；`panic("boom")` 只展开子 G 的 defer。
3. `close(done)` 会在子 G 的 panic 展开期间执行，因此 main G 可能被唤醒。
4. main G 的 defer 不是子 G panic 调用链上的 deferred function，`recover()` 返回 `nil`。
5. 子 G 未恢复的 panic 最终触发进程崩溃。

**继续追问**

- 如何隔离插件任务 panic？在每个任务 G 的最外层 defer 中 recover、记录堆栈，并转换为错误；仍需审慎评估状态是否已损坏。
- `recover` 在 defer 调用的辅助普通函数中是否有效？只有在正确的 deferred function 调用上下文中直接调用才有效。

### 6.6 在子 Goroutine 内 `WaitGroup.Add`

**题目：这段代码是否保证等待五个任务？**

```go
package main

import "sync"

func main() {
    var wg sync.WaitGroup
    for i := 0; i < 5; i++ {
        go func() {
            wg.Add(1)
            defer wg.Done()
            // work
        }()
    }
    wg.Wait()
}
```

**标准答案**

不保证，属于错误用法。`Wait` 可能在任何子 G 执行 `Add(1)` 前看到计数为零并立即返回；并发的 `Add`/`Wait` 交错还可能触发 WaitGroup misuse 检测。

**逐行分析**

1. 启动 G 不等于 G 已执行 `Add`。
2. main G 可直接进入 `Wait`，此时计数仍为零。
3. 正计数的 `Add` 在计数为零时必须发生在对应 `Wait` 之前。
4. 正确方式是在 `go` 语句前 `wg.Add(1)`，或 Go 1.25+ 使用 `wg.Go(f)`。

**继续追问**

- `wg.Go` 有什么边界？`f` 按 API 契约不得 panic；它不做错误传播。
- WaitGroup 能否复用？上一轮所有 `Wait` 返回后才开始新一轮 `Add/Go`。
- 为什么 `defer wg.Done()` 应放在任务入口最前面？确保普通 return 路径计数归还；未恢复 panic 仍会终止进程。

### 6.7 无界创建永久阻塞的 Goroutine

**题目：会触发 runtime deadlock 检测吗？**

```go
package main

func main() {
    for {
        go func() {
            select {}
        }()
    }
}
```

**标准答案**

通常不会先触发“所有 G 都睡眠”的 deadlock，因为 main G 一直可运行并持续创建任务。进程会快速累积 G、栈和元数据，造成 GC/调度压力，最终很可能 OOM、被容器杀死或因资源耗尽失效。

**逐行分析**

1. main G 的无限循环始终 runnable/running。
2. 每个新 G 执行空 `select` 后永久 park。
3. 没有任何引用路径能唤醒这些 G。
4. G 数量单调增长，调度器仍需管理它们，GC 还需扫描相关根和栈。

**继续追问**

- 为什么 goroutine profile 很适合定位？大量堆栈会聚合在同一创建点和 wait reason。
- 怎样防止？并发上限、队列上限、取消、任务完成确认和 G 数增速告警。
- 仅监控 `runtime.NumGoroutine()` 有什么缺陷？瞬时值缺少状态、堆栈和基线，合法流量峰值也会升高。

### 6.8 无退出路径的后台 Ticker

**题目：每调用一次 `start` 会发生什么？**

```go
func start() {
    go func() {
        ticker := time.NewTicker(time.Second)
        for range ticker.C {
            refresh()
        }
    }()
}
```

**标准答案**

每次调用都会创建一个没有停止条件的后台 G。即使调用者已经不需要它，该 G 仍持有 ticker 和相关闭包引用并永久运行，形成逻辑泄漏；重复调用会重复刷新并持续增加资源使用。

**逐行分析**

1. `start` 不返回停止句柄，也不接收 Context。
2. G 永久等待/消费 `ticker.C`。
3. 没有 `ticker.Stop()`，也没有取消分支。
4. 因为活跃 G 持有 ticker 并持续使用它，不能指望 GC 自动回收解决业务生命周期问题。

**继续追问**

- 修复方式？传入 Context，`defer ticker.Stop()`，并提供 `Stop/Wait`。
- 若 `refresh` 执行超过一个周期会怎样？同一循环不会并发执行多个 `refresh`，tick 可能合并/丢弃；若每 tick 再开 G，则可能无界并发。

### 6.9 空 `select{}` 是“保持服务运行”的好方法吗

**题目：下面程序结果是什么？**

```go
package main

func main() {
    select {}
}
```

**标准答案**

该 G 永久阻塞；当 runtime 发现没有任何其他可运行 G、Timer、netpoll/cgo 等可带来进展的工作时，会报 fatal deadlock。它不是通用的服务生命周期管理方式。

**逐行分析**

1. 没有 case 的 `select` 永远不能完成。
2. main G 进入 waiting。
3. 若整个进程无其他潜在进展，runtime deadlock detector 终止程序。
4. 在一个仍有其他工作来源的进程里，`select{}` 只会永久泄漏当前 G，不一定立刻报错。

**继续追问**

- 服务 main 应等待什么？等待信号、server 错误、Context 取消或任务组完成，并执行有序关闭。
- runtime deadlock 能发现互斥锁死锁的所有情况吗？不能；只在全局无可推进工作等条件下检测，不理解业务活锁和部分死锁。

### 6.10 `runtime.Gosched` 能否替代同步

**题目：是否一定打印 `1`？**

```go
package main

import (
    "fmt"
    "runtime"
)

func main() {
    var x int
    go func() { x = 1 }()
    runtime.Gosched()
    fmt.Println(x)
}
```

**标准答案**

不一定，而且对 `x` 的并发读写构成 data race。`Gosched` 只让当前 G 暂时让出执行权，不等待目标 G，也不建立 happens-before。

**逐行分析**

1. 子 G 被置为 runnable。
2. main G 调用 `Gosched` 后回到 runnable 队列。
3. 调度器可能运行子 G，也可能运行其他 G，再恢复 main G；API 不承诺特定目标被运行。
4. 即使一次实验总打印 `1`，代码仍没有内存模型保证。

**继续追问**

- 正确同步方式？Channel、Mutex、atomic 或 WaitGroup，按所需语义选择。
- `Gosched` 的合理用途？极少数主动合作让步、runtime/测试实验；不能修复竞态和锁设计。

### 6.11 `time.Sleep` 能否替代同步

**题目：把休眠改成一秒是否就“可靠”了？**

```go
package main

import (
    "fmt"
    "time"
)

func main() {
    var x int
    go func() { x = 1 }()
    time.Sleep(time.Second)
    fmt.Println(x)
}
```

**标准答案**

不可靠，仍是 data race。休眠只建立“当前 G 在某段墙钟时间内不继续执行”，不建立内存同步关系，也不保证目标 G 在期限内得到调度并完成。

**逐行分析**

1. 子 G 的写与 main G 的读没有 Channel、锁、原子或其他同步事件连接。
2. 一秒通常足够只是一种经验概率；系统暂停、CPU 配额、调试器、负载和任务阻塞都可打破假设。
3. race detector 会按并发访问报告，而不会因为 Sleep 很长就认为安全。

**继续追问**

- 测试中能否用 Sleep 等异步结果？应优先使用条件、事件或可控 fake clock；Sleep 容易造成慢且 flaky 的测试。
- `time.Sleep(0)` 是否等价于 `Gosched`？不要依赖这种实现细节；语义目的不同，均不是同步。

### 6.12 `LockOSThread` 忘记解锁

**题目：早返回有什么后果？**

```go
func callNative() error {
    runtime.LockOSThread()
    if err := initNative(); err != nil {
        return err
    }
    runtime.UnlockOSThread()
    return nil
}
```

**标准答案**

错误路径没有平衡 `UnlockOSThread`。当前 G 若在锁定状态下退出，runtime 会终止该 OS 线程，而不是把它交给其他 G 复用；频繁发生会造成线程创建/销毁、资源和延迟问题。嵌套锁定还要求相同次数解锁。

**逐行分析**

1. `LockOSThread` 绑定当前 G 与当前 M。
2. `initNative` 报错时直接 return。
3. G 退出前没有解锁；线程亲和状态不能安全转给任意 G。
4. 正确写法是在成功锁定后立即 `defer runtime.UnlockOSThread()`，除非设计上有意让线程随 G 退出。

**继续追问**

- `LockOSThread` 是否禁止内核抢占线程？否，只是 G 与 OS 线程绑定。
- 它是否把 G 固定到某个 CPU 核？否；CPU affinity 是另一层机制。

### 6.13 `GOMAXPROCS(1)` 是否只允许一个 Goroutine/线程

**题目：判断下面说法：“设置为 1 后进程最多有一个 G 和一个 M。”**

```go
func main() {
    runtime.GOMAXPROCS(1)
    for i := 0; i < 1000; i++ {
        go task(i)
    }
    select {}
}
```

**标准答案**

错误。`GOMAXPROCS(1)` 主要限制同一时刻执行用户 Go 代码的 P 数为 1；可以存在任意多的 G，也可能存在多个 M，例如阻塞 syscall、cgo、runtime 系统线程和线程亲和场景。

**逐行分析**

1. P 数被设为 1，普通 Go 代码最大并行度约为 1。
2. 1000 个 `task` 仍会被创建并排队/等待。
3. 如果某个 M 带着 G 进入阻塞 syscall，P 可交给另一个 M。
4. `select{}` 还会让 main G 永久等待；它不负责等待任务完成。

**继续追问**

- 为什么 `runtime.NumCPU()` 可能与 `GOMAXPROCS(0)` 不同？Go 1.25+ 默认还考虑 affinity 和 cgroup quota，且可自动更新。
- 显式调用 `GOMAXPROCS` 后容器 quota 改变会怎样？自动更新被关闭，除非调用 `SetDefaultGOMAXPROCS`。

### 6.14 为每个 CPU 元素创建一个 Goroutine

**题目：一百万个 CPU 小任务是否一定更快？**

```go
func HashAll(items [][]byte) [][32]byte {
    out := make([][32]byte, len(items))
    var wg sync.WaitGroup
    wg.Add(len(items))
    for i := range items {
        i := i
        go func() {
            defer wg.Done()
            out[i] = sha256.Sum256(items[i])
        }()
    }
    wg.Wait()
    return out
}
```

**标准答案**

不一定，常常更慢。CPU 算力受有效 P/CPU 配额限制；百万 G 会增加创建、栈、调度、GC root、队列和 cache 成本。应根据任务粒度分块，或使用接近 `GOMAXPROCS` 的固定 worker 数，并 benchmark。

**逐行分析**

1. 每个任务写独立 `out[i]`，数据正确性通常没问题。
2. 但每个元素都付出一次 G 生命周期成本。
3. 同时 runnable 的任务远超 P，形成 backlog 和 scheduler latency。
4. 若输入切片对象巨大或闭包逃逸，还会提高 GC 压力。

**继续追问**

- 什么时候“一任务一 G”合理？任务足够粗、数量有界、主要 I/O 等待且资源受控时。
- CPU worker 为什么也不一定恰好等于 GOMAXPROCS？内存带宽、cache、SIMD、外部库线程池和容器 throttling 都会影响最优点。

### 6.15 泄漏的 Goroutine 如何留住大对象

**题目：`leak()` 返回后 100 MiB 能否被 GC 回收？**

```go
func leak() {
    buf := make([]byte, 100<<20)
    never := make(chan struct{})

    go func() {
        <-never
        fmt.Println(buf[0])
    }()
}
```

**标准答案**

通常不能。阻塞 G 的闭包和栈仍可达，并且在接收之后还要读取 `buf[0]`，因此 `buf` 必须保持存活。泄漏的不只是 G 的初始栈，而是它可达的整个对象图。

**逐行分析**

1. `buf` 分配大底层数组。
2. 闭包捕获 `buf` 和 `never`。
3. Channel 永远不关闭/发送，G 永久等待。
4. 编译器活跃性分析看到恢复后仍使用 `buf`，不能提前判定它死亡。

**继续追问**

- 为什么示例特意在阻塞后读取 `buf`？若后续不再使用，编译器可能缩短变量活跃期，不能稳定演示保留。
- 怎样确认？结合 heap profile 的 retained objects 与 goroutine profile 的创建堆栈，而不是只看 G 数。

### 6.16 Worker Pool 输入永不关闭

**题目：为什么 `wg.Wait()` 永远不返回？**

```go
jobs := make(chan Job)
var wg sync.WaitGroup

for i := 0; i < 8; i++ {
    wg.Add(1)
    go func() {
        defer wg.Done()
        for job := range jobs {
            handle(job)
        }
    }()
}

for _, job := range batch {
    jobs <- job
}
wg.Wait()
```

**标准答案**

Worker 的 `for range jobs` 只有在 `jobs` 关闭后才结束；生产者发送完却没有 `close(jobs)`，所以八个 G 全部阻塞接收，`Done` 不会执行，`Wait` 永久阻塞。

**逐行分析**

1. 由发送方/输入所有者负责关闭 `jobs`。
2. 发送完最后一项不等于 Channel 自动关闭。
3. 每个 Worker 等待下一项。
4. `defer wg.Done()` 只在函数返回时运行。

**继续追问**

- 若任务源是长期流，不能关闭怎么办？Worker `select` 同时监听 `ctx.Done()`。
- 多个生产者谁关闭？由一个协调者在所有生产者完成后关闭，避免重复 close 和 send-on-closed-channel。
- `close` 是否是广播？对接收方是：缓冲耗尽后所有接收立即得到零值和 `ok=false`。

### 6.17 Context 创建了却没有向下传递

**题目：超时能否中止 HTTP 请求？**

```go
func load(parent context.Context, url string) error {
    ctx, cancel := context.WithTimeout(parent, 100*time.Millisecond)
    defer cancel()

    req, err := http.NewRequest(http.MethodGet, url, nil) // 没使用 ctx
    if err != nil {
        return err
    }
    resp, err := http.DefaultClient.Do(req)
    if err != nil {
        return err
    }
    defer resp.Body.Close()
    return nil
}
```

**标准答案**

不能。`ctx` 没有绑定到请求，也没有被下游读取；100ms 到期只会关闭这个无人监听的 Context。应使用 `http.NewRequestWithContext` 或 `req.WithContext(ctx)`，同时配置合适的 transport/client 超时。

**逐行分析**

1. 创建 Context 不会自动影响当前 G 或所有子调用。
2. `NewRequest` 默认使用非该超时 Context。
3. `Do` 因此不观察 `ctx.Done()`。
4. `defer cancel()` 仍应保留，用于释放定时器资源，但它不能补救未传播。

**继续追问**

- Context 应放结构体字段吗？通常不应；应作为请求作用域函数的首个参数显式传递。
- CPU 密集循环会自动响应取消吗？不会，应在合理粒度检查 `ctx.Done()`，但也要控制检查成本。

### 6.18 每次 Tick 都启动一个可能不结束的任务

**题目：当 `refresh` 比周期慢或永久阻塞时会怎样？**

```go
func run(ctx context.Context) {
    ticker := time.NewTicker(10 * time.Millisecond)
    defer ticker.Stop()

    for {
        select {
        case <-ctx.Done():
            return
        case <-ticker.C:
            go refresh() // refresh 可能永久阻塞
        }
    }
}
```

**标准答案**

会无界累积正在执行或阻塞的 `refresh` G。Ticker 本身通常不会无限缓存 tick，但循环每收到一次就创建一个新任务，没有并发上限、超时、等待或错误处理。

**逐行分析**

1. 主循环能持续消费 tick，所以持续创建 G。
2. `refresh` 的执行时间不影响下一次创建。
3. Context 只停止创建者；没有传给 `refresh`，已创建任务也不会取消。
4. 服务停止时这些任务可能仍持有资源。

**继续追问**

- 只允许一个 refresh 怎么写？在循环中同步调用，或使用容量为 1 的触发 Channel/`TryLock` 合并重复触发。
- 允许有限并发怎么写？信号量、worker pool 或 `errgroup.SetLimit`，并传递 Context。
- `time.AfterFunc` 有何相同风险？回调由独立 G 执行；递归重设或高频创建时同样必须控制生命周期和并发。

### 6.19 直接阻塞 syscall 与 `net.Conn.Read`

**题目：二者阻塞时对 M 的影响相同吗？**

```go
// A：直接对阻塞 fd 调 syscall.Read
n, err := syscall.Read(fd, buf)

// B：使用 Go net 包管理的网络连接
n, err := conn.Read(buf)
```

**标准答案**

通常不同。A 若进入真正阻塞的系统调用，会占住当前 M；runtime 可把 P handoff 给其他 M。B 对可轮询网络 fd 通常使用非阻塞模式和 netpoll：遇到未就绪时 park G，M 可立即执行其他 G。具体行为仍取决于 fd 类型、平台和调用路径。

**逐行分析**

1. 阻塞 syscall 前后有 `entersyscall/exitsyscall` 协议。
2. 内核等待期间 M 不能运行别的 G，但 P 不必一直被它占住。
3. `net.Conn` 经 `internal/poll` 注册 poller；EAGAIN 后把 G 挂到 `pollDesc`。
4. epoll/kqueue/IOCP 就绪后再把 G ready。
5. 普通文件 I/O、不可轮询 fd 或某些第三方库调用仍可能占线程。

**继续追问**

- 为什么阻塞 syscall 多时线程数会增长？为了让可用 P 继续有 M 承载，runtime 可能创建/唤醒更多 M。
- 如何观测？threadcreate profile、`/sched/threads/total`、`/sched/goroutines/not-in-go`、trace syscall 区间和系统线程视图。

### 6.20 在 main Goroutine 中调用 `runtime.Goexit`

**题目：它等价于 `return` 吗？**

```go
package main

import (
    "fmt"
    "runtime"
)

func main() {
    defer fmt.Println("main defer")
    go func() {
        fmt.Println("worker")
    }()
    runtime.Goexit()
}
```

**标准答案**

不等价。`Goexit` 终止当前 main G 并执行其 defer，但 `func main` 没有正常返回；其他 G 仍可继续。其他 G 都退出后，程序会以“main 调用了 Goexit”相关的 fatal deadlock 结束，而不是正常退出。

**逐行分析**

1. main G 注册 defer。
2. Worker 被创建，但运行顺序不确定。
3. `Goexit` 执行 main G 的 defer，因此会打印 `main defer`。
4. `Goexit` 不是 panic，defer 中 `recover()` 得到 `nil`。
5. Worker 可打印后退出；由于 main 没正常返回，runtime 最终报告异常终止。

**继续追问**

- 子 G 调用 `Goexit` 会怎样？只终止该 G并执行 defer，其他 G 不受直接影响。
- `testing.T.FailNow` 为什么与它相关？测试框架用类似机制终止当前测试 G，因此必须从运行测试的 G 调用。

### 6.21 `WaitGroup.Go` 会自动处理 panic 和错误吗

**题目：下面代码会把 panic 变成 `Wait` 的错误吗？**

```go
var wg sync.WaitGroup
wg.Go(func() {
    panic("boom")
})
wg.Wait()
```

**标准答案**

不会。`sync.WaitGroup.Go` 从 Go 1.25 起提供“增加计数、启动任务、完成时减少计数”的便利，但它没有错误返回通道，也不负责 recover；公开契约明确要求传入函数不得 panic。未恢复 panic 仍会导致进程崩溃。

**逐行分析**

1. `wg.Go` 解决的是常见 `Add` 时序错误。
2. 它不是 `errgroup`，`Wait` 没有错误返回值。
3. panic 不会转换为普通返回值。
4. 需要错误传播时使用显式结果 Channel、`errgroup` 或自定义任务组；需要隔离 panic 时在任务边界自行处理。

**继续追问**

- `WaitGroup.Go` 能否在一个任务中继续调用同一个 `wg.Go`？当 WaitGroup 非空时允许；但必须确保最终任务图有界。
- `errgroup.Group.Go` 会 recover panic 吗？不要假设；其职责是收集函数返回的 error，任务应按契约自行处理 panic。

### 6.22 `time.AfterFunc` 回调递归创建 Goroutine

**题目：调用一次 `start` 后，资源是否有界？**

```go
var never = make(chan struct{})

func start() {
    var tick func()
    tick = func() {
        go func() {
            <-never
        }()
        time.AfterFunc(10*time.Millisecond, tick)
    }
    time.AfterFunc(10*time.Millisecond, tick)
}
```

**标准答案**

无界。`AfterFunc` 到期后会在自己的 Goroutine 中调用回调；每次回调又创建一个永久阻塞 G，并安排下一次 Timer。代码没有保存 Timer 句柄、取消条件、并发上限或等待协议，G 数会持续增长。

**逐行分析**

1. `tick` 捕获自身，形成递归调度逻辑。
2. 第一个 Timer 到期后，runtime 安排回调执行。
3. 回调每次创建一个永远等不到 `never` 的子 G。
4. 回调再创建下一只 Timer；当前回调虽然会返回，但永久子 G 单调累积。
5. 即使后来想停止，也没有保存当前 Timer 或向已创建 G 提供退出信号。

**继续追问**

- 如何修复？由单一 owner 使用 `Ticker` 或重置同一 Timer，监听 Context，任务本身也接收取消，并用 semaphore/单飞策略限制重叠执行。
- `Timer.Stop` 能终止已经开始的回调吗？不能把它当作杀死正在运行函数的机制；回调必须协作取消并由调用方等待。
- 为什么 Ticker 不会自动解决任务堆积？Ticker 通道可能合并 tick，但若每个 tick 都另开无界 G，任务仍会累积。

## 七、面试高频问题

### 7.1 Goroutine 和线程的核心区别是什么

- **普通回答**：Goroutine 是 Go 的并发执行单元，由 runtime 多路复用到 OS 线程；线程由内核调度。
- **中高级回答**：G 保存用户任务现场，M 表示线程，P 提供执行 Go 代码的调度资源；大量 G 可共享较少 M，G 栈还能按需增长和收缩。
- **深挖回答**：G 切换通常先在用户态完成，但承载 G 的 M 仍由内核调度；阻塞 syscall、cgo 和线程亲和会改变 G/M 映射。轻量来自更小的初始资源和用户态调度，不是零成本。
- **错误回答**：“一个 Goroutine 就是一条 2 KiB 的线程。”这混淆抽象层，也把初始栈误当固定总成本。
- **版本/边界**：规范只定义 Goroutine 并发执行语义；GMP、栈大小和字段属于当前 gc/runtime 实现。

### 7.2 并发和并行有什么区别

- **普通回答**：并发是任务生命周期重叠，并行是同一时刻在多个执行资源上运行。
- **中高级回答**：`GOMAXPROCS=1` 仍可有大量并发 G，但普通 Go 代码通常不能多核并行；多个 I/O 等待任务仍能交替推进。
- **深挖回答**：并行度还受 cgroup quota、CPU affinity、内核调度、可运行任务数和外部库线程池影响；P 数只是 runtime 的一层上限。
- **错误回答**：“用了 Goroutine 就自动多核并行。”没有可并行工作、P 或 CPU 资源时不成立。
- **版本/边界**：概念稳定；容器中的默认 P 数在 Go 1.25 起发生重要变化。

### 7.3 `go f(x)` 的函数值和参数何时求值

- **普通回答**：在执行 `go` 语句的当前 G 中先求值，实际函数调用在新 G 中执行。
- **中高级回答**：参数求值引发的 panic、阻塞和副作用都发生在发起方；调用返回值会被丢弃。
- **深挖回答**：编译器通常生成包装调用并把已求值参数交给 `runtime.newproc`，这解释了“异步的是调用体，不是参数准备”。
- **错误回答**：“`go` 后所有内容都异步，包括参数。”
- **版本/边界**：这是语言规范保证，不依赖 GMP 实现。

### 7.4 G、M、P 分别是什么

- **普通回答**：G 是任务，M 是 OS 线程，P 是执行 Go 代码所需的处理器上下文。
- **中高级回答**：P 持有本地 runq、runnext、Timer、mcache 等资源；M 要运行普通 Go G 通常必须持有 P。
- **深挖回答**：G 保存栈、调度上下文和状态；M 有 g0、curg、线程 ID 和当前/待接管 P；P 的数量近似由 `GOMAXPROCS` 决定。
- **错误回答**：“P 就是物理 CPU，M 永远等于 P。”
- **版本/边界**：字段和队列布局会变，面试应先讲职责，再注明实现细节。

### 7.5 为什么已经有线程 M，还需要 P

- **普通回答**：P 把运行 Go 代码所需的本地调度和内存资源从线程中解耦，控制并行度并降低全局竞争。
- **中高级回答**：阻塞 syscall 时 M 可被占住，但 P 能 handoff 给别的 M；本地 runq 和 mcache 也可随 P 延续，而不是绑定死线程。
- **深挖回答**：P 让 scheduler 在“内核线程数量”和“Go 代码并行度”之间建立独立控制面，并支持 work stealing、per-P timer 与分配缓存。
- **错误回答**：“P 只是一个锁，删掉也一样。”
- **版本/边界**：这是现行 runtime 架构，不是规范术语。

### 7.6 本地队列、全局队列和 `runnext` 有什么作用

- **普通回答**：本地队列减少共享锁竞争；全局队列承接溢出和跨 P 工作；`runnext` 优先运行一个关联任务以降低延迟。
- **中高级回答**：当前本地 runq 是环形数组，满时会批量把工作转移到全局；`runnext` 可能被替换，旧值回到普通队列。
- **深挖回答**：`runnext` 可继承当前时间片，适合刚解锁/刚创建的接力，但调度器通过全局队列周期检查、继承限制和 stealing 避免永久偏置。
- **错误回答**：“新创建 G 一定下一条运行。”
- **版本/边界**：当前源码中 runq 容量和公平性常数是实现细节，不能写成 API 承诺。

### 7.7 Work stealing 如何工作，能否保证绝对公平

- **普通回答**：空闲 P 从其他 P 的本地队列窃取一批可运行 G，平衡负载。
- **中高级回答**：批量窃取可摊薄原子操作和 cache 迁移成本；调度器还会查全局队列、netpoll 和 Timer。
- **深挖回答**：stealing 需要控制自旋 M 数，避免所有空闲线程同时扫队列；随机/轮转探测和周期性全局检查只提供工程公平性，不是严格 FIFO 或最大等待时间保证。
- **错误回答**：“每个 P 只运行自己创建的 G。”
- **版本/边界**：算法和探测顺序可随 runtime 版本调整。

### 7.8 `schedule → findRunnable → execute` 分别做什么

- **普通回答**：`schedule` 驱动一轮调度，`findRunnable` 找工作，`execute` 把选中 G 切为 running 并恢复执行。
- **中高级回答**：找不到工作时会尝试 stealing、netpoll、Timer，最终释放 P 或让 M 休眠；`execute` 建立 G/M 关系并通过底层上下文切换进入 G。
- **深挖回答**：调度代码在 g0/system stack 上运行，必须遵守无写屏障、不可随意栈增长等 runtime 约束；状态转换需要与 GC、trace、profile 协调。
- **错误回答**：“schedule 每次只从全局队列 pop 一个 G。”
- **版本/边界**：函数名较稳定，但分支顺序和特殊工作类型是实现细节。

### 7.9 `gopark` 和 `goready` 是什么关系

- **普通回答**：`gopark` 让当前 G 等待，`goready` 把等待 G 重新置为 runnable。
- **中高级回答**：Channel、锁、Timer、netpoll 等先把等待节点挂到自己的队列，再 park；事件方取出节点后 ready。
- **深挖回答**：`gopark` 的解锁回调用于原子地衔接“登记等待”和“释放锁”，避免丢失唤醒；ready 需要保证目标状态合法且不会重复入队。
- **错误回答**：“gopark 会让当前 OS 线程一直睡眠。”通常只是 G 停止，M 可运行其他 G。
- **版本/边界**：内部函数不是公开 API，业务代码不应链接或调用。

### 7.10 Go 的 netpoll 为什么能支持高并发网络连接

- **普通回答**：网络 fd 采用非阻塞 I/O，未就绪时只 park G，内核 poller 在事件到达后唤醒它。
- **中高级回答**：`internal/poll` 把 fd 与 `pollDesc` 关联，runtime 用 epoll/kqueue/IOCP 等等待事件，并将读/写等待 G 注入 runnable 队列。
- **深挖回答**：netpoll 与调度器、Timer 截止时间协同决定阻塞时长；它减少“一连接一阻塞线程”，但每连接的缓冲、协议状态和业务 G 仍有成本。
- **错误回答**：“所有 I/O 都不会阻塞线程。”普通文件、不可轮询 fd、DNS/cgo 或第三方调用可能占线程。
- **版本/边界**：平台实现不同；语言规范不规定网络轮询器。

### 7.11 阻塞系统调用时 G、M、P 怎么变化

- **普通回答**：G/M 进入 syscall，M 被内核调用占住，P 可移交给别的 M 继续运行 Go 代码。
- **中高级回答**：返回时 `exitsyscall` 尝试快速重新获得 P；失败则把 G 变成 runnable，M 进入后续调度/休眠路径。
- **深挖回答**：`sysmon` 可 retake 长时间 syscall 的 P；因此 `GOMAXPROCS` 限制 P 而不限制被 syscall 占用的 M 总数。
- **错误回答**：“一个 syscall 会冻结整个 Go 程序。”
- **版本/边界**：具体快慢路径与平台封装属于 runtime 实现。

### 7.12 cgo 调用对调度器有什么影响

- **普通回答**：调用 C 时当前 M 可能长期在 C 中，P 通常可供其他 M 使用。
- **中高级回答**：runtime 必须处理 Go/C 栈切换、线程状态、GC 可见性、回调和指针规则；高频短 cgo 调用的边界成本可能明显。
- **深挖回答**：C 代码阻塞、创建线程或回调 Go 会增加 M 管理复杂度；C 内存和 C 栈通常不按普通 Go 堆方式被扫描，跨边界指针受 cgo 规则约束。
- **错误回答**：“cgo 调用就是普通 Go 函数调用，调度器无感。”
- **版本/边界**：ABI、开销和检查机制会变；应以目标版本 benchmark 和文档为准。

### 7.13 `sysmon` 的职责是什么

- **普通回答**：它是无 P 运行的 runtime 监控线程，处理抢占、syscall retake、Timer/netpoll 等系统级维护。
- **中高级回答**：它周期检查长时间运行 G 和 syscall 中的 P，触发抢占或回收；还参与 force GC、网络轮询等工作。
- **深挖回答**：sysmon 不能依赖普通 P 才运行，否则 P 全被长任务/系统调用占用时无法自救；其休眠周期会动态变化。
- **错误回答**：“sysmon 是用户可配置的 watchdog Goroutine。”它是 runtime 内部系统线程/执行路径。
- **版本/边界**：具体职责和周期常数会随版本变化，例如当前实现中的强制抢占阈值不是语言保证。

### 7.14 Goroutine 栈为什么可以很小

- **普通回答**：当前实现使用可增长、可收缩的连续栈，按实际需要扩容，不必为每个 G 一开始预留巨大固定栈。
- **中高级回答**：函数序言检查 `stackguard`，空间不足进入 `morestack/newstack`；runtime 分配更大栈并复制已用部分。
- **深挖回答**：复制后要调整 defer、panic、sudog 和栈内指针；GC/编译器提供 stack map 以区分指针。栈增长成本与已用栈量相关，不是常数。
- **错误回答**：“Goroutine 栈永远只有 2 KiB。”初始值和实现策略都不是固定总上限。
- **版本/边界**：当前 `stackMin` 等常数属于实现；语言规范不规定栈模型。

### 7.15 Go 如何实现抢占

- **普通回答**：runtime 可请求正在运行的 G 在安全点让出 P；现代版本还支持基于信号的异步抢占。
- **中高级回答**：协作式抢占可借函数调用的栈检查触发；Go 1.14 引入异步抢占，改善无函数调用长循环对 GC 和调度的阻塞。
- **深挖回答**：信号到达后 runtime 仍要判断 PC 是否是 async-safe point，以及是否处于 runtime 临界区、持锁或不可抢占状态，再重写执行上下文进入抢占路径。
- **错误回答**：“Go 能在任意指令处无条件抢占，所以有实时保证。”
- **版本/边界**：Go 1.14 是重要分界；平台支持和实现细节不同，最大调度延迟仍无硬实时保证。

### 7.16 Safe point 为什么既和调度有关，也和 GC 有关

- **普通回答**：safe point 是 runtime 能准确理解 G 状态、栈指针并安全暂停或扫描的位置。
- **中高级回答**：GC 需要准确 stack map 和可控状态，调度抢占也要确保切走后能恢复且不破坏 runtime 不变量。
- **深挖回答**：函数序言、异步可抢占 PC、阻塞调用等形成不同 safe-point 机制；unsafe point、`nosplit` 和无写屏障区会限制暂停时机。
- **错误回答**：“safe point 只在 GC 时存在，和 scheduler 无关。”
- **版本/边界**：编译器插桩与 runtime 协议属于工具链实现。

### 7.17 `runtime.Gosched` 与 `time.Sleep` 的区别

- **普通回答**：`Gosched` 主动让出执行机会但保持 runnable；`Sleep` 让当前 G 至少等待指定时间。
- **中高级回答**：`Gosched` 后 G 会自动恢复，不等待特定事件；`Sleep` 依赖 Timer 到期再 ready。二者都不是同步原语。
- **深挖回答**：Gosched 进入特定调度路径并把 G 放回队列；Sleep 通常把 G park 到 Timer 机制。调度延迟使实际恢复时间可晚于请求时间。
- **错误回答**：“调用一次 Gosched 就能保证刚创建的 G 完成。”
- **版本/边界**：API 语义稳定，内部队列放置和 Timer 实现可变。

### 7.18 `runtime.LockOSThread` 什么时候必须使用

- **普通回答**：当外部 API 或线程局部状态要求同一 OS 线程时使用，并平衡 Unlock。
- **中高级回答**：典型是 GUI 主线程、C 库 TLS、线程级 namespace/credential；锁定后 G 与 M 绑定，但线程仍由 OS 调度。
- **深挖回答**：重复调用会增加锁定计数；若 G 在锁定状态退出，runtime 不能安全复用该线程，通常会让线程终止。错误使用会导致线程膨胀和调度灵活性下降。
- **错误回答**：“LockOSThread 可以把任务固定到 CPU 核并提升缓存命中。”
- **版本/边界**：公开 API 文档保证线程绑定语义；操作系统线程状态的具体用途由平台决定。

### 7.19 Goroutine 是否有父子生命周期

- **普通回答**：没有自动父子取消、等待和错误传播；创建者退出不会自动终止子 G。
- **中高级回答**：runtime 记录创建栈和 parent goid 主要用于诊断，不是结构化任务树；业务必须显式用 Context、WaitGroup 或任务组。
- **深挖回答**：结构化并发要求作用域退出前子任务结束，且取消、错误和资源所有权可推导；裸 `go` 会把这些责任交给调用者自行设计。
- **错误回答**：“父 Goroutine return 后，runtime 会回收所有子 Goroutine。”
- **版本/边界**：Go 当前没有公开 Goroutine handle/join API；第三方库可提供任务组抽象。

### 7.20 为什么 `recover` 不能跨 Goroutine

- **普通回答**：panic 只沿发生 panic 的 G 的调用栈展开 defer，其他 G 不在该调用链上。
- **中高级回答**：必须在同一 G 的 deferred function 中按规范调用 `recover`；父任务的 defer 无法拦截子任务 panic。
- **深挖回答**：任务执行器若要隔离 panic，必须在每个 worker/task 包装层 recover，并记录 `debug.Stack`；但 recover 后共享状态可能已部分修改，不能无条件继续。
- **错误回答**：“main 里放一个全局 recover 就能保护整个进程。”
- **版本/边界**：这是规范语义，不是调度器实现选择。

### 7.21 WaitGroup、Channel 和 errgroup 应如何选择

- **普通回答**：只等完成用 WaitGroup；传值/建立所有权用 Channel；要首错传播和取消用 errgroup。
- **中高级回答**：WaitGroup 不携带结果且容易出现 Add 时序错误；Channel 需要清晰关闭责任；errgroup 可结合 Context 和 `SetLimit`，但只返回首个非 nil error。
- **深挖回答**：选择取决于协议：是否需要流式结果、全部错误、首错取消、并发上限和任务图动态扩展。不要为了“并发”强行用 Channel 传每个状态。
- **错误回答**：“WaitGroup.Go 已经等于 errgroup。”它没有 error 语义。
- **版本/边界**：`sync.WaitGroup.Go` 从 Go 1.25 提供；`errgroup` 位于 `golang.org/x/sync`，不是标准库。

### 7.22 什么是 Goroutine 泄漏

- **普通回答**：G 已经不再有业务价值，却因等待、阻塞或失去取消路径而长期不能退出。
- **中高级回答**：常见原因是 Channel 无对端、下游不读、Context 未传播、Ticker/后台循环无停止、无界任务、锁永远不释放。
- **深挖回答**：泄漏会保留 G 栈可达对象、文件描述符、Timer、连接和 trace/profile 元数据；数量稳定也可能是“固定但巨大的泄漏”。
- **错误回答**：“只要 G 在 waiting 就是泄漏。”合法的 server accept、连接等待也会长期 waiting，需结合所有权和业务生命周期判断。
- **版本/边界**：Go 1.26 有实验性 `goroutineleak` profile，需构建时启用实验，且只能检测部分永久阻塞模式，不能替代设计审计。

### 7.23 如何排查 Goroutine 泄漏

- **普通回答**：比较不同时间的 goroutine profile，按相同堆栈聚合，找数量持续增长且无合理生命周期的等待点。
- **中高级回答**：结合创建堆栈、wait reason、heap retained objects、请求率和 shutdown 行为；用最小压测复现“停止流量后不回落”。
- **深挖回答**：Go 1.26 可辅助看 `/sched/goroutines-created` 与 runnable/waiting 分类；实验性 leak profile可作为证据之一，但可达性算法有 false negative/适用边界。
- **错误回答**：“NumGoroutine 超过 100 就一定泄漏。”
- **版本/边界**：profile 类型和 runtime metrics 集合会演进，采集前检查目标版本。

### 7.24 什么是 Scheduler latency

- **普通回答**：G 已经 runnable 到真正获得 P/M 运行之间的等待时间。
- **中高级回答**：高值常见于 CPU 饱和、runnable backlog、配额 throttling、长不可抢占区或过多短任务；它不同于 I/O 等待时间。
- **深挖回答**：用 `/sched/latencies:seconds` 看分布，用 trace 关联 runnable 事件、processor 利用率、syscall、GC 和任务创建源；只看平均值会掩盖尾部。
- **错误回答**：“请求慢就是 scheduler latency 高。”请求也可能在锁、Channel、网络或下游排队。
- **版本/边界**：`/sched/latencies` 较早已有；Go 1.26 新增更细的 runnable/running/waiting 指标可辅助归因。

### 7.25 如何理解容器中的 `GOMAXPROCS`

- **普通回答**：Go 1.25 起默认会考虑 Linux cgroup CPU quota，不再简单等于宿主机逻辑 CPU 数。
- **中高级回答**：当前默认综合逻辑 CPU、affinity 和 cgroup throughput limit，通常取较小值并对小数 quota 向上取整；CPU request 不是该默认值依据。
- **深挖回答**：runtime 可周期更新默认值；显式环境变量或调用 `GOMAXPROCS` 会固定并关闭自动更新，`SetDefaultGOMAXPROCS` 可恢复。quota 是时间窗口吞吐预算，不是瞬时专属核心数。
- **错误回答**：“Pod request=500m，所以 GOMAXPROCS 一定是 1。”request 与 limit/quota 语义不同。
- **版本/边界**：Go 1.25 是默认行为分界；`go.mod` 语言版本不高于 1.24 时有兼容性 GODEBUG 默认；以目标版本 runtime 文档为准。

### 7.26 CPU-bound 和 I/O-bound 的并发度如何设置

- **普通回答**：CPU 任务从有效 `GOMAXPROCS` 附近开始；I/O 任务可更高，但受连接、内存和下游容量限制。
- **中高级回答**：CPU 任务需按粒度分块并 benchmark；I/O 可用 Little's Law 估算在途量，再用尾延迟、错误率和资源上限校准。
- **深挖回答**：混合负载应分池或分配预算，避免慢 I/O 占满业务信号量，也避免 CPU fan-out 形成 runnable backlog；动态控制可基于队列和下游反馈，而非只看 CPU。
- **错误回答**：“Goroutine 很便宜，所以并发数不用限制。”
- **版本/边界**：没有适用于所有服务的固定公式，硬件、quota、任务成本和 SLA 都是边界。

### 7.27 Goroutine 为什么轻量但不免费

- **普通回答**：它仍需要 G 元数据、栈、调度、闭包、同步和 GC 扫描成本。
- **中高级回答**：高并发的主要内存常来自每 G 持有的请求缓冲、Context、连接、Timer 和对象图，而非初始栈本身。
- **深挖回答**：大量 runnable G 增加队列和调度延迟，大量 waiting G 增加根扫描和 retained memory，大量短命 G 增加分配与 trace/profile 事件；应按系统总成本容量规划。
- **错误回答**：“开一百万个 G 只需约 2 GiB，所以一定没问题。”忽略增长栈、对象和运行时成本。
- **版本/边界**：初始栈、G 结构大小和复用策略会变化，不能用单一常数做跨版本预算。

### 7.28 False sharing 为什么会让更多 Goroutine 更慢

- **普通回答**：不同核频繁写同一 cache line 上的独立变量，会不断争夺缓存行所有权。
- **中高级回答**：它没有 data race，锁和 race detector 未必提示；症状是 CPU 高、吞吐下降、扩核负收益。
- **深挖回答**：可通过 per-worker 分片、批量聚合、减少共享写和谨慎 padding 优化；但 cache line 大小、结构布局和对象分配位置依架构与编译器而异，必须 benchmark。
- **错误回答**：“两个字段不同就不会互相影响。”硬件一致性单位是缓存行，不是 Go 字段。
- **版本/边界**：属于硬件/布局效应；不要依赖 runtime 内部 P ID 作为稳定公开分片接口。

## 八、深挖追问链

### 8.1 追问链一：从一条 `go` 语句追到 CPU 执行

**Q1：`go f(x)` 首先发生什么？**  
**答案要点**：当前 G 求值函数值和参数；调用体才异步。副作用、panic 和阻塞属于求值阶段。

**Q2：编译器怎样把它交给 runtime？**  
**答案要点**：生成适配调用的包装/参数布局，并调用 `runtime.newproc`；不要把编译器 lowering 当语言规范的一部分。

**Q3：`newproc` 与 `newproc1` 为什么分两层？**  
**答案要点**：外层取得调用方信息并切到 system stack；内层操作 G 池、栈和调度状态，不能在可能增长/移动的用户栈上随意完成。

**Q4：新 G 从哪里来？**  
**答案要点**：优先从当前 P/全局空闲 G 缓存复用 `_Gdead`，必要时分配新的 `g` 和初始栈；随后初始化 `sched`、goid、创建 PC 等。

**Q5：为什么新 G 通常进入 `runnext`？**  
**答案要点**：降低创建后通信/接力延迟；但 `runnext` 只有一个槽，可被替换或窃取，不构成立即运行保证。

**Q6：谁决定是否唤醒新 M？**  
**答案要点**：`wakep/startm` 根据是否有空闲 P、是否已有自旋 M 等判断；盲目唤醒会造成 thundering herd 和线程抖动。

**Q7：哪个函数最终选中 G？**  
**答案要点**：M 在 g0 上进入 `schedule`，由 `findRunnable` 综合本地/全局队列、Timer、netpoll、stealing 等选取，`execute` 建立 running 状态。

**Q8：真正切到 G 的机器状态怎样恢复？**  
**答案要点**：`execute` 准备 `gp.m/mp.curg`，底层通过 `gogo(&gp.sched)` 等汇编恢复 SP、PC 和 G 寄存器；新 G 的入口最终运行包装函数，返回后进入 `goexit` 清理。

**Q9：如何证明上述顺序，而不是背博客？**  
**答案要点**：固定 `go1.26.4`，从 `proc.go:newproc/newproc1/runqput/schedule/findRunnable/execute/goexit0` 纵向阅读，并用 trace 观察创建、runnable、running 与退出事件。

### 8.2 追问链二：从 `conn.Read` 阻塞追到 netpoll 唤醒

**Q1：`conn.Read` 为什么通常不占一条线程等待数据？**  
**答案要点**：Go net 层把可轮询 fd 设置为非阻塞；未就绪返回 EAGAIN 后进入 poll 等待，而不是让 read syscall 长期卡住。

**Q2：等待关系记录在哪里？**  
**答案要点**：`internal/poll.FD` 关联 runtime `pollDesc`；读和写方向有各自等待状态/G。

**Q3：当前 G 怎样停止执行？**  
**答案要点**：`poll_runtime_pollWait → netpollblock → gopark`；G 从 running 变 waiting，M/P 可调度其他工作。

**Q4：谁在等 epoll/kqueue/IOCP？**  
**答案要点**：runtime 的 netpoll 路径可由调度线程或 sysmon 进入；没有必要为每个连接单独建立等待线程。

**Q5：事件到达后怎样避免重复唤醒？**  
**答案要点**：pollDesc 状态以原子协议表示“无等待者/等待者/已就绪”；`netpollready` 原子交换并取得对应 G，避免丢失和重复 ready。

**Q6：为什么 G 被唤醒后还不能保证立刻执行？**  
**答案要点**：ready 只把 G 注入 runnable 队列；它仍需竞争 P，受 backlog、CPU quota、GC 和其他任务影响。

**Q7：Deadline 如何与这条链结合？**  
**答案要点**：读写 deadline 通过 Timer 在到期时修改 poll 状态并唤醒等待 G；G 恢复后观察超时错误，而不是 Timer 直接“杀死” G。

**Q8：什么 I/O 不适用这一模型？**  
**答案要点**：普通文件、不可 poll 的 fd、直接阻塞 syscall、部分 DNS/cgo/第三方库调用；这些可能占住 M，需用 trace/thread 指标确认。

### 8.3 追问链三：从线程数暴涨追到 syscall/cgo

**Q1：`GOMAXPROCS=4` 为什么进程仍能有几百个线程？**  
**答案要点**：P 限制同时执行 Go 代码的并行度，不限制阻塞 syscall/cgo、runtime 系统线程或 LockOSThread 造成的 M 数量。

**Q2：一个 M 进入阻塞 syscall 时 P 怎么办？**  
**答案要点**：通过 `entersyscall` 协议标记 G/M；P 可在明确路径释放，或由 sysmon 判断后 retake，交给其他 M。

**Q3：为什么 runtime 可能创建新 M？**  
**答案要点**：有可运行 G 和空闲 P，却没有可用 M 承载时，需要唤醒或创建 M 保持 Go 工作推进。

**Q4：syscall 返回时一定回到原 P 吗？**  
**答案要点**：不一定；`exitsyscall` 先走快速重取路径，失败时把 G 变 runnable，再进入普通调度。

**Q5：cgo 为什么比普通 syscall 更复杂？**  
**答案要点**：涉及 Go/C 栈、GC 指针规则、线程局部状态、C 回调 Go、C 创建线程及信号；长期 C 阻塞也占住 M。

**Q6：`LockOSThread` 怎样制造线程膨胀？**  
**答案要点**：锁定 G 需要专属 M；未平衡解锁或大量并发线程亲和任务会减少可复用 M，G 退出时锁定线程还可能被终止。

**Q7：如何区分是哪一类原因？**  
**答案要点**：看 threadcreate profile、goroutine 堆栈、trace 的 syscall/cgo 区间、`/sched/threads/total` 与 `/sched/goroutines/not-in-go`；再结合 OS 线程栈和系统调用观测。

**Q8：治理手段是什么？**  
**答案要点**：优先使用 netpoll 友好 API；给 cgo/阻塞调用建有界池；修复 LockOSThread 配对；降低跨边界频率；建立线程数和 not-in-go 告警。

### 8.4 追问链四：从无限循环追到抢占、栈和 GC

**Q1：一个无函数调用的 CPU 无限循环为什么危险？**  
**答案要点**：它持续占 P，旧式纯协作抢占难以打断，会推高其他 G 调度和 GC STW 等待。

**Q2：Go 1.14 后为什么改善？**  
**答案要点**：引入基于信号的异步抢占，可在编译器标记为安全的 PC 注入抢占入口，不再完全依赖函数调用。

**Q3：是否任何指令都可被抢占？**  
**答案要点**：否；runtime 检查 async safe point、栈状态、内部锁、写屏障和 unsafe-point，某些区间只能延后。

**Q4：栈检查与抢占有什么联系？**  
**答案要点**：`stackguard0` 可被设置为抢占哨兵；函数序言走 `morestack/newstack` 时识别抢占请求，这是协作式路径之一。

**Q5：真正需要扩栈时发生什么？**  
**答案要点**：在 g0 上分配更大连续栈，复制已用栈，并按 stack map/运行时元数据调整指针、defer、panic、sudog 等引用。

**Q6：为什么不能用 C 的任意指针扫描方式处理 Go 栈？**  
**答案要点**：Go GC 要精确识别指针；编译器产生 stack map，栈复制也只调整被证明是栈指针的槽，避免把整数误认成指针。

**Q7：异步抢占怎样帮助 GC？**  
**答案要点**：让长时间运行 G 更快到达可停止/可扫描状态，缩短 STW 握手尾部；但不可抢占区仍可造成长尾。

**Q8：如何排查长不可抢占区？**  
**答案要点**：trace 看长 running 区间和 STW 等待，CPU profile 看热点，检查 cgo/汇编/`nosplit`/runtime 临界代码；不要只用 `Gosched` 掩盖算法问题。

### 8.5 追问链五：从 Goroutine 数上涨追到泄漏根因

**Q1：G 数上涨就能判定泄漏吗？**  
**答案要点**：不能。需与请求量、连接数、批处理阶段和基线比较；关键是流量停止/任务结束后是否回落，以及生命周期是否仍有业务价值。

**Q2：第一份证据取什么？**  
**答案要点**：在相同负载阶段抓两到三份 goroutine profile，按堆栈聚合比较增长项，同时记录总数和 wait reason。

**Q3：看到大量 `[chan send]` 应怎样推理？**  
**答案要点**：找发送 Channel 的所有者、接收者是否提前返回、是否无缓冲/缓冲已满、发送是否有 Context 分支；沿创建栈回到任务入口。

**Q4：看到 `[select]` 是否就是泄漏？**  
**答案要点**：不一定。检查 select 是否含取消/关闭分支、对应 Context 是否永不取消，以及该后台 G 的 owner 是否已经退出。

**Q5：为什么还要看 heap profile？**  
**答案要点**：一个 G 可保留巨大对象图；数量不多也可能造成内存事故。按 retained memory 和创建栈关联，确认闭包、缓冲和请求对象。

**Q6：Go 1.26 的细分指标能提供什么？**  
**答案要点**：created、runnable、running、waiting、not-in-go 可区分创建速率和状态积压；总 G 平稳但 created 激增说明 churn，不一定是泄漏。

**Q7：实验性 `goroutineleak` profile 能否一锤定音？**  
**答案要点**：不能。它需 `GOEXPERIMENT=goroutineleakprofile` 构建，只能识别一部分通过可达性证明永久阻塞的 G；未报告不代表无泄漏。

**Q8：修复后如何验证？**  
**答案要点**：加入取消/关闭/等待；压测后停流量，确认 G、heap、fd、Timer 和连接回落；增加泄漏回归测试和稳定性监控，而非只看一次 profile。

### 8.6 追问链六：从容器 CPU throttling 追到 `GOMAXPROCS`

**Q1：服务只有 2 CPU limit，为何旧程序仍创建几十个 P？**  
**答案要点**：Go 1.24 及以前默认多基于宿主机逻辑 CPU；容器可能看到整机 CPU，而 cgroup quota 只有 2。

**Q2：P 过多为什么会有问题？**  
**答案要点**：瞬时并行超过 quota，cgroup 在周期内很快耗尽预算并 throttling；形成延迟锯齿、runnable 积压、更多调度/cache 成本。

**Q3：Go 1.25 默认怎样改进？**  
**答案要点**：Linux 上默认结合逻辑 CPU、affinity 和 cgroup CPU throughput limit，选择更合适的 `GOMAXPROCS`。

**Q4：1.5 CPU quota 为什么可能得到 2？**  
**答案要点**：当前规则对小数向上取整，以保留一定并行；但 quota 是时间窗口平均吞吐，2 个 P 仍可能触发部分 throttling，需要压测。

**Q5：为什么 Kubernetes request 不参与？**  
**答案要点**：request 主要用于调度/权重，不是硬吞吐上限；runtime 能可靠读取的是 affinity 和 cgroup limit/quota 等执行约束。

**Q6：为何升级 Go 后行为看起来没变化？**  
**答案要点**：模块语言版本不高于 1.24 时兼容性默认可能关闭 container-aware/auto-update；还要检查 `GOMAXPROCS` 环境变量、显式 API 和 GODEBUG。

**Q7：运行中 quota 改变会怎样？**  
**答案要点**：默认模式可周期更新，频率上限约每秒一次；显式设置 `GOMAXPROCS` 会关闭更新，可用 `SetDefaultGOMAXPROCS` 恢复。

**Q8：如何验证最优值？**  
**答案要点**：同时看 throughput、P95/P99、cgroup throttled time、CPU、`/sched/latencies`、runnable 数和 GC；用阶梯压测比较，而不是只追求 CPU 100%。

### 8.7 追问链七：从“任务池吞吐下降”追到锁与缓存

**Q1：增加 worker 后吞吐先升后降，第一反应是什么？**  
**答案要点**：检查是否超过 CPU/下游容量，并拆分调度、锁竞争、内存带宽、false sharing 和 GC 成本。

**Q2：如何区分 runnable backlog 与锁等待？**  
**答案要点**：runnable/latency 高且 CPU 满偏向 CPU 排队；mutex/block profile 高且 CPU 未满偏向同步等待；trace 看状态区间。

**Q3：为什么一个全局 atomic counter 也可能成为瓶颈？**  
**答案要点**：所有核反复争夺同一 cache line，即使无锁也发生一致性序列化。

**Q4：两个不同 atomic 字段为何仍互相干扰？**  
**答案要点**：若落在同一 cache line，分别写也造成 false sharing；字段级独立不等于硬件级独立。

**Q5：加 padding 就一定正确吗？**  
**答案要点**：不一定。cache line 大小与对象布局依平台，padding 增加内存和 cache footprint，还可能被嵌套布局破坏；必须 benchmark。

**Q6：更稳健的优化是什么？**  
**答案要点**：减少共享写、每 worker 分片、局部累加后批量合并、按 key 分区；先优化算法和所有权，再考虑手工 padding。

**Q7：为什么不能直接按 P 分片？**  
**答案要点**：runtime 不公开稳定的当前 P ID；使用内部 linkname/unsafe 会绑定实现。用显式 worker shard 更可维护。

**Q8：怎样避免“优化后数据对但更慢”？**  
**答案要点**：固定硬件和 quota 做 `go test -bench -benchmem`，采集 CPU/mutex profile 与硬件计数器，比较吞吐和尾延迟，并保留回归基线。

## 九、生产故障与排查

### 9.1 排障总原则：先判断 G 在“运行、排队还是等待”

遇到并发故障，先用一个三分法收敛问题：

```text
Running：正在消耗 CPU
Runnable：已经能运行，但在等 P/M/CPU
Waiting：在等锁、Channel、Timer、网络、系统调用或业务事件
```

对应证据：

| 现象 | 第一证据 | 第二证据 | 常见误判 |
|---|---|---|---|
| CPU 高 | CPU profile | trace、runnable/latency、GC 指标 | 把所有 CPU 高都归咎于“G 太多” |
| 请求慢、CPU 高 | trace scheduler latency | CPU/mutex profile、cgroup throttling | 只加实例，不查单实例排队 |
| 请求慢、CPU 低 | goroutine/block/mutex profile | trace netpoll/syscall、下游指标 | 盲目提高 GOMAXPROCS |
| 内存涨、G 数涨 | goroutine + heap profile | created/waiting 指标、fd/连接数 | 只看初始 G 栈估算内存 |
| 线程数涨 | threadcreate profile | not-in-go、trace syscall/cgo | 误以为 GOMAXPROCS 限制线程 |
| 吞吐扩核不升 | CPU/mutex profile | benchmark、cache/内存带宽 | 继续加 worker |
| 程序卡死 | 全量 goroutine stacks | block/mutex、trace、锁/Channel 所有权 | 等 runtime 自动报所有死锁 |

### 9.2 现场保全顺序

生产故障中，按“低扰动 → 高扰动”采证：

1. 记录精确时间、版本、实例、容器 CPU/memory limit、`GOMAXPROCS` 和流量。
2. 抓取运行指标：CPU、RSS/heap、G/M/P、GC、fd、连接池、队列和下游延迟。
3. 抓 goroutine profile，最好间隔 30～120 秒再抓一份比较趋势。
4. 对 CPU 故障抓 20～30 秒 CPU profile。
5. 必要时临时启用 block/mutex profile，再抓短窗口。
6. 对调度/延迟故障抓 5～10 秒 execution trace；trace 数据量和开销较高，窗口应短而有代表性。
7. 最后才考虑重启、core dump、SIGQUIT 等破坏性操作。

> **安全边界**：`net/http/pprof` 不应直接暴露到公网。使用仅管理网可达的端口、认证代理或按需启用，并记录取证时间与开销。

### 9.3 工具矩阵

| 工具 | 最适合回答 | 不能单独回答 |
|---|---|---|
| goroutine profile | 当前所有 G 在哪里、为何等待、哪些堆栈数量多 | 时间顺序、精确等待时长 |
| CPU profile | CPU 时间花在哪些调用路径 | G 为什么在 runnable 队列等待 |
| heap/allocs profile | 活对象保留、累计分配热点 | 对象为什么在业务上不再需要 |
| block profile | Channel、锁、WaitGroup 等阻塞累计时间 | 当前瞬时等待全貌；未启用前历史 |
| mutex profile | 哪些解锁点造成大量锁等待 | 无锁算法/Channel/netpoll 等等待 |
| threadcreate profile | 哪些调用路径导致 OS 线程创建 | 当前每个线程正做什么 |
| execution trace | G 状态时间线、P 利用率、syscall、netpoll、GC、调度延迟 | 长时间低频热点；数据量大 |
| race detector | 实际执行路径上的数据竞争 | 未覆盖路径、业务死锁、所有内存错误 |
| `schedtrace` | 周期性调度器快照 | 单请求因果链和源码热点 |
| runtime/metrics | 可持续趋势和告警 | 具体阻塞调用栈 |
| benchmark | 受控比较吞吐、延迟、分配和扩展性 | 生产下游、真实流量分布 |
| `-gcflags=-m=2` | 逃逸、内联等编译决策 | 生产实际热点；输出不是性能结论 |

### 9.5 Goroutine 泄漏排查剧本

#### 症状

- `/sched/goroutines:goroutines` 或 `runtime.NumGoroutine()` 长期单调上升；
- 停止流量后不回落；
- heap/RSS、fd、连接和 Timer 同步上涨；
- goroutine profile 某些堆栈样本持续增加。

#### 步骤

1. **建立基线**：记录空闲、正常峰值、故障时的 G 数，不用绝对阈值武断判断。
2. **抓三份堆栈**：例如 T0、T+60s、T+120s，按相同栈顶和创建栈聚合。
3. **按 wait reason 分类**：`chan send`、`chan receive`、`select`、`semacquire`、`IO wait`、`sleep`、`syscall`。
4. **找 owner**：谁创建该 G？谁应关闭 Channel/取消 Context/调用 Stop？owner 是否已经结束？
5. **看对象保留**：若 RSS/heap 同涨，用 heap profile 找闭包和 G 栈保留的大对象。
6. **检查增长速率**：Go 1.26 的 `/sched/goroutines-created` 可识别创建 churn；waiting/runnable 分类帮助判断是在等待还是排队。
7. **最小复现**：固定输入循环调用接口，停止后用最终 G 数、profile 和资源数断言回落。
8. **修复验证**：所有退出路径都要覆盖成功、错误、超时、取消、消费者提前退出和服务 shutdown。

#### 典型堆栈模式

| 堆栈状态 | 常见根因 | 修复方向 |
|---|---|---|
| 大量 `chan send` | 下游不读、缓冲满、消费者提前返回 | 发送增加取消分支；修复关闭/消费协议 |
| 大量 `chan receive` | 生产者不再发送且未关闭 | 关闭输入；Context；owner 等待 |
| 大量 `select` | 后台循环没有有效取消，或 Context 永不取消 | 传递并触发取消；提供 Stop/Wait |
| 大量 `semacquire` | Mutex/RWMutex/WaitGroup/内部 semaphore 等待 | mutex/block profile；检查持锁 I/O、计数错误 |
| 大量 `IO wait` | 可能是正常连接，也可能缺失 deadline | 与连接数、请求状态和 deadline 对照 |
| 大量 `sleep`/Timer | 重试无上限、周期任务重复启动 | 退避上限、任务去重、停止 Timer |

#### Go 1.26 实验性 leak profile

构建时开启：

```bash
GOEXPERIMENT=goroutineleakprofile go build ./cmd/service
```

运行后可通过 `pprof.Lookup("goroutineleak")` 或相应 pprof 端点读取。它通过可达性分析识别**部分可证明永久阻塞**的 G，但存在重要边界：

- 仍可被某个潜在事件唤醒的 G 不一定被报告；
- 业务上无用但技术上可唤醒的 G 可能检测不到；
- 实验功能和输出可能变化；
- 未报告不等于无泄漏。

### 9.6 内存上涨：区分 Goroutine 泄漏、对象泄漏和分配抖动

先拆成三种模式：

1. **G 数涨、in-use heap 涨**：G/闭包/栈保留对象，优先关联两类 profile。
2. **G 数稳定、in-use heap 涨**：缓存、map、队列、连接状态或普通对象生命周期问题。
3. **in-use 稳定、alloc-space/GC CPU 高**：不是保留泄漏，而是高分配率和 churn。

对比 heap 快照：

```bash
curl -sS http://127.0.0.1:6060/debug/pprof/heap > before.pb.gz
# 经过相同负载窗口
curl -sS http://127.0.0.1:6060/debug/pprof/heap > after.pb.gz

go tool pprof -http=:0 -base before.pb.gz after.pb.gz
```

注意：heap profile 是采样数据，默认主要反映最近一次已完成 GC 时的活对象；RSS 还包含栈、runtime 元数据、未归还 OS 的页、mmap/cgo 内存等，不能直接等同于 Go heap。

### 9.7 CPU 100% 排查剧本

#### 第一步：CPU profile 先找“执行了什么”

常见形态：

| CPU profile 热点 | 推理方向 |
|---|---|
| 业务纯计算 | 算法、任务量、分块、SIMD/库、是否重复计算 |
| `runtime.mallocgc`、GC worker | 分配率、逃逸、对象 churn、G/栈根数量 |
| `runtime.findRunnable`、队列/锁相关 | 超多短任务、空转/stealing、唤醒风暴 |
| sync/atomic 或锁路径 | 高竞争、false sharing、共享热点 |
| syscall/cgo | 外部调用或内核成本；Go CPU profile 可能不完整覆盖 C |
| 编解码/拷贝 | 缓冲策略、批处理、零拷贝可行性 |

#### 第二步：区分 Running 高还是 Runnable 高

- **CPU 高、running≈GOMAXPROCS、runnable 高、latency 高**：算力/配额饱和或任务过度 fan-out。
- **CPU 高、runnable 不高**：少量 G 本身很热，优化热点代码。
- **Go 进程 CPU 不高，但请求慢且 cgroup throttling 高**：配额窗口耗尽；容器视角与宿主机视角要统一。
- **CPU 高但吞吐低，mutex profile 高**：自旋、竞争或 cache contention。

#### 第三步：不要把加 `GOMAXPROCS` 当万能修复

增加 P 可能：

- 提高真正可并行 CPU 工作吞吐；
- 也可能增加 cgroup throttling、锁竞争、cache miss 和 GC 并行开销；
- 对 I/O/锁等待根因几乎无效。

必须用 throughput、P99、throttled time、scheduler latency 和 profile 联合验证。

### 9.8 延迟尖刺与 Scheduler latency 排查

#### 先分解请求时间

```text
总延迟 = 业务队列等待 + runnable 调度等待 + CPU 执行
       + 锁/Channel 等待 + 网络/磁盘/下游等待 + GC/系统暂停
```

#### 证据链

1. 用分布式 trace/业务埋点确定慢在哪个阶段。
2. 用 execution trace 检查该窗口 P 是否繁忙、G 何时 runnable、何时 running、syscall/netpoll 和 GC 何时发生。
3. 查看 `/sched/latencies:seconds` 和 runnable 数是否同步上升。
4. 若 mutex/block profile 高，定位锁持有/阻塞栈。
5. 检查 cgroup `throttled_usec`/throttled periods、CPU limit 和当前 `GOMAXPROCS`。
6. 检查是否在高峰瞬间创建海量短 G，造成队列和唤醒风暴。
7. 检查长 cgo、阻塞 syscall、不可抢占汇编或极端 runtime 临界区。

#### trace 适用边界

Execution trace 擅长时间线和调度因果，不擅长长时间热点统计。故障窗口很长时，先用指标定位短代表窗口，再抓 trace；CPU 热点仍优先用 CPU profile。

### 9.9 数据竞争排查

#### Race detector 的正确定位

```bash
go test -race ./...
go test -race -run TestName -count=100 ./pkg

go build -race -o service-race ./cmd/service
```

它通过插桩发现**本次执行实际发生**且缺少同步的冲突内存访问。使用原则：

- 覆盖不到的路径不会被发现；应增加真实并发负载和随机交错。
- `-race` 会显著改变内存、CPU 和调度，不应用它测正式性能。
- 没报告 race 不代表没有逻辑竞态，例如“两个合法请求都成功扣减一次库存”。
- Sleep、Gosched、日志输出和“通常按此顺序执行”都不是同步。
- 先修第一个可靠报告；多个后续报告可能是同一根因的连锁结果。

#### 常见修复误区

| 错误修复 | 问题 | 正确方向 |
|---|---|---|
| 加大 Sleep | 只改变概率 | 建立 Channel/锁/atomic 同步 |
| 到处加一个全局 Mutex | 可能正确但吞吐和死锁风险高 | 明确状态所有权和最小临界区 |
| 把普通变量改成 atomic | 单字段无 race，不等于多字段不变量正确 | 设计事务边界或用锁保护复合状态 |
| 关闭 race 检测 | 隐藏症状 | 修复生命周期和访问协议 |
| `recover` | recover 与 data race 无关 | 消除未同步共享访问 |

### 9.10 死锁与“程序卡住”排查

#### Runtime 能检测什么

Runtime 的 fatal deadlock 通常需要全局没有可推进的用户 G/Timer/netpoll/cgo 等工作。它不能可靠识别：

- 只有一部分请求相互死锁，其他 G 仍在工作；
- 活锁：G 一直 running，但业务没有进展；
- 长时间而非永久的锁等待；
- 依赖永远不会到来的外部响应，但 netpoll 仍认为可能就绪；
- 队列饥饿和优先级反转。

#### 非破坏性取证

优先抓 `/debug/pprof/goroutine?debug=2`；查看：

1. main G/请求入口在等谁；
2. 大量 `sync.(*Mutex).Lock`、`RWMutex`、`WaitGroup.Wait` 或 Channel 栈；
3. 持锁者是否在 I/O、发送 Channel、等待另一个锁；
4. nil Channel 是否使所有 select case 永久禁用；
5. WaitGroup 的 `Add/Done` 是否匹配；
6. Channel 的唯一关闭者是否永远到不了关闭点；
7. 是否在回调中重入同一把非可重入 Mutex；
8. 锁顺序是否形成 `A→B` 与 `B→A` 环。

`SIGQUIT` 在常见 Go/Unix 默认行为下会打印堆栈并终止进程，只应在已接受进程退出的应急场景使用，不要把它当普通在线采样手段。

#### 经典锁顺序修复

```go
// 错误：不同路径锁顺序相反。
func f() {
    muA.Lock()
    defer muA.Unlock()
    muB.Lock()
    defer muB.Unlock()
}

func g() {
    muB.Lock()
    defer muB.Unlock()
    muA.Lock()
    defer muA.Unlock()
}
```

修复方式是制定全局锁顺序，或把复合状态交给同一 owner/同一锁；不要用 `TryLock + Sleep` 把确定死锁改成概率活锁。

### 9.11 吞吐下降和扩核负收益

#### 先建立可比较基线

```bash
go test -run '^$' -bench BenchmarkServe -benchmem -count=10 ./pkg > before.txt
# 修改后在同一机器、同一 power mode、同一 GOMAXPROCS 下执行
 go test -run '^$' -bench BenchmarkServe -benchmem -count=10 ./pkg > after.txt
```

比较：

- `ns/op`、吞吐；
- `B/op`、`allocs/op`；
- P50/P95/P99，而非只看平均；
- `GOMAXPROCS=1,2,4,8...` 的 scaling curve；
- CPU/mutex/block profile 的热点是否移动；
- GC CPU、scheduler latency、runnable backlog；
- cgroup throttling 和下游限流。

#### 常见根因树

```text
增加并发但吞吐不升
├─ CPU 已饱和
│  ├─ 算法热点
│  ├─ 分配/GC
│  └─ 内存带宽/cache miss
├─ 同步串行化
│  ├─ 大锁/热点 atomic
│  ├─ 单 owner 成为瓶颈
│  └─ false sharing
├─ 外部容量上限
│  ├─ DB/连接池
│  ├─ 下游限流
│  └─ 磁盘/网络
├─ 调度开销
│  ├─ 海量短 G
│  ├─ 无界 runnable backlog
│  └─ 频繁 park/ready
└─ 容器约束
   ├─ quota throttling
   ├─ affinity
   └─ GOMAXPROCS 不匹配
```

优化顺序通常是：消除无用工作 → 降低分配 → 缩小/分片共享状态 → 调整任务粒度 → 调整并发度 → 最后才做 cache-line 级微优化。

### 9.12 OS 线程数暴涨

#### 关键指标

- OS 线程总数；
- `/sched/threads/total:threads`；
- `/sched/goroutines/not-in-go:goroutines`；
- threadcreate profile；
- cgo 调用数/耗时；
- 阻塞 syscall 堆栈；
- `LockOSThread` 调用路径。

#### 根因优先级

1. 大量不可 netpoll 的阻塞 syscall/文件 I/O；
2. cgo 调用长期阻塞或外部库内部建线程；
3. LockOSThread 未配对或线程亲和任务无界；
4. 极高并发下 runtime 为保持 P 工作而增建 M；
5. 线程本地资源或 OS 行为导致线程无法复用。

治理时不要把 `GOMAXPROCS` 调成 1 作为“线程上限”。应改用 netpoll 友好 API、有界阻塞调用池、异步 I/O、cgo 批处理和线程亲和任务限流。

### 9.13 服务 Shutdown 卡住

常见顺序错误：先等待 Worker，再关闭其输入；先关结果 Channel，再有 Worker 发送；取消了父 Context，却有下游使用 `context.Background()`。

推荐关闭协议：

```text
1. 停止接收新任务/新连接
2. 关闭或停止生产者
3. 取消任务 Context
4. 让发送和接收两端都能观察取消
5. 等待任务组，设置总 shutdown deadline
6. 关闭仅由本组件拥有的输出资源
7. 输出未退出 G 的 profile，再决定强制退出
```

对每个后台组件要求：

```go
type Lifecycle interface {
    Stop()       // 幂等地发出停止信号
    Wait() error // 等待所有内部 G 退出并返回终态
}
```

不要让 `Stop` 在持有组件内部锁时等待 G，而内部 G 退出又需要同一把锁，否则会制造 shutdown deadlock。

### 9.14 临时启用 block/mutex profile

```go
package diagnostics

import "runtime"

func EnableContentionProfiles() (restore func()) {
    // 1 表示尽可能记录每个阻塞事件，开销可能很高；生产应按需提高采样间隔。
    runtime.SetBlockProfileRate(1)

    // 1 表示采样每个 mutex contention 事件；生产通常应使用更稀疏比例。
    oldMutexRate := runtime.SetMutexProfileFraction(1)

    return func() {
        runtime.SetBlockProfileRate(0)
        runtime.SetMutexProfileFraction(oldMutexRate)
    }
}
```

注意：

- block profile 只记录启用后的采样事件；不能回溯历史。
- mutex profile 的堆栈通常指向造成等待的临界区结束/Unlock 一侧，阅读时不要误以为它一定是等待者。
- 全量采样可能显著增加开销，应先在压测/单实例/短窗口验证。

### 9.15 `GODEBUG` 与 traceback 的诊断用法

```bash
# 每秒输出调度摘要；详细模式输出会非常多
GODEBUG='schedtrace=1000,scheddetail=1' ./service

# 打印 GC 周期摘要，判断 GC 是否与尖刺相关
GODEBUG='gctrace=1' ./service

# 在堆栈中附带最多 10 层 Goroutine 创建祖先
GODEBUG='tracebackancestors=10' ./service

# 崩溃时包含 runtime G 和 runtime 栈
GOTRACEBACK=system ./service

# 仅用于 A/B 诊断异步抢占相关问题，不是生产修复
GODEBUG='asyncpreemptoff=1' ./reproducer
```

`schedtrace` 常见字段会展示 `gomaxprocs`、空闲 P、线程、自旋线程、空闲线程、全局 runqueue 和 per-P runq 等，但文本格式属于诊断实现，脚本不应把它当稳定机器接口。长期监控优先使用 `runtime/metrics`。

### 9.16 Go 1.26 Scheduler 指标采集示例

指标集合随版本演进，下面先枚举支持项，再采样，避免旧版本因硬编码名称产生错误假设：

```go
package schedmetrics

import (
    "fmt"
    "runtime/metrics"
)

var wanted = map[string]bool{
    "/sched/gomaxprocs:threads":                 true,
    "/sched/threads/total:threads":              true,
    "/sched/goroutines:goroutines":              true,
    "/sched/goroutines-created:goroutines":      true, // Go 1.26+
    "/sched/goroutines/runnable:goroutines":     true, // Go 1.26+
    "/sched/goroutines/running:goroutines":      true, // Go 1.26+
    "/sched/goroutines/waiting:goroutines":      true, // Go 1.26+
    "/sched/goroutines/not-in-go:goroutines":    true, // Go 1.26+
    "/sched/latencies:seconds":                  true,
}

func Snapshot() {
    descriptions := metrics.All()
    samples := make([]metrics.Sample, 0, len(wanted))

    for _, d := range descriptions {
        if wanted[d.Name] {
            samples = append(samples, metrics.Sample{Name: d.Name})
        }
    }

    metrics.Read(samples)
    for _, sample := range samples {
        switch sample.Value.Kind() {
        case metrics.KindUint64:
            fmt.Printf("%s=%d\n", sample.Name, sample.Value.Uint64())
        case metrics.KindFloat64:
            fmt.Printf("%s=%g\n", sample.Name, sample.Value.Float64())
        case metrics.KindFloat64Histogram:
            h := sample.Value.Float64Histogram()
            fmt.Printf("%s histogram buckets=%d counts=%d\n",
                sample.Name, len(h.Buckets), len(h.Counts))
        default:
            fmt.Printf("%s unsupported kind=%v\n", sample.Name, sample.Value.Kind())
        }
    }
}
```

生产采集建议：

- 对 counter 计算速率，例如 created/s；
- 对 gauge 观察趋势和与流量的比例；
- 对 histogram 在监控端计算分位数；
- 标签维度不要包含请求 ID 等高基数数据；
- 将 Go 版本、GOMAXPROCS、CPU quota 和实例规格作为诊断元数据。

### 9.17 用 `runtime/trace` 标记业务阶段

```go
func Handle(ctx context.Context, req Request) error {
    ctx, task := trace.NewTask(ctx, "HandleRequest")
    defer task.End()

    trace.Log(ctx, "request.id", req.ID)

    var data Data
    trace.WithRegion(ctx, "load", func() {
        data = load(ctx, req)
    })

    var err error
    trace.WithRegion(ctx, "store", func() {
        err = store(ctx, data)
    })
    return err
}
```

用户 task/region 可以把业务阶段与 runtime 调度、GC、syscall 时间线对齐。不要在每个极细操作上打大量事件，以免 trace 体积和扰动失控。

### 9.18 编译器信息如何辅助调度问题

`-gcflags='all=-m=2'` 可回答：

- 闭包是否逃逸到堆；
- 参数/局部对象是否因 Goroutine 生命周期而逃逸；
- 函数是否内联；
- 某些对象为何产生额外分配。

典型现象：

```go
for _, item := range items {
    go func() {
        use(item)
    }()

```

闭包和捕获变量可能需要活到调用方栈帧之外，因此产生堆分配或被复制到闭包对象。编译器输出只是“为什么这样编译”的证据；是否真是性能瓶颈仍需 alloc/CPU profile 和 benchmark 证明。

| 故障 | 短期止血 | 根因修复 | 回归验证 |

## 十、面试回答模板

### 10.1 30 秒模板：什么是 Goroutine 和 GMP

> Goroutine 是 Go 规范定义的并发执行单元，不等同于 OS 线程。官方 runtime 用 GMP 调度：G 保存任务、栈和执行现场，M 对应操作系统线程，P 保存本地运行队列等执行 Go 代码所需资源；M 通常持有 P 才能运行用户 Go 代码。`GOMAXPROCS` 主要控制 P 数和 Go 代码并行度，不限制 G 或总线程数。网络等待通常通过 netpoll 只挂起 G，阻塞 syscall 会占 M 但 P 可移交。G 很轻量，但仍有栈、调度和 GC 成本。

### 10.2 2 分钟模板：从创建讲到运行

> 执行 `go f(x)` 时，规范保证函数值和参数先在当前 Goroutine 求值，实际调用才在新 Goroutine 中执行，也不保证立即运行。当前 runtime 中，编译器把调用交给 `newproc`，`newproc1` 在 system stack 上取得或分配一个 G，初始化栈与调度上下文，再优先放入当前 P 的 `runnext` 或本地 runq。
>
> M 在 g0 上执行 `schedule`，`findRunnable` 会综合本地队列、周期性全局队列、Timer、netpoll 和 work stealing 找任务，再由 `execute` 把 G 从 runnable 切到 running。G 因 Channel、锁或网络等待时，通过 `gopark` 进入 waiting；事件方用 `goready` 让它重新 runnable。
>
> 阻塞系统调用会占住 M，但 P 可以 handoff 给其他 M；可轮询网络 I/O 一般只 park G。`GOMAXPROCS` 控制同时执行用户 Go 代码的上限，不等于线程上限。Go 1.25 起默认还会考虑容器 CPU quota，当前版本可动态更新默认值。生产上我会用 Context、任务组和并发上限管理生命周期，并用 goroutine profile、runtime metrics 和 trace 排查泄漏与调度延迟。

### 10.3 5 分钟模板：完整系统回答

可以按“语义—模型—流程—阻塞—抢占—工程”六段回答：

#### 第一段：语言语义

> Goroutine 是由 `go` 语句启动的独立并发调用。函数值和参数在调用方求值，返回值被丢弃，不保证启动顺序、完成顺序或固定线程。Goroutine 没有自动父子生命周期；父 G return、recover 或 defer 都不会自动管理子 G。

#### 第二段：GMP 模型

> 当前官方 runtime 里，G 保存连续可增长栈、PC/SP、状态、defer/panic 和等待信息；M 是 OS 线程及其 g0、当前 G、线程状态；P 是执行 Go 代码的逻辑资源，持有本地 runq、runnext、Timer、分配缓存等。P 数大体由 `GOMAXPROCS` 决定。把 P 从 M 解耦，使阻塞 syscall 中的 M 可以让出 P，同时本地队列降低全局锁竞争。

#### 第三段：创建和调度

> `newproc → newproc1` 准备新 G，`runqput` 将其放入 runnext/本地队列。调度循环是 `schedule → findRunnable → execute`。找任务时大致先利用本地性，但会周期检查全局队列，也会处理 Timer、netpoll、GC 工作和从其他 P 窃取，因此不是严格 FIFO。找不到任务时 M 可能自旋、释放 P、休眠或阻塞 netpoll。

#### 第四段：阻塞和唤醒

> Channel、Mutex、WaitGroup 等在条件不满足时先登记等待节点，再 `gopark`，避免阻塞整条线程；对端完成后 `goready`。网络 fd 通常非阻塞，EAGAIN 后 G 挂到 pollDesc，内核就绪事件再 ready。真正阻塞 syscall/cgo 会占 M，P 可由其他 M 接管，因此总线程数可能超过 GOMAXPROCS。

#### 第五段：栈和抢占

> G 使用连续可增长栈。函数序言检查 stack guard，空间不足通过 `morestack/newstack` 在 g0 上分配更大栈、复制并调整栈内指针。抢占既有函数调用协作式路径，也有 Go 1.14 起的异步信号路径；但只能在安全点处理，不是任意指令硬切换，也不提供实时保证。safe point 同时服务调度和 GC 精确扫描。

#### 第六段：工程边界

> G 轻量不免费。CPU 任务并发一般从有效 GOMAXPROCS 附近起步，I/O 任务按连接池、内存、下游限额和 Little's Law 限流。任务必须有 owner、取消、等待、错误传播和上限。排障先判断 G 是 running、runnable 还是 waiting，再选择 CPU/goroutine/block/mutex profile、runtime metrics、schedtrace 或 execution trace。

### 10.4 源码级模板：面试官让你沿调用链讲

> 我会固定到目标 Go tag，先看 `runtime2.go` 的 `g/m/p` 和状态，再追三条主链：
>
> 1. 创建链：`newproc → newproc1 → runqput(next=true) → wakep`；
> 2. 运行链：`schedule → findRunnable → execute → gogo`；
> 3. 阻塞链：业务原语登记等待 → `gopark → park_m → schedule`，事件到达后 `goready → ready → runqput`。
>
> 网络路径继续追 `internal/poll → netpollblock/netpollready`；系统调用追 `entersyscall/exitsyscall` 和 `sysmon/retake`；栈追 `morestack → newstack → copystack`；异步抢占追 `preemptM` 和平台信号入口。讲每条链时我会明确状态从 `_Grunning` 变成什么、代码运行在用户栈还是 g0、谁持锁、谁负责唤醒，以及这是规范还是实现。

### 10.5 60 秒模板：如何排查 Goroutine 泄漏

> 我不会只看 `NumGoroutine` 的绝对值。先确认流量停止后 G 是否回落，再间隔抓多份 goroutine profile，按相同创建栈和 wait reason 比较增长项。看到 `chan send/receive`、`select`、`semacquire` 或 `IO wait` 后，沿创建点找 owner，检查 Channel 关闭、Context 传播、消费者提前退出和 Timer/后台循环停止路径。同时对比 heap profile，因为泄漏 G 可能保留很大的对象图。Go 1.26 还可用 created/runnable/waiting 等 metrics；实验性 goroutineleak profile只能辅助，未报告不能证明没有泄漏。修复后用停流量回落测试验证 G、heap、fd 和连接都恢复基线。

### 10.6 60 秒模板：容器中的 `GOMAXPROCS`

> `GOMAXPROCS` 限制的是同时执行用户 Go 代码的线程/P 数，不是 Goroutine 或总线程数。Go 1.25 起 Linux 默认会综合逻辑 CPU、进程 affinity 和 cgroup CPU quota，当前规则通常取较小值，fractional quota 向上取整，并可周期更新。它读取的是 limit/quota，不是 Kubernetes request。显式设置环境变量或调用 `runtime.GOMAXPROCS` 会固定值并停止自动更新，`SetDefaultGOMAXPROCS` 可恢复。最终值不是越大越好，应结合 cgroup throttling、吞吐、P99、runnable backlog 和 scheduler latency 压测。

## 十一、本章速记

1. **规范先行**：`go` 的函数值和参数在调用方求值，调用体才在新 G 执行。
2. **不保证立即运行**：新 G runnable 不等于 running，更不等于完成。
3. **没有自动父子关系**：取消、等待、错误传播必须由业务显式建立。
4. **G 是任务现场**：不仅有栈，还有 PC/SP、状态、defer/panic、等待与诊断信息。
5. **M 是 OS 线程表示**：有 g0，承载用户 G，也可能被 syscall/cgo 占用。
6. **P 是 Go 执行资源**：本地 runq、runnext、Timer、mcache 等都与它相关。
7. **GOMAXPROCS ≈ P 数**：限制用户 Go 代码并行度，不限制 G 数和总 M 数。
8. **本地队列保局部性**：减少全局锁；满时批量转移，全局队列还承担公平性。
9. **`runnext` 不是承诺**：只是低延迟接力优化，可能被替换或窃取。
10. **Work stealing 做负载均衡**：空闲 P 批量窃取，但没有严格 FIFO 公平保证。
11. **调度主链**：`schedule → findRunnable → execute`。
12. **阻塞主链**：登记等待 → `gopark`；事件发生 → `goready`。
13. **Netpoll 主要 park G**：可轮询网络 fd 未就绪时通常不长期占 M。
14. **阻塞 syscall 占 M**：P 可 handoff；所以线程数可远大于 GOMAXPROCS。
15. **sysmon 无 P 运行**：负责抢占、syscall retake、Timer/netpoll 等维护。
16. **连续栈会复制增长**：`morestack/newstack/copystack`，成本与已用栈相关。
17. **Go 1.14 是抢占分界**：增加异步抢占，但 safe point 和不可抢占区仍存在。
18. **Go 1.22 是循环变量分界**：`:=` 声明的循环变量按迭代创建；外部预声明变量仍共享。
19. **Go 1.25 有两项重点**：`WaitGroup.Go`；容器感知的默认 `GOMAXPROCS`。
20. **Go 1.26 指标更细**：可观察 created/runnable/running/waiting/not-in-go 等调度状态。
21. **G 轻量不免费**：闭包、对象图、GC 根、队列、Timer、连接和调度延迟都要计入。
22. **排障先做三分法**：Running 看 CPU，Runnable 看调度/配额，Waiting 看锁、Channel、I/O 与生命周期。

### 12.4 统一答案

#### 简答题答案

**1. 规范、API 与实现边界**

规范保证 `go` 语句启动独立并发执行的函数调用、函数值和参数在调用方求值、返回值丢弃等语义。`runtime.GOMAXPROCS`、`Gosched`、`LockOSThread` 属于公开 API 契约。GMP、`g/m/p` 字段、本地队列容量、`runnext`、work stealing、初始栈和抢占常数都是当前编译器/runtime 实现，未来可变。

**2. `go f(a(), b())`**

当前 G 按普通调用规则求值函数值 `f` 与参数 `a()`、`b()`；若求值 panic 或阻塞，发生在当前 G。求值完成后，实际对 `f` 的调用体在新 G 中独立执行。新 G 何时开始、是否在当前 G 下一条语句前开始、在哪个 M 上执行都没有规范顺序；返回值被丢弃。

**3. G/M/P 与 syscall**

G 保存任务栈和运行现场；M 表示 OS 线程及 g0/当前 G；P 持有 runq、Timer、分配缓存等执行 Go 代码的资源。M 执行普通 Go 代码通常要持 P。G 进入阻塞 syscall 时 M 被内核占用，但 P 可 handoff 给另一个 M；为让 P 继续工作，runtime 可能存在或创建更多 M，所以总线程数可高于 `GOMAXPROCS`。

**4. 创建到运行调用链**

编译器准备调用包装和已求值参数，调用 `runtime.newproc`；它在 system stack 上进入 `newproc1`，从空闲 G 池复用或分配 G，准备初始栈和 `sched` 上下文，将状态变为 runnable，并通过 `runqput(..., true)` 放入 runnext/本地队列，必要时 `wakep`。某个 M 在 g0 上运行 `schedule`，`findRunnable` 选中它，`execute` 建立 running 状态并通过底层 `gogo` 恢复执行。

**5. 四类机制的目的**

本地 runq 提供局部性并减少全局锁；全局 runq 接收溢出、跨 P 工作并帮助公平；`runnext` 为刚创建/唤醒的相关 G 提供低延迟接力；work stealing 让空闲 P 从繁忙 P 批量拿任务。四者共同在吞吐、延迟、局部性和公平之间折中。

**6. park/ready**

等待原语必须避免“登记等待之前事件已经发生”造成丢失唤醒。`gopark` 的解锁回调让 runtime 在当前 G 已安全进入等待协议后再释放关联锁，从而把登记与解锁正确衔接。`goready` 只把 G 从 waiting 转为 runnable 并入队；它仍需等待 P/M/CPU，不保证立即运行。

**7. 三类阻塞**

可轮询网络 fd 通常设为非阻塞，EAGAIN 后 G 挂到 pollDesc，M/P 可做别的工作；阻塞 syscall 会让 G/M 处于 syscall，M 被占住但 P 可 handoff；cgo 同样可能长期占 M，还增加 Go/C 栈、指针规则和回调管理。具体平台和 fd 类型可能使路径不同。

**8. 栈与抢占**

函数序言检查 stack guard；空间不足进入 `morestack/newstack`，在 g0 上分配更大连续栈，复制已用部分并调整可识别的栈指针、defer、panic、sudog 等。异步抢占用信号请求切换，但只能在 runtime 能安全理解 PC、栈和指针状态的位置执行；持有内部锁、unsafe point、`nosplit` 区域等会延后。

**9. 容器默认值**

当前默认主要综合逻辑 CPU、进程 CPU affinity 和 Linux cgroup CPU quota/period 得到的 throughput limit，通常取较小值；不直接使用 Kubernetes request。显式 `GOMAXPROCS` 环境变量或 API 调用会固定值并停止自动更新；`runtime.SetDefaultGOMAXPROCS()` 可恢复默认计算与更新。Go 1.25 及语言版本兼容边界要一并检查。

**10. 任务组不变量与取证**

至少明确 owner、启动/停止、Context 传播、等待、错误策略、panic 策略、并发上限、队列上限、Channel 关闭者、结果消费和可观测性。G 数上涨时先比较停流量后是否回落，间隔抓多份 goroutine profile，按创建栈/wait reason 找增长项，再结合 heap、fd、连接和 Go 1.26 `/sched/goroutines-*` 指标判断保留与状态。

#### 代码题答案

**代码题 1**

允许的输出只有：

```text
ABC
ACB
```

`arg()` 在 main G 求值，所以 `A` 必须最先。之后子 G 的 `B` 与 main G 的 `C` 无同步，顺序可互换。main 在 `<-done` 等待，而 `close(done)` 在打印 `B` 之后，因此 `B` 一定出现，且不会有只打印 `AC` 的正常完成结果。

**代码题 2**

存在 data race，输出不受保证。`i` 在循环外声明，循环使用赋值 `=`，Go 1.22 per-iteration 语义不为它创建每次迭代的新变量。main G 写 `i`，子 G 读 `i`。

修复一：传参，参数在发起方求值：

```go
for i = 0; i < 3; i++ {
    wg.Add(1)
    go func(v int) {
        defer wg.Done()
        fmt.Println(v)
    }(i)
}
```

修复二：循环体创建副本：

```go
for i = 0; i < 3; i++ {
    v := i
    wg.Add(1)
    go func() {
        defer wg.Done()
        fmt.Println(v)
    }()
}
```

**代码题 3**

正计数 `Add` 在计数为零时必须发生在对应 `Wait` 之前。把 Add 放进子 G 后，main 可先看到零并返回；并发交错也可能触发 WaitGroup misuse 检测。

传统修复：

```go
var wg sync.WaitGroup
for range 10 {
    wg.Add(1)
    go func() {
        defer wg.Done()
        work()
    }()
}
wg.Wait()
```

Go 1.25+：

```go
var wg sync.WaitGroup
for range 10 {
    wg.Go(func() {
        work()
    })
}
wg.Wait()
```

`wg.Go` 的函数按契约不得 panic，且没有 error 返回语义。

**代码题 4**

`generate` 从不读取 `ctx.Done()`。`first` 只接收一个值后返回，生成 G 下一轮会永久阻塞在 `out <- i`，无人接收；取消 Context 对它没有作用。

正确实现：

```go
func generate(ctx context.Context) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for i := 0; ; i++ {
            select {
            case out <- i:
            case <-ctx.Done():
                return
            }
        }
    }()
    return out
}
```

调用者还应保证 `cancel()` 在所有返回路径执行，通常创建后立即 `defer cancel()`。

**代码题 5**

`initializeGUI` 错误路径没有 Unlock。当前 G 在锁定状态退出时，runtime 无法把该 M 正常交给任意 G 复用，可能终止该 OS 线程并造成线程抖动。应在锁定后立即：

```go
runtime.LockOSThread()
defer runtime.UnlockOSThread()
```

`LockOSThread` 具有嵌套计数语义；若内部又锁一次，总计两次 Lock，只 Unlock 一次仍处于锁定状态，必须配对两次。更好的设计是明确唯一负责线程亲和的层，避免跨层隐式嵌套。

#### 设计与故障题答案

**设计题 1：50 后端聚合**

推荐结构：

1. 入口 Context 带 200ms deadline，并为连接建立、单后端响应保留预算，避免把全部时间耗在排队。
2. 使用 `errgroup.WithContext` 建立**单请求**任务组；关键后端返回错误时让派生 Context 取消。
3. 用服务实例级共享 semaphore（容量 20）执行可取消的 Acquire/Release，才能保证所有并发请求合计不超过 20；`SetLimit(20)` 只能限制单个 group，可作为额外的单请求 fan-out 上限，不能替代全局限流。连接池上限 100 是更外层硬资源边界，不应作为业务并发目标。
4. 每个请求使用 `NewRequestWithContext`，响应体必须关闭，并设置大小上限。
5. 结果写入预分配 slice 的独立索引或单 owner 聚合 Channel；请求返回前 `g.Wait()`，确保所有任务退出。
6. 明确关键/非关键后端策略：若允许部分结果，应把可忽略错误转换成结果状态，而不是让所有 error 都触发首错取消。
7. 记录 active tasks、limit wait time、后端耗时/错误、取消原因、deadline exceeded、G 创建数和 scheduler latency；避免后端名之外的高基数标签。
8. 测试覆盖首错、超时、调用方取消、连接池耗尽、消费者提前退出和 50 项全部成功，断言返回后 G 数回落。

**故障题 2：G 与内存上涨**

证据已经强烈指向结果发送阻塞：Worker 产生 `r` 后向 `results` 发送，但消费者停止、过慢或提前返回，导致 Worker 堆积；每个 G 还可能保留结果对象，解释 RSS 上涨。

处理步骤：

1. 间隔抓 goroutine profile，确认 `results <- r` 栈样本持续增长，并定位创建路径和批次 ID。
2. 抓 heap profile，验证 `r`、输入批次或闭包被这些 G 保留；查看 fd/连接是否一并增长。
3. 检查消费者为何退出、`results` 缓冲是否满、Channel 谁关闭、错误路径是否取消生产者。
4. 止血：限制新批次/worker 并发、降载，必要时滚回或重启泄漏实例；仅扩大 Channel 缓冲只是延后事故。
5. 修复发送：

```go
select {
case results <- r:
case <-ctx.Done():
    return
}
```

并确保消费者退出时取消 Context；由唯一 owner 在所有 Worker 退出后关闭 `results`。
6. 若结果可丢弃，明确 drop/覆盖策略并计数；若不可丢，采用有界持久队列和背压。
7. 回归：循环运行批次并在中途取消/制造消费者错误，停流量后检查 G、heap 和资源回落；加入发送阻塞时长与队列深度告警。

**故障题 3：容器 P99 锯齿**

机制是 64 个 P 允许大量 Go 工作瞬时并行，在 100ms 左右的 cgroup quota 周期内迅速消耗 2 CPU 的预算；随后进程被 throttling，runnable G 无法获得 CPU，形成与 quota 周期相关的延迟锯齿。平均 CPU 仍约 2 核，因此只看平均利用率会漏掉问题。

方案：

1. 升级到 Go 1.25+ 的容器感知默认行为，并确认 `go.mod` 语言版本、`GODEBUG=containermaxprocs/updatemaxprocs`、环境变量和代码中没有旧的显式设置。
2. 无法升级时先显式将 `GOMAXPROCS` 设在 2 附近，做 1、2、3、4 等阶梯实验，而非盲定。
3. 若运行中 quota 会变，使用默认自动更新；若业务要固定值，记录为什么关闭更新。
4. 在相同流量下比较 throughput、P50/P99、cgroup throttled periods/time、runnable 数、`/sched/latencies`、GC 和 CPU profile。
5. 同时限制 CPU fan-out；即使 P=2，海量 runnable G 仍会形成应用级排队。
6. 建立告警：throttling 比率、scheduler latency 尾部、runnable/GOMAXPROCS 比率和请求 P99 联动。
