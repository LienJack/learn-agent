# On-Page SEO 实战：服务端渲染、关键词密度、内链与外链怎么配合

很多人做博客 SEO 时，第一反应是找关键词、改标题、堆描述。但真正能长期生效的 On-Page SEO，通常不是某一个技巧，而是一组互相配合的页面信号：

- 搜索引擎能不能稳定拿到页面正文；
- 标题、摘要、正文和链接能不能说明页面主题；
- 内链能不能把相关内容串起来；
- 外链能不能补充可信来源；
- 文章本身是否真的解决了读者的问题。

所以这篇不讲玄学，只讲一个技术博客或内容站可以落地的检查框架：服务端渲染、关键词密度、内链、外链和博客内容结构分别应该怎么做。

## 一、On-Page SEO 先解决什么问题？

On-Page SEO 指的是单个页面内部可以控制的优化项，包括标题、描述、正文结构、链接、图片、语义标签、URL、性能和可索引性。

它的目标不是“骗搜索引擎排名”，而是让页面更容易被理解：

- 这篇文章讲什么；
- 适合什么搜索意图；
- 页面中哪些段落是核心内容；
- 它和站内其他文章是什么关系；
- 它引用了哪些可信资料；
- 搜索引擎和读者能不能顺利访问正文。

如果一篇博客内容质量不错，但页面是纯客户端渲染、标题模糊、没有目录结构、没有内链、外链随意堆放，那么搜索引擎需要付出更多成本才能判断它的价值。

On-Page SEO 的第一原则是降低理解成本。

## 二、服务端渲染：先让正文出现在 HTML 里

对博客、文档站、教程站来说，服务端渲染或静态生成通常是最稳的默认选择。

原因很简单：搜索引擎抓取页面时，首先拿到的是 HTML。如果核心正文、标题、链接和图片信息已经在 HTML 里，页面就更容易被发现、解析和索引。

如果页面必须等 JavaScript 执行以后才出现正文，就会多一层不确定性：

- 爬虫是否执行了这段 JavaScript；
- 执行时接口是否可访问；
- 渲染是否超时；
- 首屏内容是否和真实正文一致；
- 内链是否能以标准 `<a href="">` 的形式被发现。

Google 的开发者文档也提醒过，JavaScript 渲染会带来额外限制；对于依赖 JS 注入正文的站点，需要验证搜索引擎最终能看到什么。可以参考 Google Search Central 的 [JavaScript SEO 渲染问题指南](https://developers.google.com/search/docs/guides/rendering) 和 [开发者 SEO 指南](https://developers.google.com/search/docs/fundamentals/get-started-developers)。

对 Astro、Next.js、Nuxt、SvelteKit 这类内容站来说，建议把博客详情页做成：

- 标题、description、正文、目录、发布时间在 HTML 中直接可见；
- 文章内链使用标准 `<a href="">`；
- 图片有稳定 `src` 和 `alt`；
- canonical、Open Graph、RSS、sitemap 等元信息由服务端或构建期生成；
- 交互组件可以客户端增强，但不要依赖它们输出正文。

一个简单判断方法是：关闭浏览器 JavaScript 后，页面是否仍然能读完正文。如果可以，博客 SEO 的技术底座通常就比较稳。

## 三、关键词密度：用来检查主题，不用来堆词

“关键词密度”容易被误解成一个固定比例，比如主关键词出现 2% 到 3%。这类数字在真实内容优化里帮助有限，甚至会诱导作者堆词。

更合理的做法是把关键词密度当成诊断指标：

- 如果主关键词几乎没出现，说明标题和正文可能没有对齐；
- 如果同一个词反复机械出现，说明内容可能在堆砌；
- 如果只有精确关键词，没有同义词和相关概念，说明语义覆盖太窄；
- 如果关键词只出现在开头，正文后半段可能已经偏题。

例如一篇讲 On-Page SEO 的文章，不应该只重复“OnPageSeo”这个词。它应该自然覆盖这些相关表达：

- On-Page SEO；
- 页面 SEO；
- 服务端渲染；
- 静态生成；
- 关键词密度；
- 内链；
- 外链；
- 锚文本；
- 博客内容结构；
- 搜索意图；
- 可索引性。

Google 的 [SEO 入门指南](https://developers.google.com/search/docs/fundamentals/seo-starter-guide) 明确把关键词堆砌列为不应该关注的方向之一。对内容站来说，关键词不是要塞满页面，而是要出现在正确的位置。

推荐检查顺序：

1. 标题中是否包含核心主题；
2. description 是否用一句话说清页面价值；
3. H1 是否和搜索意图一致；
4. 前两段是否自然出现核心关键词；
5. H2 是否覆盖用户会继续追问的问题；
6. 正文是否出现同义词、场景词和问题词；
7. 结尾是否收束成可执行步骤。

如果这些位置都自然覆盖到了主题，就不需要追求某个固定密度。

## 四、内链：让文章进入站点结构，而不是成为孤岛

内链是技术博客最容易低估的 SEO 动作。

搜索引擎发现新页面，很大程度依赖链接。读者也需要通过链接判断：这篇文章属于哪个系列，前置知识在哪里，下一篇该读什么。

对博客来说，内链至少承担四件事：

- 把新文章接入已有专题；
- 把高价值旧文章的权重传给新文章；
- 帮读者补齐前置知识；
- 让搜索引擎理解文章之间的主题关系。

例如本站已经有 [Agent Harness 教程](/blog/AI/build-harness/build-harness) 和 [RAG 整体步骤](/blog/AI/2.Rag/01.整体步骤) 这类系列内容。写新文章时，就可以把“内容结构”“系列导读”“相关概念”自然连回这些旧文章，而不是让每篇文章都从零解释一遍。

内链优化时，要注意锚文本。

弱锚文本通常是：

```text
点击这里
了解更多
这篇文章
```

更好的锚文本是：

```text
Agent Harness 教程
RAG 整体流程
Claude Code 源码解析导读
```

锚文本应该让读者在点击之前就知道目标页面讲什么。Google 的 [链接最佳实践](https://developers.google.com/search/docs/crawling-indexing/links-crawlable) 也强调，链接需要可抓取，锚文本需要能说明目标页面内容。

一个简单的博客内链规则是：

- 每篇新文章至少链接 2 到 5 篇相关旧文章；
- 每个系列入口链接到所有核心文章；
- 旧的高流量文章定期补链到新文章；
- 避免在无关段落强行加链接；
- 不要让重要文章只能通过搜索框访问。

内链不是装饰，它是站点的信息架构。

## 五、外链：引用可信来源，而不是把权重“送出去”

很多站长担心外链会让页面权重流失，所以尽量不放外链。这个想法过于机械。

对技术博客来说，适当外链有两个价值：

- 给读者提供更权威的一手资料；
- 让搜索引擎看到文章不是闭门造车，而是在可信资料基础上展开解释。

外链应该优先指向：

- 官方文档；
- 标准规范；
- 学术论文；
- 项目源码；
- 权威工具说明；
- 原始数据来源。

比如写 SEO 技术文章时，链接到 Google Search Central 的官方文档，比链接到一堆二手营销文章更合适。写框架教程时，优先链接官方文档和源码仓库，而不是只链接聚合站。

外链也要克制：

- 不要为了 SEO 批量交换友情链接；
- 不要链接到低质量、采集、欺骗或明显过期的页面；
- 商业合作、广告、赞助内容应该按平台规则使用合适的链接属性；
- 用户生成内容中的链接需要更严格处理。

外链的判断标准很简单：这个链接是否真的帮读者验证事实、理解背景或继续学习？如果答案是否定的，就不要加。

## 六、博客内容结构：让读者和搜索引擎都能扫读

内容站最常见的问题不是没有写够字，而是结构混乱。

一篇适合搜索的技术博客，最好有清晰的信息层级：

1. 标题说明具体问题；
2. 开头交代读者场景；
3. 第一屏给出文章会解决什么；
4. H2 按问题拆分，而不是按作者思路流水账；
5. 每节先讲结论，再给原因和例子；
6. 代码、表格、清单服务于解释，不要堆素材；
7. 结尾给检查清单或下一步。

以“On-Page SEO”为例，标题不要写成：

```text
SEO 优化技巧大全
```

更好的标题是：

```text
On-Page SEO 实战：服务端渲染、关键词密度、内链与外链怎么配合
```

前者太泛，搜索意图模糊；后者直接说明主题、适合对象和覆盖范围。

正文也要避免一上来就塞概念。更好的写法是先把读者带入问题：

```text
你已经写了一篇技术博客，但搜索引擎迟迟不收录，或者收录后没有展示合适摘要。
这时不要先改关键词密度，而要检查正文是否在 HTML 里、标题是否清楚、内链是否接入站点结构。
```

这类开头比“SEO 是 Search Engine Optimization 的缩写”更接近真实搜索需求。

## 七、一篇博客发布前的 On-Page SEO 清单

发布前可以按这张清单过一遍。

### 技术可索引性

- 页面返回 `200`；
- 正文在 HTML 中可见；
- 标题、description、canonical 正确；
- 不被 `robots.txt` 或 `noindex` 阻止；
- 重要链接是标准 `<a href="">`；
- 图片有 `alt`；
- 移动端可读；
- 页面加载速度没有明显问题。

### 内容相关性

- 标题包含核心主题；
- description 说明页面解决什么问题；
- H1 只有一个，并且和标题一致或高度相关；
- H2 覆盖主要子问题；
- 前两段出现核心关键词和搜索场景；
- 正文自然覆盖同义词和相关概念；
- 没有机械重复关键词。

### 链接结构

- 链接到相关系列入口；
- 链接到 2 到 5 篇相关旧文章；
- 有必要时引用官方文档或一手资料；
- 锚文本能说明目标页面；
- 没有无关外链、失效链接或明显低质量链接。

### 内容质量

- 读者读完能执行下一步；
- 有具体例子，而不只是抽象原则；
- 没有过时结论；
- 没有大段重复内容；
- 结尾有总结或检查清单。

## 八、把 On-Page SEO 做成内容生产流程

最稳定的 SEO 不是写完以后补丁式修改，而是在内容生产流程里提前设计。

可以按这个顺序写博客：

1. 先定义搜索意图：读者到底想解决什么问题；
2. 再定标题和 H2：让文章结构直接回答问题；
3. 写正文时自然覆盖关键词和相关概念；
4. 写完后补内链：接入已有专题；
5. 最后补外链：引用可信来源；
6. 发布后用 Search Console 看查询词、点击率和收录状态；
7. 根据真实数据更新标题、摘要、内链和内容缺口。

如果是 Astro 这类静态内容站，最好把这些动作固化成模板：

- frontmatter 必填 `title`、`description`、`pubDate`、`locale`、`tags`；
- 每个专题有 `_meta.json`；
- 每篇文章正文保留一个清晰 H1；
- 系列文章互相链接；
- 重要图片统一压缩并填写说明；
- 构建前运行内容检查。

这样 SEO 就不会依赖临时记忆，而会变成发布流程的一部分。

## 总结

On-Page SEO 的核心不是追求关键词密度，而是让页面更容易被理解。

对博客和内容站来说，优先级可以这样排：

1. 用服务端渲染或静态生成保证正文可见；
2. 用标题、description、H1、H2 对齐搜索意图；
3. 把关键词密度当成诊断工具，不要堆词；
4. 用内链把文章接入站点结构；
5. 用外链补充可信来源；
6. 用真实数据持续更新内容。

当这些基础动作稳定执行，博客内容才更容易被搜索引擎抓取、理解和展示，也更容易让读者在站内继续阅读。
