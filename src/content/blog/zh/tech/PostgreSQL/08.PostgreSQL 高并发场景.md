# PostgreSQL 高并发场景：连接、锁和热点数据怎么一起压垮系统

很多人第一次排查 PostgreSQL 高并发问题，会先盯着两个指标：

```text
QPS 很高
CPU 很高
```

这两个指标当然重要，但它们不是入口。

PostgreSQL 的高并发问题，常常不是"数据库不够快"这么简单，而是几类压力叠在一起：

1. 应用连接太多，数据库进程数被打爆。
2. 事务太长，锁和 MVCC 旧版本迟迟释放不了。
3. 热点行被大量请求同时更新，所有写入排队。
4. 慢 SQL 把连接占住，后续请求继续堆积。
5. 读写没有分层，所有请求都挤在主库和同一批表上。

这篇文章只回答一个问题：

**PostgreSQL 在高并发业务里，通常不是先扩机器，而是先把连接、事务、热点和读写路径治理清楚。**

为了让例子具体一点，我们继续用电商订单和库存场景：

```sql
CREATE TABLE orders (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  user_id BIGINT NOT NULL,
  status TEXT NOT NULL,
  amount NUMERIC(10, 2) NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE inventory (
  sku TEXT PRIMARY KEY,
  stock INT NOT NULL,
  version INT NOT NULL DEFAULT 0,
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

现在有一个热门商品，几万用户同时下单。你希望数据库能扛住流量，但更重要的是：库存不能扣错、请求不能把连接池耗尽、局部热点不能拖垮整个系统。

## 一、先把高并发压力拆开

高并发不是一个单点问题，而是一条链：

```text
请求变多
-> 应用连接变多
-> PostgreSQL backend 进程变多
-> 慢查询和锁等待占住连接
-> 连接池排队
-> 应用超时重试
-> 数据库压力进一步放大
```

所以排查时不要只问"数据库最多能撑多少 QPS"。更好的问题是：

**这些并发请求到底是在消耗连接、CPU、IO、锁，还是同一行数据的修改机会？**

可以先用一张表把压力来源分清楚：

| 压力类型 | 常见现象 | 首先观察 |
|----------|----------|----------|
| 连接压力 | `too many clients`、大量 idle 连接 | `pg_stat_activity`、连接池指标 |
| 锁压力 | 请求卡住但 CPU 不高 | `pg_locks`、等待事件 |
| 热点行压力 | 某个 UPDATE 延迟暴涨 | `pg_stat_activity` 中相同 SQL 排队 |
| 慢查询压力 | 连接被长时间占用 | `pg_stat_statements`、`EXPLAIN ANALYZE` |
| IO 压力 | 查询和写入一起变慢 | `pg_stat_io`、系统 `iostat` |
| VACUUM 压力 | 表膨胀、索引膨胀、WAL 增长 | `pg_stat_user_tables`、dead tuple |

高并发治理的顺序通常是：先保护连接入口，再缩短事务，再拆热点，最后才谈读写扩展和分片。

## 二、连接风暴：PostgreSQL 不是靠无限连接扛并发

前一篇架构里已经讲过，PostgreSQL 是多进程模型。每个客户端连接通常对应一个 backend 进程。

这意味着：

```text
应用连接数 = 数据库 backend 数量
```

如果应用有 100 个实例，每个实例开 100 个数据库连接，主库就可能面对 10000 个连接。即使这些连接大部分不忙，也会带来内存、进程调度和锁管理开销。

看当前连接状态：

```sql
SELECT
  state,
  wait_event_type,
  wait_event,
  count(*) AS connections
FROM pg_stat_activity
GROUP BY state, wait_event_type, wait_event
ORDER BY connections DESC;
```

再看连接来自哪里：

```sql
SELECT
  application_name,
  client_addr,
  state,
  count(*) AS connections
FROM pg_stat_activity
GROUP BY application_name, client_addr, state
ORDER BY connections DESC;
```

如果发现大量连接处于 `idle`，说明数据库连接被应用长时间占着，却没有真正执行 SQL。此时盲目调大 `max_connections`，通常只是把问题推迟。

更常见的做法是加连接池：

```text
应用大量短连接
  -> PgBouncer / 应用连接池
  -> PostgreSQL 少量稳定 backend
```

连接池的目标不是让请求变少，而是让 PostgreSQL 看到的并发执行数保持可控。

## 三、PgBouncer：降低连接成本，但不要误用池化模式

PgBouncer 常见三种池化模式：

| 模式 | 连接复用粒度 | 适用场景 | 风险 |
|------|--------------|----------|------|
| Session pooling | 一个客户端会话占用一个服务端连接 | 兼容性最好 | 降低连接数能力有限 |
| Transaction pooling | 一个事务结束就归还服务端连接 | Web 请求、短事务 | 不能依赖会话状态 |
| Statement pooling | 每条语句后归还连接 | 极少数简单场景 | 事务语义容易被破坏 |

生产里常见的是 transaction pooling，但它有一个很重要的边界：

**不要在事务池化模式下依赖连接级状态。**

例如这些东西都要小心：

- session 临时表
- `SET` 后希望一直生效的参数
- advisory lock 的会话级用法
- 没有正确释放的 prepared statement
- 应用误以为同一个请求永远绑定同一个后端连接

更稳妥的原则是：

```text
每个请求拿连接
-> 开短事务
-> 执行业务 SQL
-> 提交或回滚
-> 归还连接
```

连接池不是数据库的"外挂加速器"，它是入口限流器。它让数据库只处理自己能处理的并发量，剩下的压力在应用侧排队、超时或降级。

## 四、短事务：高并发时最怕连接拿着锁发呆

高并发场景里，事务越长，风险越大。

一个常见坏味道是：

```text
BEGIN
-> SELECT ... FOR UPDATE
-> 调第三方支付接口
-> 写支付结果
-> COMMIT
```

问题在于：事务里夹了外部网络调用。只要第三方接口慢，数据库里的行锁就会一直被占住。

更好的结构是把外部调用移出数据库锁范围：

```text
创建支付单
-> COMMIT
-> 调第三方支付接口
-> 回调或轮询结果
-> 开短事务更新订单状态
```

观察长事务：

```sql
SELECT
  pid,
  usename,
  state,
  xact_start,
  now() - xact_start AS xact_age,
  wait_event_type,
  wait_event,
  left(query, 120) AS query
FROM pg_stat_activity
WHERE xact_start IS NOT NULL
ORDER BY xact_start;
```

如果你看到很多 `idle in transaction`，要立刻警惕。它不仅占连接，还可能：

1. 持有锁。
2. 阻碍 VACUUM 清理 dead tuple。
3. 让表和索引继续膨胀。
4. 让后续 DDL 一直等。

高并发系统里，事务应该像短跑，不应该像长途占道。

## 五、热点行：库存扣减为什么会排队

库存扣减是最典型的热点行场景：

```sql
UPDATE inventory
SET stock = stock - 1,
    updated_at = now()
WHERE sku = 'iphone-17'
  AND stock > 0;
```

这条 SQL 在正确性上不错，因为 `stock > 0` 和更新发生在同一条语句里，可以避免先查后改的竞态。

但如果几万请求同时更新同一个 `sku`，它们还是会排队。原因很简单：同一行当前版本只能被一个事务成功修改，其他事务需要等待行锁。

看锁等待：

```sql
SELECT
  blocked.pid AS blocked_pid,
  blocked.query AS blocked_query,
  blocking.pid AS blocking_pid,
  blocking.query AS blocking_query
FROM pg_stat_activity blocked
JOIN pg_locks blocked_locks
  ON blocked_locks.pid = blocked.pid
JOIN pg_locks blocking_locks
  ON blocking_locks.locktype = blocked_locks.locktype
 AND blocking_locks.database IS NOT DISTINCT FROM blocked_locks.database
 AND blocking_locks.relation IS NOT DISTINCT FROM blocked_locks.relation
 AND blocking_locks.page IS NOT DISTINCT FROM blocked_locks.page
 AND blocking_locks.tuple IS NOT DISTINCT FROM blocked_locks.tuple
 AND blocking_locks.transactionid IS NOT DISTINCT FROM blocked_locks.transactionid
 AND blocking_locks.pid <> blocked_locks.pid
JOIN pg_stat_activity blocking
  ON blocking.pid = blocking_locks.pid
WHERE NOT blocked_locks.granted
  AND blocking_locks.granted;
```

热点行治理不是靠给这行再建索引。索引能帮你更快找到这行，但不能让同一行被多人同时修改。

常见方案是把单点写压力拆开：

| 场景 | 治理方式 |
|------|----------|
| 秒杀库存 | Redis 预扣、消息队列串行落库、数据库最终确认 |
| 计数器 | 分片计数，读时汇总或异步汇总 |
| 用户余额 | 按账户串行化，避免跨账户大事务 |
| 热门帖子点赞 | 批量聚合写入，异步刷新计数 |
| 状态机更新 | 幂等状态流转，避免重复请求反复写同一行 |

数据库负责最终一致的事实记录，但不一定要承接每一次前台点击的同步写压力。

## 六、`FOR UPDATE SKIP LOCKED`：让队列消费者别互相堵住

高并发不只有用户请求，还有后台任务。

假设有一张任务表：

```sql
CREATE TABLE jobs (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  status TEXT NOT NULL,
  payload JSONB NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

多个 worker 同时抢任务时，不能让所有 worker 都拿到同一条任务。一个常见写法是：

```sql
WITH picked AS (
  SELECT id
  FROM jobs
  WHERE status = 'READY'
  ORDER BY id
  LIMIT 10
  FOR UPDATE SKIP LOCKED
)
UPDATE jobs
SET status = 'RUNNING'
WHERE id IN (SELECT id FROM picked)
RETURNING *;
```

`SKIP LOCKED` 的意思是：如果某些行已经被别的事务锁住，就跳过它们，去拿还没被锁的行。

它适合队列消费，但不要把它误用成业务查询：

- 适合：后台任务抢占、批处理、可跳过的工作队列。
- 不适合：用户订单查询、余额查询、需要完整结果集的业务查询。

因为它会主动跳过被锁住的行，结果天然不是完整快照。

## 七、读写路径拆分：不是所有读都应该打主库

高并发读多写少时，常见做法是读写分离：

```text
写请求 -> 主库
普通读 -> 只读副本
强一致读 -> 主库
复杂报表 -> 独立分析库或离线链路
```

但是读写分离有一个绕不过去的问题：副本延迟。

用户刚提交订单，立刻查询订单列表，如果读到了延迟的副本，就可能看不到刚写入的数据。

常见处理方式：

| 读类型 | 推荐路径 |
|--------|----------|
| 写后立即读自己的数据 | 短时间读主库 |
| 支付状态、库存确认 | 读主库或等待副本追上 |
| 商品详情、公开列表 | 可读副本 |
| 报表和导出 | 不要打主库，走分析链路 |
| 后台大范围扫描 | 分批、限速、离峰执行 |

读写分离不是把所有 `SELECT` 都丢给从库，而是按一致性要求分层。

## 八、应用重试：高并发事故里的放大器

数据库变慢时，应用通常会超时。超时后如果立刻重试，就可能把一次压力变成三次压力：

```text
数据库慢
-> 应用超时
-> 请求重试
-> 数据库收到更多相同请求
-> 更慢
```

所以高并发系统里，重试必须有边界：

1. 对只读请求可以重试，但要有指数退避和总超时。
2. 对写请求必须有幂等键，避免重复扣款、重复下单。
3. 对热点接口要有排队、限流、熔断和降级。
4. 数据库连接池要设置合理的获取连接超时，不能无限等待。

写请求可以这样设计幂等键：

```sql
CREATE TABLE payment_requests (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  idempotency_key TEXT NOT NULL UNIQUE,
  order_id BIGINT NOT NULL,
  status TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

应用重试时先用 `idempotency_key` 查已有请求，避免重复创建业务事实。

## 九、一个高并发排查顺序

线上已经慢了，不要一上来就改参数。可以按这个顺序排：

1. **看连接**：是不是连接池打满，数据库 backend 过多？
2. **看等待**：慢在 CPU、IO，还是锁等待？
3. **看长事务**：有没有 `idle in transaction` 或超长事务？
4. **看热点 SQL**：是不是少数 SQL 占住大部分时间？
5. **看热点行**：是不是大量请求更新同一行？
6. **看副本延迟**：读写分离是否读到了旧数据？
7. **看重试风暴**：应用是否在数据库慢时继续放大压力？

几个常用入口：

```sql
-- 当前活动和等待
SELECT
  pid,
  state,
  wait_event_type,
  wait_event,
  now() - query_start AS query_age,
  left(query, 120) AS query
FROM pg_stat_activity
WHERE state <> 'idle'
ORDER BY query_start;

-- 每个数据库的连接和事务概况
SELECT
  datname,
  numbackends,
  xact_commit,
  xact_rollback,
  blks_read,
  blks_hit,
  temp_files
FROM pg_stat_database;

-- 表级 dead tuple 和 autovacuum 状态
SELECT
  relname,
  n_live_tup,
  n_dead_tup,
  last_autovacuum,
  last_autoanalyze
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC
LIMIT 20;
```

真正有效的高并发治理，往往不是一个参数，而是一组边界：

```text
连接有上限
事务足够短
热点被拆散
读写有分层
重试有节制
后台任务可削峰
```

## 十、一分钟复习

1. PostgreSQL 高并发问题常常从连接数开始放大，因为一个连接通常对应一个 backend 进程。
2. PgBouncer 能降低连接成本，但 transaction pooling 要避免依赖会话级状态。
3. 长事务会同时带来锁等待、连接占用和 VACUUM 清理受阻。
4. 热点行不是靠索引解决，而是靠分片、队列、预扣、聚合和幂等拆压力。
5. `FOR UPDATE SKIP LOCKED` 适合队列抢任务，不适合需要完整结果的业务查询。
6. 读写分离要按一致性要求分层，写后读和关键读不要随便打副本。
7. 超时重试必须有退避和幂等，否则会把数据库慢变成重试风暴。

**高并发治理的核心不是让 PostgreSQL 无限并发，而是让它只处理可控、短小、可恢复的并发。**
