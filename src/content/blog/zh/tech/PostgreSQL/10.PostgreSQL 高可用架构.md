# PostgreSQL 高可用架构：复制、故障转移和备份恢复怎么配合

单机 PostgreSQL 跑得再稳，也绕不开几个问题：

1. 主库机器挂了怎么办？
2. 数据盘坏了怎么办？
3. 应用读流量太大怎么办？
4. 主库误删数据怎么办？
5. 切主以后客户端怎么连到新主？

这些问题经常被一句"做主从"带过。但 PostgreSQL 高可用不是只有主从复制。

更准确地说，它是一组能力叠加：

```text
流复制
-> 只读副本
-> 故障检测
-> 主备切换
-> 客户端路由
-> 备份和时间点恢复
-> 演练和监控
```

这篇文章只回答一个问题：

**PostgreSQL 高可用不是有一个从库就够了，而是要同时处理 RTO、RPO、读写路由、复制延迟和恢复演练。**

## 一、先区分几个目标

高可用讨论里最容易混在一起的词是：

| 目标 | 解决什么问题 | 典型手段 |
|------|--------------|----------|
| 数据冗余 | 单机坏了还有副本 | 流复制、备份 |
| 自动切主 | 主库挂了快速恢复服务 | Patroni、repmgr、云数据库 HA |
| 读扩展 | 主库读压力太大 | 只读副本、读写分离 |
| 灾难恢复 | 整个机房或误操作出问题 | 异地备份、PITR |
| 数据不丢 | 主库宕机时少丢或不丢提交 | 同步复制、业务幂等 |

这里有两个关键指标：

- **RTO**：恢复服务需要多久。
- **RPO**：最多能丢多少数据。

比如：

```text
RTO = 30 秒
RPO = 0
```

这意味着主库故障后 30 秒内恢复，并且已提交事务不能丢。这个要求比"有一个从库"高得多。

## 二、PostgreSQL 复制的底层主线还是 WAL

前面日志篇讲过，PostgreSQL 的 WAL 不只服务本机崩溃恢复，也支撑复制。

物理流复制大致是：

```text
Primary 产生 WAL
-> walsender 发送 WAL
-> Standby walreceiver 接收 WAL
-> Standby replay WAL
-> Standby 追上主库状态
```

主库查看复制状态：

```sql
SELECT
  pid,
  application_name,
  client_addr,
  state,
  sync_state,
  sent_lsn,
  write_lsn,
  flush_lsn,
  replay_lsn,
  write_lag,
  flush_lag,
  replay_lag
FROM pg_stat_replication;
```

几个 LSN 可以这样理解：

| 字段 | 含义 |
|------|------|
| `sent_lsn` | 主库已经发送到哪里 |
| `write_lsn` | 从库已经写入操作系统缓存到哪里 |
| `flush_lsn` | 从库已经刷到磁盘到哪里 |
| `replay_lsn` | 从库已经重放到哪里 |

如果 `replay_lsn` 落后很多，说明从库虽然收到了 WAL，但还没重放完。读这个副本就可能读到旧数据。

## 三、异步复制：性能好，但故障时可能丢最后一段

PostgreSQL 默认常见配置是异步复制。

主库提交事务时，只要本机提交条件满足，就可以返回成功；从库稍后再接收和重放 WAL。

```text
应用写主库
-> 主库提交成功
-> 返回客户端
-> WAL 异步发送给从库
```

异步复制的优点是写入延迟低。从库慢、网络抖动时，主库不用等它。

代价是：主库突然宕机时，最后一小段还没发到从库或还没落盘的 WAL 可能丢失。

所以异步复制适合：

- 能接受极小概率丢最后几秒写入的业务。
- 可以通过消息、对账、幂等重放修复的业务。
- 对写入延迟敏感的业务。

如果业务要求严格 RPO=0，就不能只靠异步复制。

## 四、同步复制：降低丢数据风险，但会增加写延迟

同步复制要求主库提交时等待同步 standby 确认。

常见理解：

```text
应用写主库
-> 主库写 WAL
-> 等待同步从库确认
-> 返回客户端
```

同步复制能降低主库故障时的数据丢失风险，但它把网络和从库状态纳入写入延迟。

如果同步从库很慢，主库提交也会慢。

如果同步从库不可用，而配置又要求必须等待它，写入可能被卡住。

所以同步复制不是"开了就更高级"，而是一个取舍：

| 维度 | 异步复制 | 同步复制 |
|------|----------|----------|
| 写入延迟 | 低 | 更高 |
| 主库受从库影响 | 小 | 大 |
| 故障丢数据风险 | 更高 | 更低 |
| 运维要求 | 较低 | 较高 |
| 适合场景 | 大多数普通业务 | 金融、强一致关键写入 |

实践里常见做法是：

1. 核心交易链路使用同步复制或云数据库强一致高可用能力。
2. 普通业务使用异步复制，并用对账和幂等兜底。
3. 跨地域复制通常不做强同步，否则延迟太高。

## 五、只读副本：读扩展不是免费的

只读副本可以承担读请求：

```text
写请求 -> Primary
读请求 -> Standby
```

这能缓解主库读压力，但引入副本延迟。

一个常见事故是：

```text
用户提交订单
-> 主库写成功
-> 页面立刻查订单列表
-> 读请求打到延迟副本
-> 用户看不到刚下的订单
```

解决方式不是简单关掉读写分离，而是把读分类：

| 读类型 | 推荐策略 |
|--------|----------|
| 写后立即读 | 短时间读主库 |
| 支付、库存、权限 | 读主库或等待副本追上 |
| 商品详情、公开列表 | 可读副本 |
| 报表和导出 | 专门副本或分析库 |
| 后台扫描 | 限速，避免拖慢 replay |

从库上跑长查询还可能阻碍 WAL replay。为了服务长报表，把从库 replay 延迟放大，也会让这个副本不适合承接强一致读。

所以副本也要分角色：

```text
HA Standby：尽量追主，准备随时接管
Read Replica：承担读请求，可接受轻微延迟
Analytics Replica：承接长查询，不参与快速切主
```

## 六、故障转移：PostgreSQL 自身不会自动替你完成全部 HA

PostgreSQL 支持 standby 被提升为 primary：

```text
Primary 故障
-> 监控确认故障
-> 选择候选 Standby
-> promote 成新 Primary
-> 其他 Standby 改为跟随新 Primary
-> 应用连接切到新 Primary
```

但完整故障转移需要额外系统协调：

- 谁判断主库真的挂了？
- 谁防止两个节点同时认为自己是主库？
- 谁执行 promote？
- 谁更新 VIP、DNS、服务发现或代理？
- 旧主恢复后怎么处理？
- 客户端连接怎么重建？

常见方案包括：

| 方案 | 说明 |
|------|------|
| Patroni + etcd/Consul | 常见开源 HA 组合，依赖分布式一致性存储做 leader 选举 |
| repmgr | PostgreSQL 复制管理工具，支持 failover 管理 |
| Pgpool-II | 可做连接池、负载均衡和部分 HA 能力 |
| 云数据库 HA | 托管服务封装了复制、检测、切主和连接入口 |
| 自研脚本 | 只适合非常明确且演练充分的场景 |

最危险的是"半自动"：脚本能切主，但没人确认脑裂、DNS 缓存、旧主回归和应用连接重建。

## 七、备份和 PITR：复制不是备份

很多人以为有从库就不需要备份。

这是危险误解。

复制会同步正确数据，也会同步错误操作：

```sql
DROP TABLE orders;
```

如果这条语句在主库执行，从库也会跟着删除。只读副本不能帮你回到误删前的状态。

所以 PostgreSQL 高可用还需要备份和时间点恢复：

```text
基础备份 base backup
  + 连续 WAL 归档
  -> 恢复到指定时间点
```

常见能力叫 PITR（Point-in-Time Recovery）。

它解决的是：

- 误删表后恢复到误删前。
- 错误批量更新后恢复到某个时间点。
- 主从都不可用时从备份重建。
- 做灾难恢复演练。

检查归档和复制槽时，可以看：

```sql
SELECT
  slot_name,
  slot_type,
  active,
  restart_lsn,
  pg_size_pretty(
    pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)
  ) AS retained_wal
FROM pg_replication_slots;
```

如果归档失败或复制槽长期落后，WAL 可能无法回收，磁盘会被撑满。高可用链路本身也会成为故障源。

## 八、监控：高可用要盯住哪些指标

一套 PostgreSQL HA 至少要监控：

| 类别 | 指标 |
|------|------|
| 主库状态 | 是否可写、连接数、锁等待、WAL 生成量 |
| 复制状态 | `replay_lag`、LSN 差距、复制连接是否断开 |
| 从库状态 | 是否只读、replay 是否暂停、长查询是否阻碍 replay |
| WAL | 归档是否失败、复制槽保留量、磁盘剩余空间 |
| 备份 | 最近一次备份时间、备份是否可恢复 |
| 切主 | leader 状态、VIP/DNS/代理入口状态 |
| 客户端 | 连接错误率、重连耗时、写入失败率 |

常用观察：

```sql
-- 当前是否处于恢复状态；从库返回 true，主库返回 false
SELECT pg_is_in_recovery();

-- 从库 replay 到哪里
SELECT
  pg_last_wal_receive_lsn(),
  pg_last_wal_replay_lsn(),
  now() - pg_last_xact_replay_timestamp() AS replay_delay;
```

不要只监控"数据库进程是否活着"。高可用真正关心的是：它是否还能接受正确写入、从库是否追得上、备份是否真能恢复。

## 九、一个常见架构演进

小系统：

```text
App -> Primary PostgreSQL
```

有数据冗余需求：

```text
App -> Primary
          |
          +-> Standby
```

有读扩展需求：

```text
Write -> Primary
Read  -> Read Replica
HA    -> Standby
```

有自动切主需求：

```text
App -> Proxy / VIP / Service Discovery
              |
              v
        HA Controller
              |
        Primary + Standby
```

有灾备需求：

```text
Primary
  -> Local Standby
  -> Remote Standby
  -> Base Backup + WAL Archive
```

每一层都解决一个问题，也带来新的运维复杂度。不要为了"架构看起来完整"提前引入所有层。先明确业务的 RTO、RPO、读一致性和运维能力。

## 十、一分钟复习

1. PostgreSQL 高可用要同时看 RTO 和 RPO，不能只说"有主从"。
2. 物理流复制基于 WAL，主库发送 WAL，从库接收并重放。
3. 异步复制写入延迟低，但主库故障时可能丢最后一段 WAL。
4. 同步复制降低丢数据风险，但会把从库和网络纳入写入延迟。
5. 读写分离要按读一致性分层，写后读和关键读不能随便打延迟副本。
6. PostgreSQL 的完整自动故障转移需要外部 HA 控制器、路由入口和防脑裂机制。
7. 复制不是备份，误删会被复制；PITR 才能把数据库恢复到某个历史时间点。
8. 高可用必须定期演练，否则只是配置文件里的安全感。

**高可用不是某一个组件，而是一条从 WAL、复制、切主、路由到备份恢复的完整链路。**
