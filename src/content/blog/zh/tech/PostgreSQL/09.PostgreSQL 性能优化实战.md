# PostgreSQL 性能优化实战：慢 SQL、统计信息和 VACUUM 怎么串起来

学 PostgreSQL 性能优化，最容易陷入两个误区：

1. 一看到慢就加索引。
2. 一看到参数就调 `shared_buffers`、`work_mem`。

这两件事都可能有用，但都不应该是第一步。

性能优化真正要回答的是：

**慢在哪里？为什么优化器选了这条路？数据形状和系统状态是否已经变了？**

这篇文章不做参数大全，而是按线上排障顺序来讲：

```text
先定位慢 SQL
-> 看执行计划
-> 校验统计信息
-> 判断索引和 SQL 写法
-> 检查表膨胀和 VACUUM
-> 观察 WAL、IO、内存和临时文件
```

继续用订单表：

```sql
CREATE TABLE orders (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  user_id BIGINT NOT NULL,
  status TEXT NOT NULL,
  amount NUMERIC(10, 2) NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  paid_at TIMESTAMPTZ
);
```

业务经常查某个用户最近已支付订单：

```sql
SELECT id, amount, paid_at
FROM orders
WHERE user_id = 10086
  AND status = 'PAID'
ORDER BY paid_at DESC
LIMIT 20;
```

这条 SQL 以前很快，某天突然变慢。我们就从这个问题开始。

## 一、先找 Top SQL，不要靠感觉优化

如果系统已经接入 `pg_stat_statements`，优先看累计耗时和平均耗时：

```sql
SELECT
  calls,
  total_exec_time,
  mean_exec_time,
  rows,
  shared_blks_hit,
  shared_blks_read,
  temp_blks_written,
  left(query, 160) AS query
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 20;
```

这里要区分两类 SQL：

| 类型 | 特征 | 优先级 |
|------|------|--------|
| 总耗时高 | 单次不一定慢，但调用次数极多 | 影响整体资源 |
| 平均耗时高 | 单次很慢，可能拖垮连接池 | 影响尾延迟 |
| 读块多 | `shared_blks_read` 高 | 可能 IO 压力大 |
| 临时块多 | `temp_blks_written` 高 | 可能排序/哈希溢出 |
| 返回行多 | `rows` 高 | 可能接口分页或过滤不足 |

如果没有 `pg_stat_statements`，也可以先从当前活动查询入手：

```sql
SELECT
  pid,
  state,
  wait_event_type,
  wait_event,
  now() - query_start AS query_age,
  left(query, 200) AS query
FROM pg_stat_activity
WHERE state <> 'idle'
ORDER BY query_start;
```

优化的第一原则是：

**先优化真实消耗资源最多的 SQL，而不是优化你最熟悉的 SQL。**

## 二、用 `EXPLAIN ANALYZE` 看估算和真实执行差距

找到 SQL 后，再看执行计划。

普通 `EXPLAIN` 只展示计划，不执行：

```sql
EXPLAIN
SELECT id, amount, paid_at
FROM orders
WHERE user_id = 10086
  AND status = 'PAID'
ORDER BY paid_at DESC
LIMIT 20;
```

排查慢 SQL 时更常用：

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT id, amount, paid_at
FROM orders
WHERE user_id = 10086
  AND status = 'PAID'
ORDER BY paid_at DESC
LIMIT 20;
```

`ANALYZE` 会真实执行 SQL，所以生产环境要谨慎使用。对写语句可以放在事务里执行后回滚，或者在低峰、影子库、只读副本上验证。

看计划时，重点不是背节点名，而是问几个问题：

1. 估算行数 `rows` 和实际行数 `actual rows` 差多少？
2. 是否扫描了远超返回结果的数据？
3. 是否发生了排序、哈希、临时文件？
4. 是否有大量 `shared read` 而不是 `shared hit`？
5. 最慢的时间花在哪个节点？

一个典型慢计划可能是：

```text
Limit
  -> Sort
       Sort Key: paid_at DESC
       -> Seq Scan on orders
            Filter: user_id = 10086 AND status = 'PAID'
```

如果订单表有几千万行，这个计划就很危险：它先扫大量数据，再排序，最后才取 20 条。

## 三、统计信息不准，优化器就会走错路

PostgreSQL 优化器依赖统计信息估算成本。统计信息不准时，它可能以为某个条件能过滤很多行，实际却过滤不了；也可能以为全表扫更便宜，实际应该走索引。

手动刷新统计信息：

```sql
ANALYZE orders;
```

查看表级统计：

```sql
SELECT
  relname,
  n_live_tup,
  n_dead_tup,
  last_analyze,
  last_autoanalyze
FROM pg_stat_user_tables
WHERE relname = 'orders';
```

如果某列分布很偏，比如大部分订单是 `PAID`，少量是 `REFUNDED`，默认统计信息可能不足以描述真实分布。可以提高统计目标：

```sql
ALTER TABLE orders ALTER COLUMN status SET STATISTICS 1000;
ANALYZE orders;
```

如果多个列之间强相关，比如 `status = 'PAID'` 时 `paid_at` 才非空，单列统计也可能估错。可以考虑扩展统计信息：

```sql
CREATE STATISTICS orders_status_paid_at_stats
ON status, paid_at
FROM orders;

ANALYZE orders;
```

记住一句话：

**优化器不是神，它是拿统计信息做成本估算。统计信息错了，执行计划就可能错。**

## 四、索引优化：让查询少看数据，也少排序

对于刚才的查询，合适的索引可能是：

```sql
CREATE INDEX CONCURRENTLY idx_orders_user_status_paid_at
ON orders (user_id, status, paid_at DESC);
```

这个索引同时回答三个问题：

1. `user_id` 缩小用户范围。
2. `status` 过滤订单状态。
3. `paid_at DESC` 支持按支付时间倒序拿最近记录。

如果返回列只有 `id`、`amount`、`paid_at`，还可以考虑覆盖更多字段：

```sql
CREATE INDEX CONCURRENTLY idx_orders_user_status_paid_at_include
ON orders (user_id, status, paid_at DESC)
INCLUDE (amount);
```

但索引不是越多越好。

| 索引收益 | 索引成本 |
|----------|----------|
| 查询少扫数据 | INSERT/UPDATE/DELETE 要维护索引 |
| 可以避免排序 | 占用磁盘和缓存 |
| 可以支持唯一约束 | 增加 WAL 写入 |
| 可以让查询更稳定 | 过多索引会干扰优化器选择 |

删除不用的索引前，要先观察：

```sql
SELECT
  relname AS table_name,
  indexrelname AS index_name,
  idx_scan,
  idx_tup_read,
  idx_tup_fetch
FROM pg_stat_user_indexes
WHERE relname = 'orders'
ORDER BY idx_scan;
```

`idx_scan = 0` 不代表一定能删。可能是统计重置后没覆盖完整业务周期，也可能是某些低频但关键的后台任务在用。

## 五、SQL 写法也会让索引失效或变贵

索引存在，不代表一定能用得好。

常见问题：

### 1. 对索引列做函数

```sql
-- 不友好
WHERE date(created_at) = DATE '2026-06-19'

-- 更友好
WHERE created_at >= TIMESTAMPTZ '2026-06-19 00:00:00+08'
  AND created_at <  TIMESTAMPTZ '2026-06-20 00:00:00+08'
```

### 2. 返回太多行

```sql
SELECT *
FROM orders
WHERE status = 'PAID';
```

如果 `PAID` 占大多数，走索引再回表可能不如顺序扫描便宜。

### 3. 排序和过滤没对齐

```sql
WHERE user_id = 10086
ORDER BY paid_at DESC
LIMIT 20;
```

如果索引只有 `(user_id)`，还需要排序。索引 `(user_id, paid_at DESC)` 更贴合这个查询形状。

### 4. 分页越翻越慢

```sql
SELECT *
FROM orders
ORDER BY id
OFFSET 1000000
LIMIT 20;
```

深分页会跳过大量行。更适合用游标式分页：

```sql
SELECT *
FROM orders
WHERE id > 1000000
ORDER BY id
LIMIT 20;
```

性能优化不是"SQL 写短一点"，而是让 SQL 的过滤、排序、分页方式贴近索引和数据分布。

## 六、VACUUM：PostgreSQL 性能优化里绕不开的一层

PostgreSQL 的 MVCC 会留下 dead tuple。UPDATE 不是原地覆盖，而是写新版本，旧版本等待 VACUUM 清理。

如果 VACUUM 跟不上，会出现：

1. 表膨胀，扫描更多 page。
2. 索引膨胀，缓存命中率下降。
3. autovacuum 更频繁但效果变差。
4. 查询计划估算偏差变大。

观察 dead tuple：

```sql
SELECT
  relname,
  n_live_tup,
  n_dead_tup,
  vacuum_count,
  autovacuum_count,
  last_vacuum,
  last_autovacuum
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC
LIMIT 20;
```

长事务会阻碍清理：

```sql
SELECT
  pid,
  state,
  xact_start,
  now() - xact_start AS xact_age,
  left(query, 120) AS query
FROM pg_stat_activity
WHERE xact_start IS NOT NULL
ORDER BY xact_start;
```

高更新表通常要单独调 autovacuum 参数，例如：

```sql
ALTER TABLE orders SET (
  autovacuum_vacuum_scale_factor = 0.02,
  autovacuum_analyze_scale_factor = 0.01,
  autovacuum_vacuum_cost_limit = 2000
);
```

这不是固定模板。真正的参数要根据表大小、更新频率、IO 能力和业务低峰来调。

一句话：

**PostgreSQL 的性能优化不能只看读路径，还要看 MVCC 留下的垃圾有没有被及时收走。**

## 七、`work_mem`：别让排序和哈希偷偷落盘

查询里出现 `Sort`、`Hash Join`、`Hash Aggregate` 时，会用到 `work_mem`。

如果内存不够，就会写临时文件：

```sql
SELECT
  datname,
  temp_files,
  pg_size_pretty(temp_bytes) AS temp_size
FROM pg_stat_database
ORDER BY temp_bytes DESC;
```

执行计划里也能看到：

```text
Sort Method: external merge  Disk: 204800kB
```

这说明排序写到了磁盘。

但 `work_mem` 不能简单调很大，因为它不是全局只用一份，而是每个连接、每个排序/哈希节点都可能用。

粗略理解：

```text
最大内存风险 ≈ 并发查询数 × 每个查询的排序/哈希节点数 × work_mem
```

所以更稳的做法是：

1. 优先通过索引避免大排序。
2. 控制并发查询数量。
3. 对报表任务单独设置较大的 `work_mem`。
4. 不要为了一个 SQL 把全局 `work_mem` 调到离谱。

例如只在当前会话调：

```sql
SET work_mem = '128MB';
```

## 八、WAL 和 checkpoint：写入抖动常常在这里暴露

写入压力大时，除了表和索引，还要观察 WAL。

看 WAL 统计：

```sql
SELECT
  wal_records,
  wal_fpi,
  pg_size_pretty(wal_bytes) AS wal_size,
  wal_buffers_full,
  stats_reset
FROM pg_stat_wal;
```

看 checkpoint：

```sql
SELECT
  num_timed,
  num_requested,
  write_time,
  sync_time,
  buffers_written
FROM pg_stat_checkpointer;
```

如果 `num_requested` 很高，说明 checkpoint 经常被迫触发，可能是 WAL 增长太快或配置太紧。

常见写入抖动来源：

| 来源 | 现象 | 方向 |
|------|------|------|
| 批量 UPDATE | WAL 暴涨，复制延迟上升 | 分批、限速、离峰 |
| 频繁 checkpoint | 写入周期性抖动 | 调整 checkpoint 相关参数 |
| 索引过多 | 写入和 WAL 都增加 | 清理无用索引 |
| 随机写太散 | cache 命中低、IO 高 | 优化主键和写入模式 |
| 复制槽落后 | WAL 无法回收 | 监控 slot retained WAL |

看复制槽保留 WAL：

```sql
SELECT
  slot_name,
  slot_type,
  active,
  pg_size_pretty(
    pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)
  ) AS retained_wal
FROM pg_replication_slots;
```

性能优化里，WAL 是写入链路的温度计。WAL 异常增长，通常说明某些写入、DDL、复制或 CDC 链路正在制造压力。

## 九、缓存命中率不是唯一答案

很多监控会展示 buffer cache hit ratio。它有用，但不要迷信。

看数据库级命中：

```sql
SELECT
  datname,
  blks_hit,
  blks_read,
  round(
    blks_hit * 100.0 / nullif(blks_hit + blks_read, 0),
    2
  ) AS hit_ratio
FROM pg_stat_database;
```

命中率高不代表没有问题：

- 热点小表可能命中率很高，但锁等待严重。
- 全表扫描可能走 OS cache，不一定都表现成数据库 read。
- 查询返回太多行，即使命中内存也会消耗 CPU 和网络。
- 表膨胀会让"命中"命中到很多没价值的 page。

性能优化应该把缓存指标和执行计划、锁等待、IO、WAL、业务延迟一起看。

## 十、一个实战优化 Checklist

遇到 PostgreSQL 慢，按这个顺序走：

- [ ] 找 Top SQL：按 `total_exec_time` 和 `mean_exec_time` 排序。
- [ ] 看当前等待：区分 CPU、IO、锁、客户端等待。
- [ ] 跑 `EXPLAIN (ANALYZE, BUFFERS)`：确认慢在哪个节点。
- [ ] 对比估算行数和实际行数：判断统计信息是否失真。
- [ ] 检查索引是否贴合过滤、排序和分页。
- [ ] 检查 SQL 是否破坏索引或返回太多行。
- [ ] 看 dead tuple 和 autovacuum：确认表膨胀是否影响读写。
- [ ] 看临时文件：判断排序/哈希是否落盘。
- [ ] 看 WAL、checkpoint、复制槽：确认写入链路是否抖动。
- [ ] 看连接池和慢查询占用：确认是不是连接被耗尽。

## 十一、一分钟复习

1. PostgreSQL 性能优化第一步是找真实 Top SQL，不是凭感觉加索引。
2. `EXPLAIN (ANALYZE, BUFFERS)` 要重点看估算和实际行数差距、最慢节点和 buffer 使用。
3. 统计信息不准会让优化器选错计划，必要时提高统计目标或使用扩展统计。
4. 索引要服务过滤、排序、分页和返回列，但索引越多写入成本越高。
5. VACUUM 跟不上会造成表和索引膨胀，影响扫描、缓存和计划估算。
6. `work_mem` 要结合并发和节点数量看，不能只为一个慢 SQL 全局调大。
7. WAL、checkpoint、复制槽和临时文件能暴露写入链路、IO 和后台任务压力。

**性能优化的本质不是调一个神奇参数，而是把执行计划、数据分布、MVCC 清理和系统资源连成一条因果链。**
