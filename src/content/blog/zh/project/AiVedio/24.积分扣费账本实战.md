# 第 24 章：积分扣费账本实战

> 本章定位：第 20 章已经从平台级视角讲过计费、成本和对账，本章继续把镜头拉近，专门回答一个更落地的问题：**AI 视频生成平台里的积分扣费系统到底怎么建表、怎么扣、怎么防重复、怎么对账。**

很多人设计积分系统时，会下意识把它当成一张普通业务表：

```text
user_id
balance
```

然后在任务成功时执行：

```sql
update user_account
set balance = balance - 100
where user_id = ?
```

这个写法在小型同步业务里可能暂时能跑，但放到 AiVedio 这种长耗时、高成本、强异步的视频生成平台里，很快就会出问题：

- 用户重复点击，可能重复扣积分。
- 前端超时重试，可能重复创建任务。
- RocketMQ 重复投递，可能重复执行结算。
- 供应商已经受理但本地超时，系统可能误释放后再次提交。
- 任务失败、取消、部分成功、供应商仍收费时，退款语义会变乱。
- 出现账务差异后，很难解释“这 100 积分到底为什么少了”。

所以积分系统不要当普通 CRUD 设计，而要当成一个**准金融账本系统**设计。

本章的核心原则可以先记一句：

> **账户余额是快照，积分流水是事实，幂等操作是防线，对账任务是兜底。**

## 24.1 AiVedio 里的积分扣费到底在保护什么

AI 视频平台里，一次用户消费并不是一个瞬间完成的动作，而是一条跨系统链路：

```text
用户提交生成
-> 服务端报价
-> 积分预占
-> 创建 generation_task
-> RocketMQ 调度
-> Provider Adapter 提交供应商
-> 回调或轮询确认结果
-> 输出回源与后处理
-> 成功结算或失败释放
-> 日终对账
```

这条链路里有三类事实不能混在一起：

| 事实链 | 关注点 | 典型数据 |
|---|---|---|
| 用户积分链 | 用户可用额度、预占额度、消费、退款、补偿 | `wallet_account`、`credit_reservation`、`ledger_transaction` |
| 任务执行链 | 任务是否创建、是否提交供应商、是否完成 | `generation_task`、`provider_attempt`、`task_event` |
| 成本对账链 | 供应商是否收费、平台成本是否入账、外部账单是否匹配 | `provider_cost_event`、`invoice_line`、`reconciliation_item` |

积分扣费系统要保护的不是一个 `balance` 字段，而是这些不变量：

1. 同一个真实消费动作只能预占一次。
2. 同一个任务成功只能结算一次。
3. 同一个失败分支只能释放一次。
4. 余额变化必须能由账本流水解释。
5. 账本流水必须能关联到任务、预占、价格版本和供应商尝试。
6. 供应商状态未知时，不能把用户额度释放后盲目重提。
7. 对账发现差异后，只能追加修正事实，不能覆盖历史流水。

如果面试官问“积分扣费系统怎么做”，不要只回答“用 Redis 防重复”。更好的回答是：

```text
服务端报价
-> PostgreSQL 事务内预占积分、建任务、写 Outbox
-> RocketMQ 异步执行
-> 成功后按稳定业务键结算
-> 失败或取消后按状态机释放、退款或补偿
-> 用不可变账本和日终对账兜底
```

## 24.2 两种账户快照模型

设计积分账户时，常见有两种模型。

### 24.2.1 `balance + frozen_balance` 模型

这种模型适合讲解，也适合很多简单积分系统：

```sql
point_account
- user_id
- balance
- frozen_balance
- version
- updated_at
```

语义是：

```text
可用积分 = balance - frozen_balance
```

各动作的变化如下：

| 动作 | `balance` | `frozen_balance` | 含义 |
|---|---:|---:|---|
| 充值 | `+1000` | 不变 | 用户总积分增加 |
| 预占 | 不变 | `+120` | 可用减少，但还没真正消费 |
| 成功确认 | `-120` | `-120` | 冻结积分转为真实消费 |
| 失败释放 | 不变 | `-120` | 冻结额度回到可用 |
| 已结算退款 | `+120` | 不变 | 追加返还事实 |

这种写法的优点是直观。缺点是当系统继续扩展出促销额度、订阅额度、企业后付费、成本中心和多币种时，`balance` 的含义会越来越重。

### 24.2.2 `available_amount + reserved_amount` 模型

AiVedio 第 20 章使用的是更接近账本分录的模型：

```sql
wallet_account
- wallet_id
- tenant_id
- account_type
- unit_code
- available_amount
- reserved_amount
- version
- status
- updated_at
```

各动作的变化如下：

| 动作 | `available_amount` | `reserved_amount` | 账本含义 |
|---|---:|---:|---|
| 充值 | `+1000` | 不变 | `AVAILABLE +1000` |
| 预占 | `-120` | `+120` | `AVAILABLE -120`，`RESERVED +120` |
| 成功结算 | 不变 | `-120` | `RESERVED -120`，`CONSUMED +120` |
| 失败释放 | `+120` | `-120` | `RESERVED -120`，`AVAILABLE +120` |
| 已结算退款 | `+120` | 不变 | `CONSUMED -120`，`AVAILABLE +120` |

这两种模型本质上都能做预占和结算。差别在于：

- `balance + frozen_balance` 更像一个用户钱包快照。
- `available + reserved` 更像账本账户分录在快照层的投影。

在 AiVedio 里，推荐使用 `wallet_account + ledger_transaction + ledger_entry`。这样后续增加订阅额度、企业额度池、促销额度批次和成本中心时，不需要推翻原来的语义。

## 24.3 核心数据模型

本节只列积分扣费最小闭环。完整平台还会有 `generation_task`、`provider_attempt`、`generation_charge`、`provider_cost_event` 和 `reconciliation_item` 等表。

### 24.3.1 账户快照：`wallet_account`

账户快照用于高频读取和并发控制，不是历史真相本身。

```sql
create table wallet_account (
    wallet_id          uuid primary key,
    tenant_id          uuid not null,
    owner_user_id      uuid null,
    account_type       varchar(32) not null, -- PREPAID / SUBSCRIPTION / POSTPAID
    unit_code          varchar(32) not null, -- CREDIT
    available_amount   bigint not null default 0,
    reserved_amount    bigint not null default 0,
    version            bigint not null default 0,
    status             varchar(32) not null,
    created_at         timestamptz not null,
    updated_at         timestamptz not null,

    unique (tenant_id, owner_user_id, account_type, unit_code),
    check (available_amount >= 0),
    check (reserved_amount >= 0)
);
```

对预付费钱包来说，`available_amount` 不能扣成负数。企业后付费可以增加 `credit_limit_amount`，但也应该由预算和账期控制，而不是随意让余额为负。

### 24.3.2 幂等操作表：`billing_operation`

`billing_operation` 是所有加积分、冻结、结算、释放和退款的入口防线。

```sql
create table billing_operation (
    operation_id       varchar(160) primary key,
    wallet_id          uuid not null,
    tenant_id          uuid not null,
    biz_type           varchar(64) not null,
    biz_id             varchar(128) not null,
    action_id          varchar(128) not null,
    action_type        varchar(32) not null,
    request_hash       varchar(128) not null,
    amount             bigint not null,
    status             varchar(32) not null, -- PROCESSING / SUCCESS / FAILED
    result_json        jsonb not null default '{}'::jsonb,
    created_at         timestamptz not null,
    updated_at         timestamptz not null,

    unique (wallet_id, biz_type, biz_id, action_id, action_type)
);
```

它解决三个问题：

| 场景 | 处理方式 |
|---|---|
| 用户重复点击 | 同一个业务动作命中同一个 `operation_id`，返回第一次结果 |
| HTTP 超时重试 | `request_hash` 相同则复用结果，不重复扣费 |
| 非法重放 | 同一个幂等键但请求参数变了，返回 409 |

`request_hash` 很重要。它能区分“同一次请求的重试”和“拿着旧幂等键请求了新金额”。

### 24.3.3 预占表：`credit_reservation`

预占表把一次用户消费动作和冻结额度绑定起来。

```sql
create table credit_reservation (
    reservation_id     uuid primary key,
    wallet_id          uuid not null,
    tenant_id          uuid not null,
    task_id            uuid not null,
    consume_action_id  varchar(128) not null,
    quoted_amount      bigint not null,
    reserved_amount    bigint not null,
    settled_amount     bigint not null default 0,
    released_amount    bigint not null default 0,
    state              varchar(32) not null,
    price_version_id   uuid not null,
    expires_at         timestamptz null,
    version            bigint not null default 0,
    created_at         timestamptz not null,
    updated_at         timestamptz not null,

    unique (task_id),
    unique (wallet_id, consume_action_id),
    check (reserved_amount >= 0),
    check (settled_amount >= 0),
    check (released_amount >= 0),
    check (settled_amount + released_amount <= reserved_amount)
);
```

推荐状态：

```text
ACTIVE
PARTIALLY_SETTLED
SETTLED
RELEASED
PARTIALLY_REFUNDED
REFUNDED
```

注意，`expires_at` 不能被理解为“时间到了就自动释放”。如果任务已经提交给供应商，尤其处于 `SUBMITTED`、`RUNNING` 或 `SUBMIT_UNKNOWN`，预占到期也不能直接释放。因为外部世界可能已经产生费用。

### 24.3.4 不可变账本：`ledger_transaction` 和 `ledger_entry`

账本不要只记录一行 `amount = -120`。更好的做法是把一次业务事务和多条分录分开。

```sql
create table ledger_transaction (
    ledger_txn_id      uuid primary key,
    wallet_id          uuid not null,
    tenant_id          uuid not null,
    business_key       varchar(160) not null,
    event_type         varchar(32) not null,
    task_id            uuid null,
    reservation_id     uuid null,
    reversal_of        uuid null,
    reason_code        varchar(64) null,
    metadata           jsonb not null default '{}'::jsonb,
    created_at         timestamptz not null,

    unique (wallet_id, business_key)
);
```

```sql
create table ledger_entry (
    entry_id           uuid primary key,
    ledger_txn_id      uuid not null references ledger_transaction(ledger_txn_id),
    bucket             varchar(32) not null, -- AVAILABLE / RESERVED / CONSUMED
    amount_delta       bigint not null,
    unit_code          varchar(32) not null,
    created_at         timestamptz not null
);
```

概念上的分录如下：

```text
RESERVE 120:
  AVAILABLE -120
  RESERVED  +120

SETTLE 100:
  RESERVED  -100
  CONSUMED  +100

RELEASE 20:
  RESERVED  -20
  AVAILABLE +20

REFUND 30:
  CONSUMED  -30
  AVAILABLE +30
```

这里的关键是：账本只追加，不修改历史。业务修正应该新增一笔反向流水、退款流水或补偿流水，而不是 `update ledger_entry set amount_delta = ...`。

## 24.4 幂等键设计

幂等键不能随便取。一个常见错误是：

```text
idem_key = biz_type + biz_id
```

这只适合“一个业务单据只发生一次积分动作”的场景，不适合 AiVedio。

同一个项目可能多次生成视频，同一个任务可能有预占、结算、释放、退款多个动作，同一个导出任务还可能分段计费。如果只用 `project_id` 或 `task_id` 粗暴去重，就会误杀真实的新消费，或者漏掉同一任务下的重复结算。

推荐把业务动作拆细：

| 场景 | 推荐业务键 |
|---|---|
| 创建充值订单 | `tenant_id + user_id + client_idem_key` |
| 支付成功加积分 | `recharge:{payment_order_no}:credit` |
| 视频生成预占 | `reserve:generation:{task_id}:v1` |
| 视频生成结算 | `settle:generation:{task_id}:{charge_version}` |
| 失败释放 | `release:generation:{task_id}:{reason_code}` |
| 已结算退款 | `refund:generation:{task_id}:{refund_request_id}` |
| 客服补偿 | `compensate:{ticket_id}:{compensate_no}` |
| 最终导出扣费 | `reserve:export:{export_job_id}:v1` |
| 实时会话分片扣费 | `settle:realtime:{session_id}:{usage_slice_id}` |

这里有一个很重要的点：

> 前端可以传 `Idempotency-Key`，但最终入账应该使用后端生成的稳定业务 ID。

例如用户点击“生成视频”：

```text
1. 前端发送 POST /generation-tasks，并带 Idempotency-Key。
2. 服务端根据 tenant_id + user_id + Idempotency-Key 创建或复用 generation_task。
3. 后端生成 task_id 和 consume_action_id。
4. 积分系统使用 task_id / consume_action_id 作为账务动作键。
5. 后续所有重试都围绕同一个 task_id 收敛。
```

这样即使浏览器重试、网关重试、Go 服务超时重试，最终都会回到同一个任务和同一个预占动作。

## 24.5 提交生成任务时如何预占积分

预付费用户提交生成任务时，正确顺序是：

```text
归一化参数
-> 计算服务端报价
-> 幂等创建任务
-> PostgreSQL 事务内预占积分
-> 创建 generation_task
-> 写 outbox_event
-> 返回 task_id
```

如果 Generation 与 Billing 共用同一个 PostgreSQL，可以把“预占积分、创建任务、写 Outbox”放在同一个本地事务里。

伪代码如下：

```text
begin transaction

1. insert billing_operation(operation_id = reserve:generation:{task_id}:v1)
   on conflict do nothing

2. 如果 operation 已存在：
   - request_hash 相同且 SUCCESS：返回原结果
   - request_hash 不同：返回 409
   - PROCESSING：返回处理中或让客户端查询任务状态

3. 原子更新 wallet_account：
   available_amount -= reserved_amount
   reserved_amount  += reserved_amount
   条件：available_amount >= reserved_amount

4. 插入 credit_reservation

5. 插入 ledger_transaction 和 ledger_entry

6. 创建 generation_task

7. 插入 outbox_event(TASK_CREATED)

8. billing_operation.status = SUCCESS

commit
```

账户更新可以写成条件更新：

```sql
update wallet_account
set available_amount = available_amount - $1,
    reserved_amount = reserved_amount + $1,
    version = version + 1,
    updated_at = now()
where wallet_id = $2
  and status = 'ACTIVE'
  and available_amount >= $1
returning available_amount, reserved_amount, version;
```

这句 SQL 同时完成三件事：

1. 检查余额。
2. 冻结积分。
3. 防止并发超扣。

不要写成：

```text
select balance
-> Go 代码判断够不够
-> update balance
```

在高并发下，先查再改很容易出现两个请求都看到余额足够，然后同时扣掉的情况。要么使用 `select ... for update` 锁定账户行，要么使用上面的条件 `update` 原子完成判断和修改。

### 24.5.1 Redis 在这里能做什么

Redis 可以用于：

- 快速展示余额。
- 短期接口幂等缓存。
- 限流。
- 租户预算准入加速。
- 防止明显重复点击打到数据库。

但 Redis 不能成为最终账务事实源。

原因很简单：Redis 缓存丢失、主从切换、过期策略、网络分区都不应该导致用户积分多扣或少扣。最终正确性必须落在 PostgreSQL 的事务、唯一约束和账本流水上。

## 24.6 任务成功时如何确认扣除

任务成功后，不能再扣一次 `available_amount`。因为提交任务时已经把可用额度转成了预占额度。

正确动作是：

```text
RESERVED -> CONSUMED
```

如果预占 120 积分，最终只需要收 100 积分，则应同时完成：

```text
SETTLE 100
RELEASE 20
```

结算消费者可以由 `TASK_SUCCEEDED` 事件触发：

```text
RocketMQ -> Settlement Consumer
  1. 插入 consumer_inbox 去重。
  2. 锁定 generation_task 和 credit_reservation。
  3. 判断任务是否已经结算。
  4. 计算 final_charge_amount。
  5. 插入 settle 操作和账本。
  6. reserved_amount -= final_charge_amount。
  7. 若有差额，插入 release 操作和账本。
  8. 更新 reservation 状态。
  9. 写 outbox_event(CHARGE_SETTLED)。
```

关键约束包括：

```text
unique (wallet_id, business_key)
unique (task_id) on credit_reservation
reservation.state in allowed previous states
settled_amount + released_amount <= reserved_amount
```

这里仍然不能相信 RocketMQ 只投一次。`TASK_SUCCEEDED` 重复到达时，业务效果必须只生效一次。真正挡住重复结算的是：

- `consumer_inbox` 消费去重。
- `billing_operation.operation_id` 唯一约束。
- `ledger_transaction.business_key` 唯一约束。
- `credit_reservation` 状态条件更新。

## 24.7 失败、取消和未知结果怎么处理

AI 视频平台最容易出错的地方，不是成功路径，而是失败路径。

### 24.7.1 还没提交供应商就失败

例如参数校验失败、内容审核拒绝、内部排队前失败。

这类失败没有外部成本，可以释放全部预占：

```text
release:generation:{task_id}:validation_failed
```

账本分录：

```text
RESERVED  -120
AVAILABLE +120
```

### 24.7.2 供应商明确失败且不收费

如果供应商返回了明确终态：

```text
FAILED
not_billable
```

也可以释放全部预占。

但要注意，平台应该把供应商失败状态和平台交付失败状态分开。供应商生成成功但平台回源失败，不一定能立即释放，因为供应商成本可能已经发生，后续还可能通过补偿回源修复。

### 24.7.3 供应商已受理但本地超时

这是最危险的场景：

```text
Provider Adapter 调用供应商
-> 供应商已经创建任务并开始计费
-> 本地读取响应超时
-> 平台不知道 provider_task_id
```

这不是普通失败，而是 `SUBMIT_UNKNOWN`。

此时不能做两件事：

1. 不能直接释放用户预占。
2. 不能立刻换供应商重提。

正确做法是：

```text
1. 保留 credit_reservation。
2. 将 provider_attempt 标记为 SUBMIT_UNKNOWN。
3. 使用 external_idempotency_key 或供应商查询接口确认。
4. 查到任务后补录 provider_task_id。
5. 明确查不到并超过幂等窗口后，再按策略创建新 attempt。
6. 长时间无法确认时，进入人工或账单对账队列。
```

这条规则要和第 9 章多供应商调度、第 12 章回调轮询、第 20 章计费对账保持一致：**未知结果不是失败，不能用释放和重试掩盖不确定性。**

### 24.7.4 用户取消和供应商完成同时发生

用户点击取消时，供应商可能已经完成。系统必须通过任务状态机决定唯一结果。

```text
PENDING / QUEUED:
  可以取消，释放全部预占。

SUBMITTED / RUNNING:
  请求取消供应商任务，进入 CANCEL_REQUESTED。
  是否收费取决于供应商返回和计费政策。

SUCCEEDED:
  取消请求不改变成功事实，按成功结算。

SUBMIT_UNKNOWN:
  不能直接释放，先确认外部是否已受理。
```

如果供应商支持取消但收取取消费，可以做部分结算：

```text
reserved_amount = 120
cancel_fee = 30

SETTLE 30
RELEASE 90
```

面试时这里要讲清楚：取消不是简单删除任务，而是一条有状态、有账务影响的业务动作。

## 24.8 充值积分怎么入账

充值同样不能把“创建充值订单”和“加积分”混在一起。

完整流程是：

```text
Client
-> 创建 payment_order
-> 跳转或拉起支付
-> 支付平台回调
-> 验签
-> PostgreSQL 事务内加积分、写账本、改订单状态
-> Outbox 发布 RechargeCredited
```

关键表可以简化为：

```sql
create table payment_order (
    order_no           varchar(128) primary key,
    wallet_id          uuid not null,
    tenant_id          uuid not null,
    money_amount       bigint not null,
    point_amount       bigint not null,
    provider           varchar(64) not null,
    provider_trade_no  varchar(128) null,
    status             varchar(32) not null,
    client_idem_key    varchar(128) null,
    request_hash       varchar(128) null,
    paid_at            timestamptz null,
    credited_at        timestamptz null,
    created_at         timestamptz not null,
    updated_at         timestamptz not null,

    unique (tenant_id, client_idem_key),
    unique (provider, provider_trade_no)
);
```

支付回调一定要假设会重复到达。

加积分操作键可以设计为：

```text
recharge:{payment_order_no}:credit
```

事务逻辑：

```text
begin transaction

1. select payment_order for update

2. 如果已经 CREDITED:
   直接返回成功给支付平台，避免对方继续回调。

3. insert billing_operation(recharge:{order_no}:credit)
   on conflict do nothing

4. update wallet_account
   available_amount += point_amount

5. insert ledger_transaction / ledger_entry

6. payment_order.status = CREDITED

7. insert outbox_event(RECHARGE_CREDITED)

commit
```

这里也不要依赖 Redis 防回调重复。真正的防线是：

- `payment_order for update`。
- `provider + provider_trade_no` 唯一约束。
- `billing_operation.operation_id` 唯一约束。
- 账户更新和账本写入在同一个事务内完成。

## 24.9 RocketMQ 重复投递下如何保证一次效果

RocketMQ 在这个系统里的定位是异步传递，不是账务事实源。

必须默认：

```text
消息可能重复
消息可能延迟
消费者可能处理一半宕机
人工可能重放消息
```

所以消息消费要遵循 Outbox + Inbox 模式。

### 24.9.1 生产侧 Outbox

业务事务里写入：

```text
generation_task
credit_reservation
ledger_transaction
outbox_event
```

事务提交后，由 Outbox Relay 扫描并投递 RocketMQ。

```sql
select *
from outbox_event
where status = 'PENDING'
order by created_at
for update skip locked
limit 100;
```

这样即使 RocketMQ 短暂不可用，数据库里的事实仍然在，后续可以继续投递。

### 24.9.2 消费侧 Inbox

消费者处理消息时，先插入：

```sql
create table consumer_inbox (
    consumer_group    varchar(128) not null,
    message_key       varchar(160) not null,
    processed_at      timestamptz not null,
    primary key (consumer_group, message_key)
);
```

如果插入冲突，说明这条业务消息已经处理过，直接 ACK。

但注意，Inbox 只是消息级去重。账务操作仍然要有自己的 `operation_id` 和账本唯一约束。因为同一条消息可能触发多个业务动作，而人工修复也可能绕过原始消息入口。

## 24.10 对账任务要检查什么

积分扣费系统不应该等用户投诉才发现问题。至少要有日终对账任务。

### 24.10.1 余额快照和账本聚合对账

核心检查：

```text
wallet_account.available_amount
wallet_account.reserved_amount
```

是否等于账本分录聚合后的结果。

概念 SQL：

```sql
select
    wallet_id,
    sum(case when bucket = 'AVAILABLE' then amount_delta else 0 end) as ledger_available,
    sum(case when bucket = 'RESERVED' then amount_delta else 0 end) as ledger_reserved
from ledger_entry e
join ledger_transaction t on t.ledger_txn_id = e.ledger_txn_id
group by wallet_id;
```

发现差异后，不要直接改旧流水。应该生成 `reconciliation_item`，确认原因后追加调整流水。

### 24.10.2 预占和任务状态对账

检查以下异常：

| 异常 | 可能原因 |
|---|---|
| `reservation=ACTIVE` 但任务已失败 | 释放事件丢失或消费者失败 |
| `reservation=ACTIVE` 且任务长期 `SUBMIT_UNKNOWN` | 供应商受理状态未确认 |
| `reservation=SETTLED` 但任务不是成功或可收费终态 | 状态机或人工修复错误 |
| `settled + released < reserved` | 部分金额悬挂 |
| 任务成功但没有结算流水 | 结算消费者漏处理 |

这里不能简单把所有超时预占释放。对账任务必须先看任务是否进入可能产生外部成本的阶段。

### 24.10.3 支付订单和充值流水对账

检查：

- 支付平台显示成功，但本地没有 `CREDITED`。
- 本地 `CREDITED`，但支付平台无成功记录。
- 同一个 `provider_trade_no` 对应多个订单。
- 加积分金额和订单金额、套餐规则不匹配。

充值问题通常可以自动补偿，但必须有支付验签、原始回调、订单状态和操作流水作为证据。

### 24.10.4 供应商账单和用户结算对账

供应商账单要先不可变归档，再标准化为 `invoice_line`：

```text
external_charge_id
provider_task_id
external_idempotency_key
amount
currency
occurred_at
raw_payload_hash
```

匹配优先级：

```text
external_charge_id
-> provider_task_id
-> external_idempotency_key
-> provider account + time window + normalized params
```

常见差异包括：

| 差异 | 处理 |
|---|---|
| 供应商收费但内部无 attempt | 标记 `MISSING_INTERNAL`，查提交日志和幂等键 |
| 内部认为失败释放，但供应商收费 | 进入成本差异，不一定向用户追扣 |
| 用户已结算但供应商未收费 | 可能是账单延迟或促销额度，等待下一账期 |
| 金额不一致 | 检查价格版本、汇率、供应商计费单位和舍入 |
| 重复外部账单 | 标记 `DUPLICATE_EXTERNAL`，不要重复入成本 |

对账的原则仍然是：修正通过追加事实完成，不覆盖历史。

## 24.11 Go 服务实现边界

如果用 Go 实现，可以把模块边界拆成下面这样：

| 模块 | 职责 |
|---|---|
| `PricingService` | 归一化参数、读取价格版本、输出报价和最大预占额 |
| `BillingService` | 账户快照、预占、结算、释放、退款、账本和幂等操作 |
| `GenerationService` | 任务创建、状态机、用户可见状态 |
| `OutboxRelay` | 扫描本地 Outbox，投递 RocketMQ |
| `SettlementConsumer` | 消费任务成功、失败、取消事件，触发账务动作 |
| `ReconciliationJob` | 做账本、任务、支付、供应商账单之间的差异检查 |

一个推荐的 Go 层接口形状：

```go
type ReserveCreditsCommand struct {
    TenantID        string
    WalletID        string
    TaskID          string
    ConsumeActionID string
    Amount          int64
    UnitCode        string
    PriceVersionID  string
    RequestHash     string
}

type SettleCreditsCommand struct {
    TenantID      string
    WalletID      string
    TaskID        string
    ReservationID string
    Amount        int64
    ChargeVersion string
    ReasonCode    string
}

type ReleaseCreditsCommand struct {
    TenantID      string
    WalletID      string
    TaskID        string
    ReservationID string
    Amount        int64
    ReasonCode    string
}
```

实现时有几个细节要坚持：

1. 金额使用 `int64` 最小积分单位，不使用 `float64`。
2. 远程供应商调用不能放在 PostgreSQL 事务里。
3. `operation_id` 在进入事务前就计算好。
4. `request_hash` 要由规范化后的请求体计算，不能直接 hash 原始 JSON 字符串。
5. 所有状态变化要带条件，例如 `where state = 'ACTIVE' and version = ?`。
6. 事务内写业务事实和 Outbox，事务外发送 RocketMQ。
7. 日志里记录 `task_id`、`reservation_id`、`operation_id`、`ledger_txn_id` 和 `trace_id`，不要记录支付密钥或完整敏感参数。

## 24.12 常见错误

### 1. 只在任务成功后直接扣余额

这种方式无法处理长时间在途任务。供应商已经产生费用时，用户余额可能已经被其他任务消费完，平台就会承担信用风险。

### 2. 用 Redis setnx 当唯一防重复

Redis 可以做短期拦截，但不能替代数据库唯一约束。账务正确性必须由 PostgreSQL 事务和唯一键兜住。

### 3. 用 `task_id` 同时表示所有扣费动作

同一个任务至少有 `reserve`、`settle`、`release`、`refund` 多种动作。幂等键必须包含动作类型和必要版本。

### 4. 把 Release 和 Refund 混成一个概念

Release 是释放未消费的预占额度。Refund 是对已经结算的消费追加返还。二者账本语义不同。

### 5. 供应商超时就释放额度

本地超时不代表供应商未受理。释放后重提可能造成双份生成和双份成本。

### 6. 对账时直接改余额

对账发现差异后，应生成差异单并追加调整流水。直接改余额会让账本无法解释当前快照。

## 24.13 面试表达

如果面试官问“你们 AI 视频平台的积分扣费怎么设计”，可以这样回答：

> 我不会把积分系统设计成简单的 `user.balance -= amount`，因为 AI 视频生成是长耗时异步任务，供应商一旦受理就可能产生真实成本，而且 HTTP、MQ、回调和人工重放都会带来重复执行风险。
>
> 我们采用的是预占、结算、释放和退款分离的账本模型。用户提交任务时，服务端先按归一化参数和价格版本报价，然后在 PostgreSQL 事务里完成积分预占、任务创建和 Outbox 写入。预占成功后任务才会进入 RocketMQ 和供应商调度。
>
> 积分系统里账户余额只是快照，真正的事实是不可变账本。每个动作都有稳定业务键，比如 `reserve:generation:{task_id}:v1`、`settle:generation:{task_id}:{charge_version}` 和 `release:generation:{task_id}:{reason_code}`。即使 RocketMQ 重复投递，数据库唯一约束、Inbox 去重和状态条件更新也能保证业务效果只发生一次。
>
> 任务成功时，不会再次扣可用余额，而是把预占额度转成已消费额度；如果实际收费低于预占，再释放差额。任务失败或取消时，要看任务处于哪个阶段。还没提交供应商可以直接释放；已经提交或处于 `SUBMIT_UNKNOWN` 时不能盲目释放，要先通过供应商幂等键、查询接口或账单确认外部结果。
>
> 最后我们会做日终对账，检查余额快照和账本聚合、预占和任务状态、支付订单和充值流水、供应商账单和内部 attempt 是否一致。对账发现问题后追加补偿或调整流水，不覆盖历史记录。

压缩成一句话就是：

> **先报价，再预占；成功结算，失败释放；重复靠业务键和数据库唯一约束挡住；未知结果先确认，最终靠不可变账本和对账补偿收敛。**

## 24.14 本章总结

积分扣费系统的核心不是“扣掉多少积分”，而是让每一次积分变化都能被解释、被追踪、被重放验证。

本章最重要的结论有十条：

1. 积分系统要按准金融账本设计，不要只当普通业务表。
2. 账户余额是快照，账本流水才是事实。
3. AI 视频生成应先预占积分，再进入高成本外部执行。
4. `reserve`、`settle`、`release`、`refund` 是不同动作，幂等键要分开。
5. 前端 `Idempotency-Key` 负责请求幂等，后端 `task_id` 和 `consume_action_id` 负责账务幂等。
6. PostgreSQL 事务、唯一约束和状态条件更新是防重复扣费的核心。
7. Redis 只能做速度层，不能做最终余额事实源。
8. RocketMQ 默认至少一次投递，消费端必须有 Inbox 和业务幂等。
9. `SUBMIT_UNKNOWN` 不能当失败处理，不能直接释放后重提。
10. 对账发现差异后追加修正事实，不覆盖历史账本。

把这套设计放回 AiVedio 的主线里，它正好补上第 20 章的实战落地层：

```text
报价
-> 积分预占
-> 任务执行
-> 成功结算
-> 失败释放 / 退款 / 补偿
-> 日终对账
```

这条链路跑稳了，AI 视频平台才不会在高并发、重复消息、供应商未知结果和人工补偿里把账算乱。
