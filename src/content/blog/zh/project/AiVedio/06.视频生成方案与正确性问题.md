# 第六章视频生成方案｜最容易体现资深程度的正确性问题

> 基于《AI 生成视频项目面试 Roadmap》第六章改编。原稿中缺失的小节标题已补为 **6.3 回调重复、乱序和延迟**。

## 一、成片规格

| 项目 | 设定 |
|---|---|
| 成片时长 | 05:00 |
| 画幅 | 16:9 |
| 主控规格 | 3840×2160，24 fps；交付可降为 1920×1080 |
| 内容形态 | 技术解说 + 架构动效 + AI 视频氛围底片 |
| 叙事目标 | 用四个故障场景展示资深工程师如何处理不确定性、重复、乱序与竞态 |
| 受众 | Go / 后端 / 分布式系统 / AI 视频平台面试者 |
| 核心金句 | 分布式正确性，不靠某个中间件的承诺，而靠业务层把异常设计成可恢复的正常路径。 |

## 二、叙事结构

1. **开场立论（00:00–00:26）**：真正拉开资深度差距的是正确性设计。
2. **6.1 提交结果未知（00:26–01:42）**：HTTP 超时不等于远端失败，必须建模 `SUBMIT_UNKNOWN`。
3. **6.2 重复消费（01:42–02:39）**：至少一次投递下，正确性依赖业务幂等。
4. **6.3 回调重复、乱序和延迟（02:39–03:36）**：所有外部状态先转为内部事件，再由状态机裁决。
5. **6.4 取消与完成竞态（03:36–04:32）**：取消是请求，不是事实；最终语义决定状态与计费。
6. **总结（04:32–05:00）**：幂等、状态机、版本约束、审计补偿。

## 三、统一视觉与生成规则

### 视觉风格锁（每条 AI 视频提示词前置）

```text
Cinematic technical explainer, dark obsidian and deep navy environment, translucent glass UI panels, cyan data signals with restrained amber warning accents, clean isometric distributed-system architecture, subtle volumetric light, physically plausible reflections, crisp geometry, premium enterprise technology aesthetic, slow controlled camera movement, high temporal consistency, 16:9, 4K composition, no readable text inside the generated footage.
```

### 通用负面提示词（每条提示词后置）

```text
readable text, letters, numbers, logos, brand marks, watermark, distorted interface, random symbols, malformed geometry, flicker, jitter, unstable objects, duplicated nodes, oversaturated neon, cyberpunk clutter, cartoon style, low resolution, heavy motion blur, fast chaotic camera, people speaking to camera
```

### 生成与后期原则

- AI 模型只生成环境、节点、数据流和镜头运动；**所有技术文字、状态名、箭头和数值均在后期叠加**。
- 单条模型素材建议生成 6–8 秒；超过 10 秒的镜头用 A/B 两个变体、慢推、局部特写或 MG 插入镜头拼接。
- 精确逻辑图使用 After Effects、Blender、Rive 或前端 Canvas 制作，不让生成模型承担状态机文字排版。
- 色彩语义固定：青色=正常流转，琥珀=未知/请求中，绿色=已确认终态，红色=禁止路径或风险。

## 五、逐镜头脚本与生成提示词

### C06_S01｜00:00:00.000–00:00:14.000｜开场

**旁白**
真正体现资深程度的，不是会不会调用一个 AI 接口，而是系统进入不确定状态时，你能不能保证不重复扣费、不丢任务，也不把错误状态写成事实。

**画面设计**
黑暗中的分布式系统悬浮展开，数据包在节点之间穿行；部分链路进入雾状不确定区，中央账本与任务事实源保持稳定。

**后期叠字**
- `正确性，才是资深度`

**镜头专属英文提示词**

```text
A sprawling abstract backend system suspended in darkness, glowing data paths crossing through uncertain fog, several event packets frozen between nodes, one central transaction ledger protected behind glass, slow dolly-in, tense but precise.
```

**运动设计**
0–5 秒缓慢推近；5–10 秒四周链路出现超时、重复、乱序的轻微扰动；10–14 秒聚焦中央稳定账本。

**转场 / 音效**
从黑场中由一条青色数据线点亮全景。 低频启动脉冲、细微网络电流声。

### C06_S02｜00:00:14.000–00:00:26.000｜开场

**旁白**
这一章看四类最常见、也最能拉开面试差距的正确性问题：提交结果未知、消息重复、回调乱序，以及取消与完成的竞态。

**画面设计**
画面分为四象限：丢失的响应包、重复消息、逆序状态线、取消信号与完成信号相撞。

**后期叠字**
- `提交结果未知`
- `重复消费`
- `回调乱序`
- `取消竞态`

**镜头专属英文提示词**

```text
Four-part visual tableau: a lost response packet, duplicated message capsules, an out-of-order event timeline, and a cancel signal colliding with a completion signal, symmetrical composition, smooth camera orbit.
```

**运动设计**
四象限依次点亮，最后缩合成同一个任务状态机核心。

**转场 / 音效**
四象限中心快速收束，切入第一类问题。 四次短促定位音，最后一声下潜。

### C06_S03｜00:00:26.000–00:00:40.000｜6.1 提交结果未知

**旁白**
先看第一类。后端向供应商提交生成任务，供应商已经成功创建任务，但响应在返回途中超时。本地只看到了超时，却不知道远端究竟有没有受理。

**画面设计**
请求从后端飞向外部供应商；供应商节点成功生成任务记录，返回包却在网络隧道中消散。

**后期叠字**
- `供应商已受理 ≠ 本地已确认`

**镜头专属英文提示词**

```text
A luminous API request travels from a backend cluster to an external cloud service; the cloud accepts and creates one video job, but the returning response packet dissolves inside a turbulent network tunnel, side-tracking camera.
```

**运动设计**
镜头跟随请求向右移动；远端任务点亮；返回路径在中段断裂并冻结。

**转场 / 音效**
对断裂的返回包做瞬时放大。 请求发送音、远端确认音、随后连接中断声。

### C06_S04｜00:00:40.000–00:00:56.000｜6.1 提交结果未知

**旁白**
最危险的做法，是把所有 HTTP 超时都当成失败，然后立刻重试。第二次请求可能再创建一个付费任务，用户只想生成一次，平台却承担了两次成本。

**画面设计**
本地节点出现红色重试脉冲，第二条请求路径启动；远端出现两个并行生成任务和两条计费记录。

**后期叠字**
- `HTTP 超时后立即重试：危险`
- `1 次意图 → 2 个付费任务`

**镜头专属英文提示词**

```text
The backend automatically sends a second request after timeout; the path splits into two paid generation jobs and two glowing video outputs, duplicate cost indicators rising, rapid push-in and subtle warning pulse.
```

**运动设计**
第一次任务保持运行；重试箭头分叉；两份输出和两条扣费同时出现。

**转场 / 音效**
红色告警闪一次后，画面立即冷却为琥珀色未知状态。 重复触发音、双重计费落账声。

### C06_S05｜00:00:56.000–00:01:12.000｜6.1 提交结果未知

**旁白**
因此，提交动作必须有唯一的 attempt_id，供应商支持幂等键时必须透传。超时后，本地状态不是 FAILED，而应进入 SUBMIT_UNKNOWN。

**画面设计**
请求胶囊被压印唯一指纹；超时后进入琥珀色隔离槽，而不是落入红色失败终态。

**后期叠字**
- `attempt_id / client_request_id`
- `Idempotency-Key`
- `SUBMIT_UNKNOWN`

**镜头专属英文提示词**

```text
A unique request token is stamped onto a job capsule; an amber unknown-state container safely holds the task instead of a red failure bin, macro close-up, deliberate mechanical motion.
```

**运动设计**
先特写唯一指纹，再拉远展示 FAILED 与 SUBMIT_UNKNOWN 两条路径，前者被禁用。

**转场 / 音效**
沿唯一指纹的扫描线进入对账画面。 机械盖章声、琥珀状态锁定声。

### C06_S06｜00:01:12.000–00:01:28.000｜6.1 提交结果未知

**旁白**
系统随后通过幂等标识、任务列表或查询接口对账。只有确认远端没有创建任务，才允许重试；无法确认时，要延迟重试，而不是立即放大风险。

**画面设计**
扫描器同时读取本地任务与远端任务注册表；匹配到任务时亮绿，未确认的重试包停留在延迟轨道。

**后期叠字**
- `先对账，再决定是否重试`
- `无法确认 → 延迟重试`

**镜头专属英文提示词**

```text
A reconciliation scanner compares local records with a remote provider job registry; matching identifiers illuminate, while a retry packet waits in a delayed orbit rather than launching immediately, slow top-down move.
```

**运动设计**
扫描线从左至右完成比对；重试包围绕时钟轨道缓慢运行，不立即发射。

**转场 / 音效**
时钟轨道化为计费流水的圆形指示。 扫描声、匹配确认音、延迟计时滴答。

### C06_S07｜00:01:28.000–00:01:42.000｜6.1 提交结果未知

**旁白**
同时，重复任务成本要被监控，计费流水要支持冲正。这里的关键不是重试机制，而是把“明确失败”和“结果未知”分开建模。

**画面设计**
账本发现重复扣费后写入反向补偿流水；画面明确分出“明确失败”和“结果未知”两条状态泳道。

**后期叠字**
- `明确失败 ≠ 结果未知`
- `计费可冲正`

**镜头专属英文提示词**

```text
A financial ledger detects a duplicate charge and posts a compensating reversal; two clearly separated lanes represent confirmed failure and unknown outcome, clean isometric view, calm stabilization.
```

**运动设计**
异常费用被圈出，反向流水抵消；两条状态泳道最终保持隔离。

**转场 / 音效**
账本中的一条事件流水变形成 MQ 消息。 账本翻页声、冲正完成音。

### C06_S08｜00:01:42.000–00:01:55.000｜6.2 重复消费

**旁白**
第二类是重复消费。RocketMQ 应按至少一次投递理解：同一条消息，完全可能被消费者再次收到。

**画面设计**
消息经 Broker 输送到消费者；链路短暂中断后，同一个带相同指纹的消息再次到达。

**后期叠字**
- `RocketMQ：至少一次投递`
- `同一消息可能再次到达`

**镜头专属英文提示词**

```text
A message broker distributes one event capsule; after a brief interruption, the exact same capsule arrives again at the consumer, looped conveyor metaphor, controlled parallax.
```

**运动设计**
第一条消息到达后短暂停顿；原消息沿回环轨道再次进入消费者。

**转场 / 音效**
镜头进入消息胶囊内部，展示重复原因。 消息投递声、回环重投声。

### C06_S09｜00:01:55.000–00:02:09.000｜6.2 重复消费

**旁白**
消费超时、消费者宕机、ACK 丢失、Broker 重投，甚至人工重放，都会让业务逻辑执行第二遍。

**画面设计**
五个故障图标按节奏出现：计时器溢出、消费者熄灭、ACK 信号消失、Broker 回投、人工拉杆重放。

**后期叠字**
- `消费超时`
- `消费者宕机`
- `ACK 丢失`
- `Broker 重投`
- `人工重放`

**镜头专属英文提示词**

```text
A sequence of failure causes around a message path: consumer crash, acknowledgement signal fading, broker redelivery, manual replay lever, all as abstract icons without text, rhythmic cuts.
```

**运动设计**
每 2 秒切换一个原因，最后五个原因同时指向“重复执行”。

**转场 / 音效**

### C06_S10｜00:02:09.000–00:02:26.000｜6.2 重复消费

**旁白**

**画面设计**

**后期叠字**

- `inbox_dedup`
- `UNIQUE 业务键`
- `version / CAS`

### C06_S12｜00:02:39.000–00:02:52.000｜6.3 回调重复、乱序和延迟

## 六、配音、字幕与声音设计

### 配音

- 普通话，中性偏沉稳，技术负责人语气；语速约 1.02 倍。
- `attempt_id`、`SUBMIT_UNKNOWN`、`event_id`、`inbox_dedup`、`CANCEL_REQUESTED` 等术语应逐词读清。
- 关键对比句前留 150–250 ms 停顿，例如“明确失败——不等于——结果未知”。

### 字幕

- 主字幕最多两行，每行建议不超过 18 个汉字；术语保持英文原样。
- 重点词使用琥珀色块或描边，不整句变色。
- 状态名采用等宽字体；正文采用无衬线字体。
- 已生成 `chapter06_voiceover.srt`，时间轴与本分镜完全对齐。

### BGM 与混音

- 00:00–00:26：低频脉冲，建立严肃技术氛围。
- 00:26–01:42：轻度张力上升，突出提交未知风险。
- 01:42–02:39：机械节拍，表现消息重复与去重闸门。
- 02:39–03:36：加入轻微错拍与 Glitch，但不要干扰旁白。
- 03:36–04:32：悬念型持续音，取消与完成在边界竞争。
- 04:32–05:00：和声收束，进入确定、稳定的结论。
- 旁白为主，BGM 自动闪避约 5–7 dB；转场音效不覆盖术语。
