# Agent 设计范式专栏导读

这个专栏放的是比某一个具体 Agent 项目更长期的东西。

`Build Harness` 更适合按工程步骤往前搭：Provider、Loop、Tool Runtime、Context Policy、Session Replay、Trace、Memory、Hosted Harness。

而这里更适合沉淀横跨多个 Agent 系统都稳定成立的设计范式：

```text
上下文不是 messages[]，而是注意力治理。
记忆不是缓存，而是带作用域和来源的知识。
工具调用不是文本，而是有权限、因果和审计的事件。
压缩不是总结一下，而是可追溯的语义蒸馏。
Subagent 不是多开一个模型，而是上下文隔离和责任回收机制。
```

这个专栏的读法很简单：每篇文章都围绕一个工程抽象，先问它为什么出现，再拆它解决了什么、留下了什么边界，以及怎样落到数据模型和不变量。

## 文章路径

1. [Context Manager 新范式：Agent 的注意力操作系统](/blog/AI/agent设计范式/01-context-manager-attention-os)  
   把上下文从 `messages[]` 拼接升级成事件事实、状态投影、长期记忆、外部证据、工具环境和系统策略共同参与的注意力治理层。

2. [Agent 长期记忆与自我优化范式](/blog/AI/agent设计范式/02-agent-long-term-memory-self-upgrade)  
   把 Agent 的经历转化为可追溯的记忆、可复用的技能、可测试的优化，并通过评估、灰度和回滚机制让 Agent 稳定变好。

3. [Tool Manager 新范式：Agent 的行动操作系统](/blog/AI/agent设计范式/03-tool-manager-action-os)  
   把工具系统从 function registry 升级成围绕能力、意图、权限、凭证、沙盒、执行、副作用和审计的行动治理层。

4. [Agent Team 任务分配与通信协议](/blog/AI/agent设计范式/04-agent-team-assignment-communication)  
   把 Agent Team 从多个 agent 的拓扑图，推进到任务分配、租约、结构化通信、证据回流、冲突处理和验证合并的协作运行时。

后续这个专栏会继续补充记忆治理、协作隔离、权限策略和可审计运行等主题。
