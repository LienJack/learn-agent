1. 图解类型
circular loop，因为这一节强调自我升级不是一次性总结，而是从轨迹、候选、评估、发布到真实使用后的持续演化闭环。

2. 画面元素清单
8 个环形节点：Trajectory、Reflection、Candidate、Eval、Release、Runtime、Monitor、Patch；中央是 Skill Bank；淡黄色高亮“Eval”和“Patch”；背景有轻微电路线和版本标签。

3. 正向图片提示词
画一张文章内技术解释图，主题是 skill 从经历中生成并持续演化的生命周期。

画风：米白色纸张背景，黑色手绘签字笔线稿，线条粗细略不均，少量淡黄色高亮，editorial technical illustration，技术博客手绘循环图，清晰、克制、有工程草图感。

构图：环形流程图。画面中央是一个简洁的 Skill Bank，像带版本标签的工具书架。围绕中心画 8 个流程节点，按顺时针排列，用手绘箭头连成闭环。节点之间关系清楚，整体不要太密。

节点：
① Trajectory：事件纸带，表示完整任务轨迹。
② Reflection：便签和回看箭头，表示经验教训。
③ Candidate：草稿 skill 卡片，表示候选技能。
④ Eval：测试台和基线尺，用淡黄色高亮，表示验证是否有效。
⑤ Release：版本标签和小旗，表示 gated release。
⑥ Runtime：Agent 调用工具书，表示真实使用。
⑦ Monitor：仪表盘，表示调用效果和成本。
⑧ Patch：修补贴纸和扳手，用淡黄色高亮，表示 post-use patch 或回滚。

高亮：用淡黄色强调“Eval”和“Patch”，表达没有评估不发布，用后还要复盘。

背景：很淡的电路线、节点连线和工程草图辅助线，不要抢主体。

文字：节点标签短；底部可留出一行手写金句位置：“只有跑赢基线，才叫升级”。如果中文文字不稳定，则只留空白区域给后期添加。

整体感觉像技术博客中的手绘机制图，不是产品 UI，不是正式架构图。元素清晰，层次明确，箭头关系明确。

4. 负向提示词
不要照片写实，不要 3D，不要复杂 UI 截图，不要大量代码，不要密集小字，不要复杂表格，不要赛博朋克，不要霓虹色，不要深色背景，不要彩色卡通，不要复杂阴影，不要过多装饰，不要超过 8 个主节点，不要让背景电路线比主体更明显，不要生成长段中文文字。
