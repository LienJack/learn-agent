1. 图解类型
状态序列，因为这张图要解释支付模块重构例子里，后端、前端、审查、测试和 Lead 如何通过依赖问题、接口契约、冲突通知、决策记录和验证合并完成闭环。

2. 画面元素清单
8 个主节点：任务图、后端分析、前端扫描、依赖问答、接口契约、审查冲突、决策记录、测试验证合并。图标分别使用任务树、API 插头、购物车窗口、双向问答卡、契约卷轴、警告审查卡、印章日志、测试烧杯与合并轨道。用淡黄色高亮“接口契约”和“决策记录”。背景使用很淡的电路线和工程草图辅助线。底部留一句短句位置：“真正进入主线的不是聊天，而是证据、契约和决策”。

3. 正向图片提示词
画一张文章内技术解释图，主题是支付模块重构中的 Agent Team 协作闭环。

画风：米白色纸张背景，黑色手绘签字笔线稿，线条粗细略不均，少量淡黄色高亮，editorial technical illustration，技术博客手绘流程图，清晰、克制、有工程草图感。

构图：从左到右的状态序列。画面包含 8 个关键节点，并用手绘箭头连接。① 任务图：Root / Lead 旁边有小任务树。② 后端分析：API 插头和 payment adapter 卡片。③ 前端扫描：购物车窗口和 dependency scan 卡片。④ 依赖问答：两张问答卡双向箭头，表示 DependencyQuestion / DependencyAnswer。⑤ 接口契约：卷轴或契约卡片，淡黄色高亮，表示 InterfaceContractProposal。⑥ 审查冲突：审查卡片上有小警告三角，表示 ConflictNotice。⑦ 决策记录：DecisionLog 印章，淡黄色高亮，表示 Lead 写入决策。⑧ 测试验证合并：测试烧杯、盾牌勾选和合并轨道，表示 TestAgent / Verifier / Merge。

背景有很淡的电路线、节点连线和工程草图辅助线，但不要抢主体。每个节点只放很短的中文标签。底部留出一行手写中文金句位置：“真正进入主线的不是聊天，而是证据、契约和决策”。整体感觉像技术博客中的手绘机制图，不是产品 UI，不是正式架构图。

4. 负向提示词
不要照片写实，不要 3D，不要复杂 UI 截图，不要大量代码，不要密集小字，不要复杂表格，不要赛博朋克，不要霓虹色，不要深色背景，不要彩色卡通，不要复杂阴影，不要过多装饰，不要超过 8 个主节点，不要让背景电路线比主体更明显，不要生成长段中文文字。
