1. 图解类型
分栏对比图，因为这张图要解释星型、受控总线型和层级型在分配与通信路径上的差异。

2. 画面元素清单
3 个主栏：星型、受控总线、层级型。每栏 3-4 个小节点，整体主节点不超过 8 个视觉组。图标分别使用中心 Lead、任务看板总线、Root/Domain/Worker 层级树。用淡黄色高亮“TaskBoard / Scheduler”和“DomainLead”。背景使用很淡的电路线和工程草图辅助线。底部留一句短句位置：“拓扑决定谁能说话，协议决定说什么能落状态”。

3. 正向图片提示词
画一张文章内技术解释图，主题是三种 Agent Team 拓扑下的任务分配与通信路径。

画风：米白色纸张背景，黑色手绘签字笔线稿，线条粗细略不均，少量淡黄色高亮，editorial technical illustration，技术博客手绘流程图，清晰、克制、有工程草图感。

构图：三栏横向对比。左栏标题很短“星型”，画一个中心 LeadAgent 节点，向 3 个 Subagent 节点发箭头，Subagent 回报给 Lead，强调 subagent 之间没有横向箭头。中栏标题“总线”，画 TaskBoard / Scheduler 作为中间横向轨道，多个 agent 从轨道 claim 任务并回写 ArtifactStore，TaskBoard / Scheduler 用淡黄色高亮。右栏标题“层级”，画 RootLead、DomainLead、WorkerAgent 的树状结构，DomainLead 用淡黄色高亮，跨域箭头需要经过上级。

背景有很淡的电路线、节点连线和工程草图辅助线，但不要抢主体。每个节点只放很短的中文标签。底部留出一行手写中文金句位置：“拓扑决定谁能说话，协议决定说什么能落状态”。整体感觉像技术博客中的手绘机制图，不是产品 UI，不是正式架构图。

4. 负向提示词
不要照片写实，不要 3D，不要复杂 UI 截图，不要大量代码，不要密集小字，不要复杂表格，不要赛博朋克，不要霓虹色，不要深色背景，不要彩色卡通，不要复杂阴影，不要过多装饰，不要超过 8 个主视觉组，不要让背景电路线比主体更明显，不要生成长段中文文字。
