1. 图解类型
查询匹配图，因为Q/K/V 很像拿问题去匹配档案标签并读取内容。

2. 画面元素清单
1. Query: 问题卡片，表示当前 token 想找什么
2. Key: 档案标签，表示其他 token 的可匹配特征
3. Value: 档案内容页，表示真正取回的信息
4. 匹配分数: 几条不同粗细连线，表示相关性
5. Causal Mask: 遮住未来格子的三角网格，表示不能偷看答案
高亮：Query、Causal Mask
背景：米白纸张、很淡的电路线和工程草图辅助线。

3. 正向图片提示词
画一张文章内技术解释图，主题是 Q/K/V attention 的工程直觉。

画风：米白色纸张背景，黑色手绘签字笔线稿，线条粗细略不均，少量淡黄色高亮，editorial technical illustration，技术博客手绘流程图，清晰、克制、有工程草图感。

构图：左侧 Query，右侧一排 Key-Value 档案，中间用匹配线连接，底部有 causal mask 小三角。 画面包含 5 个关键节点，并用手绘箭头连接。

节点：
① Query：问题卡片，表示当前 token 想找什么。
② Key：档案标签，表示其他 token 的可匹配特征。
③ Value：档案内容页，表示真正取回的信息。
④ 匹配分数：几条不同粗细连线，表示相关性。
⑤ Causal Mask：遮住未来格子的三角网格，表示不能偷看答案。

高亮：用淡黄色强调 Query、Causal Mask。

背景：很淡的电路线、节点连线和工程草图辅助线，不要抢主体。

文字：节点标签短；底部可留出一行手写金句位置：“先匹配标签，再读取内容；生成时不能看未来”。如果中文文字不稳定，则只留空白区域给后期添加。

整体感觉像技术博客中的手绘机制图，不是产品 UI，不是正式架构图。元素清晰，层次明确，箭头关系明确。

4. 负向提示词
不要照片写实，不要 3D，不要复杂 UI 截图，不要大量代码，不要密集小字，不要复杂表格，不要赛博朋克，不要霓虹色，不要深色背景，不要彩色卡通，不要复杂阴影，不要过多装饰，不要超过 8 个主节点，不要让背景电路线比主体更明显，不要生成长段中文文字。
