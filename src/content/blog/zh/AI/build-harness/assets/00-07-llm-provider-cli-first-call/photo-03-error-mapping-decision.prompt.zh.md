1. 图解类型
决策路径图，因为这张图要解释 provider 原始错误如何变成 runtime 可判断的错误分类，并引出停止、重试、压缩和提示配置等分支。

2. 画面元素清单
7 个主要节点：Raw Error、Adapter Mapping、ProviderError、Runtime Decision、停止、退避重试、压缩上下文。ProviderError 和 Runtime Decision 用淡黄色高亮。Raw Error 用破裂云端接口图标，Runtime Decision 用分叉路标图标。

3. 正向图片提示词
画一张文章内技术解释图，主题是“错误映射让模型调用失败变成 Runtime 决策输入”。

画风：米白色纸张背景，黑色手绘签字笔线稿，线条粗细略不均，少量淡黄色高亮，editorial technical illustration，技术博客手绘流程图，清晰、克制、有工程草图感。

构图：决策路径图。左侧是 Raw Error，画成破裂的云端接口和错误闪电。箭头进入 Adapter Mapping，画成小漏斗或分类器。中间是 ProviderError 统一错误卡片，用淡黄色高亮，卡片上短标签包括 auth、rate_limit、context。右侧进入 Runtime Decision 分叉路标，也用淡黄色高亮。分叉到 3 个结果：① 停止并提示配置，用停止牌和钥匙图标。② 退避重试，用时钟和回旋箭头。③ 压缩上下文，用收缩盒和文档图标。

高亮：用淡黄色强调 ProviderError 和 Runtime Decision 两个节点。

背景：很淡的电路线、节点连线和工程草图辅助线，不要抢主体。

文字：节点标签短；底部留出一行手写金句位置：“错误不是字符串，而是决策输入”。如果中文文字不稳定，则只留空白区域给后期添加。

整体感觉像技术博客中的手绘机制图，不是产品 UI，不是正式架构图。元素清晰，层次明确，箭头关系明确。

4. 负向提示词
不要照片写实，不要 3D，不要复杂 UI 截图，不要大量代码，不要密集小字，不要复杂表格，不要赛博朋克，不要霓虹色，不要深色背景，不要彩色卡通，不要复杂阴影，不要过多装饰，不要超过 8 个主节点，不要让背景电路线比主体更明显，不要生成长段中文文字。
