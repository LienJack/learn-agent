1. 图解类型

Input-process-output。重点是并排比较文件、搜索、终端三类本地工具的输入、风险和控制点。

2. 画面元素清单

三条并排行：文件工具、搜索工具、终端工具；每行都有输入、主要风险、控制点、输出；文件行突出 readFileState，搜索行突出范围与预算，终端行突出权限与 sandbox；高亮三个控制点。

3. 正向图片提示词

画一张文章内技术解释图，主题是文件工具、搜索工具、终端工具的边界差异。

画风：米白色纸张背景，黑色手绘签字笔线稿，线条粗细略不均，少量淡黄色高亮，editorial technical illustration，技术博客手绘流程图，清晰、克制、有工程草图感。

构图：三条横向泳道式 input-process-output 图。第一行“文件”：左侧是路径卡片，中间是 readFileState 小档案盒，右侧是 diff/内容观察；第二行“搜索”：左侧是 glob/grep 放大镜，中间是范围边界和结果预算漏斗，右侧是候选列表；第三行“终端”：左侧是命令行窗口，中间是权限盾牌和 sandbox 盒子，右侧是 exit code 与输出 preview。每行用手绘箭头从左到右连接，三行之间用淡虚线分隔。

高亮：用淡黄色强调 readFileState、结果预算、sandbox 三个控制点。

背景：很淡的电路线、节点连线和工程草图辅助线，不要抢主体。

文字：节点标签短；底部留出一行手写金句位置：“同是本地工具，风险形态不同”。如果中文文字不稳定，则只留空白区域给后期添加。

整体感觉像技术博客中的手绘机制图，不是产品 UI，不是正式架构图。元素清晰，层次明确，箭头关系明确。

4. 负向提示词

不要照片写实，不要 3D，不要复杂 UI 截图，不要大量代码，不要密集小字，不要复杂表格，不要赛博朋克，不要霓虹色，不要深色背景，不要彩色卡通，不要复杂阴影，不要过多装饰，不要超过 8 个主节点，不要让背景电路线比主体更明显，不要生成长段中文文字。
