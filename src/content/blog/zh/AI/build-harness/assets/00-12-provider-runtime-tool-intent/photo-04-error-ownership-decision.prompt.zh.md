1. 图解类型
decision path，因为这张图要把不同错误按来源分流：provider error、intent error、validation error、permission event、tool error，各自进入不同处理路径。

2. 画面元素清单
中央一个“发生错误”决策菱形；向外分 5 条路径：ProviderError、IntentError、ValidationError、PermissionEvent、ToolError；每条路径有不同小图标和处理结果；淡黄色高亮 ProviderError 和 ToolError 的分离；背景用轻微电路线。

3. 正向图片提示词
画一张文章内技术解释图，主题是 Provider Error 不是 Tool Error，错误必须按责任层分流。

画风：米白色纸张背景，黑色手绘签字笔线稿，线条粗细略不均，少量淡黄色高亮，editorial technical illustration，技术博客手绘决策路径图，清晰、克制、有工程草图感。

构图：中心是一个决策菱形节点，图标是警告三角，标签“错误来源”。从中心向右侧和下方分出 5 条手绘箭头路径：① ProviderError：云端接口断线图标，标签“Provider”，处理结果是“重试/降级”，用淡黄色高亮。② IntentError：破损 JSON 卡片图标，标签“意图解析”，处理结果是“重发意图”。③ ValidationError：检查表和叉号图标，标签“校验”，处理结果是“拒绝执行”。④ PermissionEvent：盾牌和人形确认图标，标签“权限”，处理结果是“询问/拒绝”。⑤ ToolError：扳手冒烟图标，标签“工具失败”，处理结果是“观察回填”，用淡黄色高亮。

ProviderError 和 ToolError 两条路径之间画一条清晰分隔线，强调二者不是同一种失败。箭头末端用小回执单表示各自写入不同事件。

背景：很淡的电路线、节点连线和工程草图辅助线，不要抢主体。

文字：节点标签短；底部可留出一行手写金句位置：“同样失败，不同归属”。如果中文文字不稳定，则只留空白区域给后期添加。

整体感觉像技术博客中的手绘机制图，不是产品 UI，不是正式架构图。元素清晰，层次明确，箭头关系明确。

4. 负向提示词
不要照片写实，不要 3D，不要复杂 UI 截图，不要大量代码，不要密集小字，不要复杂表格，不要赛博朋克，不要霓虹色，不要深色背景，不要彩色卡通，不要复杂阴影，不要过多装饰，不要超过 8 个主节点，不要让背景电路线比主体更明显，不要生成长段中文文字。
