1. 图解类型
horizontal pipeline，因为这张图要解释候选能力如何一步步过滤成本轮模型可见集合，并最终进入统一工具执行管线。

2. 画面元素清单
6 个主节点：能力来源、能力目录、发现策略、轻量索引、可见集合、工具管线。图标分别使用工具箱、目录卡片、筛子、索引卡、眼睛、传送带。用淡黄色高亮“发现策略”和“可见集合”。背景使用很淡的电路线和工程草图辅助线。底部留一句短句位置：“存在不等于可见，可见不等于可执行”。

3. 正向图片提示词
画一张文章内技术解释图，主题是 Capability Discovery 如何把大量候选能力过滤成本轮最小可见集合。

画风：米白色纸张背景，黑色手绘签字笔线稿，线条粗细略不均，少量淡黄色高亮，editorial technical illustration，技术博客手绘流程图，清晰、克制、有工程草图感。

构图：从左到右的水平管线。画面包含 6 个关键节点，并用手绘箭头连接。① 能力来源：工具箱图标，旁边有小标签“Tools / Skills / MCP”。② 能力目录：目录卡片和抽屉图标，表示候选能力事实表。③ 发现策略：筛子图标，淡黄色高亮，表示按任务、路径、权限、预算过滤。④ 轻量索引：几张小索引卡，表示只暴露名称和短描述。⑤ 可见集合：眼睛和小菜单图标，淡黄色高亮，表示本轮模型能看到的能力。⑥ 工具管线：传送带和扳手图标，表示所有可执行动作仍进入 validate / permission / execute / observe。

背景有很淡的电路线、节点连线和工程草图辅助线，但不要抢主体。每个节点只放很短的中文标签。底部留出一行手写中文金句位置：“存在不等于可见，可见不等于可执行”。整体感觉像技术博客中的手绘机制图，不是产品 UI，不是正式架构图。

4. 负向提示词
不要照片写实，不要 3D，不要复杂 UI 截图，不要大量代码，不要密集小字，不要复杂表格，不要赛博朋克，不要霓虹色，不要深色背景，不要彩色卡通，不要复杂阴影，不要过多装饰，不要超过 6 个主节点，不要让背景电路线比主体更明显，不要生成长段中文文字。
