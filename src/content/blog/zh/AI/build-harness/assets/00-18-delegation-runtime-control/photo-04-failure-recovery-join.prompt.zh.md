1. 图解类型

Decision path，因为重点是子任务失败要分类回收，并统一回到父 Agent Join，而不是直接让主任务失败。

2. 画面元素清单

左侧一个“子任务异常”入口；中间分成 5 类失败：Validation、Capability、Runtime、Contract、Conflict；每类连接一个处理动作：修任务包、申请权限、重试降级、重写输出、证据仲裁；右侧所有动作汇入“父 Agent Join”。高亮父 Agent Join 和 Capability/Permission 路径。图标包括断开的齿轮、清单、锁、计时器、格式卡片、冲突箭头、汇聚漏斗。

3. 正向图片提示词

画一张文章内技术解释图，主题是 Delegation Runtime 中子 Agent 失败的分类回收机制。

画风：米白色纸张背景，黑色手绘签字笔线稿，线条粗细略不均，少量淡黄色高亮，editorial technical illustration，技术博客手绘决策图，清晰、克制、有工程草图感。

构图：decision path。左侧是“子任务异常”入口，图标是断开的齿轮。中间分成 5 个短标签分支：① Validation：清单错误，处理为“修任务包”；② Capability：锁和工具，处理为“申请权限/改派”，这一支用淡黄色轻微高亮；③ Runtime：计时器，处理为“重试/降级”；④ Contract：格式卡片，处理为“重写输出”；⑤ Conflict：相反箭头，处理为“证据仲裁”。右侧所有处理动作汇入一个“父 Agent Join”节点，用淡黄色高亮，图标是汇聚漏斗和目标旗。

背景：很淡的电路线、节点连线和工程草图辅助线，不要抢主体。节点标签短，避免长句。底部可留一句手写金句位置：“子任务失败，不等于主任务失败”。整体感觉像技术博客中的手绘机制图，不是产品 UI，不是正式架构图。
