1. 图解类型

Decision path，因为重点是子 Agent 遇到高风险动作时必须冒泡审批，而不是自己批准。

2. 画面元素清单

左侧是子 Agent 执行任务；中间遇到高风险动作分叉：安全子集继续执行，高风险动作进入 Permission Escalation；上方是父 Agent/用户审批闸门；右侧是批准后继续、拒绝后收窄范围、重新规划三个出口。高亮 Permission Escalation 和审批闸门。图标包括小工作台、红色但克制的警示三角、盾牌闸门、父 Agent 指挥台、分叉路标。

3. 正向图片提示词

画一张文章内技术解释图，主题是 Delegation Runtime 中子 Agent 高风险动作必须冒泡回父 Agent 或用户审批。

画风：米白色纸张背景，黑色手绘签字笔线稿，线条粗细略不均，少量淡黄色高亮，editorial technical illustration，技术博客手绘决策图，清晰、克制、有工程草图感。

构图：decision path。左侧是“子 Agent”小工作台，向右执行“局部任务”。中间遇到一个分叉节点“高风险动作？”旁边有小警示三角。下方安全路径写“继续安全子集”。上方高风险路径进入“权限冒泡”节点，用淡黄色高亮，图标是盾牌和向上箭头。再连接到“父 Agent / 用户审批”闸门，同样用淡黄色强调。右侧三个出口：① 批准继续，② 拒绝收窄，③ 重新规划。箭头关系要清楚，表现子 Agent 不能自批。

背景：很淡的电路线、节点连线和工程草图辅助线，不要抢主体。节点标签短，避免长句。底部可留一句手写金句位置：“子 Agent 可以请求，不能自批”。整体感觉像技术博客中的手绘机制图，不是产品 UI，不是正式架构图。

4. 负向提示词

不要照片写实，不要 3D，不要复杂 UI 截图，不要大量代码，不要密集小字，不要复杂表格，不要赛博朋克，不要霓虹色，不要深色背景，不要彩色卡通，不要复杂阴影，不要过多装饰，不要超过 8 个主节点，不要让背景电路线比主体更明显，不要生成长段中文文字。
