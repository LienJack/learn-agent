1. 图解类型
decision path，因为重点是子 Agent 发现事实以后，父 Agent join 可以采纳证据，也可能忽略 unknowns 导致失败。

2. 画面元素清单
6 个主节点：父任务、任务包、子 Agent、结构化结果、Join 闸门、最终决策；Join 后分两条路径：采纳证据和忽略 unknowns；高亮“结构化结果”和“Join 闸门”；图标使用旗帜、信封、小人、证据卡、合流开关、终点旗。

3. 正向图片提示词
画一张文章内技术解释图，主题是 Delegation Join 错误如何导致 Agent 失败。

画风：米白色纸张背景，黑色手绘签字笔线稿，线条粗细略不均，少量淡黄色高亮，editorial technical illustration，技术博客手绘流程图，清晰、克制、有工程草图感。

构图：decision path。左侧从“父任务”开始，流向“任务包”，再到“子 Agent”，然后到“结构化结果”，进入中央的“Join 闸门”。Join 闸门右侧分两条路径：上方是“采纳证据”走向“保守修复”，下方是“忽略 unknowns”走向“错误决策”。画面包含 6 个关键节点和一个明显分叉。

节点：
① 父任务：小旗帜，表示修复测试的主目标。
② 任务包：信封 + 清单，表示父 Agent 派出的 scoped task。
③ 子 Agent：小人 + 放大镜，表示局部调查。
④ 结构化结果：证据卡 + 问号，用淡黄色高亮，表示 evidence 和 unknowns。
⑤ Join 闸门：合流开关，用淡黄色高亮，表示父 Agent 合并证据。
⑥ 最终决策：终点旗，分成保守修复和错误决策两种路径。

高亮：用淡黄色强调“结构化结果”和“Join 闸门”。

背景：很淡的电路线、节点连线和工程草图辅助线，不要抢主体。

文字：节点标签短；底部可留出一行手写金句位置：“子 Agent 查到了，不等于父 Agent 用对了”。如果中文文字不稳定，则只留空白区域给后期添加。

整体感觉像技术博客中的手绘机制图，不是产品 UI，不是正式架构图。元素清晰，层次明确，箭头关系明确。

4. 负向提示词
不要照片写实，不要 3D，不要复杂 UI 截图，不要大量代码，不要密集小字，不要复杂表格，不要赛博朋克，不要霓虹色，不要深色背景，不要彩色卡通，不要复杂阴影，不要过多装饰，不要超过 8 个主节点，不要让背景电路线比主体更明显，不要生成长段中文文字。
