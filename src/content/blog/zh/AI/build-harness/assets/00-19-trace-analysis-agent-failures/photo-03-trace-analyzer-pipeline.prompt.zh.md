1. 图解类型
horizontal pipeline，因为重点是 Trace Analyzer 从 event log 到诊断报告和 eval 候选的线性处理流程。

2. 画面元素清单
7 个主节点：Event Log、Trace 投影、事实抽取、规则检查、失败分类、诊断报告、Eval 候选；图标分别用日志、分层面板、镊子、尺子、标签树、报告页、测试卡；淡黄色高亮“规则检查”和“失败分类”；背景是淡电路线。

3. 正向图片提示词
画一张文章内技术解释图，主题是 Trace Analyzer 的处理管线。

画风：米白色纸张背景，黑色手绘签字笔线稿，线条粗细略不均，少量淡黄色高亮，editorial technical illustration，技术博客手绘流程图，清晰、克制、有工程草图感。

构图：horizontal pipeline。画面从左到右排列 7 个流程节点，用手绘箭头连接，表现从事实日志进入分析器，最后产出诊断报告和 eval 候选。流程中间分成两段：前半段偏结构化，后半段偏语义判断。

节点：
① Event Log：日志纸条堆，表示事实事件。
② Trace 投影：分层面板，表示组织成诊断视图。
③ 事实抽取：镊子夹出便签，表示提取关键事实。
④ 规则检查：尺子 + 勾叉，用淡黄色高亮，表示确定性断言。
⑤ 失败分类：标签树，用淡黄色高亮，表示归因 taxonomy。
⑥ 诊断报告：报告页 + 放大镜，表示人可读复盘。
⑦ Eval 候选：测试卡片，表示失败样本可回归。

高亮：用淡黄色强调“规则检查”和“失败分类”。

背景：很淡的电路线、节点连线和工程草图辅助线，不要抢主体。

文字：节点标签短；底部可留出一行手写金句位置：“先结构化，再判断，再沉淀为回归”。如果中文文字不稳定，则只留空白区域给后期添加。

整体感觉像技术博客中的手绘机制图，不是产品 UI，不是正式架构图。元素清晰，层次明确，箭头关系明确。
