1. 图解类型
input-process-output，因为重点是同一份 event log 如何经过 trace projection 变成可诊断视图。

2. 画面元素清单
6 个主节点：事件日志、投影器、责任链、Trace 视图、诊断发现、修复建议；图标分别用日志纸卷、漏斗、链条、放大镜面板、警示标记、扳手清单；淡黄色高亮“Trace 视图”和“诊断发现”；背景是很淡的电路线和工程草图辅助线；底部留一句短 caption。

3. 正向图片提示词
画一张文章内技术解释图，主题是 event log 如何变成可诊断 trace。

画风：米白色纸张背景，黑色手绘签字笔线稿，线条粗细略不均，少量淡黄色高亮，editorial technical illustration，技术博客手绘流程图，清晰、克制、有工程草图感。

构图：input-process-output。左侧是“事件日志”节点，画成一叠带时间戳的小纸条；中间依次是“投影器”“责任链”“Trace 视图”；右侧是“诊断发现”和“修复建议”。用从左到右的手绘箭头连接，表示事实日志被组织成面向失败定位的视图。画面包含 6 个关键节点。

节点：
① 事件日志：纸卷 + 小时间戳，表示事实源。
② 投影器：漏斗 + 小齿轮，表示把事件组织成 trace。
③ 责任链：链条图标，表示目标、判断、工具、观察、验证的因果链。
④ Trace 视图：放大镜 + 面板，用淡黄色高亮，表示诊断阅读入口。
⑤ 诊断发现：警示标记 + 便签，用淡黄色高亮，表示失败归因。
⑥ 修复建议：扳手 + 清单，表示下一步改哪层。

高亮：用淡黄色强调“Trace 视图”和“诊断发现”两个节点。

背景：很淡的电路线、节点连线和工程草图辅助线，不要抢主体。

文字：节点标签短；底部可留出一行手写金句位置：“日志记录发生过什么，Trace 定位责任链断点”。如果中文文字不稳定，则只留空白区域给后期添加。

整体感觉像技术博客中的手绘机制图，不是产品 UI，不是正式架构图。元素清晰，层次明确，箭头关系明确。

4. 负向提示词
不要照片写实，不要 3D，不要复杂 UI 截图，不要大量代码，不要密集小字，不要复杂表格，不要赛博朋克，不要霓虹色，不要深色背景，不要彩色卡通，不要复杂阴影，不要过多装饰，不要超过 8 个主节点，不要让背景电路线比主体更明显，不要生成长段中文文字。
