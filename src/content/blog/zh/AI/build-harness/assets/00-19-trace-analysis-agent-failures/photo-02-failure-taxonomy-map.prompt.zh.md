1. 图解类型
decision path，因为重点是 Agent 失败如何沿责任边界分类，并落到不同修复路线。

2. 画面元素清单
中心决策节点“失败归因”，向外分出 7 个短标签：模型、上下文、工具、权限、观察、验证、Join；每个分支末端有一个小修复图标；高亮“观察”和“验证”；背景使用淡电路线；底部留一句短 caption。

3. 正向图片提示词
画一张文章内技术解释图，主题是 Agent 失败分类如何指向修复路线。

画风：米白色纸张背景，黑色手绘签字笔线稿，线条粗细略不均，少量淡黄色高亮，editorial technical illustration，技术博客手绘流程图，清晰、克制、有工程草图感。

构图：decision path。画面中央是一个菱形决策节点“失败归因”，周围向七个方向分出短分支，每条分支连接一个责任边界节点，再连接一个小修复图标。整体像一张手绘的事故归因罗盘。

节点：
① 模型：小脑图标，表示判断错。
② 上下文：文件夹 + 眼睛，表示事实没有被模型看见。
③ 工具：扳手，表示执行或 schema 问题。
④ 权限：盾牌，表示 allow / deny 误判。
⑤ 观察：放大镜 + 便签，表示 observation 投影错，用淡黄色高亮。
⑥ 验证：勾叉测试卡，表示 verification 缺失或矛盾，用淡黄色高亮。
⑦ Join：合流箭头，表示 sub-agent 结果合并错误。

高亮：用淡黄色强调“观察”和“验证”，因为它们最容易把失败伪装成成功。

背景：很淡的电路线、节点连线和工程草图辅助线，不要抢主体。

文字：节点标签短；底部可留出一行手写金句位置：“分类不是贴标签，而是决定修哪一层”。如果中文文字不稳定，则只留空白区域给后期添加。

整体感觉像技术博客中的手绘机制图，不是产品 UI，不是正式架构图。元素清晰，层次明确，箭头关系明确。

4. 负向提示词
不要照片写实，不要 3D，不要复杂 UI 截图，不要大量代码，不要密集小字，不要复杂表格，不要赛博朋克，不要霓虹色，不要深色背景，不要彩色卡通，不要复杂阴影，不要过多装饰，不要超过 8 个主节点，不要让背景电路线比主体更明显，不要生成长段中文文字。
