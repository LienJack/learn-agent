1. 图解类型
Input-process-output 图，因为这张图要说明“框架抽象”如何经过“手写最小机制”的显影，转化成“工程判断力”。

2. 画面元素清单
6 个主节点：框架抽象、快速开始、真实问题、底层机制、工程判断、Harness 思维。图标分别用积木框架、启动按钮、警告三角、放大镜下的齿轮、分叉路标、控制台外壳。高亮“底层机制”和“工程判断”。底部留一句金句空位。

3. 正向图片提示词
画一张文章内技术解释图，主题是“框架抽象背后的最小机制如何带来工程判断力”。

画风：米白色纸张背景，黑色手绘签字笔线稿，线条粗细略不均，少量淡黄色高亮，editorial technical illustration，技术博客手绘流程图，清晰、克制、有工程草图感。

构图：input-process-output 横向流程图。左侧是“框架抽象”积木框架图标，旁边接一个“快速开始”启动按钮；中间出现“真实问题”警告三角，分出四个小问题图标：工具失控、上下文爆炸、权限审批、评估回归；这些问题汇入“底层机制”节点，用放大镜观察齿轮表示，使用淡黄色高亮；再流向“工程判断”分叉路标节点，也用淡黄色高亮；最右侧是“Harness 思维”控制台外壳和护栏。

高亮：用淡黄色强调“底层机制”和“工程判断”，以及从真实问题到机制显影的箭头。

背景：很淡的电路线、节点连线和工程草图辅助线，不要抢主体。

文字：节点标签短；底部可留出一行手写金句位置：“手写不是替代框架，而是看懂边界”。如果中文文字不稳定，则只留空白区域给后期添加。

整体感觉像技术博客中的手绘机制图，不是产品 UI，不是正式架构图。元素清晰，层次明确，箭头关系明确。

4. 负向提示词
不要照片写实，不要 3D，不要复杂 UI 截图，不要大量代码，不要密集小字，不要复杂表格，不要赛博朋克，不要霓虹色，不要深色背景，不要彩色卡通，不要复杂阴影，不要过多装饰，不要超过 8 个主节点，不要让背景电路线比主体更明显，不要生成长段中文文字。
