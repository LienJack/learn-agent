1. 图解类型
Decision path 图，因为这张图要帮助读者判断什么时候直接用框架、什么时候扩展框架、什么时候绕开局部抽象。

2. 画面元素清单
入口节点是“新 Agent 需求”；决策节点依次是路径是否确定、是否有真实副作用、默认策略是否足够、框架是否有关键 hook；结果节点是直接用框架、薄封装、扩展框架、自建 Harness 边界。高亮“扩展框架”和“自建 Harness 边界”。

3. 正向图片提示词
画一张文章内技术解释图，主题是“如何判断 Agent 项目该用框架、扩展框架还是绕开抽象”。

画风：米白色纸张背景，黑色手绘签字笔线稿，线条粗细略不均，少量淡黄色高亮，editorial technical illustration，技术博客手绘流程图，清晰、克制、有工程草图感。

构图：从上到下的决策路径图。顶部是“新 Agent 需求”入口，用任务卡片图标表示。向下经过 4 个菱形判断节点：① 路径确定？② 有副作用？③ 默认策略足够？④ 有关键 hook？每个判断节点用短箭头分支。左侧结果是“直接用框架”和“薄封装”，图标用积木和对话气泡；右侧结果是“扩展框架”和“自建 Harness 边界”，分别用插槽扩展模块、控制台护栏表示，并用淡黄色高亮。

高亮：用淡黄色强调“扩展框架”和“自建 Harness 边界”，以及通向它们的风险路径。

背景：很淡的电路线、节点连线和工程草图辅助线，不要抢主体。

文字：节点标签短；底部可留出一行手写金句位置：“框架省力，边界要自己判断”。如果中文文字不稳定，则只留空白区域给后期添加。

整体感觉像技术博客中的手绘机制图，不是产品 UI，不是正式架构图。元素清晰，层次明确，箭头关系明确。

4. 负向提示词
不要照片写实，不要 3D，不要复杂 UI 截图，不要大量代码，不要密集小字，不要复杂表格，不要赛博朋克，不要霓虹色，不要深色背景，不要彩色卡通，不要复杂阴影，不要过多装饰，不要超过 8 个主节点，不要让背景电路线比主体更明显，不要生成长段中文文字。
